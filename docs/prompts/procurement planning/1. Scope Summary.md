Procurement Planning — Pre-PRD (v1.0, rigorous)

**1\. Module Definition**

**Name**

**Procurement Planning**

**Purpose**

Transform **approved demand (DIA)** into:

-   compliant procurement packages
-   scheduled procurement activities
-   method-approved execution units

**Core Principle**

No procurement should proceed without a structured, approved plan.

**2\. Position in System**

Strategy → Budget → DIA → Procurement Planning → Tendering → ...

**Inputs**

-   Approved DIA demands
-   Budget reservations (read-only)
-   Procurement rules (thresholds, policies)

**Outputs**

-   Procurement Packages
-   Procurement Plan (annual / rolling)
-   Method decisions
-   Tender-ready units

**3\. Core Design Shift (Important)**

**Without templates:**

-   every planner manually decides structure → chaos

**With templates:**

-   system enforces:
    -   structure
    -   method defaults
    -   packaging rules
    -   compliance logic

**4\. Core Entities**

**4.1 Procurement Plan**

Represents:

Institutional procurement plan (annual or rolling)

**Key fields**

-   plan\_code
-   fiscal\_year
-   procuring\_entity
-   status
-   approval\_status

**4.2 Procurement Package (CRITICAL)**

Represents:

A unit of procurement execution (future tender)

**Key fields**

-   package\_code
-   package\_name
-   procurement\_method
-   estimated\_value
-   currency
-   status
-   planning\_status
-   approval\_status
-   schedule\_window
-   template\_used

**4.3 Package Line (link to DIA)**

Represents:

A DIA demand included in a package

**Fields**

-   package\_id
-   demand\_id (DIA reference)
-   amount
-   item summary
-   department
-   priority

**4.4 Procurement Template (NEW — FIRST-CLASS)**

Represents:

Predefined structure for procurement

This is the most important addition.

**4.5 Method Rule**

Represents:

Decision logic for procurement method

Example:

-   5M → Open Tender
-   emergency → Direct Procurement

**5\. Procurement Templates (Deep Design)**

**5.1 Why templates matter**

Templates define:

-   how demands are grouped
-   what method is default
-   how schedules are set
-   what approvals are required

**5.2 Template Types**

**A. Category-based**

-   ICT Equipment
-   Medical Supplies
-   Infrastructure Works

**B. Method-driven**

-   Open Tender template
-   RFQ template
-   Direct Procurement template

**C. Use-case driven**

-   Emergency procurement
-   Framework agreements
-   Routine supplies

**5.3 Template Structure**

Each template must define:

**1\. Scope**

-   applicable demand types
-   categories

**2\. Packaging Rules**

-   group by:
    -   program
    -   location
    -   item type
-   max/min thresholds

**3\. Method Defaults**

-   procurement method
-   allowed alternatives

**4\. Approval Flow**

-   who approves
-   required stages

**5\. Scheduling Rules**

-   lead time
-   procurement cycle duration

**6\. Compliance Constraints**

-   threshold enforcement
-   emergency overrides

**5.4 Example Template**

Medical Equipment Template

Scope:

\- Healthcare equipment demands

Packaging:

\- group by county + equipment type

Method:

\- default: Open Tender

Threshold:

\- > 2M → Open Tender

\- < 2M → RFQ

Approval:

\- Planning Authority

\- Procurement Head

Schedule:

\- Q1 execution

**6\. State Machines**

**6.1 Procurement Plan**

Draft → Submitted → Approved → Locked

**6.2 Procurement Package**

Draft → Structured → Submitted → Approved → Ready for Tender → Locked

**6.3 Key rule**

Only Approved Packages can proceed to Tendering

**7\. Core Workflows**

**7.1 Create Plan**

-   create procurement plan
-   attach fiscal year
-   define scope

**7.2 Ingest Demand**

-   pull approved DIA demands
-   filter:
    -   unplanned
    -   emergency
    -   planned

**7.3 Apply Template**

User selects template → system:

-   groups demands
-   creates packages
-   assigns methods
-   estimates values

**7.4 Manual Adjustments**

Planner can:

-   split packages
-   merge packages
-   override method (with justification)

**7.5 Submit for Approval**

-   plan submitted
-   packages reviewed
-   compliance validated

**7.6 Approval**

-   approve plan
-   lock packages
-   send to Tendering

**8\. Governance & Approval**

**Roles**

| **Role** | **Responsibility** |
| --- | --- |
| Planner | structure packages |
| Procurement Officer | validate methods |
| Planning Authority | approve plan |
| Auditor | review compliance |

**Approval rules**

-   method overrides require justification
-   threshold violations must be blocked or escalated
-   emergency packages flagged

**9\. Handling Unplanned & Emergency Demand**

This must explicitly exist.

**Unplanned Demand**

Flow:

DIA (Unplanned) → Planning → justification → inclusion

Rules:

-   must be flagged
-   must not disrupt approved plan silently

**Emergency Demand**

Flow:

DIA (Emergency) → direct template → expedited package

Rules:

-   fast-track
-   strict audit trail

**With DIA**

-   reference only
-   do not mutate

**With Budget**

-   read:
    -   reservation
    -   availability

**With Tendering**

-   output:
    -   approved packages

**12\. Key Constraints**

**12.1 No orphan procurement**

Every package must:

-   trace back to DIA
-   trace back to Budget Line

**12.2 No execution without approval**

-   unapproved packages cannot proceed

**12.3 Template enforcement**

-   templates must guide planning
-   not optional decoration

**13\. Risks & Mitigation**

**Risk: Over-complex templates**

→ keep initial templates simple

**Risk: Planner bypass**

→ enforce template usage

**Risk: UI overload**

→ follow DIA workbench model

**Risk: weak compliance**

→ embed rules early

**14\. What we will NOT do (important)**

-   no tendering logic here
-   no supplier interaction
-   no contract creation
-   no payment logic

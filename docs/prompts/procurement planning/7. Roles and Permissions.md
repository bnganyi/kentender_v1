Procurement Planning — Roles & Permissions Matrix

**1\. Purpose**

This matrix defines:

-   who can see what
-   who can do what
-   who can transition which states
-   what the UI should expose by role
-   what must be blocked server-side

This is a **module control artifact**, not just a UX aid.

**2\. Core Roles**

**2.1 Planner**

Owns package structuring and template application.

**2.2 Procurement Officer**

Validates procurement method, packaging coherence, and readiness.

**2.3 Planning Authority**

Approves plan/package for downstream tendering.

**2.4 Auditor / Oversight**

Read-only oversight and traceability review.

**2.5 Administrator**

System-level support and controlled override.

**3\. Actor Table**

| **Role** | **Business Responsibility** | **Entity Scope** | **Primary Purpose** |
| --- | --- | --- | --- |
| Planner | Build plan and packages | Own entity | Structure procurement |
| Procurement Officer | Validate package and method | Own entity | Operational review |
| Planning Authority | Approve plan/package | Own entity | Governance approval |
| Auditor | Review compliance and traceability | Assigned entity / broader oversight | Read-only review |
| Administrator | System administration and support | Global | Controlled support |

**6\. Workflow / State Transition Permissions**

We need this explicitly, because this is where governance often goes soft.

**6.1 Procurement Plan transitions**

**Plan states**

-   Draft
-   Submitted
-   Approved
-   Locked
-   Rejected / Returned (if you support these explicitly)

| **From** | **To** | **Allowed Role(s)** | **Preconditions** |
| --- | --- | --- | --- |
| Draft | Submitted | Planner, Admin | Plan valid |
| Submitted | Approved | Planning Authority, Admin | All required packages valid |
| Submitted | Draft / Returned | Planning Authority, Admin | Return reason required |
| Submitted | Rejected | Planning Authority, Admin | Reject reason required |
| Approved | Locked | System / Planning Authority / Admin | Approval complete |
| Locked | Draft | Admin only, exceptional | Formal override only |

**6.2 Procurement Package transitions**

**Package states**

-   Draft
-   Structured
-   Submitted
-   Approved
-   Ready for Tender
-   Rejected / Returned (if supported explicitly)

| **From** | **To** | **Allowed Role(s)** | **Preconditions** |
| --- | --- | --- | --- |
| Draft | Structured | Planner, Admin | Template + valid lines present |
| Structured | Submitted | Planner, Admin | Package complete |
| Submitted | Approved | Planning Authority, Admin | Profiles complete, method valid |
| Submitted | Draft / Returned | Planning Authority, Admin | Return reason required |
| Submitted | Rejected | Planning Authority, Admin | Reject reason required |
| Approved | Ready for Tender | Procurement Officer or Planning Authority, Admin | Approval complete, handoff valid |
| Ready for Tender | Draft | Admin only | exceptional override only |

**8\. Template/Profile Governance**

Templates are critical enough that they need stronger control than ordinary records.

**Recommended v1 policy**

-   Templates and profiles are **centrally managed**
-   They are **not editable by planners**
-   They are selected and consumed, not improvised

**Why**

This prevents:

-   drift in procurement method defaults
-   inconsistent risk logic
-   ad hoc evaluation criteria

**Matrix**

| **Action** | **Planner** | **Procurement Officer** | **Planning Authority** | **Auditor** | **Admin** |
| --- | --- | --- | --- | --- | --- |
| View Template | Yes | Yes | Yes | Yes | Yes |
| Create Template | No | No | No | No | Yes |
| Edit Template | No | No | No | No | Yes |
| Deactivate Template | No | No | No | No | Yes |
| View Profiles | Yes | Yes | Yes | Yes | Yes |
| Edit Profiles | No | No | No | No | Yes |

If later you want procurement leadership to edit templates, make that a deliberate governance decision—not a default.

**9\. Package Line Rules by Role**

Package Lines are sensitive because they preserve traceability.

| **Action** | **Planner** | **Procurement Officer** | **Planning Authority** | **Auditor** | **Admin** |
| --- | --- | --- | --- | --- | --- |
| Add DIA Demand to Package | Yes | No | No | No | Yes |
| Remove DIA Demand from Package | Yes | No | No | No | Yes |
| Edit Package Line Amount | Limited / preferably No in v1 | No | No | No | Yes |
| View Package Lines | Yes | Yes | Yes | Yes | Yes |

**Recommendation**

For v1:

-   do **not** allow arbitrary line amount editing
-   use full-demand inclusion unless explicit split design is introduced later

That keeps the model safer.

**10\. Entity and Scope Rules**

These must apply in addition to role.

**Rule set**

A user’s ability to act depends on:

-   role
-   procuring entity
-   package/plan status
-   assignment (if used later)
-   sensitivity classification (if added later)

**Minimum enforcement**

-   Planner cannot edit another entity’s plan/package
-   Planning Authority cannot approve outside entity scope unless explicitly authorized
-   Auditor read scope must be configured, not universal by accident

**11\. Locking Rules**

**Procurement Plan**

When:

-   Submitted
-   Approved
-   Locked

then:

-   no ordinary edits by Planner
-   no package-structure changes through normal UI

**Procurement Package**

When:

-   Submitted
-   Approved
-   Ready for Tender

then:

-   no ordinary edits by Planner
-   no template changes
-   no line changes
-   no silent method changes

That should be enforced server-side, not just hidden in UI.

**12\. Exceptional Override Policy**

You will need a rule here even if you keep it minimal.

**v1 recommendation**

Only Administrator may bypass standard locks.

Examples:

-   reopen locked package
-   move Ready for Tender back to Draft
-   alter approved record for emergency correction

But every override should later become auditable through governance/compliance layers.

For now:

-   keep Admin override minimal
-   do not expose casual override paths

**13\. Recommended Role Summary by Surface**

**Planner sees**

-   New Plan
-   New Package
-   Apply Template
-   Edit Draft/Structured package
-   Submit

**Procurement Officer sees**

-   review workbench
-   method/packaging review visibility
-   Ready for Tender handoff action if policy assigns that role

**Planning Authority sees**

-   Submitted queue
-   Approve / Return / Reject actions

**Auditor sees**

-   read-only visibility
-   no action buttons

**Admin sees**

-   everything, but not as ordinary workflow substitute

**14\. Acceptance Criteria**

This roles/permissions design is correct only if:

-   planners can build but not approve
-   approvers can approve but not casually rebuild packages
-   auditors are read-only
-   templates/profiles are centrally controlled
-   submitted/approved records are locked
-   package line traceability cannot be casually tampered with
-   permissions are enforced server-side
-   UI only shows valid actions by role/state

**Procurement Templates — Deep Design**

**1\. What a template is**

A Procurement Template is **not** a document stub and **not** a saved filter.

It is a **planning control object** that tells the system how a class of approved demand should be transformed into a procurement package.

It answers:

-   what kinds of demand it applies to
-   how those demands may be grouped
-   what procurement method is default
-   what contract type is preferred
-   what risk profile applies
-   what decision criteria must be used later
-   what vendor-management posture is expected
-   what approvals or escalations are required
-   what scheduling assumptions apply

So the template sits at the exact point where the system changes from:

approved demand

to:

planned procurement unit

That is why templates are central.

**2\. Why templates matter**

Without templates, Procurement Planning becomes manual judgment every time:

-   planner decides grouping ad hoc
-   planner picks method ad hoc
-   planner picks risk treatment ad hoc
-   planner picks contract type ad hoc

That creates inconsistency, weak compliance, and rework.

With templates, the planner still has controlled discretion, but the system starts from a governed baseline.

So the rule should be:

every package must be created under a template, even if the template later allows justified override.

**3\. What level templates operate at**

Templates operate at the **package level**, not the plan level and not the DIA-demand level.

That means:

-   **Plan** = container/governance layer
-   **Template** = transformation rule set
-   **Package** = execution unit created using the template
-   **Package Line** = traceability back to DIA

This is the cleanest model.

**4\. Template design principles**

A good template must be:

**4.1 Narrow enough to govern**

It should apply to a clear class of procurement.

Good:

-   Medical Equipment Procurement
-   Routine ICT Equipment Procurement
-   Emergency Clinical Equipment Procurement
-   Minor Works Procurement

Bad:

-   Generic Procurement Template
-   Operational Needs Template

**4.2 Broad enough to be reusable**

You do not want a separate template for every tiny project.

**4.3 Explicit about defaults and overrides**

The system should know:

-   what is the normal path
-   what requires justification
-   what is forbidden

**4.4 Declarative, not procedural**

A template should define configuration and rules, not embed custom code for each case.

**6\. Supporting profiles**

To avoid bloating the template record, the template should reference profiles.

**6.1 Risk Profile**

Defines:

-   standard risks
-   mitigation expectations
-   overall risk level
-   whether escalation is needed

**6.2 KPI Profile**

Defines:

-   what success will be measured against later
-   service-level expectations
-   delivery and quality benchmarks

**6.3 Decision Criteria Profile**

Defines:

-   evaluation weighting logic
-   make-vs-buy framing if relevant
-   later tender evaluation baseline

**6.4 Vendor Management Profile**

Defines:

-   supplier monitoring posture
-   escalation posture
-   performance review cadence

**6.5 Threshold / Method Rule Profile**

Defines:

-   value bands
-   method defaults by value/type
-   exception triggers

This avoids repeating the same rule bundle across many templates.

**What templates should control**

**7\. Scope control**

A template should say what kinds of DIA demand it can absorb.

Example dimensions:

-   requisition type: Goods / Works / Services
-   demand type: Planned / Unplanned / Emergency
-   category: Medical Equipment / ICT / Works
-   budget source or line family
-   strategic program family if relevant

The system should not allow a planner to apply an obviously wrong template silently.

**8\. Grouping control**

This is one of the most important design points.

The template must define how multiple approved demand records may be grouped into a package.

Examples:

-   by item category
-   by region/county
-   by facility type
-   by department
-   by funding source
-   by urgency
-   by quarter

But keep it simple at first.

For v1, a template should support something like:

-   group\_by = \["category", "region"\]  
    or
-   group\_by = \["department"\]  
    or
-   group\_by = \["none"\] for one-demand-per-package patterns

**9\. Method control**

Templates must set a default procurement method, but method selection should not be blind.

The system should use:

-   template default
-   threshold profile
-   demand type
-   emergency/unplanned flags
-   total package value

to determine a recommended method.

That means the template is the baseline, and the package calculation may refine the final method.

**10\. Contract type control**

You specifically called this out, and it belongs here.

The template should define a preferred contract type such as:

-   Fixed Price
-   Cost Reimbursable
-   Time & Materials

This should not float as an arbitrary later choice.

Examples:

-   standard equipment purchases → Fixed Price
-   emergency technical intervention → Time & Materials
-   uncertain scope consultancy or some specialized services → Cost Reimbursable where policy allows

For v1, keep the available set small and explicit.

**11\. Risk management control**

This is also best embedded in the template.

The template should define:

-   standard risk profile
-   required mitigation posture
-   escalation threshold

Example:  
Medical Equipment template may assume:

-   risk of delivery delays
-   risk of compatibility/specification mismatch
-   mitigation via acceptance tests and supplier qualification

Emergency template may assume:

-   sourcing failure risk
-   cost escalation risk
-   mitigation via prequalified vendors and fast approvals

The planner should see risk generated automatically, not start from a blank page.

**12\. Vendor-management control**

This should also be embedded.

The template should set expected supplier-management posture:

-   standard monitoring
-   high-touch monitoring
-   escalation thresholds
-   defect management expectations

This does not mean procurement planning manages the supplier lifecycle itself; it means the package carries downstream expectations.

That is useful and realistic.

**13\. Decision criteria control**

This is a strong addition and belongs in templates.

The template should point to a later evaluation profile so that from planning time the system already knows whether the likely selection logic is:

-   price-heavy
-   technical-heavy
-   balanced weighted matrix
-   minimal emergency decision framework

This gives continuity from plan to tender to evaluation.

**14\. KPI control**

Also correct to embed early.

A package should not become tender-ready without a KPI profile attached through the template or explicitly assigned.

That allows later contract and supplier performance monitoring to inherit planning intent.

Examples:

-   delivery timeliness
-   installation completion
-   defect rate
-   lead time adherence
-   cost efficiency

**Template application — deep design**

Now the important part: what happens when a planner applies a template?

**18\. Template application is a transformation, not a label**

Applying a template should not merely stamp metadata on a package.

It should:

1.  identify eligible approved DIA records
2.  group them according to packaging rules
3.  calculate package value
4.  derive recommended method
5.  attach contract/risk/KPI/decision/vendor profiles
6.  create package records and package lines
7.  leave DIA unchanged but linked

That is the real engine.

**19\. Suggested service entry point**

A good starting service is:

apply\_template\_to\_demands(

plan\_id,

template\_id,

demand\_ids,

actor,

options=None

)

Where:

-   plan\_id = target procurement plan
-   template\_id = selected template
-   demand\_ids = approved DIA records to package
-   actor = planner
-   options = limited flags such as grouping override if allowed

Do not make it too magical initially.

**20\. Template application stages**

**Stage 1 — Eligibility validation**

For every selected demand:

-   demand must be Approved or Planning Ready per your final DIA policy
-   demand must not already be fully consumed into another package
-   demand must fit template applicability
-   demand must have valid budget linkage
-   demand must have valid strategy linkage

**Stage 2 — Grouping**

The service groups selected demand by template grouping dimensions.

**Stage 3 — Package creation**

For each group:

-   create Procurement Package
-   compute estimated total value
-   assign default method
-   attach profiles
-   assign contract type

**Stage 4 — Package Line creation**

For each source demand in the group:

-   create Package Line
-   store source demand reference
-   store amount
-   store budget line reference
-   preserve department/priority context

**Stage 5 — Validation and warnings**

Before finalizing:

-   check threshold profile
-   check emergency constraints
-   check missing mandatory profile references
-   check package has at least one line

**Stage 6 — Return result**

Return:

-   packages created
-   package lines created
-   any warnings
-   any required overrides/justifications

**What not to do**

Now that this is clear, protect against these mistakes.

**31\. Anti-patterns**

**31.1 Do not let planners create package without template**

That defeats the whole design.

**31.2 Do not let package line be free text**

It must reference DIA.

**31.3 Do not let package mutate DIA**

Planning references demand; it does not rewrite it.

**31.4 Do not let tendering consume DIA directly**

Tendering must consume approved package.

**31.5 Do not overload templates with custom one-off logic**

Keep templates declarative and reusable.

Procurement Planning — Full PRD (v1.0, Production-Grade)

**1\. Module Overview**

**Name**

**Procurement Planning**

**Purpose**

Transform approved demand into:

-   compliant procurement packages
-   structured procurement plans
-   execution-ready units governed by templates

**Core Principle**

Planning is the control layer between demand and procurement execution.

**2\. Scope**

**In Scope**

-   Procurement Plans (annual/rolling)
-   Procurement Packages
-   Template-driven planning
-   Method selection
-   Demand consolidation
-   Risk, contract, vendor, KPI frameworks (embedded)
-   Planning approval workflow

**Out of Scope**

-   Tendering / bidding
-   Supplier interaction
-   Contract execution
-   Payments

**4\. State Machines**

**4.1 Procurement Plan**

Draft → Submitted → Approved → Locked

**4.2 Procurement Package**

Draft → Structured → Submitted → Approved → Ready for Tender

**5\. Governance Rules**

**5.1 Template Enforcement**

-   every package MUST use a template
-   no “free-form” package creation

**5.2 Method Governance**

-   method auto-derived from template
-   overrides require:
    -   justification
    -   approval

**5.3 Risk Governance**

-   every package must have a risk profile
-   high-risk packages require additional approval

**5.4 Contract Governance**

-   contract type derived from template
-   cannot be empty

**5.5 KPI Governance**

-   KPI profile must exist before approval
-   used later in contract monitoring

**5.6 Decision Criteria Governance**

-   mandatory for competitive methods
-   defines evaluation framework for tendering

**5.7 Vendor Management Governance**

-   must exist for all packages
-   defines:
    -   monitoring cadence
    -   issue escalation

**6\. Core Workflows**

**6.1 Create Plan**

-   create plan
-   set fiscal year
-   load approved DIA demands

**6.2 Apply Template (CRITICAL FLOW)**

User selects template:

System:

-   filters applicable demands
-   groups demands into packages
-   assigns:
    -   method
    -   contract type
    -   risk profile
    -   KPI profile
    -   decision criteria
    -   vendor management rules

**6.3 Adjust Packages**

User can:

-   split package
-   merge packages
-   adjust schedule
-   override method (with justification)

**6.4 Submit Plan**

-   validate:
    -   all packages complete
    -   no missing profiles
-   submit

**6.5 Approve Plan**

-   planning authority approves
-   lock plan
-   packages become execution-ready

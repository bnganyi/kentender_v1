Procurement Planning — UI Specification v1.0

This is aligned to your system-wide UI pattern and to the module model already agreed:

-   **Plan** = governance container
-   **Package** = execution unit
-   **Package Line** = traceability link to DIA
-   **Template** = planning control object

**1\. Page types**

Procurement Planning should use **three page types** only:

**1.1 Planning Workbench**

Primary landing and daily working surface.

Used for:

-   viewing plans
-   seeing packages
-   reviewing package readiness
-   driving approvals

**1.2 Package Builder / Editor**

Focused surface for:

-   creating or refining one package
-   applying template
-   reviewing demand lines
-   adjusting schedule/method within policy

**1.3 Template Review / Selection Surface**

Used when:

-   selecting a template
-   comparing templates
-   understanding package defaults

This can be:

-   modal
-   side panel
-   dedicated subview

For v1, a modal or right-side selector panel is enough.

**2\. Primary workbench layout**

**2.1 Route**

Recommended route:

/desk/procurement-planning

**2.2 Header**

Left:

-   Title: **Procurement Planning**
-   Subtitle: **Structure approved demand into compliant procurement packages.**

Right:

-   Primary action: **New Plan**
-   Secondary action: **New Package**  
    Only visible if user is working inside an editable plan context

**2.3 KPI strip**

The KPI strip must sit directly below the header.

**Default KPI cards**

-   Total Packages
-   Total Planned Value
-   Pending Approval
-   High-Risk Packages

**Optional later**

-   Emergency Packages
-   Ready for Tender

**Formatting rules**

-   large numeric values
-   no truncation
-   currency rendered cleanly

Examples:

-   12
-   KES 48,500,000
-   3
-   2

**2.4 Main workbench layout**

Use the same master-detail pattern you have now validated in Budget and DIA:

Left panel: 35–40%

Right panel: 60–65%

**Left panel**

Procurement Packages list

**Right panel**

Selected Package detail/review panel

This is the correct primary working model for the module.

**3\. Top-level navigation model**

Do **not** overload the top of the page with many tabs.

**3.1 Top-level tabs**

Keep only:

-   My Work
-   All
-   Approved
-   Ready for Tender

That is enough.

**3.2 Queue selector below tabs**

Like DIA, use a secondary queue selector.

**Planner queues**

-   Draft Packages
-   Structured Packages
-   Submitted Packages
-   High-Risk Packages
-   Emergency Packages

**Approver queues**

-   Pending Approval
-   High-Risk Requiring Escalation
-   Method Override Cases

**Tendering handoff queue**

-   Ready for Tender
-   Approved Not Yet Handed Off

This is better than 10 top-level tabs.

**4\. Left panel — Packages list**

**4.1 Panel title**

Context-sensitive based on active queue, for example:

-   Draft Packages
-   Pending Approval
-   Ready for Tender

Do not always label it “Packages”.

**4.2 Row structure**

Each row should follow the system pattern:

**Primary**

-   Package Name

**Secondary**

-   Package Code · Procurement Method

**Tertiary**

-   Estimated Value · Template Name

**Meaningful badges only**

Use at most 1–2 badges:

-   High Risk
-   Emergency
-   Submitted
-   Ready

Do **not** clutter with too many equal-weight chips.

**Example**

Diagnostic Imaging Equipment - Western Region

PKG-MOH-2026-001 · Open Tender

KES 5,000,000 · Medical Equipment Procurement

\[Submitted\]

**4.3 List row rules**

-   selected row must have clear highlight
-   emergency/high-risk must stand out
-   method visible at a glance
-   code visible, but subordinate to package name

**4.4 Empty states**

Queue-specific messages:

-   “No draft packages yet.”
-   “No packages pending approval.”
-   “No packages ready for tender.”

If no packages exist in an editable plan:

-   show CTA: **Create Package**

**5\. Right panel — Package detail / review panel**

This is the most important UI surface.

It must support:

-   planning review
-   approval review
-   readiness assessment
-   action without opening full edit mode every time

**5.1 Identity header**

At the top of the right panel, render:

**Primary**

-   Package Name

**Secondary**

-   Package Code · Template Name · Procurement Method

**Badges**

-   Status badge
-   High Risk badge if applicable
-   Emergency badge if applicable

**Actions (top-right)**

Context-aware, examples:

-   Edit Package
-   Submit
-   Approve
-   Return
-   Reject
-   Mark Ready for Tender

Do **not** bury these at the bottom.

**Example**

Diagnostic Imaging Equipment - Western Region

PKG-MOH-2026-001 · Medical Equipment Procurement · Open Tender

\[Edit\] \[Submit\]

**5.2 Detail panel sections**

The right panel must be sectioned and compact.

**Section A — Package Definition**

Use a two-column layout.

Left:

-   Package Name
-   Template

Right:

-   Procurement Method
-   Contract Type

Optional below:

-   Package Status
-   Method override indicator

**Section B — Financial and Schedule Context**

Two-column layout.

Left:

-   Estimated Value
-   Currency

Right:

-   Schedule Start
-   Schedule End

If there is a plan-level schedule context, do not duplicate unnecessary metadata here.

**Section C — Demand Lines**

This is a compact table, not a full editor in review mode.

Columns:

-   Demand ID
-   Department
-   Budget Line
-   Amount
-   Priority

This is the traceability proof section.

**Rules**

-   must be easy to scan
-   no raw IDs beyond business IDs
-   if package has many lines, table scrolls inside section rather than stretching the whole page too far

**Section D — Risk & Mitigation**

Display:

-   Risk Profile
-   Risk Level
-   top risks and mitigations

Prefer compact display rows or bulletless stacked items, not a giant JSON dump.

Example:

**Risk Profile**  
Standard Medical

**Key Risks**

-   Delivery delays → penalty clauses
-   Commissioning failure → acceptance checkpoints

**Section E — KPI Profile**

Display:

-   KPI Profile name
-   metrics list

Example:

-   On-time delivery
-   Commissioning success rate
-   Defect rate at acceptance

**Section F — Decision Criteria**

Display:

-   profile name
-   weighted criteria table

Columns:

-   Criterion
-   Weight

Example:

-   Price — 35
-   Technical Compliance — 35
-   Delivery — 20
-   After-Sales Support — 10

This is important because it links planning to later evaluation.

**Section G — Vendor Management**

Display:

-   monitoring profile name
-   monitoring rules
-   escalation rules

Compact, not verbose.

**Section H — Workflow / Approval**

Display rows:

-   Status
-   Submitted By / At
-   Approved By / At
-   Returned / Rejected reason if applicable
-   Ready for Tender state if applicable

This must feel like a governed review surface, not a form.

**6\. Package Builder / Editor**

This is the second major surface.

**6.1 Route**

Recommended pattern:

/desk/procurement-package/<id>

For new package:

/desk/procurement-package/new

**6.2 Builder header**

Left:

-   Back to Procurement Planning
-   Package Name or “New Package”
-   Status badge

Right:

-   Save Draft
-   Submit
-   Cancel

Only show valid actions for current state and role.

**6.3 Builder structure**

Do **not** build this as a long one-column form.

Use sectioned layout with two-column grids where appropriate.

**Section order**

1.  Package Definition
2.  Template & Method
3.  Demand Lines
4.  Financial & Schedule
5.  Risk & Mitigation
6.  KPI Profile
7.  Decision Criteria
8.  Vendor Management
9.  Workflow / Approval
10.  Notes / Exceptions

**6.4 Section details**

**Section 1 — Package Definition**

Two columns:

-   Package Name | Package Code
-   Plan | Template

If Package Code is generated, render as read-only display row, not editable input.

**Section 2 — Template & Method**

Two columns:

-   Procurement Method | Contract Type
-   Override Flag | Override Reason

Rules:

-   if method comes from template, show it clearly as derived/default
-   if override is activated, show reason field
-   reason field only visible when override is true

**Section 3 — Demand Lines**

This is a structured table.

Columns:

-   Demand ID
-   Demand Title
-   Department
-   Budget Line
-   Amount
-   Priority
-   Remove action (only in editable state)

Actions:

-   Add Approved Demand
-   Remove Line

Important:

-   adding demand should use a selector modal, not free text
-   only eligible DIA records appear

**Section 4 — Financial & Schedule**

Two columns:

-   Estimated Value | Currency
-   Schedule Start | Schedule End

Estimated Value should be derived from lines, not manually typed in normal flow.

**Section 5 — Risk & Mitigation**

Two columns or compact stacked section:

-   Risk Profile
-   Risk Level

Then below:

-   compact risk/mitigation list

Profile is selected or derived from template; risks render as display content.

**Section 6 — KPI Profile**

Display/select:

-   KPI Profile
-   metrics preview

**Section 7 — Decision Criteria**

Display/select:

-   Decision Criteria Profile
-   weighted criteria preview

**Section 8 — Vendor Management**

Display/select:

-   Vendor Management Profile
-   monitoring/escalation preview

**Section 9 — Workflow / Approval**

Read-only section:

-   current status
-   approval metadata
-   ready-for-tender state

**Section 10 — Notes / Exceptions**

Use for:

-   planner notes
-   exception notes
-   override notes beyond method if later required

Keep textarea compact by default.

**7\. Template selection UX**

Templates are important enough that the UX must make them understandable.

**7.1 Template selection modal/panel**

When planner clicks **Apply Template** or chooses template in package builder, show options as rich rows.

Each template row should display:

**Primary**

-   Template Name

**Secondary**

-   Template Code · Default Method · Contract Type

**Supporting text**

-   brief applicability description

**Example**

Medical Equipment Procurement

TPL-MED-001 · Open Tender · Fixed Price

Applies to planned medical equipment demand

**7.2 Template preview**

When selected, show summary in side panel or inline preview:

-   default method
-   grouping rule
-   risk profile
-   KPI profile
-   decision criteria profile
-   vendor management profile

This helps users trust the automation.

**8\. Plan-level screen behavior**

Because the module includes both Plan and Package, you need a light plan context without turning the workbench into a plan admin screen.

**8.1 Current Plan context**

At top of workbench, show current plan context bar:

-   Plan Name / Code
-   Fiscal Year
-   Entity
-   Status

Example:

PP-MOH-2026 · Ministry of Health · FY2026 · Draft

If multiple plans are relevant, allow a compact plan selector.

**8.2 Plan actions**

At plan level, allow:

-   New Package
-   Submit Plan
-   Approve Plan

Do not bury package actions under plan actions.

**9\. Review / approval UX**

Approvers should not need to open full edit mode for normal review.

**9.1 Approval actions from detail panel**

For submitted package:

-   Approve
-   Return
-   Reject

If method override exists:

-   surface that prominently in Definition section or as a warning strip

**9.2 Return / reject modals**

Require reason.

Examples:

-   Return reason
-   Reject reason

Do not allow silent state changes.

**10\. Visual treatment of risk and exceptions**

This is important for Procurement Planning because some packages deserve more attention.

**10.1 High Risk**

Show:

-   badge in list
-   risk level in detail panel near top
-   optional warning strip if escalation required

**10.2 Emergency**

If emergency package exists:

-   badge in list
-   emergency marker near identity header
-   prioritised queue surface

**10.3 Method override**

Show a subtle but clear indicator:

-   “Method Override”
-   with reason visible in Definition / Method section

**11\. System-wide dropdown/reference rules here too**

Planning must follow the same standard you already identified.

**11.1 Never show internal IDs**

Dropdowns/selectors for:

-   Template
-   Risk Profile
-   KPI Profile
-   Decision Criteria Profile
-   Vendor Management Profile
-   Plan
-   Budget Line
-   DIA Demand

must show:

-   name
-   code or business identifier

not internal database IDs.

**13\. UI acceptance criteria**

Procurement Planning UI is acceptable only if:

-   the landing page behaves like a workbench, not a doctype list
-   package list is compact and scannable
-   package detail panel is decision-ready
-   builder is sectioned and compact
-   templates are understandable and visibly driving defaults
-   demand lines preserve traceability in UI
-   risk / override / emergency signals are visible without clutter
-   actions are top-placed and role-valid
-   no raw internal IDs appear anywhere

**14\. Cursor-ready UI prompt**

Implement Procurement Planning UI exactly as specified.

This module must follow the Kentender UI System Pattern:

-   workbench first
-   compact identity headers
-   two-column structured sections
-   read-only fields as display rows
-   actions near top-right
-   no long single-column forms
-   no raw database IDs

-   KPI strip

# Procurement Planning MVP Bootstrap Checklist (Sticky)

Purpose: start Procurement Planning with DIA-proven standards baked in from day one.

## 1) UX + Navigation Guardrails

- Keep one canonical module entry: Procurement Planning workbench.
- Sidebar must remain workspace-oriented (no raw DocType replacement after opening forms).
- Ensure “back” and breadcrumb behavior always returns to the planning workbench route.
- Place primary actions at the top; use sticky action zones for long detail views.
- Default views must prioritize discoverability of newly created records.

## 2) Queue-First Role Design

- Define role queues before UI build (Planner, Planning Authority, Finance, Auditor/Admin as needed).
- For each role, define:
  - `mywork` queues (actor-owned items),
  - `all` queues (role-scope visibility),
  - `approved/rejected` queue subsets where relevant.
- Define explicit default queue per role and per tab to avoid “empty by default” UX.
- Add queue empty captions that explain what the queue means.

## 3) Data Display Contract (Non-Negotiable)

- Store internal IDs, but display human labels as `Name (Code)`.
- Never expose raw DB IDs in:
  - dropdown options,
  - selected link fields,
  - list rows/cards/detail headers,
  - helper text/tooltips.
- Make search match both `name` and `code` for all reference entities.
- Add/standardize `title_field`, `search_fields`, and `show_title_field_in_link` on linked masters.

## 4) API Contract + Error Model

- Use strict envelopes for all UI APIs:
  - success: `{"ok": true, "data": ..., "message": "..."}`
  - failure: `{"ok": false, "error_code": "...", "message": "..."}`
- Publish and reuse an error taxonomy (validation, permission, state-transition, dependency, not-installed).
- Prefer non-throwing UX for expected blocked states (render actionable hints inline).

## 5) Permission + Workflow Integrity

- Enforce permission-aware aggregates and list retrieval (`frappe.get_list` paths).
- Lock editing by workflow state; allow changes only through explicit lifecycle actions.
- Hidden required audit fields must be auto-populated before save/submit.
- Optional fields must remain optional in both DocType schema and client validation.

## 6) Migration + Fixture Strategy

- Add module setup patches early (Module Def, Workspace, Workspace Sidebar, Desktop Icon where needed).
- Keep patches idempotent and safe for reruns.
- Include fixture coverage for workspaces and sidebars used by the module.
- Add backfill patches for legacy/hash-like display values before UAT.

## 7) Testing Baseline (Start Early)

- Add smoke tests for each critical scenario:
  - create draft,
  - submit,
  - approve/reject/return,
  - exception path,
  - permission-denied path.
- Add queue resolver tests:
  - role + tab + queue combinations,
  - refine filters,
  - search behavior.
- Add permission boundary tests per role (read/write/action scope).

## 8) Delivery Operating Model

- Ship in small, reversible patches.
- Validate after substantive changes:
  - lint,
  - focused tests,
  - migrate sync.
- Use screenshot-based triage for UX gaps and resolve with targeted patches.
- Keep a live tracker: `done / risky / pending` with explicit owners and closure criteria.

## 9) Day-0 Implementation Sequence (Recommended)

1. Define roles, queues, defaults, and blocked-state behavior.
2. Create workspace + sidebar + route guardrails.
3. Implement API envelope + core queue list endpoint.
4. Implement list/detail shell with sticky top actions.
5. Implement create/edit form with hidden-field autofill and validation contract.
6. Implement display formatter contract for all references (`Name (Code)`).
7. Add smoke/queue/permission tests and seed data.
8. Run migrate, UAT walkthrough, and close gaps from screenshots.

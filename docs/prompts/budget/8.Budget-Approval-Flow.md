Budget Approval Workflow  

# Design

**🧠 Let’s be precise**

From your screenshots:

-   Budgets have **Draft status**
-   No transitions
-   No role-based state changes
-   No locking behavior

👉 That means:

You have a **data state**, not a **workflow**

**⚠️ Why this matters (don’t skip this)**

Without approval flow:

-   Anyone can keep editing “final” budgets
-   No audit boundary
-   Procurement later has no trusted budget
-   Finance logic breaks downstream

👉 This will bite you hard in Procurement

**🎯 Minimum viable Budget approval (what you actually need)**

Keep this **simple and strict**. No overengineering.

**✅ States**

Draft → Submitted → Approved

Optional later:

Approved → Locked

**✅ Transitions**

| **From** | **To** | **Who** |
| --- | --- | --- |
| Draft → Submitted | Strategy Manager |  |
| Submitted → Approved | Planning Authority |  |
| Approved → (no edit) | System enforced |  |

**✅ Rules (critical)**

**1\. Draft**

-   Fully editable
-   Allocations editable

**2\. Submitted**

-   No allocation edits
-   No structural changes
-   Read-only for Strategy Manager
-   Awaiting approval

**3\. Approved**

-   Fully locked
-   Used by Procurement
-   No edits allowed

**❗ Hard constraint (non-negotiable)**

NO budget allocation edits after submission

👉 This is what makes the budget trustworthy

**🧩 UX implications (tie back to your UI)**

**Landing page**

Add:

\[DRAFT\] → yellow

\[SUBMITTED\] → blue

\[APPROVED\] → green

**Actions section**

**Draft:**

\[ Open Builder \] \[ Edit \] \[ Submit for Approval \]

**Submitted:**

\[ View Builder \]

(Status: Awaiting Approval)

**Approved:**

\[ View Builder \]

(Status: Approved)

**Builder**

**Draft:**

-   editable

**Submitted / Approved:**

-   **read-only mode**
-   disable:
    -   Add Program
    -   Add Allocation
    -   Save buttons

**🔧 Backend (what must exist)**

You need:

**Field on Budget:**

status = Select(\["Draft", "Submitted", "Approved"\])

**Validation:**

if doc.status in \["Submitted", "Approved"\]:

block\_allocation\_changes()

**Role enforcement:**

Only Planning Authority can approve

**🚫 What NOT to do**

Do NOT:

-   add complex approval chains
-   add multi-step workflows
-   add comments/approvals matrix yet

👉 Keep it **binary and strict**

**🧠 My recommendation (strong)**

👉 Implement this BEFORE Procurement

Because:

-   Procurement will depend on:
    -   Approved budgets only
-   If you skip this:
    -   you’ll have to refactor everything later

**B5.1 — Add Budget status model and transitions**

Implement the Budget approval state model.

Scope:

-   Budget status field must support these exact states:
    -   Draft
    -   Submitted
    -   Approved

Requirements:

1.  Existing Draft budgets remain valid
2.  Add transition rules:
    -   Draft → Submitted
    -   Submitted → Approved
3.  No direct Draft → Approved transition in normal UI flow
4.  Approved budget is considered final for v1
5.  Do not implement rejection, return-for-edit, or archive flow yet

Backend rules:

-   only valid transitions allowed
-   invalid transitions must raise clear errors

Acceptance criteria:

-   Budget can move Draft → Submitted
-   Budget can move Submitted → Approved
-   invalid transitions are blocked

**B5.2 — Enforce role-based approval permissions**

Implement role-based permissions for Budget state transitions.

Roles:

-   Strategy Manager
-   Planning Authority
-   Administrator

Rules:

1.  Strategy Manager

-   can create/edit Draft budgets
-   can submit Draft budgets
-   cannot approve budgets

1.  Planning Authority

-   can read Budget and Budget Allocation
-   can approve Submitted budgets
-   cannot edit Draft content directly
-   cannot create budgets

1.  Administrator

-   full access

Enforce these rules in backend, not only UI.

Acceptance criteria:

-   Strategy Manager can submit but not approve
-   Planning Authority can approve but not edit budget content
-   unauthorized transitions are blocked server-side

**B5.3 — Lock Budget and Budget Allocation after submission**

Implement edit-lock behavior for Budget and Budget Allocation based on status.

Rules:

1.  Draft

-   Budget editable
-   Budget Allocations editable
-   Builder fully interactive

1.  Submitted

-   Budget content read-only
-   Budget Allocations read-only
-   Builder read-only
-   no saves allowed

1.  Approved

-   Budget fully locked
-   Budget Allocations fully locked
-   Builder read-only
-   no saves allowed

Important:

-   This lock must be enforced in backend validation
-   UI must also reflect the lock, but backend is authoritative

Block:

-   Budget edits in Submitted/Approved
-   Allocation create/update/delete in Submitted/Approved

Acceptance criteria:

-   Builder cannot save allocation once budget is Submitted or Approved
-   direct API/doc writes are blocked

Budget Approval Workflow

# Design

**Short answer: don’t punt rejection — but don’t overbuild it either.**

**If you skip it now, you will break your own workflow within minutes of real use. But if you overdesign it, you’ll slow down momentum.**

**So the right move is:**

**✅ Implement a minimal, strict rejection flow NOW  
❌ Do NOT build a full revision/iteration system yet**

**⚠️ Why You Should NOT Punt It**

**Right now you have:**

**Draft → Submitted → Approved**

**This is a one-way pipe.**

**In reality:**

-   **Most budgets will NOT be approved on first pass**
-   **Approvers need to send things back**

**Without rejection:**

-   **Approver gets stuck**
-   **Manager creates duplicates**
-   **Data integrity collapses**

**👉 This is not a “nice-to-have” — it’s structurally required.**

**✅ Minimal Rejection Flow (Production-Safe)**

**State Model (Keep it tight)**

**Draft → Submitted → Approved**

**↓**

**Rejected → Draft**

**That’s it. No extra states.**

**Behavior Rules**

**1\. Reject Action (Approver only)**

**When status = Submitted:**

-   **Show:**
    -   **✅ Approve**
    -   **❌ Reject**

**2\. Reject Modal (MANDATORY)**

**Keep it simple but strict:**

**Reason for rejection (required)**

**\[ textarea \]**

**\[ Reject Budget \]**

**👉 No reason = no rejection**

**3\. On Reject**

**System must:**

-   **Set:**
-   **status = Rejected**
-   **Store:**
-   **rejection\_reason**
-   **rejected\_by**
-   **rejected\_at**
-   **Make record editable again**

**4\. What User Sees (Strategy Manager)**

**In My Work / Draft tab:**

**FY2027 Budget \[REJECTED\] 🔴 Needs Revision**

**And in summary panel:**

**⚠️ Rejected: Increase allocation detail for Workforce program**

**5\. Edit Behavior**

**Rejected budget:**

-   **Editable ✅**
-   **Can be re-submitted → goes back to Submitted**

**🚫 What NOT to Build Yet**

**Do NOT add:**

-   **version history**
-   **multiple rejection cycles tracking**
-   **threaded comments**
-   **approval chains**
-   **partial approvals**

**👉 That’s iteration phase, not MVP**

**🧠 Why This Is the Right Cut**

**You get:**

-   **Complete lifecycle ✔**
-   **Real usability ✔**
-   **Minimal complexity ✔**

**Without:**

-   **overengineering ❌**

**🔥 Decision Framework (Use This Going Forward)**

**When unsure, ask:**

**“Does skipping this block real user action?”**

**If YES → implement minimal version now  
If NO → defer**

**Rejection = YES**

**🧭 Final Recommendation**

**👉 Implement rejection now, but only:**

-   **one field (reason)**
-   **one state (Rejected)**
-   **one loop (back to Draft)**

**That’s your sweet spot.**

**B5.9 — Add Rejected state and valid transitions**

Extend the Budget status model to include rejection.

Status values must now be exactly:

-   Draft
-   Submitted
-   Approved
-   Rejected

Valid transitions:

-   Draft → Submitted
-   Submitted → Approved
-   Submitted → Rejected
-   Rejected → Submitted

Rules:

1.  Approved remains final in v1
2.  Rejected budget becomes editable again
3.  Do not allow direct Draft → Rejected
4.  Do not allow Approved → Rejected
5.  Invalid transitions must be blocked server-side

Acceptance criteria:

-   Submitted budget can be rejected
-   Rejected budget can be re-submitted
-   invalid transitions are blocked

**B5.10 — Add rejection metadata fields**

Add minimal rejection metadata to Budget.

Add these fields:

-   rejection\_reason
-   rejected\_by
-   rejected\_at

Rules:

1.  rejection\_reason is required when status changes from Submitted → Rejected
2.  rejected\_by is set automatically to current user
3.  rejected\_at is set automatically to current timestamp
4.  when budget is re-submitted, previous rejection metadata may remain for v1 unless that creates UX issues
5.  do not add approval history tables yet

Acceptance criteria:

-   rejected budgets store reason, actor, and timestamp
-   rejecting without a reason is blocked

**B5.11 — Add backend reject action**

Implement a backend reject\_budget(budget\_name, rejection\_reason) action.

Requirements:

1.  Current status must be Submitted
2.  Only Planning Authority or Administrator may reject
3.  rejection\_reason is required
4.  On success:
    -   status = Rejected
    -   rejected\_by = current user
    -   rejected\_at = current timestamp
    -   rejection\_reason = provided reason
5.  Return clear success/error messages
6.  Enforce all checks server-side

Do not implement comment threads or return-for-edit workflow objects.

Acceptance criteria:

-   reject action callable from UI
-   unauthorized users cannot reject
-   submitted budget becomes Rejected

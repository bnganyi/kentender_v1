Demand Intake and Approval — UI Specification

**1\. UI design principles**

**1.1 Primary UI purpose**

The UI must help users:

-   find work immediately
-   understand the status of a demand record
-   take the next valid action
-   avoid unnecessary scrolling or searching

**1.2 Core UX model**

The module must use:

Header

→ KPI strip

→ role-aware work surface

→ master-detail

→ structured builder

**1.3 What this UI must not become**

It must not become:

-   a generic record list
-   a giant requisition form
-   a tab overload surface
-   a catch-all procurement workbench

**2\. Information architecture**

The module has **two primary surfaces**:

**A. Landing / Workbench**

Purpose:

-   queue-driven work management
-   status visibility
-   action initiation
-   approval handling

**B. Builder / Record Editor**

Purpose:

-   create, edit, review, and act on one demand record

These are separate on purpose.

**3\. Landing page specification**

**3.1 Route**

Recommended route:

/desk/demand-intake-and-approval

Alias may exist if needed, but this is the canonical workspace route.

**3.2 Page header**

**Structure**

Left:

-   Title: **Demand Intake and Approval**
-   Subtitle: **Capture, approve, and prepare procurement demand for planning.**

Right:

-   Primary action button:
    -   **New Demand**

**Rules**

-   title and subtitle left-aligned
-   only one primary creation CTA in header
-   no approval buttons in header
-   header remains stable across roles; action visibility changes by role

**3.3 KPI strip**

The KPI strip is required and must sit directly below the header.

**Base KPI cards**

The exact cards shown should vary slightly by role, but the layout remains consistent.

**Requisitioner default KPI set**

-   My Drafts
-   Pending My Action
-   Rejected / Returned
-   Total Value of My Demand

**HoD default KPI set**

-   Pending HoD Approval
-   Returned This Week
-   Emergency Requests
-   Total Value Pending Approval

**Finance default KPI set**

-   Pending Finance Approval
-   Budget Failures
-   Emergency Requests
-   Total Value Pending Approval

**Procurement default KPI set**

-   Approved
-   Planning Ready
-   Emergency Approved
-   Total Value Ready for Planning

**Card behavior**

Each KPI card:

-   shows a small label
-   shows one large value
-   may be clickable to filter the queue
-   must not truncate numbers

**Formatting rules**

Examples:

-   24
-   KES 15,240,000
-   3
-   7

Do not wrap numbers awkwardly.

**3.4 Top-level work tabs**

To avoid the scaling problem you already saw, top-level tabs must remain minimal.

**Allowed top-level tabs**

-   My Work
-   All
-   Approved
-   Rejected

That is the entire top row.

**Rules**

-   do not add state-specific tabs here
-   do not add role-specific tabs here
-   these tabs are broad buckets only

**3.5 Secondary queue bar**

Below the top-level tabs, provide a **role-aware queue selector**.

This is where scalable work routing happens.

**Requisitioner queue options**

-   My Drafts
-   Submitted by Me
-   Returned to Me
-   Rejected
-   My Approved

**HoD queue options**

-   Pending HoD Approval
-   Returned Awaiting Resubmission
-   Emergency Requests
-   All Department Requests

**Finance queue options**

-   Pending Finance Approval
-   Budget Exceptions
-   Emergency Requests
-   Approved Today

**Procurement queue options**

-   Planning Ready
-   Approved Not Yet Planned
-   Emergency Approved
-   All Approved Demand

**UI form**

These should be implemented as:

-   pills
-   chips
-   segmented controls
-   or a left-side sub-filter bar

Not as another tab strip equal in weight to the main tabs.

**3.6 Filter row**

Below the queue selector, provide scalable filters.

**Filters**

-   Demand Type
-   Department
-   Budget Line
-   Priority
-   Requisition Type
-   Date Range
-   Amount Range
-   Status

**Rules**

-   filters should be collapsible on smaller screens
-   default filters should be empty
-   selected filters should appear as chips
-   filters should refine the currently active queue, not replace it

**Search**

Provide a search field for:

-   demand ID
-   title
-   requester
-   department

Search is secondary. It does not replace role queues.

**3.7 Main workspace layout**

Use the now-standard module pattern:

Left panel (master list): 35–40%

Right panel (detail panel): 60–65%

This must remain stable across roles.

**4\. Master list specification**

**4.1 List title**

Depends on active queue, for example:

-   My Drafts
-   Pending Finance Approval
-   Planning Ready

Do not always call it “Demand Records.” The queue context should be explicit.

**4.2 Each list row must show**

**Required elements**

-   Demand ID
-   Title
-   Department
-   Priority badge
-   Status badge
-   Demand Type badge
-   Requested Amount
-   Required By date

**Optional role-relevant secondary line**

Depending on role:

-   Budget Line
-   Requester
-   Performance Target summary
-   Reservation status
-   Exception flag

**Row example**

DIA-2026-00124

Procurement of ultrasound machines

Clinical Services

High Planned Pending Finance Approval

KES 5,000,000 Required: 2026-06-30

Budget Line: Medical Equipment Capex

**4.3 Visual rules**

-   selected row has stronger highlight and border
-   status badge is always visible
-   emergency and unplanned demand visually stand out
-   amount is visually scannable
-   rows should not become card-heavy; list density matters here

**4.4 Empty queue state**

If the current queue has no items, show:

-   queue-specific empty message
-   relevant next action if applicable

Examples:

-   “No drafts yet. Create one to begin.”
-   “No pending HoD approvals.”
-   “No planning-ready demand.”

**5\. Detail panel specification**

The detail panel must change based on the selected record and the role.

**5.1 If no record selected**

Show:

-   a simple empty state
-   queue-specific hint

Example:

-   “Select a demand record to view details and take action.”

**5.2 If a record is selected**

The detail panel must be sectioned.

**Section A — Demand summary**

Show:

-   Title
-   Demand ID
-   Status badge
-   Demand Type badge
-   Priority badge
-   Department
-   Requester
-   Request Date
-   Required By Date

**Section B — Budget and strategy**

Show:

-   Budget Line
-   Budget Name
-   Reservation Status
-   Strategic Plan
-   Program
-   Sub-program
-   Output Indicator
-   Performance Target

This section is critical because this module is the trust boundary between intent and spend.

**Section C — Financial summary**

Show:

-   Total Requested Amount
-   Available Budget Snapshot, if checked
-   Reservation Reference, if reserved

**Section D — Items summary**

Show:

-   line count
-   a compact item preview table or expandable list:
    -   item/service description
    -   qty
    -   unit
    -   unit cost
    -   line total

Do not show a huge full editor here; this is a summary surface.

**Section E — Workflow and audit**

Show:

-   current approval stage
-   submitted\_by / submitted\_at
-   HoD approval metadata if present
-   finance approval metadata if present
-   rejection metadata if present
-   planning status

**Section F — Actions**

Only valid actions for the current role and state appear here.

**6\. Action model in detail panel**

**6.1 Requisitioner actions**

**Draft**

-   Edit Demand
-   Submit for Approval
-   Cancel

**Pending HoD / Pending Finance**

-   View only
-   Cancel if policy permits

**Rejected / Returned**

-   Edit Demand
-   Re-submit for Approval
-   Cancel

**Approved / Planning Ready**

-   View only

**6.2 HoD actions**

**Pending HoD Approval**

-   Approve
-   Return to Draft
-   Reject

**Other states**

-   View only

**6.3 Finance actions**

**Pending Finance Approval**

-   Approve & Reserve Budget
-   Return to Draft
-   Reject

**Other states**

-   View only

**6.4 Procurement actions**

**Approved**

-   Mark Planning Ready

**Planning Ready**

-   View only

**Other states**

-   View only

**6.5 Administrator actions**

-   all valid actions, with caution controls

**7\. Builder / record editor specification**

This is the second major UI surface.

**7.1 Route**

Recommended route pattern:

/desk/demand-record/<id>

For new:

/desk/demand-record/new

**7.2 Builder header**

Left:

-   Back to Demand Intake and Approval
-   Demand ID or “New Demand”
-   status badge

Right:

-   state-aware actions only

Examples:

-   Save Draft
-   Submit for Approval
-   Re-submit
-   Cancel

Do not clutter the header with approval buttons that belong only to approvers if you are on a requisitioner editing view.

**7.3 Builder layout rules**

**Overall**

-   white background
-   section cards or section dividers
-   compact but readable
-   two-column layout for structured fields
-   full-width for narrative fields and items section

**Prohibited**

-   giant grey background slabs
-   endless vertical stacks of unrelated fields
-   three-plus-column dense grids

**8\. Builder section specification**

**Section 1 — Basic Request**

Fields:

-   Title
-   Request Date
-   Required By Date
-   Department
-   Priority
-   Requisition Type
-   Demand Type

Layout:

-   two columns for structured short fields
-   title full width

**Section 2 — Items**

Fields:

-   child item table
-   Add Item action
-   per-row fields:
    -   Description
    -   Category
    -   UOM
    -   Quantity
    -   Estimated Unit Cost
    -   Line Total
    -   Notes

Rules:

-   line totals auto-compute
-   total amount shown at section footer
-   allow row add/remove in editable states only

**Section 3 — Business Justification**

Fields:

-   Justification
-   Specification Summary
-   Beneficiary Summary (optional)

Rules:

-   full width
-   required fields clearly marked

**Section 4 — Strategy Linkage**

Preferred mode:

-   Budget Line selector
-   derived strategy fields shown read-only

Fallback mode:

-   progressive hierarchical selectors
    -   Program
    -   Sub-program
    -   Output Indicator
    -   Performance Target

Rules:

-   do not expose a single giant target dropdown by default
-   derived fields should be visible, not hidden

**Section 5 — Budget Linkage**

Fields:

-   Budget Line
-   Budget
-   Funding Source
-   Available Budget Snapshot
-   Reservation Status

Rules:

-   most fields read-only/derived
-   validation feedback clear

**Section 6 — Delivery and Fulfilment**

Fields:

-   Delivery Location
-   Requested Delivery Period
-   special delivery notes if needed

**Section 7 — Flags and Exceptions**

Fields:

-   Demand Type
-   is\_exception (derived)
-   Impact if Not Procured
-   Emergency Justification

Rules:

-   show conditional fields only when relevant
-   do not leave empty placeholders visible

**Section 8 — Workflow**

Fields:

-   current status
-   current queue/stage
-   approval history summary
-   planning status

Mostly read-only.

**Section 9 — Closure / Cancellation**

Fields:

-   cancellation reason if cancelling
-   rejection summary if rejected

**9\. Approval UX specification**

**9.1 Approval actions should not require opening edit mode**

Approvers should usually act from:

-   detail panel
-   or a dedicated review surface

not from the full requisitioner editor

**9.2 HoD approval modal**

On Approve:

-   simple confirmation

On Return:

-   mandatory return reason

On Reject:

-   mandatory rejection reason

**9.3 Finance approval modal**

On Approve & Reserve Budget:  
Show summary:

-   requested amount
-   budget line
-   available budget
-   remaining after reservation

Require confirmation.

On Return:

-   mandatory return reason

On Reject:

-   mandatory rejection reason

This is where budget-control seriousness must be visible.

**10\. Visual treatment of exceptions**

**10.1 Planned demand**

-   neutral/standard badge

**10.2 Unplanned demand**

-   visible exception badge
-   highlighted in queues
-   justification summary shown in detail panel

**10.3 Emergency demand**

-   highest-contrast badge
-   visible in KPI cards and queues
-   detail panel shows emergency justification near top
-   should be easy for approvers to identify immediately

The point is not decoration. The point is operational triage.

**11\. Role-based UI behavior summary**

**Requisitioner**

Primary needs:

-   create/edit drafts
-   see returned/rejected work
-   track submitted items

UI emphasis:

-   My Work
-   Drafts
-   Returned to Me

**HoD**

Primary needs:

-   approve/return/reject business requests
-   see key summary fast

UI emphasis:

-   Pending HoD queue
-   concise detail panel
-   high-signal action buttons

**Finance**

Primary needs:

-   validate budget and reserve
-   identify exceptions and emergency items

UI emphasis:

-   Pending Finance queue
-   budget section prominence
-   reservation summary in detail panel/action modal

**Procurement**

Primary needs:

-   consume approved demand into planning
-   identify planning-ready backlog

UI emphasis:

-   Planning Ready queue
-   compact but complete summary
-   handoff action

**12\. Navigation model**

**Entry points**

-   Desktop icon / workspace
-   global search
-   contextual links from Budget / later Planning

**Back navigation**

Every builder/review screen must clearly return to:

-   DIA landing page
-   same queue if possible
-   same selected record if possible

This lesson came directly from your earlier builder feedback.

**13\. Mobile / reduced-width behavior**

The primary target is desktop, but the layout should degrade sensibly.

**For narrow widths**

-   KPI cards wrap
-   master-detail stacks vertically
-   filter row collapses
-   detail panel appears below list
-   builder sections remain usable

Do not attempt to optimize the full approval workflow for mobile-first behavior in v1.

**15\. UI acceptance criteria**

The UI is acceptable only if:

1.  each role lands directly into actionable work
2.  work can be found without scanning all records
3.  the detail panel makes approval decisions possible without opening full edit mode in most cases
4.  requisitioner entry is structured and not scroll-heavy
5.  budget and strategy linkage are visible and understandable
6.  unplanned/emergency demand is clearly distinguished
7.  approval actions are clear, local, and state-valid
8.  queue/filter model scales better than a long tab strip
9.  approved/planning-ready records clearly show downstream readiness
10.  the UI feels like a governed workbench, not a generic form system

🧱 DIA Build Strategy

**Procurement Planning depends on DIA being real, not theoretical.**

So we focus on **building DIA to production-ready completeness first**, using the standard you just locked.

**🧭 What “Build DIA First” Actually Means**

Not just:

-   forms
-   workflows
-   UI

It means:

DIA must become a trusted upstream system

that Procurement Planning can depend on without re-validation.

If Planning has to re-check:

-   budget
-   approvals
-   legitimacy

👉 DIA has failed.

We now transition from **design → execution**, but in a controlled way.

**Phase Breakdown (Strict)**

**Phase 1 — Domain + Governance (Backend First)**

👉 No UI yet beyond minimal scaffolding

**Deliverables:**

-   Demand DocType (full schema)
-   Demand Item child table
-   Status field + transitions
-   Backend validation logic:
    -   strategy linkage
    -   budget linkage
    -   demand type rules
-   Approval actions:
    -   submit
    -   approve (HoD)
    -   approve (Finance)
    -   reject / return
-   Budget reservation integration (stub or real)
-   Locking rules enforced server-side

**Exit Criteria:**

-   Can run full lifecycle via API or console
-   No invalid state transitions possible

**Phase 2 — Work Queue + Landing (Critical UX)**

👉 This is where most systems fail—don’t rush it

**Deliverables:**

-   Landing page:
    -   KPI strip
    -   role-aware default views
    -   minimal tabs (My Work / All / Approved / Rejected)
-   Queue logic:
    -   Pending HoD
    -   Pending Finance
    -   Planning Ready
-   List + detail panel
-   Action buttons (role-aware)

**Exit Criteria:**

-   Each role lands directly in actionable work
-   No one needs to “search” for tasks

**Phase 3 — Builder (Structured Entry UX)**

**Deliverables:**

-   Sectioned form (as per PRD)
-   2-column layout
-   derived fields working (strategy, totals)
-   validation feedback (inline)

**Exit Criteria:**

-   Requisitioner can create + submit without confusion
-   No form sprawl

**Phase 4 — Budget Integration (Hard Control)**

**Deliverables:**

-   Budget validation at finance step
-   Reservation creation
-   Reservation release logic
-   Budget snapshot stored

**Exit Criteria:**

-   Cannot approve without funds
-   Reservation lifecycle correct

**Phase 5 — Exception Handling**

**Deliverables:**

-   Planned / Unplanned / Emergency logic enforced
-   Required fields by type
-   UI indicators (flags, badges)

**Exit Criteria:**

-   Emergency demand is clearly distinguishable and governed

**Phase 6 — Permissions + Security**

**Deliverables:**

-   Role-based access enforced backend
-   UI visibility aligned
-   cross-department restrictions enforced

**Exit Criteria:**

-   No unauthorized actions possible via API

**Phase 7 — Seed Data**

**Deliverables:**

-   realistic scenarios:
    -   draft
    -   pending HoD
    -   pending finance
    -   approved
    -   rejected
    -   emergency
-   linked to Strategy + Budget

**Phase 8 — Smoke Contracts**

**Deliverables:**

-   automated tests covering:
    -   full lifecycle
    -   rejection loop
    -   budget failure
    -   role restrictions
    -   planning-ready output

**⚠️ Critical Build Rules (From Lessons Learned)**

**1\. Governance first, always**

You do not build UI that implies states not enforced in backend.

**2\. No “we’ll fix it later” on:**

-   approval flow
-   budget reservation
-   rejection loop

You already saw what happens.

**3\. No overengineering**

Do NOT add:

-   workflow engines
-   versioning
-   threaded comments
-   multi-stage committees

Yet.

**4\. No module leakage**

DIA must NOT:

-   create procurement plans
-   talk to suppliers
-   create tenders

**5\. Work queues over filters**

If users need to filter to find work → UX failed.

**🧠 What Success Looks Like**

When DIA is done correctly:

-   A HoD logs in → sees approvals immediately
-   Finance logs in → sees budget decisions immediately
-   Procurement logs in → sees planning-ready demand
-   Requisitioner → can track and revise easily

And most importantly:

Planning module can trust DIA output without re-checking anything.

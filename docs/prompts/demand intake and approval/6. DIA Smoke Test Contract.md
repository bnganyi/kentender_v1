Demand Intake and Approval — Smoke Contract v1

**1\. Purpose**

This smoke contract proves that DIA works as a governed intake module, not just a form.

It must verify that:

-   demand can be created correctly
-   all approvals follow the defined path
-   budget control is enforced
-   exceptions are governed
-   queues route users to the right work
-   approved demand becomes planning-ready without ambiguity

**2\. Scope**

**Included**

-   landing/workbench
-   builder / record editor
-   state transitions
-   role-based work queues
-   budget reservation behavior
-   rejection / return / cancellation
-   emergency / unplanned validation
-   planning-ready handoff

**Excluded**

-   Procurement Planning itself
-   Tendering
-   Supplier interaction
-   Contracting
-   asset/inventory downstream flows

**3\. Required seed assumptions**

These tests assume the seed packs exist and are callable:

-   seed\_core\_minimal
-   seed\_strategy\_basic

**8\. Core smoke scenarios**

**S1 — Empty-state landing**

**Purpose**

Verify that the module behaves correctly with no demand records.

**Preconditions**

-   seed\_dia\_empty

**User**

-   Requisitioner

**Steps**

1.  Login as requisitioner
2.  Open DIA landing page
3.  Confirm workspace loads
4.  Confirm empty-state message shown
5.  Confirm New Demand CTA visible

**Expected**

-   no rows in queue
-   queue-specific empty message
-   no broken KPI cards
-   New Demand available

**S2 — Create draft demand**

**Purpose**

Verify draft creation works.

**Preconditions**

-   seed\_dia\_empty

**User**

-   Requisitioner

**Steps**

1.  Open landing page
2.  Click New Demand
3.  Fill minimum valid fields:
    -   title
    -   required by date
    -   priority
    -   demand type = Planned
    -   budget line
    -   justification
    -   specification summary
4.  Add one item row
5.  Save draft

**Expected**

-   new demand record created
-   status = Draft
-   appears in My Drafts
-   total amount derived correctly

**S3 — Cannot submit incomplete draft**

**Purpose**

Verify strong validation before workflow entry.

**Preconditions**

-   empty or fresh draft

**User**

-   Requisitioner

**Steps**

2.  Attempt Submit for Approval

**Expected**

-   status remains Draft

**S22 — Role visibility and permission enforcement**

**Purpose**

Verify users only see and do what they should.

**Preconditions**

-   seed\_dia\_extended

**Users and expectations**

**Requisitioner**

-   can create/edit drafts
-   cannot see finance approval actions
-   cannot mark planning ready

**HoD**

-   can approve/reject/return only in HoD stage
-   cannot finance approve
-   cannot edit request content directly as requisitioner

**Finance**

-   can finance approve
-   cannot create new demand as requisitioner unless explicitly allowed by role model

**Procurement**

-   can mark planning ready
-   cannot approve HoD or finance

**Admin**

-   full access

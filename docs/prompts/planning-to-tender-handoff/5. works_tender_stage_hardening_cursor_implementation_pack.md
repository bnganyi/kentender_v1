## 2. Purpose

This implementation pack converts the WORKS Tender-Stage Hardening Specification into a non-interpretive, Cursor-ready execution plan.

Cursor shall implement only the hardening foundation needed to make the integrated Works tender-stage baseline structurally reliable. This pack does **not** authorize full Tender Management v2, publication, bidder portal, bid submission, opening, evaluation, award, contract execution, final PDF generation, or production BoQ pricing.

The implementation shall harden:

1. Works Requirements;
2. section-bound drawings/specifications metadata;
3. tender-stage BoQ structure;
4. BoQ bill/item compatibility;
5. lot and package-line traceability;
6. supplier editability posture;
7. required forms linkage to future submission components;
8. derived model readiness placeholders;
9. Works hardening validation;
10. audit snapshot evidence;
11. integrated Works seed verification.

## 4. Implementation Strategy

Use a transitional, low-risk approach:

1. Keep existing `Procurement Tender`, `Tender Lot`, `Tender BoQ Item`, `Tender Required Form`, and `Tender Validation Message` where they already exist.
2. Add new hardening child DocTypes only where the current model lacks structure.
3. Extend `Tender BoQ Item` rather than replacing it in the first pass.
4. Add a parent `Tender BoQ` and child `Tender BoQ Bill` if practical. If not practical in the first pass, emulate bill/header structure through fields on `Tender BoQ Item` and document the transitional limitation.
5. Add hardening services under a dedicated service module, not scattered client logic.
6. Add deterministic validation result and audit snapshot generation.
7. Update the integrated Works seed after the schema/services are in place.
8. Add smoke tests for positive and negative hardening checks.


### 17.1 Status Calculation

Use this logic:

| Condition | Overall status |
|---|---|
| Any Critical finding | `Blocked` |
| Any service exception | `Failed` |
| No Critical and one or more High/Medium/Low findings | `Warning` |
| No findings or only Info | `Pass` |

Derived model placeholder findings should usually be `High`, not `Critical`, in this integration phase, because publication is out of scope. They must still have `blocking_for = Publication`.

## 21. BoQ Hardening Rules

`harden_works_boq_structure(tender_name)` shall:

1. Ensure BoQ rows exist; if none exist and existing POC generator is available, call it.
2. Set `boq_code = BOQ-{tender_reference_or_name}-V1` unless already present.
3. Set `boq_version = V1` unless already present.
4. Map `item_category` to `bill_code` and `bill_title`.
5. Set `item_number` deterministically if missing.
6. Set `quantity_locked = true` for all measured/daywork supplier-facing rows.
7. Set `allow_supplier_amount_edit = false` for all rows.
8. Set `supplier_rate_editable = true` only for measured/daywork rows where pricing is expected later.
9. Set `supplier_rate_editable = false` for summary/provisional/fixed rows.
10. Set `formula_type` based on category:
    - `PROVISIONAL_SUMS` → `Provisional Sum`
    - `DAYWORKS` → `Daywork`
    - `GRAND_SUMMARY` → `Summary`
    - rows with fixed amount and not bidder-priced → `Fixed`
    - otherwise → `Measured`
11. Set `boq_row_status = Active` if missing.
12. Populate `source_package_line_code` from lot mapping where possible.

Bill mapping:

| Category | Bill Code | Bill Title |
|---|---|---|
| `PRELIMINARIES` | `BILL-001-PRELIMINARIES` | Preliminaries |
| `SUBSTRUCTURE` | `BILL-002-SUBSTRUCTURE` | Substructure |
| `SUPERSTRUCTURE` | `BILL-003-SUPERSTRUCTURE` | Superstructure |
| `ROOFING` | `BILL-004-ROOFING` | Roofing |
| `FINISHES` | `BILL-005-FINISHES` | Finishes |
| `EXTERNAL_WORKS` | `BILL-006-EXTERNAL-WORKS` | External Works |
| `DAYWORKS` | `BILL-007-DAYWORKS` | Dayworks |
| `PROVISIONAL_SUMS` | `BILL-008-PROVISIONAL-SUMS` | Provisional Sums |
| `GRAND_SUMMARY` | `BILL-009-GRAND-SUMMARY` | Grand Summary |

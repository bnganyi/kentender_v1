## 2. Purpose

The original intent of this artefact was to harden the WORKS BoQ model. That remains essential, but the uploaded Tender Management v2 documentation shows that BoQ cannot be hardened in isolation.

For Works, tender-stage hardening must also cover Works Requirements, drawings, specifications, site information, ESHS requirements, STD-derived submission/opening/evaluation/contract models, addendum impact readiness, validation boundaries, audit evidence, and legacy lockout.

Therefore, this artefact defines a broader **WORKS Tender-Stage Hardening Specification**.

Its purpose is to define what must be strengthened after the successful STD/officer POC and before the integrated Works scenario can be considered structurally ready for Tender Management v2 and downstream modules.

This is not full Tender Management v2 implementation. It is a focused hardening layer around the Works tender-stage artefacts that downstream Tender v2 will later consume.

---

## 3. Design Principle

This hardening work shall implement the principle:

> A Works tender is not ready merely because a preview exists; it is ready only when its Works Requirements, BoQ, section-bound attachments, required forms, submission model, opening model, evaluation model, contract carry-forward model, validation evidence, and audit lineage are structurally consistent with the controlling STD instance.

This means:

1. BoQ is first-class structured data, not an upload or loose table.
2. Works Requirements are structured and supplier-facing where applicable.
3. Drawings and specifications may be attachments, but only as section-bound, versioned, classified supporting artefacts.
4. Required forms must derive from the STD engine.
5. Submission, opening, evaluation, and contract models must be derived from the STD engine or represented as explicit required outputs to be generated later.
6. Tender Management must not become the owner of STD-owned rules.
7. The hardening layer must fail closed at legal and cross-module transition boundaries.
8. The current POC tender implementation may be refactored where needed.

## 5. Non-Scope

This hardening step shall not implement:

1. public tender publication;
2. final legally certified PDF generation;
3. supplier portal;
4. supplier bid drafting or submission;
5. supplier BoQ rate entry UI;
6. bid sealing;
7. bid opening execution;
8. evaluation execution;
9. arithmetic correction execution;
10. award;
11. contract creation or execution;
12. payment certificates;
13. variations;
14. final account;
15. full addendum lifecycle;
16. external QS/estimation integrations;
17. full Tender Management v2 module.

However, this hardening step shall prepare the structural data and validation hooks that those modules require later.

## 6. Hardening Philosophy

The prior POC proved feasibility. This hardening step converts the POC from a demonstrator into a reliable tender-stage structural foundation.

The hardening philosophy is:

| POC posture | Hardened posture |
|---|---|
| Representative rows | Structured tender-stage model |
| Preview-visible BoQ | Validated bill/item BoQ |
| Forms generated | Forms linked to submission component readiness |
| Works scope summary | Structured Works Requirements container |
| Attachments optional | Attachments section-bound, versioned, classified |
| Validation checks local | Cross-module validation boundaries identified |
| Audit values visible | Audit/snapshot evidence defined |
| Downstream assumptions implicit | Submission/opening/evaluation/contract model outputs explicitly represented |

## 8. Works Requirements Hardening

### 8.1 Purpose

The Works Requirements section must not be a plain attachment bucket. It must be structured enough for suppliers, evaluators, and contract handoff to understand the works scope and obligations.

### 8.2 Required Works Requirements Components

| Component | Required | Purpose |
|---|---:|---|
| Scope of Works | Yes | Defines works to be delivered |
| Site Information | Yes | Site constraints, access, operating context |
| Technical Specifications | Yes | Defines technical standards/specifications |
| Drawings Register | Yes, where drawings exist | Lists drawings and versions |
| ESHS / HSE Requirements | Yes for Works baseline | Supplier-facing health, safety, environmental, social requirements |
| Work Programme Requirement | Yes | Generates supplier submission requirement |
| Key Personnel Requirement | Yes where qualification requires it | Links to required forms/submissions |
| Equipment Requirement | Yes where qualification requires it | Links to required forms/submissions |
| Completion Period | Yes | Carries to SCC/contract |
| Defects Liability Period | Yes | Carries to SCC/contract |

### 8.3 Minimum Data Model Direction

If a dedicated `Works Requirement` object does not yet exist, create a structured JSON or child-table model under the tender/STD instance with stable component codes.

Minimum fields:

| Field | Purpose |
|---|---|
| `component_code` | Stable key, e.g. `WRK_SCOPE`, `WRK_SITE`, `WRK_SPEC` |
| `component_type` | Scope, Site, Specification, Drawing, ESHS, Programme, Personnel, Equipment |
| `title` | Officer/supplier-facing title |
| `structured_text` | Main structured content |
| `supplier_facing` | Whether visible to suppliers later |
| `submission_linked` | Whether it generates supplier submission requirement |
| `contract_carry_forward` | Whether it carries to contract |
| `source_field_codes` | STD/tender config fields that generated it |
| `status` | Draft / Complete / Missing / Superseded |
| `version` | Component version |

### 8.4 Required Validation

| Validation ID | Check | Severity |
|---|---|---|
| WORKS-REQ-001 | Scope of Works exists and is structured text | Critical |
| WORKS-REQ-002 | Site information exists | High |
| WORKS-REQ-003 | Technical specification component exists | Critical |
| WORKS-REQ-004 | ESHS/HSE requirements exist or explicit not-applicable reason exists | High |
| WORKS-REQ-005 | Work programme requirement exists and is submission-linked | High |
| WORKS-REQ-006 | Contract carry-forward components are flagged | Critical |
| WORKS-REQ-007 | Supplier-facing components are classified | High |

## 10. BoQ Structural Hardening

### 10.1 Purpose

The BoQ must move from a representative flat list into a structured tender-stage bill/item model.

### 10.2 Required BoQ Object Direction

Prefer a two-level model:

```text
Tender BoQ
→ BoQ Bill
→ BoQ Item
```

If the current implementation only supports `Tender BoQ Item`, the hardening step may implement this as fields on rows first, but the model must be compatible with bill/item separation later.

### 10.3 BoQ Header Minimum Fields

| Field | Purpose |
|---|---|
| `boq_code` | Stable BoQ identifier |
| `tender` | Tender reference |
| `std_template` / `tender_std_instance` | STD source reference |
| `version` | BoQ version |
| `currency` | Pricing currency |
| `status` | Draft / Ready / Superseded |
| `source` | Seeded / Imported / STD Generated / Officer Entered |
| `quantity_authority` | Procuring Entity |
| `supplier_edit_policy` | Rates Only unless STD permits otherwise |
| `validation_status` | Not Run / Pass / Warning / Blocked |

### 10.4 BoQ Bill Minimum Fields

| Field | Purpose |
|---|---|
| `bill_code` | Stable bill code |
| `bill_number` | Ordered bill number |
| `bill_title` | Bill title |
| `lot_code` | Optional lot binding |
| `category` | Category, e.g. Substructure, Roofing |
| `display_order` | Ordering |
| `supplier_facing` | Whether shown to suppliers later |
| `status` | Active / Superseded |

### 10.5 BoQ Item Minimum Fields

| Field | Purpose |
|---|---|
| `item_code` | Stable item code |
| `bill_code` | Parent bill |
| `item_number` | Ordered item number |
| `lot_code` | Lot binding where applicable |
| `description` | Item description |
| `unit` | Unit of measurement |
| `quantity` | Procuring entity quantity |
| `quantity_locked` | Must be true for supplier-facing Works pricing |
| `pricing_required` | Whether supplier rate is required later |
| `supplier_rate_editable` | Whether supplier can enter rate |
| `fixed_amount` | Fixed/provisional amount where applicable |
| `formula_type` | Measured / Provisional Sum / Daywork / Summary / Fixed |
| `allow_supplier_amount_edit` | Usually false |
| `display_order` | Order within bill |
| `status` | Active / Superseded |

### 10.6 Category Model

The current nine categories remain valid as a baseline, but they must be represented as BoQ bills or category-backed bill groups.

Required baseline categories:

1. `PRELIMINARIES`
2. `SUBSTRUCTURE`
3. `SUPERSTRUCTURE`
4. `ROOFING`
5. `FINISHES`
6. `EXTERNAL_WORKS`
7. `DAYWORKS`
8. `PROVISIONAL_SUMS`
9. `GRAND_SUMMARY`

### 10.7 BoQ Structural Validation

| Validation ID | Check | Severity |
|---|---|---|
| WORKS-BOQ-001 | BoQ header exists | Critical |
| WORKS-BOQ-002 | At least one bill exists | Critical |
| WORKS-BOQ-003 | Each item belongs to a bill | Critical |
| WORKS-BOQ-004 | Each measured item has item number, description, unit, quantity | Critical |
| WORKS-BOQ-005 | Quantities are numeric and non-negative | Critical |
| WORKS-BOQ-006 | Quantity locked flag is true for supplier-facing Works items | Critical |
| WORKS-BOQ-007 | Supplier-editable fields are explicit and limited | Critical |
| WORKS-BOQ-008 | Fixed/provisional sums are marked fixed unless STD permits otherwise | Critical |
| WORKS-BOQ-009 | Summary rows cannot accept supplier rate | Critical |
| WORKS-BOQ-010 | Dayworks category exists when configured | High/Critical depending policy |
| WORKS-BOQ-011 | Provisional Sums category exists when configured | High/Critical depending policy |
| WORKS-BOQ-012 | BoQ version is present | High |

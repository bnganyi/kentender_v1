## 2. Executive Decision

The standalone STD and officer-facing tender configuration POC workstreams are accepted as successful.

The project shall now stop expanding the isolated POC and move into integration of the proven STD-driven tender creation capability with the broader public e-Procurement lifecycle.

The next workstream shall be:

```text
STD-Driven Tender Creation Integration Workstream
```

The purpose of this workstream is to connect upstream Strategy, Budget, Demand, and Procurement Planning data to STD-driven tender creation, using the complex WORKS template as the end-to-end integration scenario.

---

## 3. Reason for Pivot

The original research and design problem was whether Standard Tender Documents, especially complex Works STDs, could be represented tractably, configured safely, and used operationally without relying on uncontrolled document uploads.

That question has now been answered sufficiently by the prior POCs:

| Capability | Result |
|---|---|
| STD package representation | Proven |
| Works STD complexity handling | Proven at POC level |
| Admin inspection and validation | Proven |
| Rule trace and explainability | Proven |
| Officer-guided tender configuration | Proven |
| Required forms generation | Proven |
| Representative Works BoQ handling | Proven at POC boundary |
| Preview and audit linkage | Proven |
| Avoidance of raw template editing by ordinary users | Proven |

The next system risk is no longer whether STDs can be represented. The next risk is whether STD-driven tender creation fits correctly into the upstream planning and budgeting lifecycle.

---

## 4. Strategic Principle

This workstream shall implement the principle:

> A tender should be created from a planned, budget-supported, approved procurement package, and then configured through the controlled STD template engine.

This means:

1. tender creation should not begin as an isolated manual action where avoidable;
2. upstream planning data should pre-populate tender configuration where authoritative;
3. officer-entered tender values should be limited to fields that are legitimately tender-specific;
4. budget, plan, package, demand, and STD template references should remain traceable;
5. the Works scenario shall become the integrated demonstration path from strategy to tender preview;
6. preliminary tender implementation may be refactored where it conflicts with the better STD-driven design.

## 5. Current Implementation Observations

The source snapshot shows a reasonably mature upstream planning model and a successful but still preliminary tender surface.

### 5.1 Existing Upstream Objects

| Area | Existing object(s) | Current relevance |
|---|---|---|
| Strategy | Strategic Plan, Strategy Program, Sub Program, Strategy Objective, Strategy Target | Strategic justification and hierarchy |
| Budget | Budget, Budget Line, Funding Source, Budget Allocation | Funding and cost control context |
| Demand | Demand | Requisition-style intake object |
| Planning | Procurement Plan, Procurement Package, Procurement Package Line, Procurement Template | Planning and packaging objects |
| Tender | Procurement Tender, STD Template, Tender Lot, Tender BoQ Item, Tender Required Form, Tender Validation Message | STD-driven tender creation surface |

### 5.2 Existing Planning Chain

The current implementation supports this upstream chain:

```text
Strategic Plan / Strategy hierarchy
→ Budget / Budget Line
→ Demand
→ Procurement Package Line
→ Procurement Package
→ Procurement Plan
```

The current implementation does **not** yet persistently complete this chain into tender creation because `Procurement Tender` currently has no persistent link field to `Procurement Plan` or `Procurement Package`.

### 5.3 Existing Handoff Service

The planning service already contains a release-to-tender handoff surface:

```text
services/planning/tendering_handoff.py
```

It builds a release payload containing package, package code, plan ID, template ID, procurement method, estimated value, and currency.

However, the hook list for `release_procurement_package_to_tender` is currently empty. Therefore, release-to-tender does not yet create or link a `Procurement Tender` record.

### 5.4 Existing Tender Surface

The tender surface now supports:

1. `STD Template` selection;
2. guided officer fields on `Procurement Tender`;
3. `configuration_json` sync;
4. validation;
5. required forms;
6. representative BoQ;
7. preview;
8. audit values.

This tender implementation should be treated as useful POC proof, but not as untouchable architecture. It may be refactored to fit the correct integration model.

## 6. Core Integration Problem

The system currently has two successful but partially disconnected layers:

| Layer | Status | Gap |
|---|---|---|
| Upstream planning | Strategy, budget, demand, package, and plan objects exist | No completed persistent handoff to tender |
| STD-driven tender creation | Officer can configure tender from STD template | Tender is not yet anchored to approved package/plan |

The integration workstream must bridge these layers.

The central design question is:

> What exact planning object becomes the authoritative source for creating an STD-driven procurement tender?

Recommended answer:

```text
Procurement Package is the canonical tender-creation handoff object.
```

`Procurement Package` already aggregates demand and budget lines, belongs to a procurement plan, has a procurement method, contract type, estimated value, currency, template, and status values including `Ready for Tender` and `Released to Tender`.

---

## 7. Recommended Canonical Handoff Model

The canonical handoff shall be:

```text
Approved / Ready for Tender Procurement Package
→ Release to Tender
→ Create linked Procurement Tender
→ Select / map STD Template
→ Pre-populate officer configuration
→ Validate / forms / BoQ / preview
```

### 7.1 Source Object

| Source object | Role |
|---|---|
| `Procurement Package` | Primary tender-creation source |

### 7.2 Supporting Objects

| Supporting object | Role |
|---|---|
| `Procurement Plan` | Plan context and approval context |
| `Procurement Package Line` | Demand and budget composition |
| `Demand` | Original requisition/intake context |
| `Budget Line` | Funding and amount context |
| `Procurement Template` | Planning template and method/category defaults |
| `STD Template` | Legal/STD tender template package |

### 7.3 Target Object

| Target object | Role |
|---|---|
| `Procurement Tender` | STD-driven tender configuration record |

## 11. Design Decision: Do Not Retrofit a Wrong Tender Model

The current `Procurement Tender` implementation is preliminary and may be refactored.

The following rule applies:

> The STD POC should influence the future tender design; the preliminary tender DocType should not force the STD model into a weaker architecture.

Therefore:

1. keep proven STD template package logic;
2. keep officer-guided field model where useful;
3. keep validation/forms/BoQ/preview engine flow;
4. add missing upstream references deliberately;
5. do not preserve manual standalone tender creation as the primary path if package-driven creation is better;
6. allow standalone tender creation only as admin/demo fallback or exceptional path.

## 13. Required State Model Direction

The handoff should use existing package and tender states where possible.

### 13.1 Procurement Package States

Relevant existing states:

| State | Meaning for handoff |
|---|---|
| `Approved` | Package approved but not necessarily ready for tender |
| `Ready for Tender` | Cleared for tender creation |
| `Released to Tender` | Handoff has occurred |
| `Returned` | Sent back for correction |
| `Rejected` | Not available for tender |

### 13.2 Tender States

Relevant existing states:

| State | Meaning after handoff |
|---|---|
| `Draft` | Tender record created from package |
| `Configured` | Officer configuration started/synced |
| `Validation Failed` | Validation blockers/errors exist |
| `Validated` | Configuration passed validation |
| `Tender Pack Generated` | Preview generated |
| `POC Demonstrated` | POC review-ready state only |

### 13.3 Handoff Rule

A package should only create/release a tender when it is in:

```text
Ready for Tender
```

or an explicitly approved equivalent state.

After successful release, the package should become:

```text
Released to Tender
```

and the tender should start as:

```text
Draft
```

or:

```text
Configured
```

if the system has successfully pre-populated configuration values.

## 14. Upstream-to-Tender Field Mapping Direction

The integration must define a field authority model.

| Field category | Authority | Officer editability |
|---|---|---|
| Procuring entity | Upstream plan/package | Usually inherited/locked |
| Procurement method | Procurement package / planning template | Inherited; override only with governance |
| Procurement category | Package/template | Inherited/locked |
| Estimated cost | Package / package lines / budget | Inherited/controlled |
| Currency | Plan/package | Inherited |
| Works location | Demand/package/officer, depending upstream completeness | Prefer inherited if available; otherwise officer-entered |
| Lot structure | Package planning / officer completion | Mixed; must be traceable |
| Qualification thresholds | STD defaults + officer configuration | Officer configurable within validation |
| Securities | Officer configuration under STD rules | Officer configurable |
| Dates | Officer configuration | Officer configurable |
| BoQ readiness | Tender-stage Works model | Officer-managed within POC boundary |

This mapping shall be fully defined in the Planning-to-Tender Handoff Specification.

## 16. Required Mapping Between Planning Template and STD Template

The snapshot contains both:

| Object | Purpose |
|---|---|
| `Procurement Template` | Planning/package template master |
| `STD Template` | Legal STD package used for tender creation |

The integration must not confuse these two.

Recommended direction:

1. `Procurement Template` remains the planning/package template.
2. `STD Template` remains the tender document package.
3. A mapping must exist from planning template/category/method to allowed STD templates.
4. For the Works POC, the mapping shall point to `KE-PPRA-WORKS-BLDG-2022-04-POC`.

Candidate implementation options:

| Option | Description | Recommendation |
|---|---|---|
| Add `default_std_template` to `Procurement Template` | Direct link from planning template to STD template | Recommended for this phase |
| Use rule-based mapping service | Map category/method/value to STD templates | Later, more flexible |
| Keep fixed allowlist in code | Fast but brittle | Accept only as temporary POC fallback |

The existing standalone STD sample tender may still be kept for engine-level tests, but the primary demonstration seed should become the integrated lifecycle scenario.

## 18. WORKS BoQ Hardening Direction

The BoQ remains the largest Works-specific production risk.

This workstream shall not build full production BoQ/pricing, but it shall harden tender-stage BoQ enough for integration.

Immediate BoQ hardening should include:

1. stable category model;
2. lot linkage checks;
3. Dayworks and Provisional Sums readiness checks;
4. representative row source from integrated seed, not only STD sample JSON;
5. clear distinction between officer/tender-stage BoQ definition and bidder pricing;
6. readiness status used by preview generation.

Full bidder pricing, arithmetic evaluation, measurement, payment certification, variations, and final accounts remain deferred.

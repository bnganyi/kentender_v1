## 2. Purpose

The purpose of this document is to define the handoff from upstream Procurement Planning into STD-driven Tender Management.

This specification connects the proven STD-WORKS and procurement officer configuration capability to the to-be Tender Management v2 architecture. Tender Management v2 is not treated as already implemented. It is the target orchestration module that will consume a released procurement package, bind it to a governed STD template or STD instance, and begin the tender lifecycle.

The handoff shall be designed so that tender creation starts from an authorized procurement package and not from an isolated manual tender record.

---

## 3. Design Principle

The Planning-to-Tender handoff shall implement the principle:

> Procurement Planning owns the approved procurement intent and package; the STD Engine owns tender content and derived legal models; Tender Management owns the controlled tender lifecycle orchestration.

This means:

1. Planning determines what is to be procured, budget lineage, package composition, procurement method, estimated value, and readiness for tendering.
2. The STD Engine determines the structured Standard Tender Document, configurable parameters, required forms, BoQ model, submission model, opening model, evaluation model, and contract carry-forward model.
3. Tender Management v2 coordinates lifecycle states, review, publication readiness, supplier participation, submission, closing, opening handoff, evaluation handoff, and contract handoff.
4. The handoff from Planning to Tender must validate all source-of-truth references before creating or activating a tender.
5. No tender lifecycle path may bypass the STD Engine once a tender is created from a package.

## 6. Canonical Handoff Object

The canonical source object for tender creation shall be:

```text
Procurement Package
```

Rationale:

1. It is the object that aggregates package lines, demands, budget references, procurement method, contract type, estimated value, currency, and package status.
2. It is closer to tendering than strategic plan, budget line, demand, or procurement plan alone.
3. It can represent a coherent tenderable unit.
4. It can be validated against procurement plan and budget lineage before release.
5. It can become the stable source reference for Tender Management v2.

### 7.1 Source Objects

## 8. Handoff Boundary

The handoff boundary is:

```text
Procurement Package: Ready for Tender
→ Release to Tender action
→ Planning-to-Tender validation
→ Tender record created or linked
→ STD template/binding initialized
→ Officer configuration pre-populated
→ Tender starts in Draft / Configuring state
```

The handoff does not mean tender publication. It only creates or initializes a controlled tender lifecycle record.

## 9. Preconditions for Release to Tender

A procurement package may be released to tender only if the following are true.

| Precondition | Required result |
|---|---|
| Package exists | Yes |
| Package status permits release | `Ready for Tender` or approved equivalent |
| Package has approved procurement plan | Yes |
| Package has at least one package line | Yes |
| Package has procurement method | Yes |
| Package has procurement category/type | Yes |
| Package has estimated value | Yes |
| Package has currency | Yes |
| Package has budget lineage | Yes, at package or line level |
| Package has demand or planning origin | Yes, where applicable |
| Package has planning template or method/category profile | Yes |
| STD template can be resolved | Yes |
| User is authorized | Yes |
| Cross-module validation passes | Yes |

If any mandatory precondition fails, release must fail closed.

## 10. Required State Transition

### 10.1 Source Package Transition

| Current package state | Action | New package state |
|---|---|---|
| `Ready for Tender` | Release to Tender | `Released to Tender` |

If the current package state is not `Ready for Tender`, the action shall be blocked unless a formally approved equivalent state is configured.

### 10.2 Target Tender Initial State

| Condition | Initial tender state |
|---|---|
| Tender record created but configuration not synced | `Draft` |
| Tender record created and package values pre-populated into configuration | `Configured` or equivalent |
| STD binding incomplete | `STD Instance Incomplete` if using Tender v2 state model |

For current transitional `Procurement Tender`, use existing state values where available. For Tender Management v2, use the v2 lifecycle states and separate STD binding/readiness dimensions.

## 12. STD Template Selection Rules

The handoff must resolve the correct STD Template.

### 12.1 Recommended Resolution Order

| Order | Source | Behavior |
|---:|---|---|
| 1 | `Procurement Template.default_std_template` | Use direct mapping if present |
| 2 | Package method/category/contract type mapping service | Resolve allowed STD templates |
| 3 | Fixed POC Works allowlist | Temporary fallback only |

### 12.2 Works POC Rule

For the Works POC:

```text
Procurement category = Works
AND procurement method = Open Tender
AND contract type/category supports building/civil works
→ STD Template = KE-PPRA-WORKS-BLDG-2022-04-POC
```

If multiple STD templates match, release must not guess. It must either require an authorized selection or block with a clear ambiguity error.

## 13. Planning-to-Tender Field Authority Model

Each tender field must have an authority classification.

| Authority type | Meaning |
|---|---|
| `Inherited Locked` | Comes from upstream and officer should not edit without governance |
| `Inherited Editable` | Comes from upstream as default but officer may refine |
| `STD Derived` | Comes from STD package or STD instance |
| `Officer Configured` | Officer enters value during tender configuration |
| `System Derived` | Computed by service |
| `Downstream Derived` | Populated later by publication/submission/evaluation/contract modules |

---

## 15. Configuration Pre-Population Rules

When a package is released to tender, the system shall create an initial configuration JSON for the STD engine.

Required behavior:

1. include inherited planning/package values;
2. include selected STD template metadata;
3. include defaults from STD field model where safe;
4. leave officer-specific values empty if not known;
5. mark validation status as `Not Validated`;
6. mark required forms, BoQ, and preview as not generated or stale;
7. store a package handoff snapshot or hash if implemented.

The system must not silently use sample/demo values for a real package release.

## 16. Release-to-Tender Service Behavior

### 16.1 Required Service

Implement or specify a service equivalent to:

```python
release_procurement_package_to_tender(package_name: str) -> dict
```

### 16.2 Required Steps

The service shall:

1. load Procurement Package;
2. check user permission;
3. verify package state permits release;
4. load related plan, package lines, budget lineage, and planning template;
5. run Planning-to-Tender validation boundary;
6. resolve STD Template;
7. prevent duplicate active tender creation from the same package unless supersession/retender path applies;
8. create target Tender / Procurement Tender;
9. persist package, plan, template, and STD references;
10. pre-populate STD configuration JSON;
11. create or initialize STD binding if v2 binding object exists;
12. write audit event;
13. update package status to `Released to Tender`;
14. return tender name/code and route.

### 16.3 Duplicate Handoff Rule

If the package already has an active tender, release shall block and return the existing tender reference unless a governed retender/supersession path exists.

## 19. Works-Specific Handoff Requirements

For Works/Open Tender packages:

1. selected STD template must be Works-compatible;
2. `WORKS.BOQ_REQUIRED` shall default to true;
3. package description shall pre-populate `WORKS.SCOPE_SUMMARY`;
4. package estimated value shall pre-populate `TENDER.ESTIMATED_COST`;
5. package currency shall pre-populate `TENDER.CURRENCY`;
6. package lines may seed preliminary lots or Works package components;
7. representative or tender-stage BoQ must be initialized through the BoQ hardening design, not hidden in document uploads;
8. release validation must flag missing Works/BoQ prerequisites as warnings or blockers according to current maturity level.

---

## 20. BoQ Handoff Boundary

This handoff shall not build full production BoQ/pricing.

It shall establish that:

1. Works package requires structured BoQ readiness;
2. BoQ rows are tender-stage structured data;
3. package lines may inform BoQ bills/lots, but do not equal full BoQ;
4. supplier rate entry is downstream and not part of this handoff;
5. arithmetic correction is downstream evaluation logic;
6. contract price is downstream evaluation/award/contract handoff logic;
7. BoQ readiness must be visible before tender preview/publication.

Detailed BoQ model shall be defined in the WORKS Tender-Stage BoQ Hardening Specification.

## 21. Error Handling Requirements

| Scenario | Required behavior |
|---|---|
| Package not found | Clear error |
| User lacks permission | Permission error |
| Package not releasable | Block with current status |
| Missing plan | Block |
| Missing package lines | Block |
| Missing estimated value/currency | Block |
| Missing budget lineage | Block or governed warning only if accepted for POC |
| STD template unresolved | Block |
| Multiple STD templates match | Block and require authorized selection |
| Active tender already exists for package | Block and return existing tender |
| Configuration pre-population fails | Roll back tender creation or mark failed without package release |
| Audit write fails | Fail closed for legal release |
| Validation service unavailable | Fail closed |

## 22. Transitional Implementation Guidance

Because Tender Management v2 is not yet fully implemented, the first integration implementation may use current `Procurement Tender` as transitional target under strict conditions.

Allowed transitional behavior:

1. create `Procurement Tender` from `Procurement Package`;
2. add persistent plan/package references;
3. pre-populate officer fields/configuration JSON;
4. reuse existing STD validation/forms/BoQ/preview POC services;
5. record handoff audit evidence if audit framework is available;
6. preserve v2-aligned boundaries.

Not allowed:

1. treat current tender DocType as final v2 model;
2. introduce document-upload-as-source behavior;
3. create manual submission checklist builder;
4. create manual opening/evaluation criteria fields;
5. bypass STD Engine;
6. publish tenders publicly from this transitional path.

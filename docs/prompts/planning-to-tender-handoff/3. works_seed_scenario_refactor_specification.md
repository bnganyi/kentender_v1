## 2. Purpose

The purpose of this document is to define how the existing seed data shall be refactored so the WORKS STD scenario is no longer an isolated tender POC but an integrated end-to-end procurement lifecycle fixture.

The refactored seed scenario shall demonstrate the following chain:

```text
Strategic Plan
→ Strategy Program / Objective / Target
→ Budget and Budget Line
→ Demand
→ Procurement Plan
→ Procurement Template
→ Procurement Package and Package Lines
→ Package Ready for Tender
→ Release to Tender
→ Procurement Tender / Tender v2 target
→ STD Template Binding
→ Officer Configuration
→ Validation / Required Forms / BoQ / Preview
```

The seed refactor shall not attempt to implement full Tender Management v2 publication, supplier portal, bid submission, evaluation, award, or contract execution. It shall prepare a coherent integrated fixture that downstream modules can consume later.

---

## 3. Design Principle

The seed refactor shall implement the principle:

> The primary WORKS demonstration must prove that STD-driven tender creation begins from a planned, budget-supported, tenderable procurement package, not from an isolated manually created tender.

This means:

1. the WORKS tender must have visible upstream strategic, budget, demand, plan, package, and package-line lineage;
2. the procurement package must be the canonical release source;
3. package release must select or resolve the WORKS STD template;
4. tender configuration must be pre-populated from package and plan data where authoritative;
5. officer-only values must remain officer-configured, not silently filled with demo values except through explicit sample actions;
6. BoQ seed data must move toward tender-stage structured data rather than being treated as a document upload or opaque sample blob;
7. downstream Tender v2 seed scenarios may reference this integrated fixture later, but this workstream stops at STD-driven tender preview/readiness.

## 18. Release-to-Tender Seed Behavior

The seed refactor shall support one of two modes.

### 18.1 Preferred Mode: Service-Generated Tender

The seed loader creates upstream records, then calls:

```text
release_procurement_package_to_tender(PKG-MOH-2026-001)
```

Expected result:

1. package release validation passes;
2. tender is created;
3. tender links to procurement package and plan;
4. STD template is selected;
5. officer fields/configuration are pre-populated;
6. package status becomes `Released to Tender`;
7. handoff audit/snapshot is created where available.

### 18.2 Transitional Mode: Seeded Tender with Verification

If the service is not implemented yet, the seed may create the tender directly, but it must include:

1. procurement package link;
2. procurement plan link;
3. STD template link;
4. pre-populated configuration JSON;
5. package status set consistently;
6. an explicit seed note that this bypasses the service temporarily;
7. a smoke check that later service implementation must replace the shortcut.

Preferred mode is service-generated tender.

## 22. Tender-Stage BoQ Seed Requirements

The integrated seed shall contain structured tender-stage BoQ rows.

The current representative 9-row model may be retained temporarily, but it should be refactored to derive from the integrated Works package and lot structure.

Expected categories:

1. `PRELIMINARIES`
2. `SUBSTRUCTURE`
3. `SUPERSTRUCTURE`
4. `ROOFING`
5. `FINISHES`
6. `EXTERNAL_WORKS`
7. `DAYWORKS`
8. `PROVISIONAL_SUMS`
9. `GRAND_SUMMARY`

Minimum rules:

1. each non-summary row shall have a valid lot reference where applicable;
2. `DAYWORKS` row shall exist if `WORKS.DAYWORKS_INCLUDED = true`;
3. `PROVISIONAL_SUMS` row shall exist if `WORKS.PROVISIONAL_SUMS_INCLUDED = true`;
4. summary row may be unlinked to a lot;
5. row structure shall be sufficient for tender-stage readiness;
6. supplier pricing fields remain downstream and out of scope here.

Detailed BoQ row model is deferred to the BoQ hardening artefact.

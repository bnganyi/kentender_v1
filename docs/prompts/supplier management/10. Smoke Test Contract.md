Supplier Management — Smoke Contract (v1.0, Locked)

**1\. Role definitions**

| **Role** | **Description** |
| --- | --- |
| External Supplier | Owns supplier profile, uploads documents, submits for review |
| Supplier Registry Officer | Manages supplier records and onboarding flow |
| Compliance Officer | Verifies documents and compliance |
| Procurement Officer | Consumes supplier data (read-heavy role) |
| Approving Authority | Approves/rejects suppliers and category qualification |
| Auditor | Read-only, full visibility |
| Admin | Full control, including overrides and configuration |

**2\. Permission model principles**

**Principle 1 — Ownership boundary**

External Supplier → own data only

Internal Users → governed access by role

**Principle 2 — Governance separation**

-   Creation ≠ Approval
-   Upload ≠ Verification
-   Qualification ≠ Consumption

**Principle 3 — State-aware permissions**

Permissions must depend on:

approval\_status

operational\_status

compliance\_status

**12\. Critical enforcement rules**

**Rule 1 — External isolation**

External suppliers:

-   can only see their own data
-   cannot infer other suppliers
-   cannot query lists

**Rule 2 — No dual control**

-   same user cannot:
    -   upload + verify document
    -   submit + approve supplier

**Rule 3 — Status-driven visibility**

Actions must be shown only if:

role is allowed AND state is valid

**Rule 4 — Admin is controlled power**

Admin can:

-   override
-   fix data

BUT:

-   actions must still log history
-   no silent changes

**Rule 5 — Procurement Officer is read-heavy**

They:

-   do not mutate supplier data
-   consume eligibility and categories

**13\. Acceptance criteria**

Permissions are correct only if:

-   external supplier cannot access other suppliers
-   no role can bypass lifecycle rules
-   document verification is restricted
-   approval authority is enforced
-   operational status cannot be changed casually
-   admin overrides are audited
-   procurement officer cannot modify supplier data
-   auditor is strictly read-only
-   API respects same permissions as UI

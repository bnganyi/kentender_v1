Supplier Management — Governance / State Model v1.0

**1\. Purpose**

This artifact defines how supplier records move through approval, compliance, operational eligibility, and external access states.

Core rule:

Supplier status is not informational. It directly controls whether a supplier can participate in procurement.

**2\. State dimensions**

Supplier Management has **four related but separate state dimensions**.

| **State Dimension** | **Purpose** | **Example Values** |
| --- | --- | --- |
| Approval Status | Onboarding / qualification decision | Draft, Submitted, Under Review, Approved, Returned, Rejected |
| Operational Status | Participation eligibility posture | Pending, Active, Suspended, Blacklisted, Expired |
| Compliance Status | Document / requirement health | Unknown, Complete, Incomplete, Expired, Non-Compliant |
| Access Status | External API/user access posture | Pending, Active, Suspended, Revoked |

Do **not** collapse these into one field. They answer different governance questions.

**3\. Approval status model**

**3.1 Approval states**

| **State** | **Meaning** |
| --- | --- |
| Draft | Supplier profile is being created |
| Submitted | Supplier has submitted registration for review |
| Under Review | Internal review is active |
| Approved | Supplier passed onboarding review |
| Returned | Supplier must correct missing/invalid information |
| Rejected | Supplier registration was rejected |

**3.2 Approval transitions**

| **From** | **To** | **Actor** | **Preconditions** | **Reason Required** |
| --- | --- | --- | --- | --- |
| Draft | Submitted | External Supplier, Registry Officer, Admin | Required fields and required registration documents present | No |
| Returned | Submitted | External Supplier, Registry Officer, Admin | Corrections made | No |
| Submitted | Under Review | Compliance Officer, Registry Officer, Admin | Supplier is submitted | No |
| Under Review | Approved | Approving Authority, Admin | Required documents verified, category review acceptable | No |
| Under Review | Returned | Compliance Officer, Approving Authority, Admin | Corrections needed | Yes |
| Under Review | Rejected | Approving Authority, Admin | Registration not acceptable | Yes |
| Submitted | Returned | Compliance Officer, Admin | Obvious missing information | Yes |
| Submitted | Rejected | Approving Authority, Admin | Serious invalid registration | Yes |

**3.3 Approval edit locks**

| **Approval Status** | **External Supplier Edit** | **Internal Edit** | **Notes** |
| --- | --- | --- | --- |
| Draft | Yes, own record | Yes | Normal creation |
| Submitted | No | Limited review edits only | Prevent silent changes during review |
| Under Review | No | Review fields only | Supplier identity locked |
| Approved | Limited own non-critical fields only | Controlled updates only | Identity changes should be governed |
| Returned | Yes, own correction fields | Yes | Corrections allowed |
| Rejected | No | Admin only | Terminal unless reopened by Admin |

**4\. Operational status model**

**4.1 Operational states**

| **State** | **Meaning** |
| --- | --- |
| Pending | Not yet active for procurement participation |
| Active | Eligible in principle, subject to compliance/category checks |
| Suspended | Temporarily blocked |
| Blacklisted | Strong block, normally terminal or highly restricted |
| Expired | Blocked due to lapsed documents/category qualification |

**4.2 Operational transitions**

| **From** | **To** | **Actor** | **Preconditions** | **Reason Required** |
| --- | --- | --- | --- | --- |
| Pending | Active | Approving Authority, Admin, System | Approval Status = Approved and compliance complete | No |
| Active | Suspended | Approving Authority, Admin | Valid suspension basis | Yes |
| Suspended | Active | Approving Authority, Admin | Reinstatement approved and compliance valid | Yes |
| Active | Blacklisted | Authorized Authority, Admin | Serious breach / legal restriction | Yes |
| Suspended | Blacklisted | Authorized Authority, Admin | Serious breach / legal restriction | Yes |
| Active | Expired | System, Compliance Officer, Admin | Required document/category expiry | No / system reason |
| Expired | Active | Compliance Officer, Approving Authority, Admin | Renewal verified and approval still valid | No |
| Pending | Blacklisted | Authorized Authority, Admin | Serious restriction discovered | Yes |

**4.3 Operational status effects**

| **Operational Status** | **Can Be Invited** | **Can Submit Bid** | **Can Be Awarded** | **Notes** |
| --- | --- | --- | --- | --- |
| Pending | No | No | No | Not active yet |
| Active | Yes, if compliant/category-qualified | Yes, if compliant/category-qualified | Yes, if compliant/category-qualified |  |
| Suspended | No | No | No | Temporary block |
| Blacklisted | No | No | No | Strong block |
| Expired | No | No | No | Renewal required |

**5\. Compliance status model**

**5.1 Compliance states**

| **State** | **Meaning** |
| --- | --- |
| Unknown | Not yet evaluated |
| Complete | Required documents are present, verified, and current |
| Incomplete | Missing required documents |
| Expired | One or more required documents expired |
| Non-Compliant | Review found document/legal/compliance failure |

**5.2 Compliance transitions**

| **From** | **To** | **Trigger / Actor** | **Preconditions** |
| --- | --- | --- | --- |
| Unknown | Incomplete | System / Review | Missing required documents |
| Unknown | Complete | System / Compliance Officer | All required docs verified/current |
| Incomplete | Complete | System / Compliance Officer | Missing docs provided and verified |
| Complete | Expired | System | Required document expiry date passed |
| Complete | Non-Compliant | Compliance Officer | Document or legal issue discovered |
| Expired | Complete | Compliance Officer | Renewal verified |
| Non-Compliant | Complete | Compliance Officer / Approver | Issue resolved and verified |

**5.3 Compliance calculation rule**

Compliance should be computed from:

-   required registration documents
-   category-specific required documents
-   document verification status
-   document expiry
-   compliance review decisions

Do not let users manually set compliance to Complete without supporting document evidence.

**6\. External access status model**

**6.1 Access states**

| **State** | **Meaning** |
| --- | --- |
| Pending | Account/access created but not fully active |
| Active | External supplier can use allowed API/profile operations |
| Suspended | Access temporarily blocked |
| Revoked | Access permanently disabled unless reissued |

**6.2 Access transitions**

| **From** | **To** | **Actor** | **Preconditions** | **Reason Required** |
| --- | --- | --- | --- | --- |
| Pending | Active | System/Admin | Supplier account verified | No |
| Active | Suspended | Admin/System | Security issue, supplier suspended, or policy block | Yes |
| Suspended | Active | Admin | Reinstatement approved | Yes |
| Active | Revoked | Admin/System | Supplier blacklisted, access compromised, or account closed | Yes |
| Pending | Revoked | Admin/System | Registration invalid or abandoned | Yes |

**6.3 Access effects**

| **Access Status** | **External API/Profile Access** |
| --- | --- |
| Pending | Minimal onboarding only |
| Active | Own-profile operations allowed by scopes |
| Suspended | Blocked except limited status message if allowed |
| Revoked | Blocked |

**7\. Eligibility model**

**7.1 Eligibility rule**

A supplier is eligible for downstream procurement only if:

approval\_status = Approved

AND operational\_status = Active

AND compliance\_status = Complete

AND category qualification = Qualified and current

AND access/status blockers do not apply

For API/internal service purposes, return both:

-   eligible: true/false
-   reasons: \[\]

Never return only a boolean.

**7.2 Eligibility blockers**

| **Blocker Code** | **Meaning** |
| --- | --- |
| NOT\_APPROVED | Supplier approval status is not Approved |
| NOT\_ACTIVE | Operational status is not Active |
| SUSPENDED | Supplier is suspended |
| BLACKLISTED | Supplier is blacklisted |
| COMPLIANCE\_INCOMPLETE | Required documents missing |
| COMPLIANCE\_EXPIRED | Required documents expired |
| NON\_COMPLIANT | Compliance review failed |
| CATEGORY\_NOT\_QUALIFIED | Supplier not qualified for requested category |
| CATEGORY\_EXPIRED | Category qualification expired |
| ACCESS\_REVOKED | External access revoked, if relevant |

**8\. Document governance**

**8.1 Document verification transitions**

| **From** | **To** | **Actor** | **Preconditions** | **Reason Required** |
| --- | --- | --- | --- | --- |
| Pending | Verified | Compliance Officer, Admin | Document is valid | No |
| Pending | Rejected | Compliance Officer, Admin | Document invalid | Yes |
| Verified | Expired | System | Expiry date passed | No |
| Rejected | Pending | Supplier / Registry Officer | Replacement uploaded | No |
| Expired | Pending | Supplier / Registry Officer | Renewal uploaded | No |

**8.2 Document edit locks**

| **Verification Status** | **Supplier Edit** | **Internal Edit** | **Notes** |
| --- | --- | --- | --- |
| Pending | Replace allowed | Review allowed | Before verification |
| Verified | No direct edit; upload replacement | Controlled only | Preserve audit trail |
| Rejected | Replace allowed | Review notes locked | Reason preserved |
| Expired | Replace allowed | Controlled only | Renewal path |

**9\. Category qualification governance**

**9.1 Category qualification transitions**

| **From** | **To** | **Actor** | **Preconditions** | **Reason Required** |
| --- | --- | --- | --- | --- |
| Requested | Under Review | Compliance Officer, Registry Officer, Admin | Category requested | No |
| Under Review | Qualified | Compliance Officer / Approver, Admin | Category requirements met | No |
| Under Review | Rejected | Compliance Officer / Approver, Admin | Category requirements not met | Yes |
| Qualified | Expired | System | Qualification date expired | No |
| Expired | Under Review | Supplier / Registry Officer | Renewal requested | No |
| Rejected | Requested | Supplier / Registry Officer | Resubmission allowed | No |

**10\. Status history rules**

Every transition must write a Supplier Status History record for:

-   approval\_status changes
-   operational\_status changes
-   compliance\_status changes
-   access\_status changes, if using the same history model or a related access audit
-   negative document verification decisions, if not separately audited

History must capture:

-   previous status
-   new status
-   status type
-   actor
-   timestamp
-   reason where required

**11\. Role-to-transition matrix**

| **Transition Type** | **External Supplier** | **Registry Officer** | **Compliance Officer** | **Approver** | **Auditor** | **Admin** |
| --- | --- | --- | --- | --- | --- | --- |
| Draft/Returned → Submitted | Own only | Yes | No | No | No | Yes |
| Submitted → Under Review | No | Yes | Yes | No | No | Yes |
| Under Review → Approved | No | No | No | Yes | No | Yes |
| Under Review → Returned | No | Limited | Yes | Yes | No | Yes |
| Under Review → Rejected | No | No | No | Yes | No | Yes |
| Pending → Active | No | No | No | Yes/System | No | Yes |
| Active → Suspended | No | No | No | Yes | No | Yes |
| Suspended → Active | No | No | No | Yes | No | Yes |
| Active/Suspended → Blacklisted | No | No | No | Authorized only | No | Yes |
| Document Pending → Verified/Rejected | No | No | Yes | No | No | Yes |
| Category Requested → Qualified/Rejected | No | No | Yes | Yes if required | No | Yes |

**12\. UI implications**

**Workbench queues**

Recommended queues:

-   Draft / Pending Submission
-   Submitted
-   Under Review
-   Returned
-   Approved
-   Active
-   Expiring Documents
-   Suspended
-   Blacklisted

**Detail panel status display**

Show four status chips separately:

-   Approval
-   Operational
-   Compliance
-   Access

Do not merge them into one ambiguous status.

**Action placement**

Valid actions appear in the top-right of supplier detail panel:

-   Submit
-   Start Review
-   Verify Documents
-   Approve
-   Return
-   Reject
-   Suspend
-   Reinstate
-   Blacklist

Only show actions valid for current role/state.

**13\. External API implications**

External supplier APIs must enforce state rules:

| **Endpoint** | **Allowed States** |
| --- | --- |
| Register | New / no existing active account |
| Update Own Profile | Draft, Returned |
| Upload Document | Draft, Returned, Submitted if correction window allowed |
| Submit for Review | Draft, Returned |
| Get Own Status | Any own supplier record |
| Get Own Documents | Any own supplier record |

External APIs must not allow:

-   approval status mutation except submit
-   operational status mutation
-   compliance status mutation
-   document verification
-   category qualification

**14\. Acceptance criteria**

Governance model is complete only if:

-   approval, operational, compliance, and access statuses are separate
-   valid transitions are explicitly enforced
-   invalid transitions are blocked server-side
-   reasons are required for negative/exception transitions
-   every lifecycle change writes history
-   supplier eligibility is computed from statuses/documents/categories
-   external supplier users cannot mutate internal governance fields
-   suspended/blacklisted/expired suppliers are ineligible downstream
-   UI shows valid actions only by role/state

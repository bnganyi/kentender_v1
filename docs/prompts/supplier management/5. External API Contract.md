Supplier Management — External API Contract v1.0

This locks how external suppliers interact with any system.  

**1\. Core principles**

**1\. Supplier-scoped access**

Every request must be scoped to:

supplier\_id (resolved from auth)

-   No endpoint should allow access to another supplier’s data.

**2\. No governance mutation externally**

External APIs must NEVER allow:

-   approval\_status changes (except submit)
-   operational\_status changes
-   compliance\_status changes
-   document verification
-   category qualification

**3\. State-aware APIs**

Every endpoint must enforce:

allowed\_states per operation

**4\. Business IDs only**

Expose:

supplier\_code

category\_code

document\_type\_code

Never expose raw internal IDs.

**2\. Authentication model**

**Required**

-   token-based authentication (JWT or equivalent)
-   token resolves:
    -   user\_id
    -   supplier\_id
    -   scopes

**Scopes**

\[

"supplier:read",

"supplier:update",

"supplier:submit",

"document:upload",

"document:read",

"status:read"

\]

**3\. API endpoints**

**3.1 Register Supplier**

**Endpoint**

POST /api/v1/suppliers/register

**Allowed**

-   unauthenticated OR authenticated (depending on design)

**Request**

{

"supplier\_name": "AfyaMed Supplies Ltd",

"supplier\_type": "Company",

"registration\_number": "REG-12345",

"tax\_identifier": "PIN-67890",

"country": "KE",

"primary\_contact": {

"name": "John Doe",

"email": "john@afyamed.com",

"phone": "+254700000000"

},

"categories\_requested": \[

"CAT-MED-EQUIP"

\]

}

**Response**

{

"supplier\_code": "SUP-KE-2026-0001",

"approval\_status": "Draft",

"operational\_status": "Pending",

"next\_steps": \[

"Upload required documents",

"Submit for review"

\]

}

**Rules**

-   email must be unique
-   creates:
    -   Supplier
    -   External user (if not exists)
    -   API access grant (Pending)

**3.2 Get Own Supplier Profile**

**Endpoint**

GET /api/v1/suppliers/me

**Response**

{

"supplier\_code": "SUP-KE-2026-0001",

"supplier\_name": "AfyaMed Supplies Ltd",

"supplier\_type": "Company",

"country": "KE",

"approval\_status": "Draft",

"operational\_status": "Pending",

"compliance\_status": "Incomplete",

"categories": \[

{

"category\_code": "CAT-MED-EQUIP",

"qualification\_status": "Requested"

}

\]

}

**Rules**

-   must return only own supplier
-   no internal notes
-   no audit history
-   no internal IDs

**3.3 Update Own Profile**

**Endpoint**

PATCH /api/v1/suppliers/me

**Allowed states**

Draft, Returned

**Request**

{

"supplier\_name": "AfyaMed Supplies Ltd Updated",

"primary\_contact": {

"name": "Jane Doe"

}

}

**Rules**

-   block if:
    -   Submitted
    -   Under Review
    -   Approved (except limited fields if allowed)
-   ignore restricted fields silently or reject

**3.4 Upload Document**

**Endpoint**

POST /api/v1/suppliers/me/documents

**Allowed states**

Draft, Returned

(optional: Submitted if correction window allowed)

**Request (multipart or JSON ref)**

{

"document\_type\_code": "DOC-TAX",

"document\_name": "Tax Compliance Certificate",

"document\_number": "TAX-12345",

"issue\_date": "2025-01-01",

"expiry\_date": "2026-01-01",

"file": "<binary or file reference>"

}

**Response**

{

"document\_id": "DOC-001",

"verification\_status": "Pending"

}

**Rules**

-   validate document\_type exists
-   enforce expiry if required
-   mark:
    -   external\_uploaded = true
    -   verification\_status = Pending
-   replace current document of same type (set previous is\_current = 0)

**3.5 List Own Documents**

**Endpoint**

GET /api/v1/suppliers/me/documents

**Response**

\[

{

"document\_type\_code": "DOC-TAX",

"document\_name": "Tax Certificate",

"verification\_status": "Verified",

"expiry\_date": "2026-01-01",

"is\_current": true

}

\]

**Rules**

-   return only current documents unless versioning explicitly needed

**3.6 Submit for Review**

**Endpoint**

POST /api/v1/suppliers/me/submit

**Allowed states**

Draft, Returned

**Behavior**

System validates:

-   required fields
-   required documents present
-   required categories requested

**Response (success)**

{

"approval\_status": "Submitted"

}

**Response (failure)**

{

"error": "VALIDATION\_FAILED",

"missing": \[

"DOC-TAX",

"registration\_number"

\]

}

**3.7 Get Own Status**

**Endpoint**

GET /api/v1/suppliers/me/status

**Response**

{

"approval\_status": "Under Review",

"operational\_status": "Pending",

"compliance\_status": "Incomplete",

"issues": \[

"Tax certificate missing",

"Registration document not verified"

\],

"review\_comments": \[

"Please upload updated tax certificate"

\]

}

**Rules**

-   include:
    -   missing items
    -   rejection/return reasons
-   exclude:
    -   internal-only notes

**3.8 Request Category Assignment (optional endpoint)**

**Endpoint**

POST /api/v1/suppliers/me/categories

**Request**

{

"category\_code": "CAT-ICT"

}

**Response**

{

"category\_code": "CAT-ICT",

"qualification\_status": "Requested"

}

**Rules**

-   cannot set to Qualified
-   creates/updates Supplier Category Assignment

**4\. Error model**

Standardized error format:

{

"error": "ERROR\_CODE",

"message": "Human-readable message",

"details": {}

}

**Common error codes**

| **Code** | **Meaning** |
| --- | --- |
| UNAUTHORIZED | Invalid/missing token |
| FORBIDDEN | Access outside supplier scope |
| INVALID\_STATE | Operation not allowed in current state |
| VALIDATION\_FAILED | Missing/invalid fields |
| NOT\_FOUND | Resource not found |
| FILE\_INVALID | Invalid file upload |
| DUPLICATE | Duplicate data (email, registration) |

**5\. State enforcement matrix**

| **Endpoint** | **Draft** | **Returned** | **Submitted** | **Under Review** | **Approved** |
| --- | --- | --- | --- | --- | --- |
| Register | — | — | — | — | — |
| Get profile | Yes | Yes | Yes | Yes | Yes |
| Update profile | Yes | Yes | No | No | Limited/No |
| Upload document | Yes | Yes | Limited | No | No |
| Submit | Yes | Yes | No | No | No |
| Get status | Yes | Yes | Yes | Yes | Yes |
| Request category | Yes | Yes | Yes | No | No |

**6\. Security rules**

Must enforce:

-   supplier-scoped data access
-   no cross-supplier leakage
-   no internal IDs exposed
-   file validation (size/type)
-   rate limiting (if infra supports)
-   audit log for:
    -   registration
    -   updates
    -   document uploads
    -   submissions

**7\. Audit requirements**

Every external action must log:

-   supplier\_code
-   external\_user
-   endpoint
-   payload summary (not full file)
-   timestamp
-   result (success/failure)

**8\. Acceptance criteria**

API is correct only if:

-   supplier can register via API
-   supplier can only access own data
-   supplier cannot mutate governance fields
-   document uploads are validated and versioned
-   submission enforces required fields/docs
-   status endpoint shows actionable feedback
-   invalid state operations are blocked
-   no internal IDs exposed
-   audit logs exist for all external actions

Supplier Management — Pre-PRD (v1.0)

**1\. Purpose of the Module**

Supplier Management provides a **controlled, auditable registry of suppliers** that can:

-   be invited to tenders
-   be evaluated and awarded
-   be monitored over time

It ensures that **only eligible, qualified, and compliant suppliers** participate in procurement.

**2\. What problem this module solves**

Without this module:

-   suppliers are inconsistent or duplicated
-   no standard qualification or onboarding
-   no control over eligibility (e.g. suspended vendors still bidding)
-   no structured supplier performance memory
-   tendering becomes chaotic and risky

This module introduces **governance before market interaction**

**3\. Core responsibilities (strict scope)**

This module MUST handle:

**A. Supplier Identity & Registry**

-   unique supplier record
-   legal identity (name, registration, tax ID)
-   organization type (company, individual, NGO, etc.)

**B. Supplier Onboarding**

-   registration process (internal or self-service later)
-   required documents:
    -   certificates
    -   licenses
    -   tax compliance
-   completeness tracking

**C. Supplier Classification**

-   categories (e.g. ICT, Medical Equipment)
-   capability areas
-   geographic coverage
-   size/type classification

This feeds later:

-   tender targeting
-   eligibility filtering

**D. Supplier Qualification / Approval**

-   approval workflow:
    -   Draft → Submitted → Reviewed → Approved / Rejected
-   qualification validity (expiry dates)

**E. Supplier Status Control (CRITICAL)**

-   Active
-   Suspended
-   Blacklisted
-   Expired

This directly controls:

whether supplier can participate in procurement

**F. Supplier Documents & Compliance**

-   document storage
-   expiry tracking
-   compliance flags

**G. Supplier Performance (light in v1)**

-   basic rating or flags
-   issues/incidents
-   future-ready for contract performance integration

**4\. What this module must NOT do**

Avoid scope creep:

❌ No tender creation  
❌ No bid submission  
❌ No evaluation logic  
❌ No contract execution  
❌ No financial transactions

This is **supplier master + governance**, not procurement execution.

**5\. Key design principles**

**Principle 1: Governance first, not just CRUD**

Every supplier must have:

-   lifecycle state
-   approval flow
-   audit trail

**Principle 2: Status must control eligibility**

Supplier status is not informational.

It must actively determine:

-   can be invited?
-   can submit?
-   can be awarded?

**Principle 3: Documents are first-class citizens**

Do NOT treat documents as attachments.

They must have:

-   type
-   expiry
-   validation status

**Principle 4: Classification must be usable**

Categories must:

-   integrate with Procurement Planning / Packages
-   later drive supplier targeting

**Principle 5: Avoid over-engineering performance (for now)**

Performance tracking should exist, but:

-   lightweight in v1
-   extensible later (contract module)

**6\. Core domain objects (preview)**

We will formalize later, but at Pre-PRD level:

-   Supplier
-   Supplier Document
-   Supplier Category Assignment
-   Supplier Status History
-   Supplier Qualification Review
-   Supplier Performance Record (light)

**7\. Lifecycle (high-level)**

Draft → Submitted → Under Review → Approved → Active

↓

Rejected

Active → Suspended → Active

Active → Blacklisted (terminal)

Active → Expired (if compliance lapses)

**8\. Key user roles**

-   Supplier (external — later phase)
-   Procurement Officer (creates/edits suppliers)
-   Compliance Officer (reviews documents)
-   Approver (final approval)
-   Admin (status override, blacklisting)

**9\. Critical integration points**

**With Procurement Planning**

-   categories align with package categories

**With Tendering (future)**

-   only Approved + Active suppliers eligible

**With Evaluation / Contracts (future)**

-   performance feedback loops back

**10\. Risks if designed poorly**

-   duplicate suppliers → data chaos
-   no status enforcement → governance failure
-   documents unmanaged → compliance risk
-   categories inconsistent → unusable targeting
-   approval missing → audit failure

**11\. Non-negotiables (carry-over discipline)**

We MUST include:

-   governance/state model from day one
-   role-based permissions
-   structured document model
-   seed data
-   UI with clear lifecycle visibility
-   no “just forms” — must be workflow-driven

Supplier Management — Eligibility Service Contract v1.0

This is one of the most critical contracts in the entire system.

-   **Tendering, Evaluation, and Award MUST depend on this service**
-   No module should “guess” supplier eligibility

**1\. Purpose**

Determine whether a supplier is eligible to:

-   be invited to a sourcing event
-   submit a bid
-   be evaluated
-   be awarded

**2\. Core principle**

Return:

eligible = true/false

AND

reasons = \[\]

-   Never return only a boolean

**3\. Service definition**

**Function**

check\_supplier\_eligibility(

supplier\_code,

category\_code = None,

context = None

)

**4\. Input contract**

| **Field** | **Required** | **Description** |
| --- | --- | --- |
| supplier\_code | Yes | Business ID |
| category\_code | Optional | Required for category-specific procurement |
| context | Optional | Future: tender, package, geography, etc. |

**5\. Output contract**

{

"supplier\_code": "SUP-KE-2026-0001",

"eligible": false,

"approval\_status": "Approved",

"operational\_status": "Suspended",

"compliance\_status": "Complete",

"category\_status": {

"category\_code": "CAT-MED-EQUIP",

"qualification\_status": "Qualified",

"valid": true

},

"reasons": \[

{

"code": "SUSPENDED",

"message": "Supplier is currently suspended"

}

\]

}

**6\. Eligibility rule (authoritative)**

A supplier is eligible only if:

approval\_status = Approved

AND operational\_status = Active

AND compliance\_status = Complete

AND category qualification (if required) is Qualified and valid

**7\. Evaluation order (strict)**

Evaluate in this order (fail-fast but collect reasons):

**7.1 Approval check**

| **Condition** | **Blocker** |
| --- | --- |
| approval\_status != Approved | NOT\_APPROVED |

**7.2 Operational check**

| **Condition** | **Blocker** |
| --- | --- |
| Pending | NOT\_ACTIVE |
| Suspended | SUSPENDED |
| Blacklisted | BLACKLISTED |
| Expired | EXPIRED |

**7.3 Compliance check**

| **Condition** | **Blocker** |
| --- | --- |
| Incomplete | COMPLIANCE\_INCOMPLETE |
| Expired | COMPLIANCE\_EXPIRED |
| Non-Compliant | NON\_COMPLIANT |

**7.4 Category qualification check (if category provided)**

| **Condition** | **Blocker** |
| --- | --- |
| No assignment | CATEGORY\_NOT\_ASSIGNED |
| Requested / Under Review | CATEGORY\_NOT\_QUALIFIED |
| Rejected | CATEGORY\_REJECTED |
| Expired | CATEGORY\_EXPIRED |

**8\. Blocker codes (standardized)**

| **Code** | **Meaning** |
| --- | --- |
| NOT\_APPROVED | Supplier not approved |
| NOT\_ACTIVE | Supplier not yet active |
| SUSPENDED | Supplier suspended |
| BLACKLISTED | Supplier blacklisted |
| EXPIRED | Supplier operational status expired |
| COMPLIANCE\_INCOMPLETE | Missing documents |
| COMPLIANCE\_EXPIRED | Documents expired |
| NON\_COMPLIANT | Compliance failure |
| CATEGORY\_NOT\_ASSIGNED | No category mapping |
| CATEGORY\_NOT\_QUALIFIED | Category not yet qualified |
| CATEGORY\_REJECTED | Category rejected |
| CATEGORY\_EXPIRED | Category qualification expired |

**9\. Multi-reason behavior**

Do NOT stop at first failure.

Return all blockers:

{

"eligible": false,

"reasons": \[

{ "code": "SUSPENDED" },

{ "code": "COMPLIANCE\_EXPIRED" }

\]

}

This is critical for:

-   UI messaging
-   audit
-   troubleshooting

**10\. Category-aware vs global eligibility**

**Case A: No category\_code provided**

-   check only:
    -   approval
    -   operational
    -   compliance

**Case B: category\_code provided**

-   also enforce:
    -   category assignment exists
    -   qualification is Qualified
    -   qualification is not expired

**11\. Service consumers**

This service will be used by:

| **Module** | **Usage** |
| --- | --- |
| Tendering | Filter invite list |
| Bid Submission | Validate submission eligibility |
| Evaluation | Confirm supplier is still valid |
| Award | Final validation before award |
| Reporting | Supplier eligibility dashboards |

**12\. Performance requirements**

-   must be callable at scale (batch)
-   must support:

check\_multiple\_suppliers(\[supplier\_codes\], category\_code)

**13\. Caching rules**

Allowed:

-   short-lived cache (e.g., 5–15 minutes)

Not allowed:

-   stale eligibility decisions during:
    -   suspension
    -   blacklist
    -   document expiry

These events should invalidate cache

**14\. Edge cases**

**14.1 Approved but compliance expired**

{

"eligible": false,

"reasons": \["COMPLIANCE\_EXPIRED"\]

}

**14.2 Active but no category**

{

"eligible": false,

"reasons": \["CATEGORY\_NOT\_ASSIGNED"\]

}

**14.3 Suspended but otherwise valid**

{

"eligible": false,

"reasons": \["SUSPENDED"\]

}

**15\. UI usage rules**

When displayed in UI:

-   show eligibility badge:
    -   Eligible
    -   Not Eligible
-   show top blocker:
    -   “Suspended”
    -   “Documents expired”
-   allow drill-down to full reason list

**16\. Audit requirements**

Every eligibility check (especially in Tendering) must log:

-   supplier\_code
-   category\_code
-   result
-   reasons
-   caller (module)
-   timestamp

**17\. Acceptance criteria**

Eligibility service is correct only if:

-   returns both boolean and reasons
-   enforces approval, operational, compliance, category checks
-   returns multiple blockers
-   works with and without category\_code
-   supports batch checks
-   does not expose internal IDs
-   used by downstream modules instead of duplicated logic
-   handles expired documents correctly
-   handles suspended/blacklisted correctly
-   logs eligibility checks

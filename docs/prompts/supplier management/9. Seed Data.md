**5\. seed\_supplier\_states**

**5.1 Approval transitions**

Include history records:

-   Draft → Submitted
-   Submitted → Under Review
-   Under Review → Approved
-   Under Review → Returned
-   Under Review → Rejected

**5.2 Operational transitions**

Include:

-   Pending → Active
-   Active → Suspended
-   Suspended → Active
-   Active → Blacklisted
-   Active → Expired

**5.3 Compliance states**

Ensure presence of:

-   Complete
-   Incomplete
-   Expired
-   Non-Compliant

**7\. seed\_supplier\_edge\_cases**

These are critical for testing.

**7.1 Suspended but compliant**

-   SUP-0003  
    → eligible = false  
    → reason = SUSPENDED

**7.2 Approved but expired documents**

-   SUP-0005  
    → eligible = false  
    → reason = COMPLIANCE\_EXPIRED

**7.3 Category missing**

Create:

-   supplier approved + active + compliant
-   no category assignment

→ eligible = false  
→ reason = CATEGORY\_NOT\_ASSIGNED

**7.4 Category rejected**

-   SUP-0006

→ eligible = false  
→ reason = CATEGORY\_REJECTED

**7.5 Multiple blockers**

Create:

-   supplier suspended + expired docs

→ eligible = false  
→ reasons = \[SUSPENDED, COMPLIANCE\_EXPIRED\]

**8\. Naming conventions**

All seeds must use:

SUP-KE-YYYY-####

CAT-XXXX

DOC-XXXX

**9\. Data integrity rules**

Seed must ensure:

-   unique supplier\_code
-   unique email
-   correct foreign keys
-   valid state combinations
-   valid history records

**10\. Acceptance criteria**

Seed data is correct only if:

-   every lifecycle path is represented
-   every eligibility blocker is testable
-   every role can perform at least one action
-   external flows can be simulated
-   UI queues have data
-   eligibility service returns varied outcomes
-   document expiry scenarios exist
-   category qualification scenarios exist

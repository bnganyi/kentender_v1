Supplier Management — Full PRD (v1.0)

**1\. Module purpose**

Supplier Management is the governed supplier registry for the procurement system.

It manages:

-   supplier identity
-   onboarding
-   documents and compliance
-   category qualification
-   supplier status
-   supplier eligibility
-   supplier-facing API access
-   audit history

Core rule:

Tendering must never rely on free-text supplier data. It must consume governed supplier records.

**2\. Scope**

**In scope**

-   Supplier master record
-   Supplier registration/onboarding
-   Supplier document management
-   Supplier category assignment
-   Supplier compliance review
-   Supplier qualification approval
-   Supplier lifecycle status
-   External supplier API access
-   Basic supplier performance/risk flags
-   Eligibility checks for downstream tendering

**Out of scope**

-   Tender creation
-   Bid submission
-   Bid evaluation
-   Contract execution
-   Payments
-   Full supplier portal UI, unless later added
-   Complex performance scoring from contract delivery, except lightweight readiness fields

**3\. Primary users**

| **User** | **Purpose** |
| --- | --- |
| Supplier external user | Register, update profile, upload documents, view own status |
| Supplier Registry Officer | Create/edit supplier records, review completeness |
| Compliance Officer | Verify documents and compliance |
| Procurement Officer | View supplier eligibility and categories |
| Approving Authority | Approve/reject supplier qualification |
| Auditor/Oversight | Read-only traceability review |
| Administrator | Configure categories, document types, controlled overrides |

**4\. Core lifecycle**

Draft

→ Submitted

→ Under Review

→ Approved

→ Active

Alternative paths:

Submitted / Under Review → Returned

Submitted / Under Review → Rejected

Active → Suspended

Suspended → Active

Active / Suspended → Blacklisted

Active → Expired

Expired → Active after renewal

Important distinction:

-   **Approval status** = onboarding/qualification decision
-   **Operational status** = whether supplier can participate

A supplier may be approved historically but not currently eligible if suspended, blacklisted, expired, or non-compliant.

**5\. Key business rules**

**5.1 Supplier eligibility rule**

A supplier is eligible only if:

approval\_status = Approved

AND operational\_status = Active

AND required documents are valid

AND category qualification is active

AND not suspended or blacklisted

**5.2 External access rule**

External supplier users may only:

-   create their own registration
-   view their own supplier profile
-   update allowed fields in draft/returned states
-   upload/replace documents
-   submit for review
-   view status and review comments

They must not:

-   approve themselves
-   change operational status
-   edit verified compliance decisions
-   view other suppliers
-   access internal notes
-   access other procurement modules unless explicitly exposed

**5.3 Document rule**

Supplier documents are not casual attachments. Each document must have:

-   document type
-   file reference
-   issue date
-   expiry date where applicable
-   verification status
-   verified by/at
-   review notes

**5.4 Category rule**

Supplier categories must be structured and controlled. They drive future tender eligibility and invitation targeting.

**7\. API requirements**

External APIs must be first-class.

**7.1 API design principles**

-   Use authenticated access only
-   Every external API call must be supplier-scoped
-   No external API may return another supplier’s data
-   Internal-only fields must be excluded
-   Mutating APIs must respect lifecycle state
-   APIs must return stable business IDs, not raw internal IDs

**7.2 External supplier API endpoints**

**Register supplier**

POST /api/suppliers/register

Creates draft supplier registration.

Must accept:

-   supplier identity
-   primary contact
-   supplier type
-   registration/tax details
-   categories requested optionally

Returns:

-   supplier\_code
-   approval\_status
-   next required steps

**Get own supplier profile**

GET /api/suppliers/me

Returns only own supplier profile.

**Update own profile**

PATCH /api/suppliers/me

Allowed only if:

-   approval\_status in Draft or Returned
-   operational\_status not Blacklisted

**Upload document**

POST /api/suppliers/me/documents

Must accept:

-   document\_type
-   file
-   issue\_date
-   expiry\_date
-   document\_number

**List own documents**

GET /api/suppliers/me/documents

**Submit for review**

POST /api/suppliers/me/submit

Validates required fields/documents and changes status to Submitted.

**Get own status**

GET /api/suppliers/me/status

Returns:

-   approval\_status
-   operational\_status
-   compliance\_status
-   missing documents
-   review comments visible to supplier

**7.3 Internal service endpoints**

**Check supplier eligibility**

check\_supplier\_eligibility(supplier\_id, category\_id=None)

Returns:

-   eligible true/false
-   reasons
-   missing/expired documents
-   status blockers

This will be consumed later by Tendering.

**8\. Governance and state rules**

**8.1 Approval lifecycle**

| **From** | **To** | **Actor** | **Rule** |
| --- | --- | --- | --- |
| Draft | Submitted | Supplier / Registry Officer | Required fields/docs present |
| Submitted | Under Review | Compliance Officer | Review starts |
| Under Review | Approved | Approving Authority | Documents valid |
| Under Review | Returned | Compliance Officer / Approver | Reason required |
| Under Review | Rejected | Approver | Reason required |
| Returned | Submitted | Supplier / Registry Officer | Corrections made |

**8.2 Operational status lifecycle**

| **From** | **To** | **Actor** | **Rule** |
| --- | --- | --- | --- |
| Approved/Pending | Active | Approver/System | Qualification complete |
| Active | Suspended | Approver/Admin | Reason required |
| Suspended | Active | Approver/Admin | Reinstatement reason required |
| Active/Suspended | Blacklisted | Authorized Authority/Admin | Strong reason required |
| Active | Expired | System | Required document/category expiry |

**12\. Smoke contract**

Must verify:

-   external supplier can register
-   external supplier can only view own profile
-   supplier can upload documents
-   supplier can submit when required docs present
-   missing required docs block submission
-   compliance officer can verify/reject documents
-   approver can approve supplier
-   active supplier becomes eligible
-   suspended supplier becomes ineligible
-   blacklisted supplier cannot be eligible
-   expired document makes supplier non-compliant
-   category eligibility works
-   internal users can view supplier status history
-   auditor is read-only
-   no raw internal IDs are exposed in UI/API responses

**13\. API security requirements**

Mandatory:

-   supplier-scoped authorization
-   role-based authorization
-   audit log for external submissions and uploads
-   rate limiting where available
-   file type/size validation for uploads
-   no internal notes in external responses
-   no direct status mutation by external users
-   no mass supplier listing via external API

**15\. Module architecture note**

ERPNext Supplier will store basic supplier identity, but all onboarding, compliance, status, category qualification, eligibility, and external access rules remain in KenTender.

-   **ERPNext Supplier = identity anchor**
-   **KenTender Supplier Management = governance authority**

**16\. Acceptance criteria**

Supplier Management is acceptable only if:

-   suppliers have governed lifecycle
-   external supplier access is scoped and safe
-   documents are structured and verifiable
-   eligibility is computed, not guessed
-   supplier status controls downstream participation
-   category qualification is explicit
-   APIs support supplier registration/update/document upload/status
-   UI follows workbench pattern
-   roles and permissions are enforced server-side
-   seed data supports core flows
-   smoke tests cover eligibility and external access

**Implementation recommendation**

Build in this order:

Supplier domain model

→ Document/category model

→ Lifecycle/governance services

→ External supplier APIs

→ Workbench UI

→ Seed data

→ Smoke tests

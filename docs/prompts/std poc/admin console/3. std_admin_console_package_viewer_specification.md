## 2. Purpose

The purpose of this document is to define the read-only Template Package Viewer for the STD Administration Console POC.

The viewer shall make an imported STD template package understandable without forcing reviewers to inspect raw JSON. It shall expose the package structure, metadata, sections, fields, rules, forms, sample tender, render map, and raw JSON fallback in a controlled and review-friendly way.

The viewer shall not become a template editor. Its job is to explain what is inside the package and how the package is intended to drive tender creation.

---

## 3. Design Principle

The Template Package Viewer shall implement the principle:

> Show the controlled STD package clearly; do not make legal or rule content casually editable through the UI.

The viewer shall therefore be:

1. read-only,
2. structured,
3. grouped by package component,
4. traceable to package source and hash,
5. understandable to non-developer reviewers,
6. useful to developers for troubleshooting,
7. explicit about POC boundaries.

---

## 6. Viewer Entry Points

The Template Package Viewer shall be reachable from the `STD Template` record.

Minimum entry points:

| Entry point | Required behavior |
|---|---|
| `STD Template` list row | Open the template detail form |
| `STD Template` form | Display package viewer sections directly or provide a `View Package` action |
| Admin Console dashboard, if implemented | Link to the same package viewer |

Recommended first implementation:

Use the `STD Template` form as the primary package viewer, enhanced with read-only HTML/table sections and actions.

A dedicated Frappe Page may be added later if the form becomes too constrained.

## 7. Viewer Layout

The viewer shall use the following logical tabs or sections:

| Order | Tab / Section | Purpose |
|---:|---|---|
| 10 | Overview | Show package identity and source metadata |
| 20 | Package Structure | Show component presence and high-level counts |
| 30 | Sections | Show STD section model |
| 40 | Fields | Show admin configuration field model |
| 50 | Rules | Show package rule model summary |
| 60 | Forms | Show bidder/contract form model summary |
| 70 | Sample Tender | Show sample tender data and variants |
| 80 | Render Map | Show rendering metadata and preview coverage |
| 90 | Raw JSON | Read-only developer fallback |
| 100 | Actions | Controlled package actions and links |

If Frappe form tabs are not practical in the first implementation, the sections may be implemented as collapsible read-only sections in the `STD Template` form.

---

## 8. General Display Rules

The viewer shall follow these general display rules:

1. All package-derived values are read-only.
2. Empty values shall show `Not specified`.
3. Boolean values shall show `Yes` or `No`.
4. Long values such as package hashes may be shown shortened in summary, with full value available on copy or detail.
5. Counts shall be shown for major arrays.
6. Each table shall clearly state its source component.
7. The viewer shall not require users to understand JSON structure to inspect the package.
8. Raw JSON shall be secondary and explicitly labelled as technical read-only detail.
9. POC warnings shall be visible where relevant.
10. Viewer actions shall not mutate package internals.

## 9. Required POC Warning

The viewer shall show a clear warning near the top of the package viewer:

```text
POC Administration View: This screen exposes a controlled STD template package for inspection and demonstration. Package internals are read-only in this POC and are not editable by ordinary users through the administration console.
```

This warning may be shown in the Overview section or a persistent banner.

## 20. Server Method Requirements

The package viewer shall be supported by server methods or controller methods.

### 20.1 Required Method: `get_template_package_summary(template_name)`

Purpose:

Return all structured summary data required by the package viewer.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `template_name` | String | Yes |

Required behavior:

1. Load `STD Template`.
2. Check read permission.
3. Parse `package_json`.
4. Build structured summaries for overview, structure, sections, fields, rules, forms, sample tender, render map, and raw JSON metadata.
5. Return dictionary suitable for UI rendering.

Required return shape:

```json
{
  "ok": true,
  "template": {},
  "overview": {},
  "structure": {},
  "sections": [],
  "fields": [],
  "rules": [],
  "forms": [],
  "sample_tender": {},
  "render_map": {},
  "raw_components": {}
}
```

### 20.2 Required Method: `get_template_package_component(template_name, component_name)`

Purpose:

Return one raw package component read-only.

Allowed `component_name` values:

1. `manifest`
2. `sections`
3. `fields`
4. `rules`
5. `forms`
6. `render_map`
7. `sample_tender`
8. `full_package`

Required behavior:

1. Load `STD Template`.
2. Check read permission.
3. Validate component name.
4. Return pretty-printed JSON string or parsed object.

### 20.3 Required Method: `reimport_std_template_package(template_name=None)`

Purpose:

Run the controlled seed loader for the STD-WORKS-POC package.

Required behavior:

1. Check user has appropriate permission.
2. Call existing seed loader.
3. Return structured result.
4. Do not accept arbitrary filesystem paths.

This method may reuse the Step 10 loader and may be limited to the known POC package.

## 22. Permission and Safety Requirements

The package viewer shall enforce:

| Requirement | Details |
|---|---|
| Read permission | User must be allowed to read `STD Template` |
| Re-import permission | Restricted to System Manager or technical admin |
| No package editing | No UI path to save package JSON modifications |
| No arbitrary path import | Re-import uses known controlled package path only |
| Raw JSON read-only | Raw JSON cannot be edited or saved |
| Action logging | Use Frappe logs or response output where practical |
| Clear POC labels | User must understand this is POC admin functionality |

---

## 23. Error Handling Requirements

The viewer shall handle these errors clearly:

| Error scenario | Required behavior |
|---|---|
| STD Template not found | Show clear error |
| User lacks read permission | Show permission error |
| Package JSON missing | Show clear package error |
| Package JSON invalid | Show clear parse error |
| Component missing | Show component-specific warning/error |
| Unsupported component requested | Show clear error |
| Re-import fails | Show loader error and do not hide failure |
| Summary generation fails | Show clear error identifying component |

Silent failures are not acceptable because this console exists to improve observability.

---

## 24. Minimal Implementation Strategy

Recommended minimal implementation sequence:

1. Add server method to produce package summary.
2. Add server method to fetch raw component JSON.
3. Add read-only HTML summary area to `STD Template` form or create a lightweight custom page.
4. Add buttons under `STD Admin` group.
5. Render Overview and Package Structure first.
6. Add Sections, Fields, Rules, Forms tables.
7. Add Sample Tender and Render Map summaries.
8. Add Raw JSON viewer last.
9. Add re-import action.
10. Defer validation and rule trace details to the next artefact if needed.

This ordering delivers visible value quickly without expanding scope.

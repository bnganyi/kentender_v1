## 2. Purpose

The purpose of this document is to define the formal decision point after completion of the STD Administration Console POC.

The decision determines whether the project is ready to proceed from an internal administration and observability layer into procurement officer-facing use cases.

This document shall not define the procurement officer workflow itself. It defines the readiness criteria, decision evidence, residual risks, required conditions, and recommended next workstream.

---

## 3. Decision Principle

The project shall proceed to procurement officer use cases only if the controlled STD package model is sufficiently visible, explainable, and demonstrable to internal reviewers.

The decision shall follow this principle:

> Do not expose procurement officers to a template-driven tender creation workflow until administrators and reviewers can inspect the template package, validate package health, explain rule behavior, verify generated forms and BoQ rows, and confirm preview/audit outputs.

This protects the project from repeating the earlier intractable route where template complexity was pushed into unclear user-facing administration or tender creation screens.

---

## 4. Decision Question

The decision question is:

> Is the STD Administration Console POC sufficiently complete, usable, and safe to support moving next into procurement officer tender-creation use cases?

The possible decisions are:

| Decision | Meaning |
|---|---|
| `Proceed` | Move to procurement officer use-case design immediately |
| `Proceed with Conditions` | Move forward only after listed conditions are resolved |
| `Extend Admin Console POC` | Continue improving admin observability before officer-facing design |
| `Stop / Reconsider` | Do not proceed because the package-driven model or admin visibility remains inadequate |

## 6. Minimum Go Criteria

The project may proceed to procurement officer use cases only if all minimum go criteria are satisfied.

| ID | Go criterion | Required result |
|---|---|---|
| GO-001 | Package viewer works | Imported STD package can be inspected through UI |
| GO-002 | Package internals remain read-only | No casual package/rule/clause editing path exists |
| GO-003 | Package validation works | Package health can be checked and displayed |
| GO-004 | Rule trace works | Rule behavior can be explained for primary sample and key variants |
| GO-005 | Demo tender workspace works | Demo tender can be created/opened and run through sample path |
| GO-006 | Required forms inspector works | Generated forms are visible with activation source and evidence policy |
| GO-007 | BoQ inspector works | BoQ rows are visible with category coverage and lot-reference checks |
| GO-008 | Preview/audit viewer works | Generated preview and audit hashes/status/timestamps are visible |
| GO-009 | Negative scenarios block | Invalid variants show blockers and prevent preview generation |
| GO-010 | Positive variants explain activation | Tender-securing declaration and international variants show expected form behavior |
| GO-011 | Permission/safety spot checks pass | Dangerous actions are server-side permission checked |
| GO-012 | No scope drift | No officer workflow, bidder portal, PDF, evaluation, award, contract, production BoQ, or integration scope was introduced |
| GO-013 | Evidence captured | Smoke evidence and issue log exist |
| GO-014 | Blocker issues resolved | No unresolved blocker issues remain |

## 7. Conditional Go Criteria

The project may proceed with conditions if one or more non-blocking gaps remain, provided they are explicitly accepted and do not undermine the next workstream.

Acceptable conditional gaps may include:

| Gap | Acceptable if |
|---|---|
| UI styling is basic | Data is correct and readable |
| Filters are not fully implemented | Tables remain small and usable |
| Raw JSON component selector is basic | Raw JSON remains read-only |
| Output completeness detection is simple | Limitations are visible |
| Demo tender reuse policy is simple | Evidence records are not accidentally overwritten |
| Package validation has warnings | Warnings are understood and not blockers |
| Rule trace row expansion is basic | Condition/effect summaries are readable |

These conditions shall be recorded before proceeding.

---

## 8. No-Go Criteria

The project shall not proceed to procurement officer use cases if any of the following remain unresolved.

| ID | No-go condition | Reason |
|---|---|---|
| NOGO-001 | Package internals can be edited casually through admin UI | Reopens legally risky route |
| NOGO-002 | Package validation cannot run | Package health is not visible |
| NOGO-003 | Rule trace cannot explain form activation or blockers | Engine remains a black box |
| NOGO-004 | Demo tender path cannot run end-to-end | Officer workflow would be premature |
| NOGO-005 | Required forms inspector cannot explain generated forms | Upload-elimination strategy is not reviewable |
| NOGO-006 | BoQ inspector cannot show category coverage | Works template output remains opaque |
| NOGO-007 | Preview/audit viewer cannot show hashes/status | Reproducibility is not visible |
| NOGO-008 | Invalid scenarios can generate preview as valid | Safety controls are unreliable |
| NOGO-009 | Re-import or dangerous actions lack permission checks | Security/governance risk |
| NOGO-010 | Major scope drift occurred | Project is losing tractability |

## 11. Residual Risks Before Officer Use Cases

Even if the Admin Console POC passes, the following risks remain and shall be acknowledged.

| Risk | Relevance to next workstream | Required handling |
|---|---|---|
| Procurement officer UI may accidentally expose raw configuration complexity | Officer workflow must use guided forms, not raw JSON |
| Legal-sensitive values may need approval | Officer workflow must include maker/checker or approval hooks later |
| Full STD legal text still not implemented | Officer workflow must remain POC/minimal until full rendering strategy is addressed |
| BoQ remains representative | Officer use cases must not pretend production BoQ is complete |
| Bidder response remains pending | Officer workflow should not enter bidder submission scope prematurely |
| PDF/publication remains pending | Officer workflow should stop at preview/configuration stage unless separately designed |
| Governance lifecycle remains minimal | Do not enable production template use without governance design |

## 12. Recommended Next Workstream if Proceeding

If the decision is `Proceed` or `Proceed with Conditions`, the next workstream shall be:


## 15. Governance Reminder Before Next Workstream

The next workstream may remain POC-level, but it shall not ignore governance/state transitions.

At minimum, the officer-facing design shall identify:

1. who can create a tender,
2. who can modify configuration,
3. who can validate and generate preview,
4. whether generated preview is draft-only,
5. what status transitions exist,
6. what actions are blocked after validation or preview generation,
7. what audit fields are updated,
8. what approvals are deferred and why.

Full production governance may remain deferred, but missing state-transition design shall be explicitly documented.

## 16. Decision Options

### 16.1 Option A — Proceed

Use this decision if:

1. smoke test passed,
2. no blockers remain,
3. reviewer confidence is high,
4. admin visibility is sufficient.

Decision statement:

```text
Proceed to Procurement Officer Tender Configuration POC. The Admin Console POC has sufficiently exposed the STD package model, validation, rule trace, forms, BoQ, preview, and audit outputs to support officer-facing workflow design.
```

### 16.2 Option B — Proceed with Conditions

Use this decision if:

1. smoke test mostly passed,
2. only non-blocking issues remain,
3. conditions are clear and time-bounded.

Decision statement:

```text
Proceed to Procurement Officer Tender Configuration POC with conditions. The conditions listed in this decision record must be resolved before or during the first officer-facing design phase.
```

### 16.3 Option C — Extend Admin Console POC

Use this decision if:

1. admin visibility is still insufficient,
2. rule trace or inspector outputs are not understandable,
3. package validation is incomplete,
4. reviewers cannot confidently explain the POC.

Decision statement:

```text
Extend the STD Administration Console POC before moving to procurement officer use cases. The current administration layer is not yet sufficient to support safe officer-facing workflow design.
```

### 16.4 Option D — Stop / Reconsider

Use this decision only if:

1. package-driven model is rejected,
2. core safety behavior cannot be made visible or reliable,
3. the direction is no longer viable.

Decision statement:

```text
Stop and reconsider the STD package-driven approach. The current POC evidence does not support proceeding safely.
```

## 19. Recommended Decision if Smoke Test Passed

If the Admin Console smoke test passed without unresolved blocker or major issues, the recommended decision is:

```text
Proceed with Conditions to the Procurement Officer Tender Configuration POC.
```

The recommended conditions are:

1. the procurement officer workflow must use guided UI, not raw JSON;
2. package internals must remain hidden/read-only to officers;
3. the first officer workflow must stop before bidder submission, publication, PDF generation, evaluation, award, contract execution, and production BoQ;
4. state transitions and approval assumptions must be documented before implementation;
5. validation and audit outputs must remain visible in the officer workflow;
6. any admin console smoke issues marked Major or Blocker must be resolved before implementation begins.

This is the safest path because it permits forward progress while preserving the controlled architecture.

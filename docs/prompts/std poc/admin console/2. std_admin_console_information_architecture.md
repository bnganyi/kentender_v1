## 2. Purpose

The purpose of this document is to define the information architecture for the STD Administration Console POC.

The information architecture determines how imported Standard Tender Document packages, package metadata, package components, validation outputs, rule traces, forms, BoQ rows, sample demonstrations, generated previews, and audit values shall be surfaced to internal administrative and review users.

This document does not define detailed implementation code. It defines the screens, navigation structure, data groupings, information hierarchy, user journeys, and display responsibilities needed before implementation.

---

## 3. Product Principle

The Admin Console information architecture shall implement the principle:

> Make the package-driven STD engine understandable without making template internals casually editable.

The interface shall therefore prioritize:

1. visibility,
2. traceability,
3. explainability,
4. auditability,
5. safe administrative actions,
6. clear POC boundaries.

It shall not prioritize template authoring or no-code legal document editing.

---

## 4. Target Users

The information architecture shall support the following users.

| User | Primary need |
|---|---|
| Technical Administrator | Import, validate, inspect, and troubleshoot packages |
| Frappe Developer | Verify package structure, engine behavior, and generated outputs |
| Procurement SME | Understand STD sections, fields, rules, forms, and sample scenarios |
| Legal/Compliance Reviewer | Inspect locked/configurable boundaries, audit values, and risk posture |
| QA Reviewer | Execute repeatable checks and compare expected/observed outputs |
| Product Owner | Review POC readiness and decide next workstream |

The information architecture shall not target bidders, procurement officers, evaluation committee members, or contract managers in this workstream.

## 5. Navigation Model

The Admin Console POC shall use a simple hub-and-detail model.

```text
STD Administration Console
│
├── STD Template Dashboard
│   └── STD Template Detail / Package Viewer
│       ├── Overview
│       ├── Package Health
│       ├── Sections
│       ├── Fields
│       ├── Rules
│       ├── Forms
│       ├── Sample Tender
│       ├── Render Map
│       ├── Raw JSON
│       └── Actions
│
├── Package Validation
│   └── Validation Result Detail
│
├── Rule Trace
│   └── Rule Trace Result Detail
│
├── Sample Tender Demo Workspace
│   ├── Sample Configuration
│   ├── Lots
│   ├── BoQ
│   ├── Validation Messages
│   ├── Required Forms
│   ├── Preview
│   └── Audit
│
└── Evidence / POC Status
    ├── Smoke Summary
    ├── Known Limitations
    └── Links to Evidence Pack
```

For implementation speed, this model may be delivered through enhanced Frappe DocType forms and buttons rather than a single custom page. The navigation model remains the target mental model even if the first implementation is split across existing forms.

## 6. Recommended First Implementation Shape

The fastest tractable implementation shall use the existing Frappe objects before creating a complex custom application page.

Recommended first implementation:

| UI surface | Responsibility |
|---|---|
| `STD Template` List View | Template dashboard |
| `STD Template` Form | Package detail, component summaries, package actions |
| `Procurement Tender` Form | Sample demo tender, validation, forms, BoQ, preview |
| Custom server dialogs or HTML fields | Package validation output and rule trace output |
| Optional Frappe Page | Later consolidated dashboard if form layout becomes insufficient |

This keeps the implementation minimal while still solving the main problem: reviewers can see what the POC is doing.

## 8. STD Template Dashboard

### 8.1 Purpose

The STD Template Dashboard shall provide the entry point for template administration and review.

It shall answer:

1. Which STD templates exist in the system?
2. What version and status is each template?
3. When was it imported?
4. Is its package hash available?
5. Can it be used for POC demonstration?

### 8.2 Required Columns

| Column | Source | Required | Notes |
|---|---|---:|---|
| Template Code | `STD Template.template_code` | Yes | Stable identifier |
| Template Short Name | `STD Template.template_short_name` | Yes | Compact display name |

| Procurement Category | `STD Template.procurement_category` | Yes | `WORKS` for current POC |
| Template Family | `STD Template.template_family` | Yes | Family/classification |

## 17. Information Hierarchy

The console shall prioritize information in this order:

1. Template identity and status.
2. Package health and validation state.
3. Sample/demo state.
4. Validation and rule trace outputs.
5. Required forms and BoQ outputs.
6. Preview output.
7. Audit/reproducibility metadata.
8. Raw JSON fallback.

Raw JSON shall never be the first or only way to understand the package.

## 18. State Indicators

The console shall use simple state indicators.

### 18.1 Package State Indicators

| Indicator | Meaning |
|---|---|
| Not Imported | No `STD Template` record exists |
| Imported | Package exists in database |
| Package Valid | Package consistency checks passed |
| Package Has Warnings | Package checks passed with warnings |
| Package Invalid | Package has errors/blockers |
| POC Demo Ready | Package can run sample demo |

### 18.2 Demo Tender State Indicators

| Indicator | Meaning |
|---|---|
| No Demo Tender | No linked demo tender exists |
| Configured | Sample configuration loaded |
| Validated | Validation passed |
| Validation Blocked | Validation blocks generation |
| Forms Generated | Required forms generated |
| BoQ Generated | Sample BoQ generated |
| Preview Generated | Tender-pack preview generated |

These indicators may be derived at runtime rather than stored as formal workflow states.

## 19. Required User Journeys

### 19.1 Journey A — Technical Administrator Re-Imports and Validates Package

1. Open STD Template Dashboard.
2. Select STD-WORKS-POC package.
3. Click Re-import Package.
4. Confirm package hash is updated or unchanged.
5. Click Validate Package.
6. Review validation checks.
7. Confirm no blockers.

Expected outcome:

The administrator can confirm the imported package is valid and traceable.

---

### 19.2 Journey B — Procurement SME Reviews Template Structure

1. Open package viewer.
2. Review Overview.
3. Review Sections.
4. Review Fields grouped by functional area.
5. Review Rules summary.
6. Review Forms summary.
7. Review Sample Tender variants.

Expected outcome:

The procurement SME can understand what the template package controls and what the procurement officer will eventually configure.

---

### 19.3 Journey C — Legal/Compliance Reviewer Checks Locked/Configurable Boundary

1. Open package viewer.
2. Review Sections tab.
3. Identify locked sections and configurable surfaces.
4. Review Rules affecting legal-sensitive outputs.
5. Review Preview and Audit output.
6. Confirm POC limitations are visible.

Expected outcome:

The reviewer can see that ordinary users are not editing legal STD text and that controlled configuration is traceable.

---

### 19.4 Journey D — QA Reviewer Runs Sample Demo

1. Open STD Template.
2. Open or create sample demo tender.
3. Load primary sample.
4. Validate configuration.
5. Generate required forms.
6. Generate sample BoQ.
7. Generate preview.
8. Confirm expected outputs.

Expected outcome:

The reviewer can run a repeatable demonstration without inspecting raw code.

---

### 19.5 Journey E — Developer Explains Rule Activation

1. Open Rule Trace View.
2. Select primary sample or variant.
3. Filter for applied rules.
4. Show which rules activated forms or blockers.
5. Open Required Forms Inspector to confirm output.

Expected outcome:

The developer can explain why the engine produced a specific result.

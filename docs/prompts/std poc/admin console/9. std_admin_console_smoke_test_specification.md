## 2. Purpose

The purpose of this document is to define the smoke test for the STD Administration Console POC.

The smoke test shall verify that the administration and observability layer works end-to-end over the existing STD-WORKS-POC. It shall prove that an internal reviewer can inspect the imported template package, validate package health, trace rule behavior, create or open a demo tender, inspect generated required forms and BoQ rows, view generated preview output, and confirm audit/reproducibility values without reading raw repository files or backend code.

This smoke test is not a full regression suite and does not certify production readiness. It verifies that the Admin Console POC is fit to support the next decision: whether to proceed to procurement officer use cases.

---

## 3. Smoke Test Principle

The smoke test shall implement the principle:

> The Admin Console POC passes only if the existing STD-WORKS-POC becomes visible, inspectable, explainable, and demonstrable through controlled administration UI.

The test shall therefore confirm:

1. package contents are readable through UI,
2. package validation produces clear results,
3. rule trace explains primary and variant behavior,
4. demo tender flow can be run from admin UI,
5. required forms are visible with activation reasons,
6. BoQ rows are visible with category and lot checks,
7. preview output is visible with POC warning,
8. audit values are visible and consistent,
9. invalid scenarios remain blocked,
10. no out-of-scope production workflows were introduced.

---

## 4. Scope of the Smoke Test

This smoke test covers:

1. `STD Template` admin entry point.
2. Package summary and package component viewer.
3. Raw JSON read-only fallback.
4. Package validation.
5. Rule trace for primary sample.
6. Rule trace for selected variants.
7. Demo tender creation/opening.
8. Demo tender action sequence.
9. Required Forms Inspector.
10. BoQ Inspector.
11. Preview and Audit Viewer.
12. Permission/safety spot checks.
13. Negative scenario visibility.
14. Evidence capture.

---

## 5. Non-Scope

The smoke test shall not test or require:

1. procurement officer production tender wizard,
2. bidder portal,
3. bid submission workflow,
4. bid opening workflow,
5. evaluation workflow,
6. award workflow,
7. contract execution workflow,
8. production BoQ module,
9. PDF generation,
10. tender publication workflow,
11. external integrations,
12. production permission matrix certification,
13. legal certification of the tender preview.

If any of those appear during the smoke test, they shall be recorded as scope drift.

## 28. Pass / Fail Criteria

### 28.1 Pass Criteria

The Admin Console POC smoke test passes if:

1. all required UI entry points are accessible,
2. package summary views are readable,
3. raw JSON is read-only,
4. package validation works,
5. rule trace explains primary and variant behavior,
6. demo tender can be created/opened,
7. primary demo path passes,
8. required forms inspector shows expected form behavior,
9. BoQ inspector shows expected row/category behavior,
10. preview/audit viewer shows generated output and hashes,
11. negative variants show blockers and block preview generation,
12. permission/safety checks do not reveal dangerous uncontrolled actions,
13. no out-of-scope workflows are introduced,
14. smoke evidence is captured.

### 28.2 Fail Criteria

The smoke test fails if any of the following occurs:

1. package viewer is not accessible,
2. package internals are editable through the console,
3. package validation cannot run,
4. rule trace cannot explain primary or variant behavior,
5. demo tender cannot be created/opened,
6. primary demo path fails,
7. required forms inspector does not show activation/evidence data,
8. BoQ inspector does not show category coverage,
9. preview/audit viewer does not show hashes/status,
10. invalid variant generates preview as if valid,
11. re-import is available without permission check,
12. out-of-scope workflows are introduced.

## 2. Purpose

The purpose of this document is to define the Required Forms Inspector and BoQ Inspector for the STD Administration Console POC.

These inspectors shall make two central outputs of the STD-WORKS-POC clearly visible and explainable:

1. the required bidder and contract forms generated from `forms.json`, `rules.json`, and tender configuration; and
2. the representative structured BoQ rows generated from `sample_tender.json` and stored as tender child data.

The inspectors shall help reviewers understand how the system reduces uncontrolled document uploads by replacing manually assembled tender packs with structured, traceable system data.

---

## 3. Design Principle

The Required Forms and BoQ Inspectors shall implement the principle:

> Generated tender requirements must be visible, explainable, and traceable to their source configuration and rules.

The inspectors shall therefore show not only **what** was generated, but also:

1. why it was generated,
2. which rule or default requirement caused it,
3. what evidence policy applies,
4. what workflow stage it belongs to,
5. how it links to the tender configuration,
6. whether expected scenario outcomes are present,
7. whether generated rows are complete and non-duplicative.

## 8. Required Forms Inspector — Purpose

The Required Forms Inspector shall show the generated required forms checklist for a tender or demo tender.

It shall answer:

1. Which forms are required?
2. Which forms are default required?
3. Which forms are activated by rules?
4. Which configuration branch caused each conditional form?
5. What evidence policy applies to each form?
6. Which forms are excluded under the current configuration?
7. Are there duplicate or missing expected forms?
8. Does the generated checklist match the selected scenario?

---

## 11. Required Forms Inspector — Activation Explanation

Each form row shall have an explanation.

Required explanation fields:

| Field | Meaning |
|---|---|
| Activation Type | `DEFAULT`, `RULE_ACTIVATED`, or `DEFAULT_AND_RULE_ACTIVATED` |
| Activation Source | Rule code(s) or default |
| Triggering Configuration | Human-readable summary if rule-derived |
| Rule Label | Rule label from `rules.json`, if available |
| Scenario Relevance | Whether the form is expected for the current sample/variant |

Example explanation:

```text
FORM_TENDER_SECURITY is required because SECURITY.TENDER_SECURITY_MODE is TENDER_SECURITY. The form was activated by RULE_TENDER_SECURITY_ACTIVATE_FORM.
```

Example default explanation:

```text
FORM_OF_TENDER is required by default for all tenders using this STD package.
```

The first implementation may display this explanation in an expandable details row or a read-only detail panel.

## 14. Required Forms Inspector — Evidence Policy Display

Evidence policy is central to the strategic objective of reducing uncontrolled uploads.

The inspector shall show evidence posture in plain language.

Example mapping:

| Evidence policy | Display meaning |
|---|---|
| `STRUCTURED_ONLY` | The form should be captured as structured system data, not as a routine uploaded document |
| `STRUCTURED_WITH_OPTIONAL_EVIDENCE` | Structured data is primary; evidence attachment may be allowed where needed |
| `STRUCTURED_WITH_ISSUER_VERIFICATION` | Structured data is primary; issuer verification or controlled instrument evidence may be required |
| `POST_AWARD_SECURITY_INSTRUMENT` | Applies after award/contract stage and may require controlled security instrument handling |

If exact policy values differ in `forms.json`, the implementation shall map the existing values to clear reviewer-facing explanations.

## 15. BoQ Inspector — Purpose

The BoQ Inspector shall show structured BoQ rows for a tender or demo tender.

It shall answer:

1. Are BoQ rows present?
2. Are required categories represented?
3. Are dayworks and provisional sums present when configured?
4. Are item codes unique?
5. Are lot references valid?
6. Which rows are bidder-priced and which are fixed/summary rows?
7. Is the BoQ sample representative rather than production-complete?

---

## 16. BoQ Inspector — Required Warning

The BoQ Inspector shall show this warning or materially equivalent text:

```text
POC BoQ Inspector: The BoQ rows shown here are representative structured POC data. They prove that Works BoQ data can be handled as system data, but they are not a production-grade BoQ authoring, pricing, or measurement module.
```

This warning shall be visible near the top of the BoQ Inspector.

## 20. BoQ Inspector — Lot Reference Checks

The inspector shall check BoQ row lot references against `Tender Lot` rows.

Required checks:

| Check | Expected behavior |
|---|---|
| Lot code exists | If `lot_code` is populated, it should match a tender lot |
| Grand summary may be unlinked | `GRAND_SUMMARY` may have no lot code |
| Missing lot reference | Show warning or error in validation note |
| Lot title resolution | Show lot title where matched |

This does not create lots or alter BoQ rows. It is inspect-only.

---

## 21. BoQ Inspector — Row Type Classification

The inspector shall classify each row.

| Row condition | Row type |
|---|---|
| `is_priced_by_bidder = true` | `Bidder priced` |
| `is_priced_by_bidder = false` and `amount` is present | `Fixed amount / provisional` |
| `item_category = GRAND_SUMMARY` | `Summary` |
| Otherwise | `Other` |

This helps non-technical reviewers understand how pricing responsibility is represented.

---

## 22. BoQ Inspector — Rule Relationship

The inspector shall show how BoQ rows relate to validation rules.

Required rule relationship display:

| Rule concern | Expected support from BoQ rows |
|---|---|
| BoQ required | At least one BoQ row exists |
| Dayworks included | At least one `DAYWORKS` row exists |
| Provisional sums included | At least one `PROVISIONAL_SUMS` row exists |
| Multiple lots | Rows reference expected lot codes where applicable |

If validation messages exist for BoQ-related rules, the inspector shall show them or link to the Validation Messages section.

## 23. Required Inspector Server Methods

The inspectors may be supported by server methods that produce ready-to-render summaries.

### 23.1 `get_required_forms_inspection(tender_name)`

Purpose:

Return required forms inspection data for a `Procurement Tender`.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `tender_name` | String | Yes |

Required behavior:

1. Load `Procurement Tender`.
2. Check read permission.
3. Load linked `STD Template` package if available.
4. Read `required_forms` child rows.
5. Resolve form metadata from `forms.json`.
6. Detect duplicates.
7. Build summary counts.
8. Build row explanations.
9. Build expected/observed comparison if sample or variant context is available.
10. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "tender": "PT-2026-00001",
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC",
  "summary": {},
  "rows": [],
  "expected_comparison": [],
  "warnings": []
}
```

### 23.2 `get_boq_inspection(tender_name)`

Purpose:

Return BoQ inspection data for a `Procurement Tender`.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `tender_name` | String | Yes |

Required behavior:

1. Load `Procurement Tender`.
2. Check read permission.
3. Read `boq_items` child rows.
4. Read `lots` child rows.
5. Compute category coverage.
6. Detect duplicate item codes.
7. Detect invalid lot references.
8. Classify row types.
9. Build summary counts.
10. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "tender": "PT-2026-00001",
  "summary": {},
  "rows": [],
  "category_coverage": [],
  "lot_reference_checks": [],
  "warnings": []
}
```

### 23.3 Optional Combined Method

A combined method may be provided for demo workspace efficiency:

```python
def get_demo_inspector_summary(tender_name) -> dict:
    ...
```

It may return both required forms and BoQ inspection data. This is optional.

## 25. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| Required forms inspection | Requires read permission on `Procurement Tender` |
| BoQ inspection | Requires read permission on `Procurement Tender` |
| No mutation | Inspection methods shall not change tender data |
| No package mutation | Inspectors shall not edit package internals |
| No form schema editing | Inspectors shall not edit `forms.json` |
| No BoQ authoring | BoQ Inspector shall not become production BoQ editor |
| Clear POC labelling | BoQ and forms limitations shall be visible |

---

## 26. Error Handling Requirements

The inspectors shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Tender not found | Clear error |
| User lacks permission | Permission error |
| Tender missing linked template | Show warning; inspect child rows where possible |
| Package JSON missing | Show package warning; inspect child rows where possible |
| Forms metadata missing | Show warning and mark source metadata unavailable |
| Required forms empty | Show empty state and instruction to generate forms |
| BoQ rows empty | Show empty state and instruction to generate BoQ |
| Duplicate form codes | Show warning/error in summary |
| Duplicate BoQ item codes | Show warning/error in summary |
| Invalid lot references | Show warning/error in BoQ row notes |

Silent failure is not acceptable.

## 33. Exit Condition

This step is complete when administrators and reviewers can inspect generated required forms and representative BoQ rows in a readable interface, understand why forms are required, verify evidence policies, detect missing or duplicate outputs, confirm BoQ category coverage, and capture evidence without reading raw JSON or backend code.

Once accepted, the next artefact shall be the **Preview and Audit Viewer Specification**.

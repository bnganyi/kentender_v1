## 2. Purpose

The purpose of this document is to define the package validation and rule trace capabilities required by the STD Administration Console POC.

The package validation capability shall confirm that an imported STD template package is internally consistent, structurally complete, and safe to use for POC demonstration.

The rule trace capability shall explain how `rules.json` behaves against a selected configuration, including which rules apply, which rules do not apply, which fields/forms/sections are activated, which validation messages are produced, and which rules block tender-pack generation.

Together, these features make the STD engine inspectable and defensible. They turn the package-driven model from a black box into an explainable administration artefact.

---

## 3. Design Principle

The package validation and rule trace features shall implement the principle:

> A controlled STD package must be not only executable, but explainable, verifiable, and auditable before it can safely drive tender creation.

The console shall therefore show:

1. whether the package is structurally valid,
2. where package inconsistencies exist,
3. whether all references resolve,
4. whether all rule operators and rule types are supported,
5. how a configuration triggers rules,
6. why forms become required,
7. why tender-pack generation is blocked or allowed,
8. what evidence supports the package's current health status.

## 6. Required Capability Overview

The Admin Console shall provide two distinct but related capabilities:

| Capability | Purpose | Primary user question answered |
|---|---|---|
| Package Validation | Validate package structure and references | Is this template package internally consistent and safe to run? |
| Rule Trace | Explain rule execution against a configuration | Why did this configuration produce these required fields, forms, messages, and blockers? |

These capabilities shall be available from the `STD Template` context and, where applicable, from a linked demo `Procurement Tender` context.

## 7. Package Validation Inputs

Package validation shall use the imported package stored on `STD Template.package_json`.

Required inputs:

| Input | Source |
|---|---|
| `STD Template` record | Frappe database |
| `package_json` | `STD Template.package_json` |
| `package_hash` | `STD Template.package_hash` |
| Manifest component | `package_json.manifest` |
| Sections component | `package_json.sections` |
| Fields component | `package_json.fields` |
| Rules component | `package_json.rules` |
| Forms component | `package_json.forms` |
| Render map component | `package_json.render_map` |
| Sample tender component | `package_json.sample_tender` |

Validation shall not read arbitrary files from disk. The seed loader is responsible for importing package files. The admin validation feature validates the imported runtime package.

## 8. Package Validation Output Summary

Package validation shall return a structured result.

Required top-level shape:

```json
{
  "ok": true,
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC",
  "template_name": "...",
  "package_hash": "...",
  "overall_status": "PASSED",
  "summary": {
    "total_checks": 0,
    "passed": 0,
    "warnings": 0,
    "errors": 0,
    "blockers": 0,
    "skipped": 0
  },
  "checks": []
}
```

Supported `overall_status` values:

| Value | Meaning |
|---|---|
| `PASSED` | All checks passed or only informational checks present |
| `PASSED_WITH_WARNINGS` | No errors or blockers, but warnings exist |
| `FAILED` | One or more errors exist, no blockers |
| `BLOCKED` | One or more blockers exist |

## 9. Package Validation Check Row Shape

Each validation check shall use this shape:

```json
{
  "check_code": "PKG_COMPONENTS_PRESENT",
  "label": "Required package components are present",
  "category": "PACKAGE_STRUCTURE",
  "severity": "BLOCKER",
  "status": "PASSED",
  "message": "All required package components are present.",
  "reference": "package_json",
  "details": {}
}
```

Supported `status` values:

| Status | Meaning |
|---|---|
| `PASSED` | Check passed |
| `WARNING` | Check produced warning |
| `FAILED` | Check failed with error severity |
| `BLOCKED` | Check failed with blocker severity |
| `SKIPPED` | Check was not run due to missing prerequisite |

Supported `severity` values:

| Severity | Meaning |
|---|---|
| `INFO` | Informational check |
| `WARNING` | Non-blocking issue |
| `ERROR` | Error that should be fixed before relying on the package |
| `BLOCKER` | Serious inconsistency that prevents safe package operation |

## 11. Required Package Validation Checks

### 11.1 Package Structure Checks

| Check code | Severity | Requirement |
|---|---|---|
| `PKG_COMPONENTS_PRESENT` | BLOCKER | Package contains manifest, sections, fields, rules, forms, render_map, sample_tender |
| `PKG_COMPONENTS_ARE_OBJECTS` | BLOCKER | Each major component is a JSON object |
| `PKG_REQUIRED_TOP_LEVEL_KEYS` | ERROR | Each component has expected top-level keys |

### 11.2 Template Identity Checks

| Check code | Severity | Requirement |
|---|---|---|
| `TPL_MANIFEST_TEMPLATE_CODE_PRESENT` | BLOCKER | Manifest has `template_code` |
| `TPL_TEMPLATE_CODE_CONSISTENT` | BLOCKER | All components with `template_code` match manifest |
| `TPL_SAMPLE_TEMPLATE_CODE_CONSISTENT` | BLOCKER | `sample_tender.template_code` matches manifest |
| `TPL_STD_TEMPLATE_RECORD_MATCHES_PACKAGE` | ERROR | `STD Template.template_code` matches package manifest |
| `TPL_PACKAGE_VERSION_PRESENT` | ERROR | Package version exists in manifest |
| `TPL_SOURCE_DOCUMENT_PRESENT` | WARNING | Source document metadata exists |

### 11.3 Sections Checks

| Check code | Severity | Requirement |
|---|---|---|
| `SEC_SECTIONS_ARRAY_PRESENT` | BLOCKER | `sections.sections` exists and is an array |
| `SEC_SECTION_CODES_UNIQUE` | BLOCKER | Section codes are unique |
| `SEC_REQUIRED_SECTION_FIELDS_PRESENT` | ERROR | Each section has required fields |
| `SEC_RENDER_ORDER_VALID` | WARNING | Render order exists where included in POC render |

### 11.4 Fields Checks

| Check code | Severity | Requirement |
|---|---|---|
| `FLD_FIELDS_ARRAY_PRESENT` | BLOCKER | `fields.fields` exists and is an array |
| `FLD_FIELD_CODES_UNIQUE` | BLOCKER | Field codes are unique |
| `FLD_REQUIRED_FIELD_METADATA_PRESENT` | ERROR | Each field has required metadata |
| `FLD_SELECT_OPTIONS_VALID` | ERROR | Select fields have valid options where required |
| `FLD_SYSTEM_FIELDS_PRESENT` | WARNING | Expected system/audit fields exist |

### 11.5 Rules Checks

| Check code | Severity | Requirement |
|---|---|---|
| `RUL_RULES_ARRAY_PRESENT` | BLOCKER | `rules.rules` exists and is an array |
| `RUL_RULE_CODES_UNIQUE` | BLOCKER | Rule codes are unique |
| `RUL_REQUIRED_RULE_METADATA_PRESENT` | ERROR | Each rule has required metadata |
| `RUL_RULE_TYPES_SUPPORTED` | BLOCKER | Rule types are supported by engine |
| `RUL_OPERATORS_SUPPORTED` | BLOCKER | Operators are supported by engine |
| `RUL_SEVERITIES_SUPPORTED` | ERROR | Severities are supported |
| `RUL_FIELD_REFERENCES_EXIST` | BLOCKER | All rule field references exist in `fields.json` |
| `RUL_SECTION_REFERENCES_EXIST` | BLOCKER | All rule section references exist in `sections.json` |
| `RUL_FORM_REFERENCES_EXIST` | BLOCKER | All rule form references exist in `forms.json` |

### 11.6 Forms Checks

| Check code | Severity | Requirement |
|---|---|---|
| `FRM_FORMS_ARRAY_PRESENT` | BLOCKER | `forms.forms` exists and is an array |
| `FRM_FORM_CODES_UNIQUE` | BLOCKER | Form codes are unique |
| `FRM_REQUIRED_FORM_METADATA_PRESENT` | ERROR | Each form has required metadata |
| `FRM_ACTIVATION_RULES_EXIST` | ERROR | Activation rule codes reference existing rules |
| `FRM_CHECKLIST_DISPLAY_VALID` | ERROR | Each form has display group and display order |
| `FRM_ORDINARY_USER_SCHEMA_LOCKED` | WARNING | Form schemas are not editable by ordinary users |

### 11.7 Sample Tender Checks

| Check code | Severity | Requirement |
|---|---|---|
| `SMP_SAMPLE_IDENTITY_PRESENT` | ERROR | Sample code/name/template code exist |
| `SMP_CONFIGURATION_PRESENT` | BLOCKER | Sample configuration exists and is an object |
| `SMP_CONFIGURATION_FIELDS_EXIST` | BLOCKER | Every sample configuration key exists in `fields.json` |
| `SMP_EXPECTED_FORMS_EXIST` | BLOCKER | Expected activated forms exist in `forms.json` |
| `SMP_LOTS_VALID` | ERROR | Sample lots have required fields and unique lot codes |
| `SMP_BOQ_ROWS_VALID` | ERROR | Sample BoQ rows have required fields and unique item codes |
| `SMP_BOQ_CATEGORIES_SUPPORTED` | ERROR | BoQ categories are supported by POC model |
| `SMP_VARIANTS_VALID` | ERROR | Scenario variants have required shape and valid overrides |
| `SMP_VARIANT_FORMS_EXIST` | ERROR | Variant expected forms exist in `forms.json` |

### 11.8 Render Map Checks

| Check code | Severity | Requirement |
|---|---|---|
| `RND_RENDER_MAP_PRESENT` | WARNING | Render map exists |
| `RND_SECTION_REFERENCES_EXIST` | ERROR | Render map section references exist where present |
| `RND_POC_RENDER_COVERAGE` | WARNING | Required POC preview sections are represented or warning emitted |

Because the current render map may be skeletal, warnings are acceptable if the preview renderer works through fixed Jinja templates.

### 11.9 Audit Checks

| Check code | Severity | Requirement |
|---|---|---|
| `AUD_PACKAGE_HASH_PRESENT` | ERROR | `STD Template.package_hash` is populated |
| `AUD_PACKAGE_HASH_FORMAT` | WARNING | Package hash appears to be SHA-256 hex |
| `AUD_IMPORTED_AT_PRESENT` | WARNING | Import timestamp exists |
| `AUD_PACKAGE_JSON_PRESENT` | BLOCKER | Package JSON exists |

## 13. Rule Trace Inputs

Rule trace shall support these input modes.

| Mode | Required? | Source | Purpose |
|---|---:|---|---|
| Primary sample | Yes | `sample_tender.configuration` | Explain normal successful scenario |
| Sample variant | Yes | Sample configuration plus selected variant overrides | Explain scenario branches |
| Demo tender | Yes | `Procurement Tender.configuration_json`, lots, BoQ | Explain actual demo record state |
| Arbitrary pasted config | No | User-provided JSON | Deferred; avoid raw editing path |

The first implementation shall support primary sample, sample variant, and demo tender trace. Arbitrary pasted configuration is not required and should be avoided unless controlled later.

## 16. Condition Trace Requirements

Each rule trace row shall include condition-level explainability.

Condition object shape:

```json
{
  "field_code": "SECURITY.TENDER_SECURITY_MODE",
  "operator": "EQUALS",
  "expected_value": "TENDER_SECURITY",
  "actual_value": "TENDER_SECURITY",
  "passed": true,
  "message": "SECURITY.TENDER_SECURITY_MODE equals TENDER_SECURITY."
}
```

For conditions without expected values, such as `IS_TRUE`, the trace shall still show:

1. field code,
2. operator,
3. actual value,
4. passed result,
5. readable message.

---

## 17. Effect Trace Requirements

Each rule trace row shall summarize rule effects.

Supported effect categories:

| Effect category | Example |
|---|---|
| Required fields | `then.require_fields` |
| Activated fields | `then.activate_fields` |
| Activated sections | `then.activate_sections` |
| Activated forms | `then.activate_forms` |
| Child row requirements | `then.require_child_rows` |
| Date validations | `then.validate_date_order` |
| Numeric validations | `then.validate_numeric_limit` |

Effect trace shall show both the rule intent and the result where evaluated.

Example:

```text
Activates FORM_TENDER_SECURITY because tender security mode is TENDER_SECURITY.
```

Example:

```text
Requires at least one Tender BoQ Item row with item_category = DAYWORKS; found 1.
```

## 19. Rule Trace Required Scenarios

The Admin Console shall support trace for these scenarios.

### 19.1 Primary Sample Trace

Expected behavior:

1. Most configured positive branch rules apply.
2. No blockers for primary sample.
3. Active forms include the primary expected forms.
4. `FORM_TENDER_SECURITY` is active.
5. `FORM_TENDER_SECURING_DECLARATION` is not active.
6. `FORM_FOREIGN_TENDERER_LOCAL_INPUT` is not active.

### 19.2 Tender-Securing Declaration Variant Trace

Variant code:

```text
VARIANT-TENDER-SECURING-DECLARATION
```

Expected behavior:

1. Tender-securing declaration rule applies.
2. `FORM_TENDER_SECURING_DECLARATION` is active.
3. `FORM_TENDER_SECURITY` is not active.

### 19.3 International Variant Trace

Variant code:

```text
VARIANT-INTERNATIONAL
```

Expected behavior:

1. International/local input rule applies.
2. `FORM_FOREIGN_TENDERER_LOCAL_INPUT` is active.

### 19.4 Missing Site Visit Variant Trace

Variant code:

```text
VARIANT-MISSING-SITE-VISIT-DATE
```

Expected behavior:

1. Site visit rule applies.
2. Required site visit fields fail.
3. Validation status is blocked.
4. Rule trace shows missing field messages.

### 19.5 Missing Alternative Scope Variant Trace

Variant code:

```text
VARIANT-MISSING-ALTERNATIVE-SCOPE
```

Expected behavior:

1. Alternative tender rule applies.
2. Alternative scope field is missing.
3. Validation status is blocked.

## 20. Rule Trace Relationship to Existing Engine

The trace feature shall reuse the existing `std_template_engine.py` rule evaluation logic as much as possible.

However, the existing engine may return only aggregate validation results. The trace feature may require one additional engine helper that evaluates rules with explainability.

Recommended helper:

```python
def trace_rules(template, config, lots=None, boq_items=None) -> dict:
    ...
```

Required behavior:

1. Load all rules.
2. Evaluate every rule, not only applied rules.
3. Return condition-level trace.
4. Return effect-level trace.
5. Return messages consistent with `validate_config()`.
6. Return active fields, sections, and forms.
7. Return configuration hash.
8. Do not write to database.

The trace helper shall not replace `validate_config()`. It shall explain it.

## 21. Server Method Requirements

### 21.1 Required Method: `validate_std_package(template_name)`

Purpose:

Run package validation checks against an imported `STD Template` package.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `template_name` | String | Yes |

Expected behavior:

1. Load `STD Template`.
2. Check read permission.
3. Parse `package_json`.
4. Run all package validation checks.
5. Return structured package validation result.
6. Do not modify package JSON.
7. Optionally store latest validation summary only if a field exists and implementation intentionally supports it.

### 21.2 Required Method: `trace_std_rules_for_sample(template_name, variant_code=None)`

Purpose:

Generate rule trace using primary sample or sample variant configuration.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `template_name` | String | Yes |
| `variant_code` | String | No |

Expected behavior:

1. Load `STD Template`.
2. Check read permission.
3. Load sample configuration through engine.
4. Apply variant if provided.
5. Load sample lots and BoQ rows.
6. Generate rule trace.
7. Return structured trace result.

### 21.3 Required Method: `trace_std_rules_for_tender(tender_name)`

Purpose:

Generate rule trace using an existing `Procurement Tender` configuration.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `tender_name` | String | Yes |

Expected behavior:

1. Load `Procurement Tender`.
2. Check read permission.
3. Confirm linked `std_template` exists.
4. Parse `configuration_json`.
5. Convert lots and BoQ rows to dictionaries.
6. Generate rule trace.
7. Return structured trace result.

## 23. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| Package validation | Requires read permission on `STD Template` |
| Rule trace for sample | Requires read permission on `STD Template` |
| Rule trace for tender | Requires read permission on `Procurement Tender` |
| No package mutation | Validation and trace do not edit package JSON |
| No arbitrary code execution | Rule trace interprets only supported declarative rules |
| No arbitrary file access | Validation uses imported package JSON only |
| No raw rule editing | UI exposes rules read-only |

---

## 24. Error Handling Requirements

The validation and trace features shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Template not found | Clear error |
| User lacks permission | Permission error |
| Package JSON missing | Clear package error |
| Package JSON invalid | Clear parse error |
| Required component missing | Validation blocker or clear error |
| Unsupported rule type | Validation blocker |
| Unsupported operator | Validation blocker |
| Rule references unknown field | Validation blocker |
| Rule references unknown form | Validation blocker |
| Rule references unknown section | Validation blocker |
| Variant code unknown | Clear error |
| Tender configuration missing | Clear tender error |
| Tender configuration invalid JSON | Clear parse error |

Silent skipping of rule or reference errors is not acceptable.

## 25. Minimal Implementation Strategy

Recommended implementation order:

1. Add package validation helper in admin console module or extend engine with read-only validation utilities.
2. Add `validate_std_package(template_name)` server method.
3. Add UI button and table output for package validation.
4. Add `trace_rules()` helper to engine or admin service.
5. Add trace server methods for primary sample and variant.
6. Add trace server method for demo tender.
7. Add UI buttons and trace output table.
8. Add row expansion/details only after basic trace table works.
9. Add optional filters if table becomes large.

Do not block the workstream on advanced UI components. Correct output is more important than visual polish at this stage.

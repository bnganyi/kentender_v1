## 2. Purpose

The purpose of this document is to define the Sample Tender Demo Workspace for the STD Administration Console POC.

The workspace shall provide a clear administration-facing way to run, inspect, and explain the STD-WORKS-POC end-to-end demonstration without requiring reviewers to jump between raw JSON, backend code, console commands, and multiple technical artefacts.

The workspace shall make the successful sample tender path and key scenario variants visible through a controlled UI.

It shall not become the procurement officer tender creation workflow. It is an internal administration and demonstration surface for proving the template package and engine behavior.

---

## 3. Design Principle

The Sample Tender Demo Workspace shall implement the principle:

> Demonstrate the STD engine using controlled sample data before exposing procurement officer workflows.

The workspace shall therefore:

1. use the existing `sample_tender.json` as the source of truth for demo data,
2. create or open a controlled demo `Procurement Tender`,
3. run the same engine actions used by the POC,
4. surface validation, forms, BoQ, preview, and audit outputs clearly,
5. support selected positive and negative variants,
6. preserve package internals as read-only,
7. clearly label itself as POC/demo functionality.

## 6. Workspace Entry Points

The Sample Tender Demo Workspace shall be reachable from at least one of the following contexts.

| Entry point | Required behavior |
|---|---|
| `STD Template` form | Action to create or open demo tender for the selected template |
| Template Package Viewer | Link/action to open demo workspace |
| `Procurement Tender` form | Existing demo actions visible when tender is linked to STD-WORKS-POC template |
| Admin Console dashboard, if implemented | Shortcut to demo workspace |

Recommended first implementation:

1. Add `Create/Open Demo Tender` action on `STD Template`.
2. Reuse the `Procurement Tender` form as the workspace.
3. Add clearer admin/demo summary sections and actions to the `Procurement Tender` form.

A dedicated custom Frappe Page may be added later if the `Procurement Tender` form becomes too constrained.

## 7. Workspace Object Model

The demo workspace shall use the existing `Procurement Tender` DocType as the demo record.

### 7.1 Demo Tender Identification

A demo tender shall be identifiable as an internal POC/demo record.

Minimum recommended fields/markers:

| Marker | Required behavior |
|---|---|
| Linked STD Template | `std_template` links to `KE-PPRA-WORKS-BLDG-2022-04-POC` |
| Tender Reference | Uses sample reference or prefixed demo reference |
| POC Notes | Indicates demo/sample use |
| Tender Status | Shows current demo execution state |
| Validation Status | Shows current validation state |

If a dedicated `is_demo` field does not exist, the demo status may be expressed through `poc_notes` and naming convention. A dedicated field may be added only if the team decides it is necessary and does not expand scope.

### 7.2 Demo Tender Reuse Policy

The console may either:

1. create a new demo tender each time, or
2. find and reuse an existing demo tender for the template.

Recommended POC behavior:

| Behavior | Recommendation |
|---|---|
| First run | Create a new demo `Procurement Tender` |
| Later runs | Open the most recent demo tender or ask user whether to create a new one |
| Variant tests | Prefer separate demo records per variant if evidence comparison is needed |
| Reset | Optional; may reload sample data into the current demo tender |

For simplicity, the first implementation may create a new demo tender on each action and return a link to it.

## 12. Required Action Sequencing

The workspace shall support the following recommended sequence.

```text
Create/Open Demo Tender
→ Load Primary Sample
→ Generate Sample BoQ, if needed
→ Validate Configuration
→ Generate Required Forms
→ Generate Tender-Pack Preview
→ Review Audit Summary
```

The UI shall not require users to know this sequence from memory. It should show either:

1. numbered action hints,
2. current state indicators,
3. disabled actions until prerequisites are met, or
4. clear error messages explaining missing prerequisites.

Recommended POC approach:

Use clear error messages and current state indicators. Do not overbuild complex action disabling yet.

## 13. Supported Sample Variants

The workspace shall support these variants from `sample_tender.json`.

| Variant code | Purpose | Expected result |
|---|---|---|
| `VARIANT-INTERNATIONAL` | Tests international tender and foreign tenderer local input | Foreign tenderer local input form active |
| `VARIANT-TENDER-SECURING-DECLARATION` | Tests tender-securing declaration instead of tender security | Tender-securing declaration active; tender security inactive |
| `VARIANT-RESERVED-TENDER` | Tests reservation fields | Reserved tender fields active/valid |
| `VARIANT-MISSING-SITE-VISIT-DATE` | Tests missing site visit blockers | Validation blocked |
| `VARIANT-MISSING-ALTERNATIVE-SCOPE` | Tests missing alternative scope blocker | Validation blocked |
| `VARIANT-SINGLE-LOT` | Tests single-lot scenario | Multiple-lot requirements inactive |
| `VARIANT-RETENTION-MONEY-SECURITY` | Tests retention money security | Retention money security form active |

The `Load Sample Variant` action shall prompt the user to select one of these variant codes.

## 23. Required Server Methods

The workspace may reuse existing `Procurement Tender` methods from STD-WORKS-POC. It also requires admin-specific helper methods for smoother navigation.

### 23.1 `create_or_open_std_demo_tender(template_name, variant_code=None)`

Purpose:

Create or open a demo `Procurement Tender` for a selected `STD Template`.

Expected behavior:

1. Load `STD Template`.
2. Check permission.
3. Create a new `Procurement Tender` or open an existing demo tender according to implementation policy.
4. Link tender to selected template.
5. If `variant_code` is supplied, load sample variant.
6. If no variant is supplied, load primary sample.
7. Save tender.
8. Return structured result with tender name and route/link.

Required return shape:

```json
{
  "ok": true,
  "message": "Demo tender ready.",
  "tender": "PT-2026-00001",
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC",
  "variant_code": null
}
```

### 23.2 `run_std_demo_path(tender_name)` Optional

Purpose:

Run the full demo path in sequence.

This is optional for first implementation.

Expected sequence if implemented:

1. Validate configuration.
2. Generate sample BoQ if rows are missing.
3. Generate required forms.
4. Generate tender-pack preview if validation does not block.
5. Return summary.

If implemented, it must stop before preview generation when validation blocks generation.

### 23.3 Existing Methods to Reuse

The workspace shall reuse existing methods where possible:

1. `load_sample_tender(tender_name)`.
2. `load_sample_variant(tender_name, variant_code)`.
3. `validate_tender_configuration(tender_name)`.
4. `generate_required_forms(tender_name)`.
5. `generate_sample_boq(tender_name)`.
6. `generate_tender_pack_preview(tender_name)`.
7. `trace_std_rules_for_tender(tender_name)`.

Do not duplicate business logic in the workspace UI.

## 26. Error Handling Requirements

The workspace shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Template missing | Clear error |
| User lacks permission | Permission error |
| Sample tender missing from package | Clear package error |
| Unknown variant code | Clear error |
| Configuration JSON missing | Prompt user to load sample first |
| Validation blockers exist | Block preview and show messages |
| BoQ generation fails | Show row/category error |
| Required form generation fails | Show form/rule error |
| Preview generation fails | Show render error |
| Package JSON invalid | Show package parse error |

Silent failure is not acceptable.

## 27. Required Demo Scenarios

The workspace shall support demonstration of these scenarios.

### 27.1 Primary Successful Scenario

Expected result:

1. Primary sample loads.
2. Validation passes.
3. Required forms count is 15.
4. BoQ row count is 9.
5. Preview generates.
6. Audit values are visible.

### 27.2 Tender-Securing Declaration Scenario

Expected result:

1. `FORM_TENDER_SECURING_DECLARATION` is active.
2. `FORM_TENDER_SECURITY` is inactive.
3. Rule trace explains the switch.

### 27.3 International Tender Scenario

Expected result:

1. `FORM_FOREIGN_TENDERER_LOCAL_INPUT` is active.
2. Rule trace explains international/local input activation.

### 27.4 Missing Site Visit Negative Scenario

Expected result:

1. Validation status is blocked.
2. Validation messages identify missing site visit data.
3. Preview generation is blocked.

### 27.5 Missing Alternative Scope Negative Scenario

Expected result:

1. Validation status is blocked.
2. Validation messages identify missing alternative scope.
3. Preview generation is blocked.

## 28. State Transition Expectations

The demo tender shall move through simple observable states.

| State | Trigger | Expected indicators |
|---|---|---|
| Created | Demo tender created | Linked template present |
| Configured | Sample or variant loaded | Configuration JSON populated |
| BoQ Generated | Generate Sample BoQ run | BoQ rows populated |
| Validated | Validate run and passed | Validation status passed |
| Validation Blocked | Validate run and blocked | Validation messages show blockers |
| Forms Generated | Generate Required Forms run | Required forms populated |
| Preview Generated | Preview generated | HTML preview and last generated timestamp present |

These are not formal workflow states. They are derived or represented through existing tender status and fields.

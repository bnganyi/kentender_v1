## 2. Purpose

The purpose of this document is to define the Preview and Audit Viewer for the STD Administration Console POC.

The viewer shall make the generated tender-pack preview and its reproducibility metadata easy to inspect, explain, and capture as evidence. It shall allow reviewers to see the output generated from the controlled STD template package, while also seeing the package hash, configuration hash, validation status, required form count, BoQ row count, and relevant timestamps that prove how the preview was produced.

The viewer shall not convert the POC preview into a final legally certified tender document. It remains an administration and evidence surface for the STD-WORKS-POC.

---

## 3. Design Principle

The Preview and Audit Viewer shall implement the principle:

> Generated tender output must be visibly tied to the exact template package, tender configuration, validation result, and generation timestamp that produced it.

The viewer shall therefore show:

1. generated preview content,
2. clear POC/non-production warning,
3. template identity,
4. package version,
5. source document reference,
6. package hash,
7. configuration hash,
8. validation status,
9. generated forms and BoQ summary,
10. validation messages where relevant,
11. generation timestamps,
12. links back to package viewer, rule trace, forms inspector, and BoQ inspector.

## 6. Viewer Entry Points

The Preview and Audit Viewer shall be reachable from the Sample Tender Demo Workspace and the `Procurement Tender` form.

Required entry points:

| Entry point | Required behavior |
|---|---|
| Demo Workspace Preview section | Shows generated preview and audit summary |
| `Procurement Tender` form | Shows or links to preview and audit viewer |
| Required Forms Inspector | Links to preview section where checklist appears |
| BoQ Inspector | Links to preview section where BoQ summary appears |
| Rule Trace View | Links to validation/audit output where relevant |
| STD Template Package Viewer | Links to sample demo tender and generated preview if available |

Recommended first implementation:

Use the existing `Procurement Tender.generated_tender_pack_html` field for preview display, enhanced with a read-only audit summary section and a server method that returns preview/audit status.

The viewer shall include the following logical sections.

| Order | Section | Purpose |

| 10 | POC Warning Banner | Prevent misunderstanding of preview status |
| 20 | Preview Status Summary | Show whether preview exists and is current enough for POC review |
| 30 | Generated Preview | Display generated tender-pack HTML |
| 40 | Output Completeness Checklist | Confirm required preview sections are present |
| 50 | Validation and Blocker Summary | Show validation status and blocking messages |
| 60 | Required Forms and BoQ Summary | Show generated counts and key outputs |

The viewer may be implemented as sections on the `Procurement Tender` form or as a dedicated read-only admin page.

## 8. Required POC Warning Banner

The viewer shall display a warning near the top:

```text
POC Preview Only: This generated preview demonstrates STD-WORKS-POC rendering from a controlled template package and tender configuration. It is not a final legally certified tender document, not a published tender, and not a production PDF.
```

This warning shall be visible even if the generated preview already includes a POC warning internally.

## 9. Preview Status Summary

### 9.1 Purpose

The Preview Status Summary shall immediately tell reviewers whether preview output exists, whether generation was blocked, and whether the output is supported by validation/audit values.

### 9.2 Required Summary Items

| Summary item | Source | Required |
|---|---|---:|
| Preview Generated | `generated_tender_pack_html` present | Yes |
| Tender Status | `tender_status` | Yes |
| Validation Status | `validation_status` | Yes |
| Blocks Generation | Derived from validation messages | Yes |
| Last Generated At | `last_generated_at` | Yes |
| Last Validated At | `last_validated_at` | Yes |
| Configuration Hash Present | `configuration_hash` present | Yes |
| Package Hash Present | `package_hash` present | Yes |
| Required Forms Count | `required_forms` row count | Yes |
| BoQ Row Count | `boq_items` row count | Yes |

### 9.3 Required Status Values

Preview status shall be displayed as one of:

| Status | Meaning |
|---|---|
| `Not Generated` | No preview HTML exists |
| `Generated` | Preview HTML exists and validation does not currently block generation |
| `Blocked` | Validation blocks generation |
| `Possibly Stale` | Preview exists but validation/configuration has changed or timestamps indicate possible mismatch |
| `Unknown` | Status cannot be determined from available fields |

For the first implementation, `Possibly Stale` may be detected conservatively by comparing whether `last_generated_at` is older than `last_validated_at` or whether configuration hash is missing.

## 10. Generated Preview Section

### 10.1 Purpose

The Generated Preview section shall display the HTML tender-pack preview generated by the STD-WORKS-POC renderer.

### 10.2 Required Behavior

1. If `generated_tender_pack_html` exists, render it in a read-only preview area.
2. If preview does not exist, show a clear empty state.
3. If validation blocks generation, show blocker warning and do not imply preview is current.
4. Do not allow editing of generated preview HTML from the viewer.
5. Do not expose a final publish or PDF action.
6. Preserve internal POC warning inside the generated preview.
7. Provide a `Generate Tender-Pack Preview` action only if the user has write permission and validation prerequisites can be checked.

### 10.3 Empty State

If no preview exists, show:

```text
No tender-pack preview has been generated for this demo tender. Load a sample configuration, validate it, generate required forms and BoQ rows, then run Generate Tender-Pack Preview.
```

### 10.4 Blocked State

If validation blocks generation, show:

```text
Tender-pack preview generation is blocked by validation messages. Review the Validation and Blocker Summary before attempting generation again.
```

## 11. Output Completeness Checklist

### 11.1 Purpose

The Output Completeness Checklist shall confirm that the generated preview contains the required POC preview sections.

This is not a legal completeness check. It is a POC rendering coverage check.

### 11.2 Required Checklist Items

| Preview section | Expected present? |
|---|---:|
| POC warning | Yes |
| Tender Cover and Identity | Yes |
| Invitation to Tender summary | Yes |
| Active STD Sections | Yes |
| Locked STD Sections Notice | Yes |
| Tender Data Sheet Summary | Yes |
| Evaluation and Qualification Criteria Summary | Yes |
| Required Bidder Forms Checklist | Yes |
| Bills of Quantities Summary | Yes |
| Specifications and Works Requirements Summary | Yes |
| Drawings Summary | Yes |
| Special Conditions of Contract Summary | Yes |
| Contract Forms Checklist | Yes |
| Template and Configuration Audit Summary | Yes |

### 11.3 Detection Method

For the POC, detection may use one of:

1. structured render context metadata if available,
2. section identifiers/classes in generated HTML,
3. heading text search,
4. conservative manual display of expected sections with note that automated detection is limited.

Recommended first implementation:

Use simple heading/text detection and show a warning that this is a POC coverage check.

### 11.4 Checklist Row Shape

| Column | Meaning |
|---|---|
| Section | Expected preview section |
| Expected | Yes/No |
| Observed | Present/Missing/Unknown |
| Status | Pass/Warning/Fail |
| Evidence | Heading or marker found |

## 12. Validation and Blocker Summary

### 12.1 Purpose

This section shall show whether preview generation is safe under the current validation state.

### 12.2 Required Summary Items

| Item | Source |
|---|---|
| Validation Status | `validation_status` |
| Validation Message Count | `validation_messages` count |
| Blocker Count | Count severity `BLOCKER` |
| Error Count | Count severity `ERROR` |
| Warning Count | Count severity `WARNING` |
| Info Count | Count severity `INFO` |
| Blocks Generation | Derived from blocker/error messages |

### 12.3 Required Message Table

| Column | Required |
|---|---:|
| Severity | Yes |
| Rule Code | Yes |
| Message | Yes |
| Affected Field | Yes |
| Affected Section | Yes |
| Affected Form | Yes |
| Blocks Generation | Yes |
| Resolution Hint | Yes |

### 12.4 Required Behavior

1. If blockers or errors exist, the viewer shall clearly state that preview generation is blocked.
2. If warnings exist but no blockers/errors, the viewer shall state that preview may be generated with warnings.
3. If no messages exist and validation status is passed, the viewer shall show a pass state.
4. If validation has not been run, the viewer shall prompt the user to validate first.

### 14.3 Hash Display Rules

1. Show full hash in copyable form.
2. Optional shortened display may be used in cards.
3. Do not truncate without access to full value.

### 14.4 Audit Warning Conditions

Show warnings if:

| Condition | Warning |
|---|---|
| Package hash missing | Package reproducibility is incomplete |
| Configuration hash missing | Configuration reproducibility is incomplete |
| Validation not run | Preview may not be reliable |
| Preview generated before latest validation | Preview may be stale |
| Required forms missing | Preview may not reflect form checklist |
| BoQ rows missing while BoQ required | Preview may not reflect BoQ requirements |

## 17. Required Server Methods

The viewer may use existing methods and add summary-specific read methods.

### 17.1 `get_preview_audit_summary(tender_name)`

Purpose:

Return preview status, output completeness, validation summary, forms summary, BoQ summary, and audit metadata for a `Procurement Tender`.

Input:

| Parameter | Type | Required |
|---|---|---:|
| `tender_name` | String | Yes |

Required behavior:

1. Load `Procurement Tender`.
2. Check read permission.
3. Load linked `STD Template` summary where available.
4. Inspect `generated_tender_pack_html`.
5. Compute preview status.
6. Compute output completeness checklist.
7. Compute validation/blocker summary.
8. Compute required forms summary.
9. Compute BoQ summary.
10. Build audit metadata.
11. Return structured result.
12. Do not mutate tender or package data.

Required return shape:

```json
{
  "ok": true,
  "tender": "PT-2026-00001",
  "preview_status": {},
  "output_completeness": [],
  "validation_summary": {},
  "required_forms_summary": {},
  "boq_summary": {},
  "audit": {},
  "warnings": [],
  "links": {}
}
```

### 17.2 Existing Method to Reuse: `generate_tender_pack_preview(tender_name)`

The viewer shall reuse the existing preview generation method. It shall not duplicate rendering logic.

### 17.3 Optional Method: `get_preview_html(tender_name)`

Purpose:

Return generated preview HTML for read-only display if the UI surface cannot directly bind to `generated_tender_pack_html`.

This method is optional.

## 19. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| View preview/audit | Requires read permission on `Procurement Tender` |
| Generate preview | Requires write permission on `Procurement Tender` |
| No editing preview HTML | Viewer is read-only |
| No final publication | No publish action in this workstream |
| No PDF generation | No PDF action in this workstream |
| No package mutation | Viewer shall not edit package data |
| No external exposure | Viewer is internal/admin only |
| POC labelling | POC warning must be visible |

---

## 20. Error Handling Requirements

The viewer shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Tender not found | Clear error |
| User lacks read permission | Permission error |
| Linked template missing | Show warning and available tender data |
| Package hash missing | Show audit warning |
| Configuration hash missing | Show audit warning |
| Preview missing | Show empty state |
| Validation not run | Show prompt/warning |
| Validation blockers exist | Show blocked state |
| Generated HTML cannot be displayed | Show render/display error |
| Output completeness detection unavailable | Show `Unknown` rather than false pass |

Silent failures are not acceptable.

## 21. Required Manual Verification Scenarios

### 21.1 Valid Primary Sample Preview

Expected result:

1. Preview status is `Generated`.
2. Validation status is `Passed`.
3. Generated preview is visible.
4. POC warning is visible.
5. Output completeness checklist passes or shows expected POC warnings only.
6. Audit summary contains package hash and configuration hash.
7. Required form count is 15.
8. BoQ row count is 9.

### 21.2 Preview Missing State

Expected result:

1. Preview status is `Not Generated`.
2. Empty-state message is shown.
3. Generate preview action is available only if user has permission.

### 21.3 Blocked Negative Variant

Use one of:

1. `VARIANT-MISSING-SITE-VISIT-DATE`.
2. `VARIANT-MISSING-ALTERNATIVE-SCOPE`.

Expected result:

1. Validation status is `Blocked`.
2. Preview generation is blocked.
3. Blocker messages are visible.
4. Viewer does not imply generated preview is valid.

### 21.4 Stale Preview Warning

If practical to simulate:

1. Generate preview.
2. Change configuration or re-run validation after generation.

Expected result:

1. Viewer shows `Possibly Stale` or warning.
2. User is prompted to regenerate preview.

### 21.5 Missing Audit Hash Warning

If practical to simulate in development:

1. Remove or clear configuration hash on a test tender.

Expected result:

1. Viewer shows audit warning.
2. It does not claim full reproducibility.

The following prompt shall be used for this step only.

```text
Implement Admin Console Step 7 only: Preview and Audit Viewer for the STD Administration Console POC.

Context:
- This is a Frappe/ERPNext custom app.

- Show output completeness checklist.
- Show validation/blocker summary.

- Show audit and reproducibility summary with full copyable hashes.
```

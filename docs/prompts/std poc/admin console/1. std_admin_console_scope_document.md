## 2. Background

The STD-WORKS-POC has successfully demonstrated that a Standard Tender Document can be represented as a developer-controlled, metadata-driven template package and used to drive a tender-creation process.

The POC proved that the following architectural direction is viable:

1. Standard Tender Documents can be stored as controlled template packages.
2. Tender configuration can be driven by structured field metadata.
3. Safety checks can be declared as rules.
4. Required bidder and contract forms can be derived from template metadata and tender configuration.
5. Representative BoQ data can be treated as structured child data.
6. A tender-pack preview can be generated from the package and configuration.
7. Template code, package hash, configuration hash, and validation status can support auditability and reproducibility.

The POC also exposed an important next-stage need: although the engine works, the results are not yet sufficiently visible, explainable, or inspectable through the user interface.

Before expanding into procurement officer workflows, the system requires a minimal administration interface that allows technical administrators, procurement SMEs, legal/compliance reviewers, and implementation reviewers to inspect and understand the template package, rules, forms, sample tenders, validation outcomes, and generated preview outputs.

---

## 3. Strategic Decision

The next workstream shall not be procurement officer tender creation.

The next workstream shall be a minimal **STD Administration Console POC**.

The purpose of this workstream is to make the existing package-driven STD architecture visible, inspectable, explainable, and demonstrable through controlled Frappe administration screens and actions.

This decision is based on the following findings from STD-WORKS-POC:

| Finding | Implication |
|---|---|
| The package-driven approach works | It should be extended, not replaced |
| A broad no-code template editor would be risky | Template internals should remain developer-controlled for now |
| The POC is technically sound but hard to surface | Administration and observability are now the bottleneck |
| Procurement officer workflows depend on trust in the template engine | Admin visibility should precede officer-facing workflow expansion |
| Reviewers need to understand rules, forms, validation, and audit outputs | The next UI should expose these transparently |

---

## 4. Purpose of the Admin Console POC

The STD Administration Console POC shall provide a minimal interface for inspecting, validating, demonstrating, and explaining imported STD template packages.

It shall answer the following questions:

1. What STD templates have been imported into the system?
2. What version, source, status, and package hash does each template have?
3. What sections, fields, rules, forms, sample tenders, and render mappings exist inside the package?
4. Are the package files internally consistent?
5. Which rules apply to a sample tender configuration?
6. Which fields are required, active, missing, or invalid?
7. Which forms are required and why?
8. Which BoQ rows are generated and how are they categorized?
9. What tender-pack preview is generated?
10. What audit values prove reproducibility?
11. What known limitations remain before production use?

---

## 5. Core Product Principle

The Admin Console shall implement the principle:

> The STD template package remains controlled code; the administration interface exposes, validates, and demonstrates it, but does not turn ordinary users into legal template authors.

This means:

1. The console shall be inspect-first.
2. The console shall be action-limited.
3. The console shall expose package internals read-only.
4. The console shall not permit ordinary users to edit legal template text.
5. The console shall not permit ordinary users to edit package JSON directly.
6. The console shall make rule and form activation traceable.
7. The console shall improve understanding without weakening governance.

---

## 6. Scope Statement

The STD Administration Console POC shall provide a minimal but complete administration and observability layer over the existing STD-WORKS-POC package and engine.

The workstream shall include:

1. A dashboard or list view for imported STD templates.
2. A read-only template package viewer.
3. A package validation panel.
4. A rule trace viewer.
5. A required forms inspector.
6. A sample BoQ inspector.
7. A sample tender demonstration workspace.
8. A tender-pack preview and audit viewer.
9. Minimal admin actions for re-importing, validating, and demonstrating the package.
10. A smoke path proving the administration console works end-to-end.

The workstream shall not introduce full template editing, procurement officer workflows, bidder workflows, or production governance workflows.

### 7.2 Read-Only Template Package Viewer

The console shall expose package contents in a structured read-only format.

Required tabs or sections:

| Area | Required content |
|---|---|
| Overview | Manifest, source, authority, version, package status |
| Sections | Section codes, titles, mutability, render inclusion, POC treatment |
| Fields | Field groups, field codes, labels, types, defaults, validation metadata |
| Rules | Rule codes, labels, types, severity, conditions, effects |
| Forms | Form codes, titles, default required flag, activation rules, evidence policy |
| Sample Tender | Sample configuration, lots, BoQ, expected forms, scenario variants |
| Render Map | Render targets and package rendering metadata |
| Raw JSON | Read-only developer fallback view |

The viewer shall not permit editing of package internals.

---

### 7.3 Package Validation Panel

The console shall provide a validation panel that checks package consistency.

Minimum validation checks:

| Check | Purpose |
|---|---|
| Required files/components exist | Confirm package completeness |
| JSON/package payload parses | Confirm readable package data |
| Template code consistency | Confirm all components reference same template |
| Section references valid | Confirm rules/render references point to defined sections |
| Field references valid | Confirm rules/sample configuration point to defined fields |
| Form references valid | Confirm rules and expected outputs point to defined forms |
| Rule operators supported | Confirm rule engine can interpret rules |
| Rule severities supported | Confirm messages can be classified |
| Sample tender fields valid | Confirm sample configuration does not invent fields |
| Expected activated forms valid | Confirm sample expected forms exist |
| BoQ categories valid | Confirm sample BoQ rows use supported categories |
| Package hash available | Confirm audit integrity value exists |

Validation output shall be shown in a structured table with severity, check code, result, message, and affected reference.

---

### 7.4 Rule Trace Viewer

The console shall expose rule behavior for a selected sample tender or tender configuration.

Required columns:

| Column | Purpose |
|---|---|
| Rule Code | Stable rule identifier |
| Rule Label | Human-readable rule meaning |
| Rule Type | Required field, activate form, validate date, etc. |
| Enabled | Whether rule is enabled |
| Applies | Whether rule applies to selected configuration |
| Severity | INFO, WARNING, ERROR, BLOCKER |
| Trigger Summary | Summary of conditions |
| Effects Summary | Required fields/forms/sections/validations |
| Messages | Validation messages produced |
| Blocks Generation | Whether rule blocks tender-pack generation |

The rule trace viewer is mandatory because it makes the rule engine explainable to technical and non-technical reviewers.

Required columns:

| Column | Purpose |

## 10. Minimal Role and Permission Position

The POC shall use minimal guardrails.

Recommended position:

| Area | POC requirement |
|---|---|
| Package viewing | Restricted to internal administrative/reviewer users |
| Package import/re-import | Restricted to System Manager or technical administrator |
| Package JSON editing | Not allowed through console |
| Sample demo actions | Restricted to internal users with POC admin rights |
| Tender preview generation | Restricted to internal users for POC |
| Raw JSON display | Read-only and preferably limited to technical users |

Full production role/permission design is out of scope, but the console shall not expose dangerous write actions to all users.

## 13. Candidate Implementation Approach

The implementation may use one or more of the following Frappe constructs:

| Construct | Use |
|---|---|
| Enhanced `STD Template` form | Show package summary, validation, and actions |
| Custom Frappe Page | Provide richer Admin Console dashboard and tabs |
| Server methods | Run package validation, sample demo, rule trace |
| Child tables or HTML fields | Display validation results and summaries |
| Read-only HTML renderers | Display package components and trace output |
| Existing `Procurement Tender` POC actions | Reuse sample/demo generation |

Recommended approach for speed:

1. Start by enhancing the `STD Template` form with read-only summary sections and actions.
2. Add server methods that return package summary, validation results, and rule traces.
3. Add a dedicated custom page only if the form becomes too constrained.

This avoids prematurely overbuilding a complex administration module.

## 17. Package Validation Requirements

The package validation feature shall output one row per validation check.

Required result shape:

```json
{
  "ok": true,
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC",
  "package_hash": "...",
  "summary": {
    "total_checks": 0,
    "passed": 0,
    "warnings": 0,
    "errors": 0,
    "blockers": 0
  },
  "checks": []
}
```

Each check row shall have:

```json
{
  "check_code": "PKG_FIELD_REFERENCES_VALID",
  "label": "Field references are valid",
  "severity": "ERROR",
  "status": "PASSED",
  "message": "All rule field references exist in fields.json.",
  "reference": "rules.json"
}
```

Supported statuses:

1. `PASSED`
2. `WARNING`
3. `FAILED`
4. `SKIPPED`

Supported severities:

1. `INFO`
2. `WARNING`
3. `ERROR`
4. `BLOCKER`

## 18. Sample Demo Requirements

The console shall support running or opening a sample demo tender.

Minimum demo flow:

1. Select or open STD Template.
2. Click `Create Sample Demo Tender` or equivalent.
3. System creates or opens a `Procurement Tender` linked to the template.
4. System loads primary sample tender configuration.
5. System validates configuration.
6. System generates required forms.
7. System generates sample BoQ if needed.
8. System generates preview if validation passes.
9. System shows summary results.

The demo flow may be implemented as separate actions rather than one all-in-one action, but the console must make the sequence easy to follow.

## 21. Risks and Controls

| Risk | Impact | Control |
|---|---|---|
| Console drifts into template editor | Reintroduces rejected complex route | Keep package internals read-only |
| Users misunderstand POC as production-ready | Legal/compliance risk | Clear POC labels and limitations |
| Raw JSON remains the only useful view | Poor usability | Provide structured summary views |
| Rule engine remains opaque | Low trust from reviewers | Implement rule trace viewer |
| Required forms lack explanation | Weak support for upload-elimination strategy | Show activation source and evidence policy |
| Package validation is incomplete | Hidden defects in template package | Provide explicit validation checks and statuses |
| Admin actions bypass permissions | Security risk | Server-side permission checks |
| Workstream expands into procurement workflows too early | Scope loss | Defer officer workflows until admin console accepted |

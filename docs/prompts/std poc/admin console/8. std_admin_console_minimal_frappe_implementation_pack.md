## 2. Purpose

The purpose of this implementation pack is to give Cursor and the developer a precise, bounded implementation plan for the STD Administration Console POC.

This pack consolidates Admin Console Steps 1 through 7 into an implementation-ready sequence using the existing STD-WORKS-POC foundation.

The implementation shall make the package-driven STD model visible, inspectable, explainable, and demonstrable through minimal Frappe administration UI and server methods.

This pack shall not expand the product into template editing, procurement officer workflows, bidder workflows, PDF generation, evaluation, award, or contract execution.

---

## 3. Implementation Principle

The implementation shall follow this principle:

> Add an administration and observability layer over the existing STD engine; do not duplicate the engine and do not convert the package model into a UI-authored template system.

This means:

1. reuse existing `STD Template` and `Procurement Tender` DocTypes,
2. reuse existing package loader and template engine services,
3. expose read-only package summaries,
4. expose validation and rule trace outputs,
5. expose demo tender status and generated artefacts,
6. keep package internals read-only,
7. add only minimal fields if absolutely necessary,
8. avoid new workflows and production scope.

---

## 4. Required Implementation Outcome

At the end of this step, an internal reviewer shall be able to:

1. open an imported `STD Template`,
2. inspect package overview, sections, fields, rules, forms, sample tender, render map, and raw JSON fallback,
3. run package validation,
4. run rule trace for primary sample and variants,
5. create or open a demo `Procurement Tender`,
6. load sample tender or variant,
7. validate configuration,
8. generate required forms,
9. generate sample BoQ,
10. generate tender-pack preview,
11. inspect required forms and BoQ outputs,
12. inspect preview and audit metadata,
13. capture evidence without reading backend code or raw package files.

## 13. Server Method Behavioral Requirements

### 13.1 General Requirements

Every server method shall:

1. check permissions,
2. validate inputs,
3. fail clearly,
4. return structured output,
5. avoid mutating data unless the method is explicitly an action method,
6. avoid arbitrary file paths,
7. avoid editing package internals,
8. reuse existing loader/engine functions.

### 13.2 Read-Only Methods

The following must not write to database:

1. `get_template_package_summary`
2. `get_template_package_component`
3. `validate_std_package`, unless explicitly persisting latest summary is approved
4. `trace_std_rules_for_sample`
5. `trace_std_rules_for_tender`
6. `get_required_forms_inspection`
7. `get_boq_inspection`
8. `get_preview_audit_summary`
9. `get_preview_html`

### 13.3 Write Methods

The following may write data:

1. `reimport_std_template_package`
2. `create_or_open_std_demo_tender`
3. `run_std_demo_path`, if implemented
4. existing tender generation actions

Write methods must check permissions and return clear action results.

## 14. Package Summary Implementation Requirements

`get_template_package_summary(template_name)` shall return:

| Key | Required content |
|---|---|
| `template` | STD Template fields and audit metadata |
| `overview` | Manifest identity and source metadata |
| `structure` | Component presence and counts |
| `sections` | Section summary rows |
| `fields` | Field summary rows |
| `rules` | Rule summary rows |
| `forms` | Form summary rows |
| `sample_tender` | Sample summary, lots, BoQ, variants, expected forms |
| `render_map` | Render summary or warning |
| `raw_components` | Component names and availability, not necessarily full JSON |
| `warnings` | Package summary warnings |

The method shall not return massive raw JSON by default unless explicitly requested through `get_template_package_component`.

## 15. Package Validation Implementation Requirements

`validate_std_package(template_name)` shall implement the checks specified in Admin Console Step 4.

Minimum required validation categories:

1. package structure,
2. template identity,
3. sections,
4. fields,
5. rules,
6. forms,
7. sample tender,
8. render map,
9. audit.

The method shall return:

1. `overall_status`,
2. summary counts,
3. check rows,
4. warnings.

Unknown rule operators, unknown referenced fields, unknown forms, and missing core components shall be blocker-level issues.

## 16. Rule Trace Implementation Requirements

The implementation shall add an explainability helper, preferably in `std_template_engine.py` or `std_admin_console.py`.

Recommended helper:

```python
def trace_rules(template, config, lots=None, boq_items=None) -> dict:
    ...
```

Required behavior:

1. Evaluate every enabled and disabled rule.
2. Show whether the rule applies.
3. Show condition-level actual/expected values.
4. Show effect summaries.
5. Show produced validation messages.
6. Show affected fields, forms, and sections.
7. Show whether generation is blocked.
8. Produce configuration hash.
9. Not write to database.

The trace result must be consistent with existing `validate_config()` results.

## 17. Demo Tender Implementation Requirements

`create_or_open_std_demo_tender(template_name, variant_code=None)` shall:

1. load `STD Template`,
2. check permission,
3. create a `Procurement Tender` linked to the template or open an existing demo tender according to implementation policy,
4. load primary sample or selected variant,
5. save tender,
6. return tender name and route/link.

Recommended first implementation:

Create a new demo tender each time to avoid accidental overwrite of prior evidence.

Set `poc_notes` to clearly state:

```text
Created by STD Administration Console POC demo action. Not a production tender.
```

## 18. Required Forms Inspector Implementation Requirements

`get_required_forms_inspection(tender_name)` shall:

1. load tender,
2. check read permission,
3. read `required_forms`,
4. load linked package forms metadata where possible,
5. compute counts,
6. detect duplicates,
7. build readable activation explanations,
8. build expected/observed comparison where sample context is derivable,
9. return warnings for missing metadata.

It shall not generate forms. Generation remains the responsibility of existing action methods.

---

## 19. BoQ Inspector Implementation Requirements

`get_boq_inspection(tender_name)` shall:

1. load tender,
2. check read permission,
3. read `boq_items`,
4. read `lots`,
5. compute category coverage,
6. detect duplicate item codes,
7. check lot references,
8. classify row types,
9. return warnings for missing rows or invalid references.

It shall not generate or edit BoQ rows.

---

## 20. Preview/Audit Implementation Requirements

`get_preview_audit_summary(tender_name)` shall:

1. load tender,

3. inspect preview HTML,

5. compute output completeness checklist,
6. summarize validation messages,
7. summarize required forms,
8. summarize BoQ,
9. build audit metadata,
10. return warnings for missing hashes or stale preview indicators.

It shall not generate preview. Generation remains the responsibility of existing preview method.

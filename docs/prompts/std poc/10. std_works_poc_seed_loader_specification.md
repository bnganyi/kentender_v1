## 2. Purpose

The purpose of this step is to define the template seed loader that imports the developer-controlled STD-WORKS-POC JSON package into Frappe as an `STD Template` record.

The seed loader is the bridge between the repository package files and the runtime Frappe database. It shall read the package files, validate their presence and internal consistency, compute a deterministic package hash, and create or update the corresponding `STD Template` record.

This step shall not implement tender initialization, rule evaluation, required-form generation, BoQ generation, tender-pack rendering, or Desk buttons. Those belong to later steps.

---

## 3. Design Principle

The seed loader shall implement the principle:

> The STD template package is maintained as version-controlled code and imported into Frappe as a controlled runtime asset.

The loader shall be deterministic, repeatable, and safe to run multiple times during development.

The loader shall use `template_code` as the stable upsert key. If a matching `STD Template` exists, the loader updates it. If no matching record exists, the loader creates it.

The loader shall not let ordinary procurement officers edit the underlying package. It simply makes the approved or draft package available to the application runtime.

## 5. Loader Responsibilities

The loader shall perform the following responsibilities:

1. Locate the STD-WORKS-POC template package folder.
2. Confirm all required files exist.
3. Parse all required JSON files.
4. Validate template code consistency across package files.
5. Validate that required top-level keys exist in each package file.
6. Validate that `sample_tender.json` references the same template code as `manifest.json`.
7. Validate that `sections.json`, `fields.json`, `rules.json`, and `forms.json` reference the same template code as `manifest.json`.
8. Compute a deterministic package hash from the required runtime JSON files.
9. Build a combined package JSON payload.
10. Create or update the `STD Template` record.
11. Populate key metadata fields on `STD Template` from `manifest.json`.
12. Store the full package JSON payload on `STD Template.package_json`.
13. Store the manifest JSON on `STD Template.manifest_json`.
14. Store package hash on `STD Template.package_hash`.
15. Store import timestamp and importing user where possible.
16. Return a structured result describing success, update/create action, template code, and package hash.

## 6. Required Package Files

The loader shall require the following files:

| File | Required | Included in package hash |
|---|---:|---:|
| `manifest.json` | Yes | Yes |
| `sections.json` | Yes | Yes |
| `fields.json` | Yes | Yes |

| `README.md` | Yes | No |

`README.md` must exist but shall not be included in the package hash.

## 7. Package Hash Policy

The package hash shall be computed using SHA-256.

The hash input shall include the raw UTF-8 bytes of the following files in the exact order listed below:

1. `manifest.json`
2. `sections.json`
3. `fields.json`
4. `rules.json`
5. `forms.json`
6. `render_map.json`
7. `sample_tender.json`

Between files, the loader shall insert a deterministic separator to avoid accidental concatenation ambiguity.

Required separator:

```text
\n---FILE:{file_name}---\n
```

The loader shall not include `README.md` in the package hash.

For the POC, raw-byte hashing is acceptable. Later production implementation may normalize JSON before hashing if required.

  "render_map": {},

## 10. Status Mapping

The manifest stores machine-oriented package status. The `STD Template` DocType stores user-facing status values.

Required mapping:

| Manifest status | `STD Template.status` |
|---|---|
| `DRAFT_PACKAGE` | `Draft Package` |
| `IMPORTED` | `Imported` |
| `POC_APPROVED` | `POC Approved` |
| `SUSPENDED` | `Suspended` |
| `SUPERSEDED` | `Superseded` |
| `RETIRED` | `Retired` |

If an unknown manifest status is encountered, the loader shall fail with a clear error message.

## 11. Required Functions

The loader module shall expose the following functions.

### 11.1 `get_template_package_path()`

Purpose:

Returns the absolute path to the STD-WORKS-POC template package folder.

Expected behavior:

1. Uses the module file path to resolve the app-local package path.
2. Does not rely on the current working directory.
3. Returns a `pathlib.Path` object.

---

### 11.2 `load_json_file(path)`

Purpose:

Loads and parses one JSON file.

Expected behavior:

1. Opens the file in UTF-8.
2. Parses JSON.
3. Raises a clear exception if parsing fails.
4. Returns the parsed object.

---

### 11.3 `validate_required_files(package_path)`

Purpose:

Confirms required package files exist.

Expected behavior:

1. Checks all required files from Section 6.
2. Raises a clear exception listing missing files if any are absent.
3. Returns true or the list of resolved file paths.

---

### 11.4 `load_template_package(package_path=None)`

Purpose:

Loads all required package files into memory.

Expected behavior:

1. Resolves the package path.
2. Validates required files.
3. Parses all JSON files.
4. Returns a dictionary with keys:
   - `manifest`
   - `sections`
   - `fields`
   - `rules`
   - `forms`
   - `render_map`
   - `sample_tender`

---

### 11.5 `validate_template_package(package)`

Purpose:

Validates package-level consistency.

Expected behavior:

1. Confirms manifest has `template_code`.
2. Confirms all package JSON files with `template_code` match the manifest template code.
3. Confirms `sample_tender.template_code` matches the manifest template code.
4. Confirms required top-level keys exist in each package file.
5. Confirms required files listed in manifest are present in the package model.
6. Returns true if validation passes.
7. Raises a clear exception if validation fails.

---

### 11.6 `compute_package_hash(package_path=None)`

Purpose:

Computes deterministic SHA-256 package hash.

Expected behavior:

1. Uses the exact file order defined in Section 7.
2. Includes file separators.
3. Excludes `README.md`.
4. Returns a lowercase hex digest string.

---

### 11.7 `build_combined_package_payload(package, package_hash, package_path)`

Purpose:

Builds the combined package JSON object stored on `STD Template`.

Expected behavior:

1. Includes all parsed package components.
2. Includes package hash.
3. Includes import metadata.
4. Returns a Python dictionary.

---

### 11.8 `map_manifest_status_to_template_status(manifest_status)`

Purpose:

Maps manifest status to Frappe `STD Template.status` value.

Expected behavior:

1. Uses mapping from Section 10.
2. Raises a clear error for unknown status.

---

### 11.9 `upsert_std_template(package=None, package_path=None)`

Purpose:

Creates or updates the `STD Template` record.

Expected behavior:

1. Loads package if not supplied.
2. Validates package.
3. Computes package hash.
4. Builds combined package payload.
5. Finds existing `STD Template` by `template_code`.
6. Creates a new record if none exists.
7. Updates existing record if found.
8. Saves the record.
9. Commits transaction only where appropriate for CLI/manual execution.
10. Returns a structured result.

Required return shape:

```json
{
  "ok": true,
  "action": "created_or_updated",
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC",
  "std_template_name": "...",
  "package_hash": "...",
  "status": "Draft Package"
}
```

---

### 11.10 `import_std_works_poc_template()`

Purpose:

Convenience wrapper for importing the STD-WORKS-POC package.

Expected behavior:

1. Calls `upsert_std_template()` with the default package path.
2. Is safe to expose as a whitelisted method later if needed.
3. Returns the structured result.

For Step 10, this function may be implemented without whitelisting.

## 12. Error Handling Requirements

The seed loader shall fail clearly and early.

Required error scenarios:

| Scenario | Required behavior |
|---|---|
| Package folder missing | Raise clear error with expected path |
| Required file missing | Raise clear error listing missing files |
| Invalid JSON | Raise clear error identifying file and parse issue |
| Missing `template_code` | Raise clear error |
| Template code mismatch | Raise clear error identifying offending file |
| Unknown manifest status | Raise clear error |
| `STD Template` DocType missing | Raise clear Frappe error indicating Step 9 must be completed |
| Frappe save failure | Allow Frappe exception to surface with context |

Do not silently ignore errors.

## 17. Minimal Implementation Skeleton

The following implementation shape is required. The exact code may vary, but the public functions and behavior must remain consistent.

```python
from __future__ import annotations

import hashlib
import json
from pathlib import Path
from typing import Any

import frappe
from frappe.utils import now_datetime

PACKAGE_FOLDER_NAME = "ke_ppra_works_building_2022_04_poc"
REQUIRED_JSON_FILES = [
    "manifest.json",
    "sections.json",
    "fields.json",
    "rules.json",
    "forms.json",
    "render_map.json",
    "sample_tender.json",
]
REQUIRED_NON_HASH_FILES = ["README.md"]
TEMPLATE_CODE = "KE-PPRA-WORKS-BLDG-2022-04-POC"


def get_template_package_path() -> Path:
    # Resolve relative to this file: {app_name}/procurement/std_template_loader.py
    return Path(__file__).resolve().parent / "std_templates" / PACKAGE_FOLDER_NAME


def load_json_file(path: Path) -> dict[str, Any]:
    try:
        with path.open("r", encoding="utf-8") as handle:
            return json.load(handle)
    except json.JSONDecodeError as exc:
        raise ValueError(f"Invalid JSON in {path}: {exc}") from exc


def validate_required_files(package_path: Path) -> None:
    expected = REQUIRED_JSON_FILES + REQUIRED_NON_HASH_FILES
    missing = [file_name for file_name in expected if not (package_path / file_name).exists()]
    if missing:
        raise FileNotFoundError(
            f"STD template package is missing required files at {package_path}: {', '.join(missing)}"
        )


def load_template_package(package_path: Path | None = None) -> dict[str, Any]:
    package_path = package_path or get_template_package_path()
    validate_required_files(package_path)
    return {
        "manifest": load_json_file(package_path / "manifest.json"),
        "sections": load_json_file(package_path / "sections.json"),
        "fields": load_json_file(package_path / "fields.json"),
        "rules": load_json_file(package_path / "rules.json"),
        "forms": load_json_file(package_path / "forms.json"),
        "render_map": load_json_file(package_path / "render_map.json"),
        "sample_tender": load_json_file(package_path / "sample_tender.json"),
    }
```

The full implementation shall complete the remaining functions defined in Section 11.

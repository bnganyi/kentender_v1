## 2. Purpose

The purpose of this step is to expose the Step 11 template engine through controlled server methods and Frappe Desk form actions on `Procurement Tender`.

This step shall allow a developer or procurement officer using the POC Desk form to:

1. load default configuration from the selected STD template,
2. load the sample Works tender configuration,
3. validate tender configuration,
4. generate the required-forms checklist,
5. generate sample BoQ rows,
6. prepare render context for later rendering,
7. store validation messages and required forms on the tender document.

This step shall not create final Jinja tender-pack templates or PDF output. Rendering belongs to Step 13 and later.

---

## 3. Design Principle

The form actions shall be thin wrappers over the template engine.

The Desk/UI layer shall not duplicate business logic already defined in `std_template_engine.py`. Each button shall call a small server-side function that:

1. loads the `Procurement Tender` document,
2. calls the appropriate engine service,
3. writes results back to the document,
4. saves the document,
5. returns a clear user-facing result.

The procurement officer shall not edit package internals. All actions operate only on the tender instance and its controlled configuration/child tables.

## 6. Required Server Methods

The following server methods shall be implemented as whitelisted methods or DocType controller methods callable from the client script.

### 6.1 `load_template_defaults(tender_name)`

Purpose:

Initialize tender configuration from the selected STD template defaults.

Input:

| Parameter | Type | Required | Description |
|---|---|---:|---|
| `tender_name` | String | Yes | Name of the `Procurement Tender` document |

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` is selected.
3. Load linked template through the Step 11 engine.
4. Call `initialize_config(template)`.
5. Apply configuration to tender using `apply_config_to_tender_doc()`.
6. Set `tender_status` to `Configured`.
7. Set `validation_status` to `Not Validated`.
8. Clear existing validation messages.
9. Save document.
10. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "message": "Template defaults loaded.",
  "tender": "...",
  "template_code": "KE-PPRA-WORKS-BLDG-2022-04-POC"
}
```

---

### 6.2 `load_sample_tender(tender_name)`

Purpose:

Populate a tender document with the primary sample Works tender configuration, lots, and BoQ rows.

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` is selected.
3. Call `populate_sample_tender(tender_doc)` from the Step 11 engine.
4. Set `tender_status` to `Configured`.
5. Set `validation_status` to `Not Validated`.
6. Clear required forms and validation messages.
7. Save document.
8. Return structured result.

---

### 6.3 `load_sample_variant(tender_name, variant_code)`

Purpose:

Populate a tender document with a sample scenario variant from `sample_tender.json`.

Input:

| Parameter | Type | Required | Description |
|---|---|---:|---|
| `tender_name` | String | Yes | Name of the `Procurement Tender` document |
| `variant_code` | String | Yes | Variant code from `sample_tender.json` |

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` is selected.
3. Call `populate_sample_tender(tender_doc, variant_code=variant_code)`.
4. Set `tender_status` to `Configured`.
5. Set `validation_status` to `Not Validated`.
6. Clear required forms and validation messages.
7. Save document.
8. Return structured result.

Required variant codes from Step 8:

1. `VARIANT-INTERNATIONAL`
2. `VARIANT-TENDER-SECURING-DECLARATION`
3. `VARIANT-RESERVED-TENDER`
4. `VARIANT-MISSING-SITE-VISIT-DATE`
5. `VARIANT-MISSING-ALTERNATIVE-SCOPE`
6. `VARIANT-SINGLE-LOT`
7. `VARIANT-RETENTION-MONEY-SECURITY`

---

### 6.4 `validate_tender_configuration(tender_name)`

Purpose:

Evaluate tender configuration against `rules.json` and store validation messages.

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` is selected.
3. Confirm `configuration_json` exists.
4. Parse `configuration_json`.
5. Convert child table rows to dictionaries.
6. Call `validate_config(template, config, lots, boq_items)`.
7. Call `write_validation_to_tender(tender_doc, validation_result)`.
8. Save document.
9. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "status": "Passed",
  "blocks_generation": false,
  "message_count": 0,
  "configuration_hash": "..."
}
```

---

### 6.5 `generate_required_forms(tender_name)`

Purpose:

Resolve and populate the required bidder/contract forms checklist.

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` and `configuration_json` exist.
3. Parse configuration.
4. Run validation or reuse existing validation result.
5. Call `resolve_required_forms(template, config, validation_result)`.
6. Call `write_required_forms_to_tender(tender_doc, required_forms)`.
7. Save document.
8. Return structured result.

Important behavior:

- This method may run validation internally to ensure active forms are current.
- It shall not block form generation merely because validation has blockers; however, the result shall include whether the configuration blocks tender-pack generation.
- Required forms generated under invalid configuration shall be treated as provisional.

---

### 6.6 `generate_sample_boq(tender_name)`

Purpose:

Populate sample BoQ rows from `sample_tender.json`.

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm linked `std_template` exists.
3. Load sample BoQ rows through `load_sample_boq_rows(template)`.
4. Clear existing `boq_items` rows.
5. Append sample BoQ rows.
6. Save document.
7. Return structured result.

This action is for POC demonstration only. Production BoQ authoring will require a separate module design.

---

### 6.7 `prepare_render_context(tender_name)`

Purpose:

Build render context for later Step 13 rendering and store a JSON placeholder or audit preview.

Expected behavior:

1. Load the `Procurement Tender` document.
2. Confirm `std_template` and `configuration_json` exist.
3. Parse configuration.
4. Convert child table rows to dictionaries.
5. Validate configuration.
6. Resolve required forms.
7. Build render context using `build_render_context()`.
8. Store pretty-printed JSON render context in `generated_tender_pack_html` temporarily, or in a dedicated internal variable if a render-context field exists.
9. Set `last_generated_at`.
10. Save document.
11. Return structured result.

Important:

This is not final tender-pack rendering. It is a bridge action before Step 13. If `generated_tender_pack_html` is used temporarily, it must clearly label the content as render-context JSON, not final HTML.

## 8. Button Safety Rules

All client-side buttons shall enforce these rules:

1. If the document is unsaved, prompt the user to save first.
2. If `std_template` is missing, show a clear error and do not call the server.
3. Do not expose buttons on submitted/cancelled documents if those states are added later.
4. Do not allow editing of `STD Template.package_json` from the `Procurement Tender` form.
5. Do not perform heavy logic client-side.
6. Server methods remain the source of truth.

---

## 9. Permission and Role Requirements

For the POC, methods may be available to users who can write `Procurement Tender` documents.

Each server method shall:

1. load the target document,
2. check write permission using Frappe where practical,
3. fail clearly if the user lacks permission.

Recommended check:

```python
tender_doc.check_permission("write")
```

Production permission hardening is outside Step 12, but methods shall not bypass Frappe document permissions unnecessarily.

| Scenario | Required behavior |
|---|---|

| Tender not found | Raise clear error |

| Missing `std_template` | Raise clear user-facing error |

| Invalid JSON in `configuration_json` | Raise clear error |
| Template package cannot be loaded | Raise clear error |
| Unknown sample variant | Raise clear error |
| Engine validation error | Raise or return clear error |

Client script shall show server errors through normal Frappe error display.

## 12. Status Update Rules

Server methods shall update tender status and validation status as follows:

| Method | Tender Status | Validation Status |
|---|---|---|
| Load Template Defaults | `Configured` | `Not Validated` |
| Load Sample Tender | `Configured` | `Not Validated` |
| Load Sample Variant | `Configured` | `Not Validated` |
| Validate Configuration with Passed | `Validated` | `Passed` |
| Validate Configuration with Passed With Warnings | `Validated` | `Passed With Warnings` |
| Validate Configuration with Failed | `Validation Failed` | `Failed` |
| Validate Configuration with Blocked | `Validation Failed` | `Blocked` |
| Generate Required Forms | No change unless currently Draft | No change unless validation is run internally |
| Generate Sample BoQ | No change unless currently Draft | No change |
| Prepare Render Context without blockers | `Tender Pack Generated` | latest validation status |
| Prepare Render Context with blockers | No generation; remain `Validation Failed` | `Blocked` |

For Step 12, `Prepare Render Context` may still store render-context JSON only if blockers are absent. If blockers exist, it shall refuse and return the blocker status.

## 15. Required Server Method Skeleton

The exact code may vary, but the controller shall follow this structure:

```python
import json

import frappe
from frappe.utils import now_datetime

from {app_name}.procurement import std_template_engine as engine


@frappe.whitelist()
def load_template_defaults(tender_name: str) -> dict:
    tender = _get_tender_doc(tender_name)
    if not tender.std_template:
        frappe.throw("Select an STD Template before loading defaults.")
    template = engine.load_template(tender.std_template)
    config = engine.initialize_config(template)
    engine.apply_config_to_tender_doc(tender, config)
    tender.tender_status = "Configured"
    tender.validation_status = "Not Validated"
    tender.set("validation_messages", [])
    tender.save()
    return {
        "ok": True,
        "message": "Template defaults loaded.",
        "tender": tender.name,
        "template_code": template.get("template_code")
    }
```

The implementation shall complete all required methods listed in Section 6.

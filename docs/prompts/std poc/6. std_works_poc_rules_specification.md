## 2. Purpose

The purpose of this step is to define the full content of the `rules.json` file for the PPRA Works Standard Tender Document POC template package.

`rules.json` shall define the validation, conditional requirement, activation, blocking, warning, derivation, and scenario-control rules used by the template engine when a procurement officer configures a Works tender.

This file shall not contain executable Python code. It shall declare rules in a predictable JSON structure that the future template engine can interpret.

---

## 3. Design Principle

The `rules.json` file shall implement the principle:

> The system must make safe configuration easy and unsafe configuration difficult or impossible.

The rules shall ensure that procurement officers cannot generate a tender-pack preview when critical configuration is incomplete, contradictory, or inconsistent with the intended POC model.

The POC rule model shall be deliberately explicit. It shall not attempt to be a universal rule engine for all future STD types. It shall be sufficiently structured for Cursor and the Frappe implementation to build a simple deterministic evaluator.

## 7. Rule Types

The POC shall define the following rule types:

| Rule type | Meaning |
|---|---|
| `REQUIRE_FIELD` | Requires a field to have a value when a condition is met |
| `REQUIRE_CHILD_ROWS` | Requires child-table rows, such as lots or BoQ rows, when a condition is met |
| `ACTIVATE_FIELD` | Indicates that a field becomes active/relevant when a condition is met |
| `ACTIVATE_SECTION` | Indicates that a section becomes active/relevant when a condition is met |
| `ACTIVATE_FORM` | Indicates that a bidder or contract form becomes required when a condition is met |
| `DERIVE_FIELD` | Derives a field value from another field or condition |
| `VALIDATE_DATE_ORDER` | Validates chronological ordering between date or datetime fields |
| `VALIDATE_NUMERIC_LIMIT` | Validates numeric or monetary values against a limit |
| `VALIDATE_ALLOWED_COMBINATION` | Validates that a combination of values is permitted |
| `WARNING_ONLY` | Produces a warning without blocking generation |
| `INFO_ONLY` | Produces informational output without blocking generation |

---

## 8. Message Severities

The POC shall define the following message severities:

| Severity | Meaning | Blocks tender-pack generation? |
|---|---|---:|
| `INFO` | Advisory information | No |
| `WARNING` | Potential issue requiring review | No |
| `ERROR` | Configuration error that should be corrected | Yes for POC rendering |
| `BLOCKER` | Critical issue that must be corrected | Yes |

For the POC, both `ERROR` and `BLOCKER` shall prevent tender-pack generation.

---

## 9. Operator Definitions

The POC rule evaluator shall support a small set of operators.

| Operator | Meaning |
|---|---|
| `EQUALS` | Field value equals expected value |
| `NOT_EQUALS` | Field value does not equal expected value |
| `IS_TRUE` | Boolean field is true |
| `IS_FALSE` | Boolean field is false |
| `IS_EMPTY` | Field value is null, empty string, or empty list |
| `IS_NOT_EMPTY` | Field value exists and is not empty |
| `GREATER_THAN` | Numeric/date value is greater than comparison value |
| `GREATER_THAN_OR_EQUALS` | Numeric/date value is greater than or equal to comparison value |
| `LESS_THAN` | Numeric/date value is less than comparison value |
| `LESS_THAN_OR_EQUALS` | Numeric/date value is less than or equal to comparison value |
| `IN` | Field value exists in a supplied list |
| `NOT_IN` | Field value does not exist in a supplied list |

The first implementation may support only the operators used by the rules in this file. Unsupported operators shall cause a validation error during package import or rule evaluation.

## 10. Rule Object Schema

Each rule object in the `rules` array shall follow this structure:

```json
{
  "rule_code": "...",
  "rule_type": "...",
  "label": "...",
  "description": "...",
  "enabled": true,
  "severity": "...",
  "when": {
    "all": []
  },
  "then": {
    "require_fields": [],
    "require_child_rows": [],
    "activate_fields": [],
    "activate_sections": [],
    "activate_forms": [],
    "derive_fields": [],
    "validate": []

  "message": "...",
  "affected_sections": [],
  "affected_fields": [],
  "affected_forms": [],
  "notes": []
```

### 10.1 Required Rule Attributes

| Attribute | Type | Required | Purpose |
|---|---:|---:|---|
| `rule_code` | String | Yes | Unique rule identifier |
| `rule_type` | String | Yes | One of the defined rule types |
| `label` | String | Yes | Short human-readable rule label |
| `description` | String | Yes | Explanation of the rule |
| `enabled` | Boolean | Yes | Whether rule is active |
| `severity` | String | Yes | INFO, WARNING, ERROR, or BLOCKER |
| `when` | Object | Yes | Rule trigger conditions |
| `then` | Object | Yes | Rule effects |
| `message` | String | Yes | Message to show if the rule fails or applies |
| `affected_sections` | Array | Yes | Related section codes |
| `affected_fields` | Array | Yes | Related field codes |
| `affected_forms` | Array | Yes | Related form codes |
| `notes` | Array | Yes | Implementation notes |

## 11. Condition Object Schema

Conditions inside `when.all` shall follow this structure:

```json
{
  "field": "FIELD.CODE",
  "operator": "EQUALS",
  "value": "EXPECTED_VALUE"
}
```

For boolean checks, the preferred format is:

```json
{
  "field": "FIELD.CODE",
  "operator": "IS_TRUE"
}
```

For empty checks:

```json
{
  "field": "FIELD.CODE",
  "operator": "IS_EMPTY"
}
```

For unconditional rules, `when.all` shall be an empty array.

---

## 12. Rule Evaluation Semantics

The POC evaluator shall use these semantics:

1. A rule is evaluated only if `enabled` is `true`.
2. A rule applies when all conditions in `when.all` evaluate to true.
3. For `REQUIRE_FIELD`, if the rule applies and any required field is empty, the rule produces a message.
4. For `REQUIRE_CHILD_ROWS`, if the rule applies and the required child row condition is not met, the rule produces a message.
5. For `ACTIVATE_FIELD`, `ACTIVATE_SECTION`, and `ACTIVATE_FORM`, the rule contributes activation metadata. These rules normally produce `INFO` messages or no visible error unless the engine chooses to show them.
6. For `DERIVE_FIELD`, the engine may set or suggest derived values. In the POC, derivation may be implemented as metadata first and applied later.
7. For `VALIDATE_DATE_ORDER`, if the date order is invalid, the rule produces an error or blocker.
8. For `VALIDATE_NUMERIC_LIMIT`, if the limit is breached, the rule produces an error or blocker.
9. `ERROR` and `BLOCKER` messages prevent tender-pack generation.
10. `WARNING` and `INFO` messages do not prevent tender-pack generation.

## 14. Child Row Requirement Model

Some rules require child table rows rather than scalar fields.

The POC shall use this child-row model inside `then.require_child_rows`:

```json
{
  "child_table": "Tender Lot",
  "minimum_rows": 2,
  "condition_label": "At least two lots are required when multiple lots are enabled."
}
```

For BoQ row subsets, the POC shall allow a `row_filter` object:

```json
{
  "child_table": "Tender BoQ Item",
  "minimum_rows": 1,
  "row_filter": {
    "field": "item_category",
    "operator": "EQUALS",
    "value": "DAYWORKS"
  },
  "condition_label": "At least one daywork row is required when dayworks are enabled."
}
```

The future implementation may map child table names to actual Frappe child DocTypes.

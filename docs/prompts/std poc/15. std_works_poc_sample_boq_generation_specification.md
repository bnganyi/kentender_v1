## 2. Purpose

The purpose of this step is to make sample BoQ generation a validated operating feature of the STD-WORKS-POC.

Earlier steps established that Works tenders should not rely on manually uploaded BoQ spreadsheets or PDFs as the primary system record. Instead, the BoQ must be treated as structured system data. Step 15 formalizes the POC behavior that populates representative BoQ rows from `sample_tender.json` into the `Tender BoQ Item` child table and validates that those rows satisfy the BoQ-related rules in `rules.json`.

This step does not implement a full production BoQ authoring module. It proves the architectural direction: BoQ content can be represented, validated, rendered, and audited as structured data linked to the tender.

---

## 3. Design Principle

The sample BoQ generation feature shall implement the principle:

> For Works tenders, the Bill of Quantities should be a controlled structured data model, not an uncontrolled uploaded document.

The POC shall use representative BoQ rows to prove:

1. structured BoQ storage,
2. lot-linked BoQ rows,
3. ordinary priced rows,
4. bidder-priced rows,
5. fixed provisional-sum rows,
6. dayworks rows,
7. grand summary rows,
8. validation against BoQ rules,
9. rendering into tender-pack preview,
10. repeatable regeneration without duplicate rows.

## 4. Scope of Step 15

Step 15 shall refine and verify the sample BoQ generation feature.

This step includes:

1. Review and harden `load_sample_boq_rows()` in `std_template_engine.py`.
2. Review and harden `generate_sample_boq(tender_name)` in the `Procurement Tender` controller.
3. Ensure sample BoQ rows are copied from `sample_tender.json` into `Tender BoQ Item` child rows.
4. Ensure generated BoQ rows include categories required by `rules.json`.
5. Ensure BoQ row generation is deterministic and idempotent.
6. Ensure BoQ rows can be validated by the rule engine.
7. Ensure the tender-pack preview displays generated BoQ rows correctly.
8. Ensure multiple-lot references are consistent with the tender lots.
9. Provide manual verification scenarios.

## 9. Required Row-Level Behavior

### 9.1 Item Code

Each generated row shall have a stable `item_code`.

Rules:

1. `item_code` shall be copied from the sample row.
2. `item_code` shall be unique within the tender's generated BoQ child table.
3. Regeneration shall not create duplicate `item_code` values.

### 9.2 Lot Code

Each generated row may reference a `lot_code`.

Rules:

1. Where `lot_code` is not null, it should match an existing `Tender Lot.lot_code` in the same tender.
2. `GRAND_SUMMARY` rows may have `lot_code = null`.
3. If a row references a missing lot code, the generation method shall either raise a clear warning or validation message. For Step 15, a clear server-side error is acceptable because the sample data should be internally consistent.

### 9.3 Item Category

Each generated row shall have an `item_category` that exists in the Step 9 allowed category list.

If an unknown category is encountered, the method shall fail clearly.

### 9.4 Quantity

Each generated row shall have a numeric quantity.

For POC sample rows, `quantity = 1` is acceptable because the sample is representative rather than a full measurement schedule.

### 9.5 Rate and Amount

Rules:

1. Bidder-priced rows may have `rate = null` and `amount = null`.
2. Fixed provisional-sum rows may have `amount` populated and `is_priced_by_bidder = false`.
3. Grand summary rows may have `rate = null` and `amount = null`.
4. The POC shall not calculate totals in Step 15.

### 9.6 Bidder-Priced Flag

`is_priced_by_bidder` shall be copied from the sample row.

The generated BoQ summary shall make it visible which rows are priced by the bidder and which are fixed or summary rows.

## 10. Required Engine Behavior

The Step 11 function `load_sample_boq_rows(template, variant_code=None)` shall be confirmed or updated to satisfy this behavior:

1. Load BoQ rows from `sample_tender.boq.rows`.
2. If `variant_code` is supplied and the variant has supported BoQ overrides, apply them.
3. If `boq_overrides` is empty, use the primary sample rows.
4. Return a list of dictionaries.
5. Do not mutate the package object.
6. Do not write to the database.
7. Raise a clear error if `sample_tender.boq.rows` is missing or invalid.

For Step 15, no variant-specific BoQ override is required beyond supporting the current empty override shape.

---

## 11. Required Server Method Behavior

The Step 12 server method `generate_sample_boq(tender_name)` shall be confirmed or updated to satisfy this behavior:

1. Load the `Procurement Tender` document.
2. Check write permission.
3. Confirm `std_template` is selected.
4. Load the linked template package.
5. Load sample BoQ rows through `load_sample_boq_rows(template)`.
6. Validate BoQ row structure before writing.
7. Validate BoQ item categories against allowed POC categories.
8. Validate lot references against `Tender Lot` rows when lots exist.
9. Clear existing `boq_items` child rows.
10. Append generated BoQ rows in source order.
11. Set a clear generated note where useful, for example `Generated from STD-WORKS-POC sample_tender.json`.
12. Set `validation_status` to `Not Validated` if BoQ replacement invalidates prior validation.
13. Save the tender document.
14. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "message": "Sample BoQ generated.",
  "tender": "PT-2026-00001",
  "boq_row_count": 9,
  "categories": [
    "PRELIMINARIES",
    "SUBSTRUCTURE",
    "SUPERSTRUCTURE",
    "ROOFING",
    "FINISHES",
    "EXTERNAL_WORKS",
    "DAYWORKS",
    "PROVISIONAL_SUMS",
    "GRAND_SUMMARY"
  ]
}
```

---

## 12. Idempotency Requirement

Sample BoQ generation shall be idempotent.

Running `generate_sample_boq(tender_name)` repeatedly against the same tender shall produce:

1. the same number of rows,
2. the same `item_code` values,
3. the same row order,
4. the same categories,
5. no duplicate rows.

The method shall clear and regenerate rows rather than appending to existing rows.

---

## 13. Relationship to Validation Rules

After sample BoQ generation, the following rules from `rules.json` should pass when the primary sample configuration is used:

| Rule | Expected result after sample BoQ generation |
|---|---|
| `RULE_BOQ_REQUIRE_ROWS` | Passes because at least one `Tender BoQ Item` row exists |
| `RULE_DAYWORKS_REQUIRE_ROWS` | Passes because at least one row has `item_category = DAYWORKS` |
| `RULE_PROVISIONAL_SUMS_REQUIRE_ROWS` | Passes because at least one row has `item_category = PROVISIONAL_SUMS` |

If the primary sample configuration has `WORKS.BOQ_REQUIRED = true`, `WORKS.DAYWORKS_INCLUDED = true`, and `WORKS.PROVISIONAL_SUMS_INCLUDED = true`, these BoQ rows are necessary for validation to pass.

## 14. Relationship to Lots

The primary sample has two lots:

| Lot code | Title |
|---|---|
| `LOT-001` | Main Building Works |
| `LOT-002` | External Works and Services |

The sample BoQ rows shall reference these lots as follows:

| BoQ item category | Expected lot behavior |
|---|---|
| `PRELIMINARIES` | May reference `LOT-001` in sample |
| `SUBSTRUCTURE` | References `LOT-001` |
| `SUPERSTRUCTURE` | References `LOT-001` |
| `ROOFING` | References `LOT-001` |
| `FINISHES` | References `LOT-001` |
| `EXTERNAL_WORKS` | References `LOT-002` |
| `DAYWORKS` | May reference `LOT-001` in sample |
| `PROVISIONAL_SUMS` | May reference `LOT-002` in sample |
| `GRAND_SUMMARY` | May have no lot code |

The generation method shall not create lots. Lot creation is handled by loading the sample tender or by the tender's lot authoring process.

If no lots exist, the generation method may still create BoQ rows, but it shall include a warning or return note that lot references could not be verified. For strict POC consistency, the recommended user flow is:

1. Load Sample Tender.
2. Generate Sample BoQ only if BoQ was cleared or needs regeneration.
3. Validate Configuration.

## 16. Manual Verification Scenarios

### 16.1 Primary Sample BoQ Generation

Input:

1. Create or open a `Procurement Tender`.
2. Select the STD-WORKS-POC `STD Template`.
3. Load Sample Tender.
4. Clear BoQ rows manually, if needed.
5. Click Generate Sample BoQ.

Expected result:

1. 9 BoQ rows are generated.
2. Required categories are present.
3. `DAYWORKS` row exists.
4. `PROVISIONAL_SUMS` row exists.
5. `GRAND_SUMMARY` row exists.
6. No duplicate `item_code` values exist.

### 16.2 Idempotency Check

Input:

1. Run Generate Sample BoQ twice.

Expected result:

1. Row count remains 9.
2. No duplicate rows appear.
3. Row order remains stable.

### 16.3 Validation After BoQ Generation

Input:

1. Load Sample Tender.
2. Generate Sample BoQ.
3. Validate Configuration.

Expected result:

1. BoQ-related blocker messages do not appear.
2. `RULE_BOQ_REQUIRE_ROWS` passes.
3. `RULE_DAYWORKS_REQUIRE_ROWS` passes.
4. `RULE_PROVISIONAL_SUMS_REQUIRE_ROWS` passes.

### 16.4 Missing Dayworks Negative Check

Input:

1. Load Sample Tender.
2. Delete the `DAYWORKS` BoQ row.
3. Validate Configuration.

Expected result:

1. Validation blocks generation.
2. A validation message indicates that a daywork BoQ row is required.

### 16.5 Missing Provisional Sums Negative Check

Input:

1. Load Sample Tender.
2. Delete the `PROVISIONAL_SUMS` BoQ row.
3. Validate Configuration.

Expected result:

1. Validation blocks generation.
2. A validation message indicates that a provisional sum BoQ row is required.

### 16.6 Tender Pack Preview Check

Input:

1. Load Sample Tender.
2. Generate Sample BoQ.
3. Validate Configuration.
4. Generate Tender Pack Preview.

Expected result:

1. BoQ summary appears in the preview.
2. All 9 representative BoQ rows are visible.
3. Dayworks and provisional sums rows are visible.
4. The preview includes the POC BoQ warning.

## 17. Error Handling Requirements

The feature shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Missing tender name | Raise clear error |
| Tender not found | Raise clear error |
| User lacks write permission | Raise permission error |
| Missing `std_template` | Raise clear message to select template |
| Template package cannot be loaded | Raise clear package error |
| Missing `sample_tender.boq.rows` | Raise clear error |
| BoQ rows not an array | Raise clear error |
| Missing required BoQ row field | Raise clear error identifying row and field |
| Unknown BoQ item category | Raise clear error identifying category |
| Duplicate `item_code` in sample BoQ | Raise clear error |
| Lot reference missing when strict verification is enabled | Raise clear error or warning, as defined by implementation |

Silent omission of BoQ rows is not acceptable.

---

## 18. Required BoQ Row Validation Before Writing

Before writing BoQ rows to the tender, the server method or engine helper shall validate each row has:

1. `item_code`,
2. `item_category`,
3. `description`,
4. `unit`,
5. `quantity`,
6. `is_priced_by_bidder`.

Optional fields:

1. `lot_code`,
2. `rate`,
3. `amount`.

Validation shall reject rows with missing required values, except that numeric zero is valid where applicable.

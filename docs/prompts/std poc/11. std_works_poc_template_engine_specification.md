## 2. Purpose

The purpose of this step is to define the template engine service for STD-WORKS-POC.

The template engine is the service layer that reads an imported `STD Template` package and performs deterministic operations needed by later Frappe actions:

1. load the template package from the `STD Template` record,
2. initialize a tender configuration from `fields.json` defaults,
3. merge sample tender data for demonstration,
4. evaluate declarative rules from `rules.json`,
5. resolve active sections,
6. resolve required forms,
7. compute configuration hash,
8. prepare render context for later Jinja rendering.

This step shall implement the core service functions, but it shall not add Desk buttons, client scripts, render templates, PDF generation, bidder portal functionality, or workflow automation.

---

## 3. Design Principle

The template engine shall implement the principle:

> The official tender configuration must be reproducible from the imported template package plus the tender configuration JSON.

The engine shall be deterministic and side-effect-light. Helper functions shall generally return data structures. Functions that write to Frappe documents shall be clearly separated from pure evaluation functions.

The engine shall be intentionally simple for the POC. It does not need to become a universal business-rule platform. It only needs to interpret the specific JSON structures defined in Steps 3 through 8.

## 6. Required Responsibilities

The template engine shall be responsible for:

1. Loading an imported `STD Template` by template code or document name.
2. Parsing the stored `package_json` into a Python dictionary.
3. Initializing default configuration values from `fields.json`.
4. Loading sample tender configuration from `sample_tender.json`.
5. Applying configuration overrides where required.
6. Computing deterministic configuration hash.
7. Evaluating field requirement rules.
8. Evaluating child-row requirement rules.
9. Evaluating date-order validation rules.
10. Evaluating numeric limit validation rules.
11. Resolving active sections.
12. Resolving required forms.
13. Building validation messages suitable for `Tender Validation Message` rows.
14. Building required-form rows suitable for `Tender Required Form` rows.
15. Building a render context for later tender-pack preview rendering.

---

## 7. Non-Responsibilities

The template engine shall not be responsible for:

1. Importing the template package from disk. That is handled by Step 10.
2. Creating Frappe DocTypes. That was handled by Step 9.
3. Creating Desk buttons. That belongs to Step 12.
4. Creating Jinja templates. That belongs to Step 13.
5. Producing final legal PDFs.
6. Implementing bidder submission.
7. Implementing bid sealing or opening.
8. Implementing full evaluation workflow.
9. Implementing contract execution workflow.
10. Performing external registry verification.

## 10. Required Public Functions

The module shall expose the following public functions.

### 10.1 `load_template(template_code_or_name)`

Purpose:

Loads an `STD Template` record and parses its stored package JSON.

Input:

| Parameter | Type | Required | Description |
|---|---|---:|---|
| `template_code_or_name` | String | Yes | Template code or Frappe document name |

Expected behavior:

1. Finds `STD Template` by `template_code` first.
2. If not found, attempts to load by document name.
3. Raises a clear error if not found.
4. Parses `package_json`.
5. Returns a dictionary with:
   - `doc`
   - `package`
   - `template_code`
   - `package_hash`

---

### 10.2 `get_package_component(template, component_name)`

Purpose:

Returns a named component from loaded package data.

Allowed component names:

1. `manifest`
2. `sections`
3. `fields`
4. `rules`
5. `forms`
6. `render_map`
7. `sample_tender`

Expected behavior:

1. Raises a clear error if component is missing.
2. Returns the component dictionary.

---

### 10.3 `initialize_config(template)`

Purpose:

Creates a default tender configuration dictionary from `fields.json`.

Expected behavior:

1. Reads `fields.fields` from the package.
2. For each field, sets `field_code` to `default_value`.
3. Includes system fields such as `SYSTEM.TEMPLATE_CODE` and `SYSTEM.PACKAGE_VERSION` from package data.
4. Does not write to the database.
5. Returns a dictionary mapping field codes to values.

---

### 10.4 `load_sample_config(template, variant_code=None)`

Purpose:

Loads the sample tender configuration from the package.

Expected behavior:

1. Reads `sample_tender.configuration`.
2. If `variant_code` is supplied, applies the variant's `configuration_overrides`.
3. Returns a dictionary mapping field codes to values.
4. Does not write to the database.

Variant behavior:

1. If `variant_code` is not found, raise a clear error.
2. If variant includes `lot_overrides`, caller may use helper function `load_sample_lots()`.
3. If variant includes `boq_overrides`, caller may use helper function `load_sample_boq_rows()`.

---

### 10.5 `load_sample_lots(template, variant_code=None)`

Purpose:

Loads sample lot rows from the package.

Expected behavior:

1. Returns primary `sample_tender.lots` by default.
2. If variant has non-empty `lot_overrides`, returns the variant lot rows.
3. Does not write to the database.

---

### 10.6 `load_sample_boq_rows(template, variant_code=None)`

Purpose:

Loads sample BoQ rows from the package.

Expected behavior:

1. Returns primary `sample_tender.boq.rows` by default.
2. Applies future `boq_overrides` only if present and supported.
3. For this POC, empty `boq_overrides` means use the primary BoQ rows.
4. Does not write to the database.

---

### 10.7 `hash_config(config)`

Purpose:

Computes a deterministic SHA-256 hash for tender configuration.

Expected behavior:

1. Serializes configuration using sorted JSON keys.
2. Uses UTF-8 encoding.
3. Returns lowercase SHA-256 hex digest.
4. Ignores transient fields if explicitly defined later; for Step 11, hash the complete dictionary passed in.

---

### 10.8 `evaluate_condition(condition, config)`

Purpose:

Evaluates one condition object from `rules.json`.

Supported operators:

1. `EQUALS`
2. `NOT_EQUALS`
3. `IS_TRUE`
4. `IS_FALSE`
5. `IS_EMPTY`
6. `IS_NOT_EMPTY`
7. `GREATER_THAN`
8. `GREATER_THAN_OR_EQUALS`
9. `LESS_THAN`
10. `LESS_THAN_OR_EQUALS`
11. `IN`
12. `NOT_IN`

Expected behavior:

1. Reads field value from `config`.
2. Applies operator.
3. Returns Boolean.
4. Raises clear error for unsupported operator.

---

### 10.9 `rule_applies(rule, config)`

Purpose:

Determines whether a rule applies.

Expected behavior:

1. If `rule.enabled` is false, returns false.
2. Evaluates all conditions in `rule.when.all`.
3. Returns true only if all conditions pass.
4. Empty `when.all` means the rule applies unconditionally.

---

### 10.10 `validate_config(template, config, lots=None, boq_items=None)`

Purpose:

Evaluates `rules.json` against a configuration and child rows.

Expected behavior:

1. Evaluates all enabled rules.
2. Handles `REQUIRE_FIELD` rules.
3. Handles `REQUIRE_CHILD_ROWS` rules.
4. Handles `VALIDATE_DATE_ORDER` rules.
5. Handles `VALIDATE_NUMERIC_LIMIT` rules.
6. Captures activation metadata from `ACTIVATE_FIELD`, `ACTIVATE_SECTION`, and `ACTIVATE_FORM` rules.
7. Returns a validation result object.

Required return shape:

```json
{
  "ok": true,
  "status": "Passed",
  "blocks_generation": false,
  "messages": [],
  "active_fields": [],
  "active_sections": [],
  "active_forms": [],
  "required_fields": [],
  "configuration_hash": "..."
}
```

Status values:

| Status | Meaning |
|---|---|
| `Passed` | No warnings, errors, or blockers |
| `Passed With Warnings` | Warnings only |
| `Failed` | One or more errors |
| `Blocked` | One or more blockers |

For this POC, `ERROR` and `BLOCKER` both set `blocks_generation = true`.

---

### 10.11 `resolve_active_sections(template, config, validation_result=None)`

Purpose:

Returns active sections for rendering.

Expected behavior:

1. Starts with all sections where `included_in_poc_render` is true.
2. Adds any sections activated by rules.
3. Sorts by `render_order`.
4. Returns section objects or a simplified list with code/title/order.

For the POC, rendering all included POC sections is acceptable. Rule activation metadata should still be preserved.

---

### 10.12 `resolve_required_forms(template, config, validation_result=None)`

Purpose:

Determines which forms are required for the tender.

Expected behavior:

1. Reads `forms.forms` from package.
2. Includes forms where `default_required = true`.
3. Includes forms activated by validation result `active_forms`.
4. Excludes duplicates.
5. Sorts by `checklist_display.display_order`.
6. Returns form objects or row dictionaries suitable for `Tender Required Form`.

Required row shape for child table population:

```json
{
  "form_code": "FORM_OF_TENDER",
  "form_title": "Form of Tender",
  "display_group": "Core Bid Forms",
  "workflow_stage": "BID_SUBMISSION",
  "respondent_type": "BIDDER_OR_JV",
  "required": true,
  "activation_source": "DEFAULT_OR_RULE_CODE",
  "evidence_policy": "STRUCTURED_ONLY",
  "display_order": 10,
  "notes": ""
}
```

---

### 10.13 `build_validation_message_rows(validation_result)`

Purpose:

Converts validation messages into child-table row dictionaries for `Tender Validation Message`.

Expected behavior:

1. Reads `validation_result.messages`.
2. Returns list of row dictionaries with fields matching Step 9.
3. Sets `blocks_generation` based on severity.

---

### 10.14 `build_render_context(template, config, lots=None, boq_items=None, validation_result=None, required_forms=None)`

Purpose:

Builds a context dictionary for later Jinja rendering.

Expected behavior:

1. Includes manifest metadata.
2. Includes template code and package hash.
3. Includes configuration dictionary.
4. Includes active sections.
5. Includes lots.
6. Includes BoQ items.
7. Includes required forms.
8. Includes validation messages.
9. Includes configuration hash.
10. Does not render HTML.

Required return shape:

```json
{
  "template": {},
  "manifest": {},
  "configuration": {},
  "active_sections": [],
  "lots": [],
  "boq_items": [],
  "required_forms": [],
  "validation": {},
  "audit": {}
}
```

---

### 10.15 `apply_config_to_tender_doc(tender_doc, config)`

Purpose:

Writes configuration JSON and summary fields to a `Procurement Tender` document.

Expected behavior:

1. Stores pretty-printed JSON into `configuration_json`.
2. Updates summary fields such as:
   - `tender_title`
   - `tender_reference`
   - `procurement_method`
   - `tender_scope`
   - `procurement_category`
   - `template_code`
   - `template_version`
3. Does not save automatically unless explicitly requested by caller.

---

### 10.16 `populate_sample_tender(tender_doc, variant_code=None)`

Purpose:

Populates a `Procurement Tender` document with the sample tender configuration, sample lots, and sample BoQ rows.

Expected behavior:

1. Loads linked `STD Template`.
2. Loads sample config and optional variant.
3. Applies configuration JSON to the tender document.
4. Clears and fills `lots` child table.
5. Clears and fills `boq_items` child table.
6. Does not save automatically unless explicitly requested by caller.

This function may be used by Step 12 buttons later.

## 11. Validation Message Object Shape

Validation messages returned by `validate_config()` shall follow this shape:

```json
{
  "severity": "BLOCKER",
  "rule_code": "RULE_CODE",
  "message": "User-facing message",
  "affected_field": "FIELD.CODE",
  "affected_section": "SECTION_CODE",
  "affected_form": "FORM_CODE",
  "blocks_generation": true,
  "resolution_hint": "Complete the required field or adjust the configuration.",
  "details": {}
}
```

If multiple required fields are missing under one rule, the engine may return either:

1. one message per missing field, or
2. one grouped message listing all missing fields.

For better Desk usability, the preferred POC behavior is **one message per missing field**.

## 12. Required Rule Handling Details

### 12.1 `REQUIRE_FIELD`

If the rule applies, every field in `then.require_fields` must be non-empty.

Empty values are:

1. `None`,
2. empty string,
3. empty list,
4. empty dictionary.

Boolean `false` is not empty.

Numeric `0` is not empty.

### 12.2 `REQUIRE_CHILD_ROWS`

If the rule applies, every child-row condition in `then.require_child_rows` must pass.

Supported child tables for POC:

1. `Tender Lot`
2. `Tender BoQ Item`

If `row_filter` exists, only matching rows count.

### 12.3 `ACTIVATE_FORM`

If the rule applies, every form in `then.activate_forms` shall be added to `active_forms`.

### 12.4 `ACTIVATE_SECTION`

If the rule applies, every section in `then.activate_sections` shall be added to `active_sections`.

### 12.5 `ACTIVATE_FIELD`

If the rule applies, every field in `then.activate_fields` shall be added to `active_fields`.

### 12.6 `VALIDATE_DATE_ORDER`

Supported validation types:

| Validation type | Meaning |
|---|---|
| `FIELD_BEFORE_FIELD` | Earlier field must be before later field |
| `FIELD_ON_OR_AFTER_FIELD` | First field must be same as or after second field |

Datetime strings shall be parsed using Frappe/dateutil-compatible parsing where possible.

### 12.7 `VALIDATE_NUMERIC_LIMIT`

Supported validation type:

| Validation type | Meaning |
|---|---|
| `MAX_PERCENT_OF_FIELD` | Field value must not exceed a percentage of base field value |

Example:

Tender security amount must not exceed 2 percent of estimated cost.

## 13. Configuration Hash Requirements

The configuration hash shall be computed from tender configuration only.

Required method:

1. Convert config dictionary to JSON.
2. Sort keys.
3. Use compact separators.
4. Encode as UTF-8.
5. Compute SHA-256.

Example implementation pattern:

```python
payload = json.dumps(config, sort_keys=True, separators=(",", ":"), default=str)
config_hash = hashlib.sha256(payload.encode("utf-8")).hexdigest()
```

---

## 14. Required Status Mapping

`validate_config()` shall map message severities to validation status as follows:

| Highest severity present | Validation status | Blocks generation |
|---|---|---:|
| None | `Passed` | No |
| `INFO` only | `Passed` | No |
| `WARNING` | `Passed With Warnings` | No |
| `ERROR` | `Failed` | Yes |
| `BLOCKER` | `Blocked` | Yes |

If both `ERROR` and `BLOCKER` exist, status shall be `Blocked`.

## 17. Error Handling Requirements

The engine shall raise clear errors for:

| Scenario | Required behavior |
|---|---|
| `STD Template` not found | Raise clear Frappe error or exception |
| Empty `package_json` | Raise clear error |
| Invalid package JSON | Raise clear error |
| Missing package component | Raise clear error |
| Unsupported operator | Raise clear error naming operator |
| Unsupported validation type | Raise clear error naming validation type |
| Unknown child table in rule | Raise clear error naming child table |
| Missing linked template on tender | Raise clear error |
| Invalid date value | Produce validation message or raise clear error depending on context |

Do not silently ignore unknown operators, rule types, or validation types.

---

## 18. Transaction Behavior

Most engine functions shall not commit database transactions.

Recommended behavior:

| Function type | May save? | May commit? |
|---|---:|---:|
| Pure helpers | No | No |
| Validation and resolution functions | No | No |
| `apply_config_to_tender_doc` | No by default | No |
| `populate_sample_tender` | No by default | No |
| `write_validation_to_tender` | No by default | No |
| `write_required_forms_to_tender` | No by default | No |

Step 12 button actions may save and commit as needed.

## 19. Security Notes

The engine shall not execute any code contained in JSON package files.

The rule model is declarative. The engine shall interpret only the supported operators and validation types.

The engine shall not accept arbitrary file paths from users.

The engine shall not expose package modification functionality.

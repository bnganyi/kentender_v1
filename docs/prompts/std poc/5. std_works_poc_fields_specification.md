## 2. Purpose

The purpose of this step is to define the full content of the `fields.json` file for the PPRA Works Standard Tender Document POC template package.

`fields.json` shall define the controlled procurement-officer configuration fields used to create a Works tender from the approved POC template package. These fields are the safe configuration surface exposed to ordinary procurement officers.

This file shall not define validation logic, conditional activation logic, bidder form schemas, rendering templates, Frappe DocTypes, or server-side implementation. Those are handled in later steps.

---

## 3. Design Principle

The `fields.json` file shall implement the principle:

> Procurement officers configure tenders by answering controlled business questions; they do not edit legal template text.

Each field shall be:

1. business-readable,
2. legally traceable to one or more template sections,
3. typed,
4. constrained where possible,
5. grouped for guided setup,
6. renderable into later tender-pack sections,
7. suitable for rule evaluation in `rules.json`,
8. suitable for storage in the `Procurement Tender` configuration JSON.

The field model shall be deliberately finite and tractable for the POC. It shall cover the major Works STD configuration scenarios without attempting to represent every blank, clause, schedule, or form field in the full source document.

The file shall exist at:

```text
{app_name}/procurement/std_templates/ke_ppra_works_building_2022_04_poc/fields.json
```

The placeholder `{app_name}` shall be replaced by the actual Frappe custom app name in the repository.

## 5. Relationship to Previous Package Files

`fields.json` shall use the same template code as `manifest.json` and `sections.json`:

```text
KE-PPRA-WORKS-BLDG-2022-04-POC
```

Field `section_targets` shall refer only to section codes defined in `sections.json`.

Field codes referenced in `sections.json.depends_on_fields` shall be defined in `fields.json` unless they are explicitly deferred to a later module. For this POC, all such field codes shall be defined in `fields.json`.

No additional top-level keys shall be added in Step 5 unless explicitly reviewed and accepted.

## 7. Field Type Definitions

The file shall define the supported field types used in the POC.

Required field types:

| Field type | Meaning |
|---|---|
| `TEXT` | Short free-text value |
| `LONG_TEXT` | Longer controlled text value |
| `SELECT` | Single value selected from controlled options |
| `MULTI_SELECT` | Multiple values selected from controlled options |
| `BOOLEAN` | True/false value |
| `INTEGER` | Whole number |
| `DECIMAL` | Numeric value that may contain decimals |
| `MONEY` | Monetary value; currency captured separately where required |
| `PERCENT` | Percentage value |
| `DATE` | Calendar date |
| `DATETIME` | Date and time |
| `DURATION_DAYS` | Duration expressed in calendar days |
| `DURATION_MONTHS` | Duration expressed in months |
| `EMAIL` | Email address |
| `URL` | Web link |
| `ORGANIZATION_REF` | Reference to a procuring entity or organization record, if available |
| `USER_REF` | Reference to a user, officer, engineer, or project manager, if available |
| `JSON_OBJECT` | Structured object reserved for future expansion |

These field types shall be represented in `field_types` as structured objects.

---

## 8. Field Group Model

Fields shall be organized into guided setup groups.

Required groups:

| Group code | Label | Purpose |
|---|---|---|
| `TENDER_IDENTITY` | Tender Identity | Procuring entity, tender name, reference, description, location, estimated cost |
| `METHOD_PARTICIPATION` | Method and Participation | Procurement method, tender scope, reservations, preference, prequalification, JV, foreign tenderer rules |
| `DATES_MEETINGS` | Dates and Meetings | Publication, clarification, site visit, pre-tender meeting, submission, opening, validity |
| `SECURITIES` | Securities | Tender security, securing declaration, performance security, advance payment security, retention money security |
| `ALTERNATIVES_LOTS` | Alternatives and Lots | Alternative tenders, lots, lot evaluation and award method |
| `QUALIFICATION` | Qualification Requirements | Financial, experience, personnel, equipment, statutory and ownership requirements |
| `WORKS_REQUIREMENTS` | Works Requirements | Scope, specifications, drawings, BoQ, dayworks, provisional sums, ESHS requirements |
| `CONTRACT_SCC` | Contract and SCC Parameters | Engineer/project manager, defects liability, damages, payment, insurance, retention, disputes, completion |
| `SYSTEM_AUDIT` | System and Audit Fields | Template, hashes, generation, and system-derived control fields |

Each group shall include a render/order value so the guided setup UI can display groups in a stable sequence.

## 9. Field Object Schema

Each field object in the `fields` array shall follow this structure:

```json
{
  "field_code": "...",
  "group_code": "...",
  "label": "...",
  "business_question": "...",
  "field_type": "...",
  "required_by_default": true,
  "default_value": null,
  "options": [],
  "section_targets": [],
  "render_targets": [],
  "help_text": "...",
  "ordinary_user_editable": true,
  "poc_required": true,
  "notes": []
}
```

### 9.1 Required Field Object Attributes

| Attribute | Type | Required | Purpose |
|---|---:|---:|---|
| `field_code` | String | Yes | Stable system identifier for the field |
| `group_code` | String | Yes | Field group identifier |
| `label` | String | Yes | Short display label |
| `business_question` | String | Yes | User-facing setup question |
| `field_type` | String | Yes | One of the defined field types |
| `required_by_default` | Boolean | Yes | Whether field is required before conditional rules are applied |
| `default_value` | Any/null | Yes | Default value where applicable |
| `options` | Array | Yes | Controlled options for select/multi-select fields; empty for other types |
| `section_targets` | Array | Yes | Section codes affected by the field |
| `render_targets` | Array | Yes | Logical render targets used later by render map |
| `help_text` | String | Yes | Plain explanation for procurement officers |
| `ordinary_user_editable` | Boolean | Yes | Whether procurement officer may enter/change the value |
| `poc_required` | Boolean | Yes | Whether field is part of this POC scope |
| `notes` | Array | Yes | Implementation notes |

## 10. Field Coding Standard

Field codes shall follow this format:

```text
DOMAIN.FIELD_NAME
```

Examples:

```text
TENDER.TENDER_NAME
METHOD.PROCUREMENT_METHOD
DATES.SUBMISSION_DEADLINE
SECURITY.TENDER_SECURITY_MODE
QUALIFICATION.MIN_AVERAGE_ANNUAL_TURNOVER
CONTRACT.DEFECTS_LIABILITY_PERIOD_MONTHS
```

Rules:

1. Domain shall be uppercase.
2. Field name shall be uppercase snake case.
3. Field code shall not contain spaces.
4. Field code shall be stable after introduction.
5. Field code shall not encode UI order.
6. Field code shall not encode implementation-specific Frappe field names.

---

## 11. Required Field Coverage

The POC shall include **64 fields**.

This count is intentionally higher than the original 40–60 estimate because Step 4 introduced explicit field dependencies for system audit, contract forms, and scenario activation. The additional fields prevent ambiguity and avoid hidden undeclared values in later steps.

The field count remains tractable and is acceptable for the POC because many fields are simple booleans or controlled selects.

Required field distribution:

| Group | Field count |
|---|---:|
| Tender Identity | 9 |
| Method and Participation | 10 |
| Dates and Meetings | 11 |
| Securities | 10 |
| Alternatives and Lots | 6 |
| Qualification Requirements | 9 |
| Works Requirements | 7 |
| Contract and SCC Parameters | 9 |
| System and Audit Fields | 4 |
| Total | 75 |

Note: The total is 75 after formalizing all scenario, rendering, and audit fields required for downstream consistency. This is still acceptable for the POC because the template is the most complex Works template and the fields are grouped for guided setup.

1. tender security branching,
2. contract form activation,
3. audit summary output,
4. SCC parameters,
5. foreign tenderer participation,
6. alternative tender treatment,
7. date and meeting validation.

The field set remains minimal relative to the full Works STD and avoids hidden implicit values in later implementation.

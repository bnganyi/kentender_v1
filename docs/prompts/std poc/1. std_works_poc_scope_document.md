## 2. Purpose

The purpose of this POC is to prove that a complex PPRA Works Standard Tender Document can be represented in the system as a controlled, versioned, developer-maintained template package and used by procurement officers to create tender packages through safe guided configuration.

The POC shall demonstrate that ordinary procurement officers can create a Works tender without editing the underlying legal template text, while the system enforces required fields, activates relevant sections and bidder forms, validates configuration choices, and generates a structured tender-pack preview.

This POC is not intended to fully digitize the entire PPRA Works STD. It is intended to prove the architecture, workflow, configuration model, validation model, and rendering model required to scale to a complete production implementation.

---

## 3. Background

The public e-Procurement system is intended to support end-to-end procurement workflows, including strategy, budgeting, planning, tender management, bid management, contracts, execution, and closing.

A foundational requirement is that standard tender documents must be systemized in a way that supports fairness, transparency, auditability, consistency, and legal defensibility. The system should reduce or eliminate the need for stakeholders to upload manually completed tender documents where structured digital forms, controlled templates, registry verification, and system-generated documents can be used instead.

The Works STD is selected for this POC because it is one of the most complex standard tender documents. It includes tendering procedures, tender data, evaluation and qualification criteria, bidder forms, bills of quantities, specifications, drawings, general conditions of contract, special conditions of contract, and contract forms. If this template type can be successfully represented in the system, simpler procurement templates are expected to be tractable using the same architectural pattern.

---

## 4. Core Design Position

The POC shall use a **template-package-as-code** approach.

The PPRA Works STD shall not be configured through a live administrative template editor. Instead, the STD shall be represented as a developer-controlled JSON package committed to the application repository and imported into Frappe as an approved POC template.

The system shall separate:

| Layer | Description | Editable by ordinary procurement officer? |
|---|---|---:|
| Source STD | Original PPRA legal source document | No |
| Template package | Developer-controlled structured representation of the STD | No |
| Template runtime record | Imported Frappe record containing package metadata and JSON | No |
| Tender configuration | Tender-specific values entered by procurement officer | Yes, through controlled fields only |
| Generated tender pack | System-generated output from template and configuration | No direct editing |
| Bidder form checklist | System-generated list of required bidder response forms | No direct editing, except through configuration changes |

The POC shall prove that procurement officers configure a tender by answering structured business questions rather than editing legal clauses.

---

## 5. POC Objectives

The POC has the following objectives:

1. Represent the PPRA Works STD as a versioned template package.
2. Import the template package into Frappe/ERPNext as a controlled `STD Template` record.
3. Allow a procurement officer to create a `Procurement Tender` from the approved Works template.
4. Capture tender-specific configuration in structured fields or JSON.
5. Validate configuration using defined rules.
6. Activate or deactivate sections, fields, and forms based on configuration choices.
7. Generate a required bidder-forms checklist from the template package.
8. Generate a sample BoQ structure for the tender.
9. Generate a readable tender-pack preview using Jinja/HTML rendering.
10. Record validation messages and basic configuration audit information.
11. Demonstrate major Works STD scenarios without systemizing the full 159-page document.

---

## 6. Non-Objectives

The POC shall not attempt to deliver the following:

1. Full digitization of every clause, paragraph, form, and schedule in the full PPRA Works STD.
2. A live administrative UI for editing STD clauses, sections, forms, or legal template text.
3. A visual template builder, drag-and-drop form builder, or clause-authoring studio.
4. A complete bidder portal.
5. Complete electronic bid submission, bid sealing, bid opening, and bid encryption.
6. Full evaluation committee workflow.
7. Full contract award, standstill, complaint, review, and contract execution workflow.
8. Integration with external registries such as tax, company registry, PPRA debarment, banks, insurers, or NCA.
9. Production-grade PDF generation.
10. Complete OCDS publication mapping.
11. Multi-language support.
12. Full legal certification of the generated tender document.

These items may be added in later phases only after the POC validates the core template-package and safe-configuration architecture.

## 7. In-Scope Functional Areas

### 7.1 Template Package

The POC shall include a developer-controlled template package located in the Frappe app repository.

The package shall include:

| File | Purpose |
|---|---|
| `manifest.json` | Template identity, authority, category, version, and applicability |
| `sections.json` | Major STD sections and mutability classification |
| `fields.json` | Procurement-officer configuration fields |
| `rules.json` | Validation and activation rules |
| `forms.json` | Bidder response form schemas/checklist definitions |
| `render_map.json` | Mapping from template data to renderable sections |
| `sample_tender.json` | Sample tender configuration used for testing |

### 7.2 Frappe Template Import

The POC shall include a seed/import mechanism that reads the template package and creates or updates a corresponding Frappe `STD Template` record.

The import mechanism shall:

1. Confirm all required package files exist.
2. Load JSON content.
3. Compute a package hash.
4. Create or update the template record.
5. Store the package JSON or package payload.
6. Mark the template as available for POC use.

### 7.3 Procurement Tender Creation

The POC shall allow a procurement officer to create a tender from the imported Works STD template.

The tender shall capture:

1. Template reference.
2. Tender identity.
3. Procurement method and participation settings.
4. Key dates and meetings.
5. Security settings.
6. Alternative tender settings.
7. Lot settings.
8. Qualification requirements.
9. Works requirements.
10. SCC/contract parameters.
11. BoQ rows.
12. Required bidder forms.
13. Validation messages.
14. Generated tender-pack preview.

### 7.4 Configuration Validation

The POC shall validate tender configuration before generating a tender pack.

Validation shall produce messages classified as:

| Type | Meaning |
|---|---|
| Info | Advisory information that does not require action |
| Warning | Potential issue that should be reviewed but may not block generation |
| Error | Configuration problem that should be corrected |
| Blocker | Configuration problem that prevents tender-pack generation |

### 7.5 Scenario Activation

The POC shall demonstrate that tender configuration choices activate or deactivate fields, forms, sections, or validation requirements.

At minimum, scenario activation shall cover:

1. National tender.
2. International tender.
3. Reserved tender.
4. Margin of preference.
5. Prequalification.
6. Joint venture participation.
7. Alternative tenders.
8. Single lot.
9. Multiple lots.
10. Tender security.
11. Tender-securing declaration.
12. Site visit.
13. Pre-tender meeting.
14. BoQ.
15. Dayworks.
16. Provisional sums.
17. Performance security.
18. Advance payment security.
19. Beneficial ownership disclosure.

### 7.6 Generated Tender Pack Preview

The POC shall generate a readable HTML preview of the tender pack.

The preview shall include at minimum:

1. Cover / tender identity summary.
2. Invitation to Tender summary.
3. Active STD sections summary.
4. Tender Data Sheet summary.
5. Evaluation and qualification summary.
6. Required bidder forms checklist.
7. BoQ summary.
8. Specifications and drawings summary.
9. SCC / contract parameters summary.
10. Contract forms checklist.
11. Validation and audit summary.

## 8. Out-of-Scope Functional Areas

The following are explicitly out of scope for this POC:

| Area | Reason for exclusion |
|---|---|
| Full legal rendering of the entire PPRA Works STD | Would turn the POC into a document-conversion project |
| Clause-level editing UI | High risk and low value for rarely changing PPRA templates |
| Production bidder portal | Requires separate bid submission, security, identity, and sealing design |
| Electronic bid encryption/sealing | Requires separate security architecture |
| Full evaluation workflow | Depends on later bid-management and evaluator modules |
| Full contract execution workflow | Belongs to downstream contract module |
| Registry/API integrations | Can be simulated through placeholders in this POC |
| Advanced PDF layout | HTML preview is sufficient for proof of concept |
| Template supersession workflow | Can be represented minimally by status and version fields |
| Multi-STD catalogue | POC focuses only on Works — Building and Associated Civil Engineering Works |

## 9. POC Scenario Coverage

The POC shall cover scenario types rather than every possible legal or clause permutation.

| Scenario | Required POC behavior |
|---|---|
| National tender | International-only fields are inactive or not required |
| International tender | Foreign tenderer/local input fields become active |
| Reserved tender | Reserved group becomes mandatory |
| Margin of preference | Preference-related configuration becomes active |
| Prequalification used | Prequalification reference becomes mandatory |
| No prequalification | Post-qualification forms remain active |
| JV allowed | JV form and maximum-member field become active |
| JV not allowed | JV form is not required |
| Alternative tenders allowed | Alternative scope and alternative form become active |
| Alternative tenders not allowed | Alternative response forms are not required |
| Single lot | Lot table may contain one lot or be represented by default lot |
| Multiple lots | Lot table and lot award method become mandatory |
| Tender security selected | Security amount, currency, type, and validity become required |
| Tender-securing declaration selected | Tender-securing declaration form becomes required instead of security amount |
| Site visit enabled | Site visit date and location become mandatory |
| Pre-tender meeting enabled | Meeting date and location become mandatory |
| BoQ enabled | BoQ rows are generated or required |
| Dayworks enabled | Daywork-related BoQ rows are generated |
| Provisional sums enabled | Provisional-sum BoQ rows are generated |
| Performance security enabled | Performance security contract form becomes required |
| Advance payment enabled | Advance payment security form becomes required |
| Beneficial ownership required | Beneficial ownership disclosure form becomes required |

## 10. Template Package Scope

The template package shall be minimal but complete enough to prove the architecture.

### 10.1 Included Template Sections

| STD section | POC representation |
|---|---|
| Invitation to Tender | Parameterized and rendered |
| Instructions to Tenderers | Representative locked section summary |
| Tender Data Sheet | Configurable field-driven summary |
| Evaluation and Qualification Criteria | Controlled criteria summary |
| Tendering Forms | Structured required-form checklist |
| Bills of Quantities | Native child-table representation |
| Specifications | Structured tender configuration summary |
| Drawings | Controlled attachment/reference placeholder |
| General Conditions of Contract | Representative locked section summary |
| Special Conditions of Contract | Configurable field-driven summary |
| Contract Forms | Checklist/sample generated forms |

### 10.2 Excluded Template Detail

The POC shall not include:

1. Full verbatim ITT text.
2. Full verbatim GCC text.
3. Every page of the original STD.
4. Every qualification subform field.
5. Every BoQ preamble or measurement rule.
6. Every contract form in final legal wording.
7. Full document pagination and formatting parity with PPRA PDF.

## 12. Frappe/ERPNext Scope

### 12.1 Required DocTypes

The POC shall include the following minimum DocTypes:

| DocType | Purpose |
|---|---|
| `STD Template` | Stores template identity, version, package JSON, status, and package hash |
| `Procurement Tender` | Stores tender instance created from an STD template |
| `Tender Lot` | Child table for lot configuration |
| `Tender BoQ Item` | Child table for sample BoQ rows |
| `Tender Required Form` | Child table for generated bidder form checklist |
| `Tender Validation Message` | Child table for validation output |

### 12.2 Required Server-Side Functions

The POC shall include the following functions:

| Function | Purpose |
|---|---|
| `load_template(template_code)` | Load imported STD template package |
| `initialize_config(template)` | Create default tender configuration |
| `validate_config(template, config)` | Evaluate rules and return validation messages |
| `resolve_active_sections(template, config)` | Determine active sections |
| `resolve_required_forms(template, config)` | Determine required bidder forms |
| `generate_sample_boq(tender)` | Create representative BoQ rows |
| `render_tender_pack(template, config)` | Generate HTML preview |
| `hash_config(config)` | Generate configuration hash for audit |

### 12.3 Required Frappe Actions

The `Procurement Tender` form shall expose the following actions, either as buttons or server actions:

1. Load Template Defaults.
2. Validate Configuration.
3. Generate Required Forms.
4. Generate Sample BoQ.
5. Generate Tender Pack.
6. View Configuration Audit.

## 13. Data Storage Approach

The POC shall use a hybrid storage approach.

| Data | Storage approach |
|---|---|
| Template package | Stored as JSON in the `STD Template` record after import |
| Tender configuration | Stored as JSON in the `Procurement Tender` record |
| Lots | Stored as child table rows |
| BoQ items | Stored as child table rows |
| Required forms | Stored as child table rows |
| Validation messages | Stored as child table rows |
| Generated tender preview | Stored as HTML or long text field |

The POC shall avoid over-normalizing the template package into many administrative DocTypes. Full normalization may be considered later if required for production reporting, governance, or editing.

## 14. Development Constraints

The following constraints shall guide development:

1. Development shall be performed in small Cursor-assisted steps.
2. Each step shall have an explicit acceptance condition.
3. Cursor prompts shall be narrow and shall not ask for future-step implementation.
4. The POC shall prioritize clarity and correctness over generic flexibility.
5. The POC shall avoid building a template editor.
6. The POC shall not assume full bidder submission workflow exists.
7. The POC shall be implementable inside a custom Frappe app.
8. The template package shall be version-controlled with the codebase.
9. Generated outputs shall be reproducible from template package plus tender configuration.
10. Existing generated output shall not be silently overwritten without explicit regeneration.

## 19. Risks and Controls

| Risk | Control |
|---|---|
| POC drifts into full STD digitization | Keep full-text rendering out of scope |
| Developers build a template editor prematurely | Use JSON package-as-code only |
| Procurement officers can alter legal text | Ordinary officers only edit tender configuration |
| Rules become too generic too early | Implement explicit POC rules first |
| Frappe DocTypes become over-normalized | Store template package as JSON for POC |
| Generated preview is mistaken for certified legal document | Label as POC preview |
| Scenario coverage becomes ambiguous | Use scenario matrix and test scenarios in this document |
| Cursor implements too much at once | Use one step per Cursor prompt |

## 22. Final Scope Statement

The STD-WORKS-POC shall prove that the PPRA Works Standard Tender Document can be safely onboarded as a versioned developer-controlled template package and used by procurement officers through structured, validated configuration to generate a representative tender pack, required bidder forms checklist, sample BoQ, and audit summary.

The POC shall deliberately avoid full document digitization, template editing UI, and bidder submission workflow. Its purpose is to validate the foundational architecture before expanding to full Works STD coverage and additional STD families.

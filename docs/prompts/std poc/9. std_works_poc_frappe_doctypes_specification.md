## 2. Purpose

The purpose of this step is to define the minimum Frappe DocTypes required to persist the STD-WORKS-POC template package and create a Works tender instance from it.

This step establishes the database and Desk-form foundation for later steps. It shall not implement the seed loader, template engine, validation logic, rendering, required-form generation, or BoQ generation. Those capabilities belong to later steps.

The DocTypes defined in this step shall support:

1. storing an imported STD template package,
2. creating a procurement tender linked to the STD template,
3. storing tender configuration as JSON,
4. storing lots,
5. storing BoQ rows,
6. storing generated required form checklist rows,
7. storing validation messages,
8. storing generated tender-pack preview metadata or HTML placeholder.

---

## 3. Design Principle

The POC shall use a deliberately small Frappe data model.

The template package shall remain mostly JSON-based at this stage. The system shall not over-normalize every section, field, rule, form, and render mapping into separate DocTypes during the POC.

This approach preserves development speed while still allowing the system to prove:

1. template import,
2. template versioning,
3. tender creation,
4. configuration capture,
5. generated child records,
6. audit traceability.

The production system may later normalize selected parts of the package if needed for governance, reporting, UI inspection, or template administration. That is explicitly outside Step 9.

## 4. Required DocTypes

Step 9 shall create the following DocTypes:

| DocType | Type | Purpose |
|---|---|---|
| `STD Template` | Parent DocType | Stores imported template package identity, status, package JSON, and hash |
| `Procurement Tender` | Parent DocType | Stores a tender instance created from an STD template |
| `Tender Lot` | Child Table | Stores tender lot rows |
| `Tender BoQ Item` | Child Table | Stores representative BoQ rows |
| `Tender Required Form` | Child Table | Stores generated required bidder/contract form checklist rows |
| `Tender Validation Message` | Child Table | Stores validation messages produced by the rule engine |

No other DocTypes shall be created in Step 9 unless explicitly reviewed and accepted.

---

## 5. Naming and Module Placement

The DocTypes shall be created inside the custom Frappe app used for the public procurement system.

The recommended Frappe module name is:

```text
Procurement
```

If the application already has a module with a more specific name, the existing module may be used, but all DocTypes from this step shall be placed consistently in the same module unless there is a clear architectural reason not to.

DocType names shall be exactly:

1. `STD Template`
2. `Procurement Tender`
3. `Tender Lot`
4. `Tender BoQ Item`
5. `Tender Required Form`
6. `Tender Validation Message`

---

## 6. General DocType Requirements

All DocTypes shall follow these general requirements:

1. Use clear labels and fieldnames.
2. Avoid abbreviations except where already standard, such as `STD` and `BoQ`.
3. Do not add business logic in Step 9.
4. Do not add custom buttons in Step 9.
5. Do not add client scripts in Step 9.
6. Do not implement server methods in Step 9.
7. Do not create Workflow records in Step 9.
8. Do not create permissions beyond basic role-aware defaults unless the app already has a role model.
9. Use JSON/Code fields where package payload or tender configuration is stored.
10. Use child tables for lot, BoQ, required forms, and validation messages.

## 13. Field Type Notes for Frappe Implementation

The following Frappe field type choices are recommended:

| Logical need | Frappe field type |
|---|---|
| JSON package or configuration | Code |
| Generated HTML preview | HTML Editor or Long Text |
| Boolean | Check |
| Date and time | Datetime |
| Monetary amount | Currency |
| Percentage | Percent or Float |
| Child table | Table |
| User reference | Link to User |
| Template link | Link to STD Template |

If the installed Frappe version lacks a specific field type, the developer may choose the nearest stable field type, but the fieldname and intended behavior shall remain consistent.

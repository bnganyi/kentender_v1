## 2. Purpose

The purpose of this document is to define how conditional fields, rule-driven requirements, validation messages, and officer-facing feedback shall behave in the Procurement Officer Tender Configuration POC.

The guided configuration field model defines what fields the officer sees. This specification defines how those fields respond to officer choices and how the system explains validation outcomes in procurement-officer language.

The officer shall not be required to understand rule codes, JSON conditions, internal template mechanics, or developer-oriented trace output. The officer shall receive clear guidance about what is required, why it is required, what is blocking progress, and how to fix it.

---

## 3. Design Principle

The conditional field and rule feedback UX shall implement the principle:

> Rules must protect the tender process without making the procurement officer reason about the rule engine.

This means:

1. conditional fields appear when they become relevant;
2. hidden fields do not remove server-side obligations where rules still apply;
3. validation messages identify the business area to fix;
4. blockers are separated from warnings and information;
5. the system explains the consequence of major configuration choices;
6. preview generation is blocked when validation blockers exist;
7. officer-facing text is practical, not technical;
8. server-side validation remains authoritative.

## 6. UX Layers

Conditional and rule feedback shall operate across four layers.

| Layer | Purpose | Authoritative? |
|---|---|---:|
| Client conditional visibility | Improves officer usability by showing relevant fields | No |
| Client required indicators | Helps officer identify expected input | No |
| Server validation | Determines whether configuration is acceptable | Yes |
| Officer feedback display | Explains blockers, warnings, and next actions | Reflects server result |

The UI may guide, but the server decides.

## 7. Conditional Visibility Model

### 7.1 Purpose

Conditional visibility shall reduce officer confusion by hiding irrelevant fields and showing relevant fields when controlling choices make them applicable.

### 7.2 Required Behavior

| Rule | Requirement |
|---|---|
| Initial load | Conditional fields shall reflect current saved configuration |
| On field change | Dependent fields shall show/hide dynamically where practical |
| On save | Hidden field values shall be handled deterministically |
| On validation | Server shall validate based on current saved configuration, not only visible fields |
| On blocked result | Related hidden/inactive field issues shall still be shown if server considers them required |

### 7.3 Hidden Field Value Policy

For the POC, use this policy:

| Scenario | Behavior |
|---|---|
| Field hidden because controlling condition is false | Value may remain stored but shall not be considered required |
| Field becomes visible because controlling condition becomes true | Existing value may be reused if present; otherwise officer must complete it |
| Officer switches away from a branch | Do not silently delete values in POC; validation determines relevance |
| Preview generation | Uses current controlling field values and server validation result |

This avoids accidental loss of input during POC while preserving rule safety.

## 9. Rule Consequence Notices

### 9.1 Purpose

Rule consequence notices shall explain important downstream effects of officer choices before validation is run.

### 9.2 Required Notice Types

| Notice type | Purpose |
|---|---|
| Form impact notice | Explains that a form will be added or removed |
| Required-field notice | Explains that additional fields are required |
| BoQ requirement notice | Explains BoQ category obligation |
| Date rule notice | Explains deadline/order implications |
| Security limit notice | Explains amount validation implication |
| POC boundary notice | Explains feature is representative, not production-complete |

### 9.3 Display Position

Rule consequence notices shall appear near the controlling field or dependent group.

They shall not replace validation. They are guidance.

## 10. Validation Feedback Model

### 10.1 Validation Summary

After validation, the officer shall see a summary:

| Summary item | Meaning |
|---|---|
| Validation Status | Passed / Passed with Warnings / Blocked / Not Validated |
| Blockers | Count of issues preventing preview |
| Errors | Count of invalid items |
| Warnings | Count of non-blocking concerns |
| Information | Count of informational messages |
| Last Validated At | Timestamp |
| Configuration Hash | Snapshot identifier |

### 10.2 Validation Message Table

Officer-facing validation messages shall be displayed with these columns:

| Column | Officer-facing label | Required |
|---|---|---:|
| `severity` | Severity | Yes |
| `group_code` | Area | Yes |
| `field_label` | Field | Yes, where available |
| `message` | Message | Yes |
| `resolution_hint` | What to do | Yes |
| `blocks_generation` | Blocks preview | Yes |
| `technical_rule_code` | Rule code | Optional/details |

### 10.3 Severity Meaning

| Severity | Officer meaning |
|---|---|
| `BLOCKER` | Must be fixed before preview generation |
| `ERROR` | Invalid configuration; must be fixed before preview generation |
| `WARNING` | Should be reviewed, but may not block preview |
| `INFO` | Helpful information |

## 11. Officer-Friendly Message Requirements

Validation messages shall be translated or wrapped into officer-friendly language.

### 11.1 Message Requirements

Each blocker/error message shall:

1. identify the affected group;
2. identify the affected field where possible;
3. explain what is missing or invalid;
4. provide a resolution hint;
5. avoid internal JSON syntax unless shown in technical detail;
6. avoid exposing raw rule expressions as the primary message.

### 11.2 Example Message Transformations

| Technical condition | Officer-facing message |
|---|---|
| `MEETINGS.SITE_VISIT_MANDATORY = true` and missing date | Site visit is mandatory, but the site visit date and time have not been provided. Enter the site visit date and time in Key Dates and Meetings. |
| Missing alternative scope | Alternative tenders are allowed, but the permitted alternative scope has not been described. Enter the scope description in Alternatives. |
| Tender security above limit | The tender security amount appears to exceed the allowed limit for the estimated cost. Review the amount in Securities and Guarantees. |
| Missing Dayworks row | Dayworks are included, but the BoQ does not contain a Dayworks row. Generate or correct the representative BoQ rows. |
| Missing lot rows | This tender is divided into lots, but no lot rows have been provided. Add at least one lot in the Lots section. |

## 12. Group-Level Status Model

Each configuration group should show a status.

| Status | Meaning | Derived from |
|---|---|---|
| `Not Started` | No relevant fields completed | Field values empty |
| `In Progress` | Some fields completed | Partial completion |
| `Needs Attention` | Validation blockers/errors exist in group | Validation messages |
| `Complete` | Required visible fields completed and no blockers for group | Field values + validation result |
| `Not Applicable` | Group inactive under current configuration | Conditional rules |

### 12.1 First Implementation Position

The first implementation may derive group status only from validation messages and simple field completeness checks. Full dynamic completeness scoring is not required.

## 13. Preview Blocking UX

### 13.1 Required Behavior

Preview generation shall be blocked if validation has blockers or errors.

If the officer attempts preview generation before validation, the system shall either:

1. run validation first and proceed only if validation passes; or
2. prompt the officer to validate configuration first.

Recommended POC behavior:

Run validation first during preview generation. If blockers/errors exist, stop and display validation messages.

### 13.2 Blocked Preview Message

```text
Tender-pack preview cannot be generated because the configuration has unresolved blockers. Review the validation messages, correct the highlighted areas, and validate again.
```

### 13.3 Passed with Warnings

If validation has warnings but no blockers/errors, preview may be generated with warning:

```text
The tender configuration has warnings. Review them before relying on the preview. Preview generation may continue for this POC.
```

## 14. Relationship to Admin Rule Trace

The officer UX shall not show full developer-style rule trace by default.

| User | Rule visibility |
|---|---|
| Procurement officer | Officer-friendly validation and consequence messages |
| Technical administrator | Full rule trace in Admin Console |
| Procurement SME/legal reviewer | May use Admin Console trace for review |

The officer UI may include a link or note:

```text
Technical rule trace is available to administrators in the STD Administration Console.
```

This link shall be restricted to users with appropriate access.

---

## 15. Save, Sync, and Validation Timing

### 15.1 Required Timing

| Event | Required behavior |
|---|---|
| Field change | Update client visibility for dependent fields |
| Save draft | Sync guided fields into `configuration_json` |
| Validate | Sync first, then run server validation |
| Generate forms | Sync/validate state must be current |
| Generate preview | Sync and validate first; block on blockers/errors |

### 15.2 Stale Validation Handling

If configuration changes after validation, the system shall mark validation as stale or reset validation status to `Not Validated`.

Recommended POC behavior:

When guided fields change after a passed validation, set `validation_status = Not Validated` on save or before next generation action.

## 17. Server Method Requirements

This UX requires the following server capabilities.

| Method / capability | Purpose |
|---|---|
| `get_officer_conditional_state(tender_name)` | Return visible/required field state for current configuration |
| `sync_officer_configuration(tender_name)` | Save guided values into canonical config |
| `validate_officer_configuration(tender_name)` | Sync and run validation |
| `get_officer_validation_feedback(tender_name)` | Return grouped officer-facing validation messages |
| `can_generate_officer_preview(tender_name)` | Return whether preview generation should proceed |
| `generate_officer_preview(tender_name)` | Validate, block if needed, then call existing preview generator |

Methods may be merged where practical, but capabilities must exist.

## 18. Client/UI Requirements

The client/UI shall:

1. update conditional visibility when controlling fields change;
2. show consequence notices near controlling fields;
3. show required indicators for visible required fields;
4. show group-level status where practical;
5. show validation messages grouped by area;
6. provide direct navigation from validation message to field/group where practical;
7. block or intercept preview generation when validation fails;
8. hide technical rule codes by default;
9. avoid raw JSON exposure in officer workflow.

---

## 19. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| Officer can see conditional guidance | Yes, for tenders they may edit/read |
| Officer cannot edit package rules | No rule editing path |
| Officer cannot bypass server validation | Preview action must validate/block server-side |
| Officer cannot publish preview | No publication action |
| Officer cannot edit generated preview HTML | Preview read-only |
| Technical trace access | Restricted to Admin Console users |

---

## 20. Error Handling Requirements

| Scenario | Required behavior |
|---|---|
| Conditional state cannot be computed | Show clear error and allow server validation to explain blockers |
| Unknown field dependency | Log/admin warning; do not crash officer UI |
| Validation method fails | Show clear validation error |
| Message cannot map to group | Show under General Validation with technical detail available |
| Preview blocked | Show blocker summary and resolution hints |
| Configuration stale | Prompt validation before generation |

## 21. Manual Verification Scenarios

### 21.1 JV Conditional Behavior

Steps:

1. Set JV allowed to Yes.
2. Confirm maximum JV members appears.
3. Generate required forms after validation.

Expected:

1. JV field appears.
2. JV form is included.
3. Officer sees explanation.

### 21.2 Tender Security Mode Switch

Steps:

1. Select Tender Security.
2. Confirm security type/amount/currency appear.
3. Generate forms.
4. Switch to Tender-Securing Declaration.
5. Revalidate and generate forms.

Expected:

1. Tender Security form appears in first mode.
2. Tender-Securing Declaration form appears in second mode.
3. Forms do not both appear incorrectly.

### 21.3 Missing Site Visit Blocker

Steps:

1. Set site visit mandatory to Yes.
2. Leave site visit date/location blank.
3. Validate.

Expected:

1. Validation blocks preview.
2. Message appears under Key Dates and Meetings.
3. Resolution hint tells officer to enter site visit date/time/location.

### 21.4 Missing Alternative Scope Blocker

Steps:

1. Allow alternatives.
2. Leave alternative scope blank.
3. Validate.

Expected:

1. Validation blocks preview.
2. Message appears under Alternatives.
3. Resolution hint tells officer to describe permitted alternative scope.

### 21.5 Preview Blocked Until Validation Passes

Steps:

1. Leave a known blocker unresolved.
2. Attempt preview generation.

Expected:

1. Preview is not generated as valid.
2. Blocker summary appears.
3. Officer is directed to validation messages.

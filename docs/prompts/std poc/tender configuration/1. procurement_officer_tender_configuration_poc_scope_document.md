## 2. Background

The STD-WORKS-POC demonstrated that a Works Standard Tender Document can be represented as a controlled, metadata-driven template package and used to drive tender creation outputs.

The STD Administration Console POC then demonstrated that the package-driven model can be surfaced through internal administration UI. The project can now inspect template package identity, sections, fields, rules, forms, sample tender scenarios, package validation, rule trace, required forms, BoQ rows, tender-pack preview, and audit values.

The next logical workstream is to test whether this controlled package model can support a procurement officer-facing workflow.

The procurement officer should not see package internals, raw JSON, rule definitions, or developer-oriented package structures. Instead, the procurement officer should experience a guided tender configuration workflow that is understandable, safe, validated, and auditable.

---

## 3. Strategic Decision

The project shall proceed to a **Procurement Officer Tender Configuration POC**.

This workstream shall test whether a procurement officer can create and configure a tender from a controlled STD template package using guided UI, validation feedback, generated required forms, representative BoQ handling, tender-pack preview, and audit outputs.

This workstream shall remain a POC. It shall not become full production tender management, bidder submission, tender publication, evaluation, award, or contract execution.

---

## 4. Core Hypothesis

The core hypothesis of this POC is:

> A procurement officer can safely create and configure a Works tender from a controlled STD template package through guided configuration screens, without editing template internals or raw JSON, and the system can validate the configuration, generate required forms, reflect structured BoQ requirements, produce a tender-pack preview, and record audit metadata.

The POC succeeds if this hypothesis is proven through a minimal, understandable, end-to-end officer workflow.

---

## 5. Product Principle

The Procurement Officer Tender Configuration POC shall implement the principle:

> Procurement officers configure safe tender parameters; they do not author or edit the underlying Standard Tender Document.

This means:

1. Officers shall not edit `package_json`.
2. Officers shall not edit `rules.json`.
3. Officers shall not edit `forms.json`.
4. Officers shall not edit legal STD clause text.
5. Officers shall not manually assemble tender documents for upload.
6. Officers shall configure guided fields exposed by the STD package.
7. The system shall validate configuration before preview generation.
8. The system shall generate required forms based on rules and configuration.
9. The system shall show audit and reproducibility metadata.
10. The system shall clearly distinguish POC preview from final legal publication.

---

## 6. Purpose of This POC

The purpose of this POC is to design and implement a minimal procurement officer-facing tender configuration workflow that proves the transition from admin-observable package engine to operational tender creation experience.

The POC shall answer these questions:

1. Can an officer select or start from an approved/imported STD template?
2. Can the officer configure tender identity, method, dates, participation, securities, qualifications, works requirements, and contract parameters through guided UI?
3. Can the system hide or show configuration fields based on rule-driven conditions?
4. Can the system validate configuration in officer-friendly language?
5. Can the officer resolve validation blockers without reading JSON or rule definitions?
6. Can the system generate a required forms checklist from configuration?
7. Can the system surface representative BoQ requirements without pretending to be a full production BoQ module?
8. Can the officer generate a tender-pack preview after validation passes?
9. Can the system show template code, package hash, configuration hash, validation status, and timestamps?
10. Can the workflow remain bounded before publication, bidder submission, evaluation, or contract execution?

---

## 7. Scope Statement

This workstream shall provide a minimal procurement officer-facing POC workflow for configuring a Works tender from an imported STD template package.

It shall include:

1. officer-facing tender creation entry point,
2. template selection from available/imported templates,
3. guided configuration UI grouped by procurement concepts,
4. rule-aware conditional field behavior,
5. officer-friendly validation feedback,
6. generated required forms checklist display,
7. POC-level BoQ handling boundary and display,
8. tender-pack preview generation,
9. preview and audit summary,
10. draft state/status indicators,
11. minimal maker/checker awareness or deferred governance statement,
12. officer workflow smoke test.

It shall not include tender publication, bidder portal, bid submission, bid opening, evaluation, award, contract execution, production BoQ authoring, PDF generation, or external integrations.

## 8. In-Scope Capabilities

### 8.1 Officer Tender Creation Entry Point

The system shall provide a clear officer-facing entry point to create a tender from an STD template.

Minimum behavior:

| Capability | Requirement |
|---|---|
| Start new tender | Officer can start a new tender configuration record |
| Select template | Officer selects an available STD template |
| Show template summary | System shows template name, category, version, source, and status |
| Prevent unsupported templates | Only allowed/imported POC templates may be used |
| Initialize defaults | System loads safe default configuration values from package |

---

### 8.2 Guided Configuration UI

The officer shall configure the tender through grouped fields, not raw JSON.

Required configuration groups:

1. Tender Identity.
2. Procurement Method and Scope.
3. Participation and Eligibility.
4. Key Dates and Meetings.
5. Tender Securities and Guarantees.
6. Alternatives.
7. Lots.
8. Qualification Requirements.
9. Works Requirements.
10. Contract/SCC Parameters.
11. Review and Audit.

Each group shall show officer-friendly labels, help text, required indicators, and validation feedback.

---

### 8.3 Conditional Field Behavior

The UI shall respond to configuration choices.

Examples:

| Officer choice | Expected behavior |
|---|---|
| JV allowed = Yes | Show maximum JV members and JV-related form impact |
| Tender security mode = Tender Security | Show tender security amount, currency, type |
| Tender security mode = Tender-Securing Declaration | Hide tender security amount fields where appropriate and show declaration impact |
| Tender scope = International | Show or explain foreign tenderer local input requirement |
| Alternatives allowed = Yes | Show alternative tender type and scope description |
| Multiple lots = Yes | Show lot-related requirements and lot evaluation method |
| Site visit mandatory = Yes | Require site visit date/time/location |
| Pre-tender meeting mandatory = Yes | Require meeting date/time/location |
| Dayworks included = Yes | Require Dayworks BoQ category in POC BoQ check |
| Provisional sums included = Yes | Require Provisional Sums BoQ category in POC BoQ check |

This POC may use client-side conditional visibility plus server-side validation. Server-side validation remains authoritative.

---

### 8.4 Officer-Friendly Validation

The system shall show validation results in language useful to procurement officers.

Validation feedback shall include:

| Field | Purpose |
|---|---|
| Severity | Info, Warning, Error, Blocker |
| Message | Clear officer-facing explanation |
| Affected configuration area | The section/group to fix |
| Affected field | Field requiring attention |
| Resolution hint | What the officer should do |
| Blocks preview | Whether preview generation is blocked |

The officer should not need to understand rule codes to correct blockers, although rule codes may be visible in technical detail.

---

### 8.5 Required Forms Checklist

The system shall show the generated required forms checklist after validation/configuration.

Minimum display:

| Column | Requirement |
|---|---|
| Display group | Show logical grouping |
| Form title | Officer-friendly form name |
| Required | Yes/No |
| Why required | Default or configuration-triggered explanation |
| Respondent type | Bidder, JV, successful bidder, etc. |
| Workflow stage | Bid submission, contract signature, etc. |
| Evidence policy | Structured response / controlled evidence posture |

This is officer-facing and should be simpler than the admin inspector.

---

### 8.6 BoQ Handling Boundary

The POC shall include BoQ awareness but shall not become a production BoQ module.

In scope:

1. Show whether BoQ is required.
2. Show representative POC BoQ rows if generated.
3. Show category coverage.
4. Warn if Dayworks or Provisional Sums rows are missing when configured.
5. Show that BoQ handling is structured system data, not an uploaded tender document.

Out of scope:

1. production BoQ editor,
2. Excel import/export,
3. bidder pricing,
4. arithmetic totals,
5. engineer's estimate,
6. contract execution measurement,
7. payment certificates.

---

### 8.7 Tender-Pack Preview

The officer shall be able to generate a tender-pack preview after validation passes.

Preview behavior:

1. Preview generation shall be blocked if validation blockers exist.
2. Preview shall clearly state that it is POC output, not final legal publication.
3. Preview shall include tender identity, TDS summary, required forms, BoQ summary, SCC summary, and audit summary as already supported by the POC renderer.
4. Officer shall not edit generated preview HTML.
5. Officer shall not publish or generate final PDF in this POC.

---

### 8.8 Audit Summary

The officer workflow shall show audit values relevant to the configured tender.

Required values:

| Audit value | Purpose |
|---|---|
| Template Code | Identifies selected template |
| Template Version | Shows template version used |
| Package Hash | Confirms package integrity |
| Configuration Hash | Confirms configured tender snapshot |
| Validation Status | Shows safety status |
| Last Validated At | Validation timestamp |
| Last Preview Generated At | Preview timestamp |
| Required Form Count | Output summary |
| BoQ Row Count | Output summary |

---

### 8.9 Minimal Status / State Indicators

The POC shall show simple officer-facing states.

Expected states:

| State | Meaning |
|---|---|
| Draft | Tender record started |
| Configuring | Officer is completing guided fields |
| Validation Failed | Blocking issues exist |
| Validated | Configuration passed validation |
| Preview Generated | POC preview generated |
| Ready for Review | Optional state indicating draft is ready for internal review |

These states do not constitute full production workflow. They are POC-level state indicators only.

## 11. User Journey Summary

The officer-facing POC shall support this journey:

```text
Start tender
→ Select STD template
→ Load safe defaults
→ Complete guided configuration groups
→ Review validation feedback
→ Resolve blockers
→ Generate required forms checklist
→ Confirm POC BoQ requirements/rows
→ Generate tender-pack preview
→ Review audit summary
→ Mark ready for review or keep draft
```

The journey shall not proceed to publication, bidder access, bid submission, or evaluation.

## 13. Required Governance and State-Transition Position

This POC shall not implement full approval governance, but it shall not ignore state transitions.

### 13.1 Minimal POC State Model

| State | Allowed actions |
|---|---|
| Draft | Select template, initialize configuration |
| Configuring | Edit guided fields, save draft |
| Validation Failed | Edit fields, revalidate |
| Validated | Generate required forms, generate preview |
| Preview Generated | Review preview, update fields if needed, regenerate |
| Ready for Review | Optional internal marker; no publication |

### 13.2 Explicitly Deferred Governance

The following are deferred:

1. formal maker/checker approval,
2. procurement head approval,
3. budget approval linkage,
4. legal approval,
5. publication approval,
6. addenda approval,
7. clarification approval,
8. template promotion lifecycle.

### 13.3 Non-Negotiable Constraint

The POC shall not create a path from officer configuration directly to public publication. Publication requires a separate workflow and governance design.

## 15. Candidate Implementation Approach

Recommended implementation approach:

1. Reuse `Procurement Tender` as the officer-facing tender record.
2. Add or refine officer-facing form sections on `Procurement Tender`.
3. Store guided field values into `configuration_json` through controlled server/client helpers.
4. Do not expose raw `configuration_json` as the primary editing surface.
5. Use existing engine validation and generation functions.
6. Add officer-friendly validation display and grouped field UI.
7. Use existing preview renderer but wrap it in officer-facing POC messaging.

Alternative:

Create a dedicated Frappe Page for guided configuration if the DocType form becomes too constrained. This should be considered only after the grouped DocType approach is tested.

## 18. Risks and Controls

| Risk | Impact | Control |
|---|---|---|
| Officer UI exposes too much technical detail | Poor usability and risk of misconfiguration | Guided grouped fields; hide raw JSON |
| Workflow drifts into publication | Governance/legal risk | Explicit boundary: preview only, no publication |
| Validation messages are too technical | Officer cannot resolve blockers | Officer-friendly message layer |
| Conditional fields become inconsistent with server validation | Incorrect configuration | Server validation remains authoritative |
| Required forms are misunderstood as upload list | Undermines structured response strategy | Show evidence policy and structured-response posture |
| BoQ POC is mistaken for production BoQ | Scope/legal risk | Clear BoQ boundary warnings |
| Governance deferred too loosely | Future rework | Minimal state model and explicit deferred governance list |
| Template internals become editable | Legal/control risk | Package internals read-only; admin-only inspection |

## 2. Purpose

The purpose of this document is to define the procurement officer-facing user journey and information architecture for the Procurement Officer Tender Configuration POC.

The design shall translate the validated STD package-driven model into a guided officer workflow. The officer shall be able to create and configure a draft Works tender from a controlled STD template package without seeing package internals, editing raw JSON, or manually assembling tender documents.

This document defines the screens, navigation, workflow sequence, information hierarchy, state indicators, user actions, and data responsibilities required before implementation.

---

## 3. Design Principle

The officer-facing information architecture shall implement the principle:

> The procurement officer sees a guided tender configuration workflow; the system hides template mechanics but preserves validation, traceability, and auditability.

This means:

1. officer screens shall use procurement language, not package-engine language;
2. package internals shall remain hidden from ordinary officer workflow;
3. configuration shall be grouped into logical tender sections;
4. validation shall be surfaced where the officer can act on it;
5. generated forms, BoQ status, preview, and audit values shall be visible;
6. the workflow shall stop before publication, bidder submission, evaluation, award, and contract execution.

## 5. Primary User Journey

The primary officer journey shall be:

```text
Open officer tender workspace
→ Start new tender
→ Select STD template
→ Review template summary
→ Initialize guided configuration
→ Complete tender identity
→ Complete procurement method and scope
→ Complete participation and eligibility
→ Complete dates and meetings
→ Complete securities and guarantees
→ Complete alternatives, if applicable
→ Complete lots, if applicable
→ Complete qualification requirements
→ Complete works requirements
→ Complete contract/SCC parameters
→ Validate configuration
→ Resolve blockers
→ Generate required forms checklist
→ Review BoQ status and representative rows
→ Generate tender-pack preview
→ Review preview and audit summary
→ Mark ready for internal review or keep draft
```

The workflow shall not continue to public publication, bidder access, bid submission, bid opening, evaluation, award, or contract execution.

## 7. Recommended First Implementation Shape

The first implementation shall prioritize speed, tractability, and minimal scope.

Recommended approach:

| Surface | Responsibility |
|---|---|
| `Procurement Tender` List View | Officer sees draft tenders and statuses |
| `Procurement Tender` Form | Main officer configuration workspace |
| Guided form sections | Officer enters safe configuration values |
| Client-side helpers | Show/hide fields and group progress indicators |
| Server methods | Sync guided fields to configuration JSON and validate |
| Existing child tables | Lots, BoQ, required forms, validation messages |
| Existing preview field | Tender-pack preview display |
| Admin Console | Troubleshooting and package internals, not ordinary officer UI |

The first implementation should avoid a large custom page unless Frappe form ergonomics fail.

## 12. Information Hierarchy

The officer workflow shall prioritize information in this order:

1. tender identity and selected template;
2. current workflow/status state;
3. next required action;
4. guided configuration groups;
5. validation blockers and resolution hints;
6. required forms checklist;
7. BoQ status;
8. preview output;
9. audit/reproducibility values;
10. technical details, only if needed.

Raw JSON and package internals shall not appear in the normal officer information hierarchy.

## 13. Conditional Visibility Model

The officer UI shall support conditional visibility derived from package rules and known field dependencies.

### 13.1 Required Conditional Examples

| Condition | Show / require |
|---|---|
| `PARTICIPATION.JV_ALLOWED = true` | Show max JV members and JV form impact |
| `SECURITY.TENDER_SECURITY_MODE = TENDER_SECURITY` | Show tender security type, amount, currency |
| `SECURITY.TENDER_SECURITY_MODE = TENDER_SECURING_DECLARATION` | Show declaration explanation; hide tender security amount where appropriate |
| `METHOD.TENDER_SCOPE = INTERNATIONAL` | Show foreign tenderer/local input explanation |
| `ALTERNATIVES.ALTERNATIVE_TENDERS_ALLOWED = true` | Show alternative type and scope description |
| `LOTS.MULTIPLE_LOTS_ALLOWED = true` | Show lot table and lot evaluation/award method |
| `MEETINGS.SITE_VISIT_MANDATORY = true` | Show site visit date/time/location |
| `MEETINGS.PRE_TENDER_MEETING_MANDATORY = true` | Show meeting date/time/location |
| `WORKS.DAYWORKS_INCLUDED = true` | Show Dayworks BoQ requirement |
| `WORKS.PROVISIONAL_SUMS_INCLUDED = true` | Show Provisional Sums BoQ requirement |

### 13.2 Authority Rule

Client-side visibility is convenience only. Server-side validation remains authoritative.

## 17. Preview Panel

### 17.1 Purpose

The Preview Panel shall show the generated tender-pack preview after validation passes.

### 17.2 Required Behavior

1. Preview is generated only after validation passes or passes with acceptable warnings.
2. Preview generation is blocked by blockers/errors.
3. Preview is read-only.
4. Preview is clearly labelled POC output.
5. No PDF or publication action exists.

### 17.3 Required Empty State

```text
No tender-pack preview has been generated. Complete configuration, validate the tender, resolve blockers, then generate preview.
```

## 19. Minimal State Indicators

The officer workflow shall show simple state indicators.

| State | Trigger | Officer meaning |
|---|---|---|
| Draft | Tender created | Configuration has started |
| Configuring | Fields are being completed | Continue completing sections |
| Validation Failed | Validation has blockers/errors | Fix highlighted issues |
| Validated | Validation passed | Outputs may be generated |
| Preview Generated | Preview exists | Review generated tender pack |
| Ready for Review | Officer marks draft ready | Internal review can begin later |

These are POC-level states only, not production governance workflow.

## 20. Required User Journeys

### 20.1 Journey A — Create Valid Primary Tender Configuration

1. Officer starts new tender.
2. Officer selects STD-WORKS-POC template.
3. System initializes guided configuration.
4. Officer completes all groups using valid primary Works values.
5. Officer validates configuration.
6. System shows passed validation.
7. Officer generates required forms.
8. Officer reviews BoQ status.
9. Officer generates preview.
10. Officer reviews audit summary.

Expected result:

Valid configured draft tender with preview generated.

---

### 20.2 Journey B — Resolve Validation Blocker

1. Officer sets site visit mandatory but leaves site visit date/location blank.
2. Officer validates configuration.
3. System shows blocker in Key Dates and Meetings group.
4. Officer enters missing site visit data.
5. Officer revalidates.
6. System passes validation.

Expected result:

Officer can resolve blocker without understanding rule internals.

---

### 20.3 Journey C — Tender Security Mode Switch

1. Officer selects Tender Security.
2. System shows tender security fields and generated form impact.
3. Officer switches to Tender-Securing Declaration.
4. System hides/unrequires tender security amount fields where appropriate.
5. Required forms checklist switches from Tender Security form to Tender-Securing Declaration form.

Expected result:

Officer sees understandable effect of security mode choice.

---

### 20.4 Journey D — International Tender Configuration

1. Officer selects International tender scope.
2. System shows local input / foreign tenderer requirement explanation.
3. Officer validates and generates forms.
4. Required forms include foreign tenderer local input form.

Expected result:

Officer sees international-scope consequences.

---

### 20.5 Journey E — Preview Blocked Until Validation Passes

1. Officer leaves required fields incomplete.
2. Officer attempts preview generation.
3. System blocks preview.
4. System directs officer to validation messages.

Expected result:

Unsafe/incomplete configuration cannot produce a valid-looking preview.

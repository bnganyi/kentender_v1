## 2. Purpose

The purpose of this document is to define the procurement officer-facing Required Forms Checklist UX for the Procurement Officer Tender Configuration POC.

The Required Forms Checklist shall show the officer which bidder and downstream contract forms the system has generated from the selected STD template, tender configuration, and rule engine. It shall explain the checklist in procurement language without exposing the officer to internal rule mechanics.

This UX is central to the broader system objective: replacing uncontrolled document uploads and manually assembled tender packs with structured, system-generated requirements.

---

## 3. Design Principle

The Required Forms Checklist UX shall implement the principle:

> The procurement officer should understand which forms are required and why, without manually assembling or uploading the tender document package.

This means:

1. forms are generated by the system from controlled template metadata and configuration;
2. the officer sees a readable checklist, not raw `forms.json`;
3. each form has a plain-language reason;
4. each form shows who must complete it and when;
5. evidence/upload posture is expressed clearly;
6. conditional form changes are visible after configuration changes;
7. form generation is tied to validation and audit state;
8. the checklist remains read-only as generated output, except through configuration changes that legitimately affect it.

## 6. Required Forms Checklist Purpose

The officer-facing checklist shall answer these questions:

1. Which forms will bidders or successful bidders be required to provide?
2. Which forms are always required under the selected STD template?
3. Which forms are required because of this tender's configuration?
4. Which forms apply at bid submission stage?
5. Which forms apply later at contract signature or post-award stage?
6. Which forms are structured system-response forms?
7. Which forms may require controlled evidence or issuer verification later?
8. Has the checklist changed because the officer changed configuration values?
9. Is the checklist ready to be included in the tender-pack preview?

---

## 7. Data Sources

The checklist shall use existing STD-WORKS-POC structures.

| Data | Source | Purpose |
|---|---|---|
| Form catalogue | `STD Template.package_json.forms` | Source form metadata |
| Activation rules | `STD Template.package_json.rules` | Determine conditional forms |
| Tender configuration | `Procurement Tender.configuration_json` | Input to rules |
| Generated checklist | `Tender Required Form` child table | Officer-facing generated output |
| Validation status | `Procurement Tender.validation_status` and messages | Determine whether generation is safe/current |
| Configuration hash | `Procurement Tender.configuration_hash` | Determine traceability/staleness |

The officer shall not edit the form catalogue or activation rules.

## 8. Checklist Generation Timing

### 8.1 Required Generation Points

The checklist may be generated:

| Trigger | Behavior |
|---|---|
| Officer clicks `Generate Required Forms` | Sync configuration, run or check validation, generate checklist |
| Officer runs full validation/generation path | Checklist generated after validation passes or passes with acceptable warnings |
| Officer changes configuration after checklist generation | Checklist marked stale until regenerated |
| Officer generates preview | System must ensure checklist is current or regenerate/check before preview |

### 8.2 Recommended POC Behavior

For the POC:

1. `Generate Required Forms` shall sync latest guided fields into `configuration_json`.
2. It shall run validation or require a current validation result.
3. If blockers/errors exist, the system may still show expected form impacts, but it shall not mark the checklist as final/current.
4. If validation passes, it shall populate `Tender Required Form` rows and update checklist summary.
5. If officer changes relevant configuration later, checklist status shall become `Stale` or `Needs Regeneration`.

---

## 9. Checklist Status Model

The checklist shall show a status.

| Status | Meaning |
|---|---|
| `Not Generated` | No required forms have been generated for this tender |
| `Generated` | Checklist exists and is current with latest validated configuration |
| `Generated with Warnings` | Checklist exists but validation has warnings |
| `Blocked` | Checklist cannot be treated as final because validation blockers exist |
| `Possibly Stale` | Configuration changed after generation or validation hash is missing/mismatched |
| `Unknown` | System cannot determine checklist state |

The first implementation may use conservative stale detection based on last modified time, validation status, and configuration hash presence.

The checklist view shall show summary metrics.

| Summary item | Officer-facing meaning | Required |
|---|---|---:|
| Checklist Status | Whether forms are current and usable | Yes |
| Required Form Count | Total generated required forms | Yes |
| Bid Submission Forms | Forms expected from bidders at bid submission stage | Yes |
| Contract Signature Forms | Forms expected from successful bidder/contract stage | Yes |
| Default Required Forms | Forms required for all tenders using this STD | Yes |
| Configuration-Triggered Forms | Forms required because of officer choices | Yes |
| Structured Response Forms | Forms expected to be captured as system data | Yes |
| Controlled Evidence Forms | Forms or instruments that may require controlled evidence later | Yes |
| Last Generated At | Timestamp when checklist was generated | Recommended |
| Configuration Hash | Snapshot used for generation | Recommended |

## 11. Main Checklist Table

The officer-facing checklist table shall include these columns.

| Column | Officer-facing label | Required | Notes |
|---|---|---:|---|
| `display_group` | Group | Yes | Example: Bidder Forms, Security Forms |
| `form_title` | Form | Yes | Officer-friendly title |
| `required_because` | Required because | Yes | Plain-language reason |
| `respondent_type` | Completed by | Yes | Bidder, JV, successful bidder |
| `workflow_stage` | Stage | Yes | Bid submission, contract signature, post-award |
| `evidence_policy_label` | Evidence approach | Yes | Plain-language evidence posture |
| `status` | Status | Recommended | Required/current/stale where useful |
| `notes` | Notes | Optional | POC limitation or guidance |

Internal fields such as `form_code`, `activation_source`, and rule codes may be available in a technical details expansion but shall not dominate the officer-facing table.

## 12. Plain-Language Required-Because Mapping

The checklist shall translate default and rule-activated forms into officer-readable reasons.

### 12.1 Default Required Forms

Default form reason:

```text
Required for all tenders using this Standard Tender Document.
```

### 12.2 Configuration-Triggered Forms

Required mapping examples:

| Form | Plain-language reason |
|---|---|
| `FORM_TENDER_SECURITY` | Tender security is selected for this tender. |
| `FORM_TENDER_SECURING_DECLARATION` | Tender-securing declaration is selected instead of tender security. |
| `FORM_JV_INFORMATION` | Joint ventures are allowed for this tender. |
| `FORM_ALTERNATIVE_TECHNICAL_PROPOSAL` | Alternative tenders are allowed and must be submitted using the required format. |
| `FORM_FOREIGN_TENDERER_LOCAL_INPUT` | This tender has international scope and requires foreign tenderer/local input information where applicable. |
| `FORM_RETENTION_MONEY_SECURITY` | Retention money security is required at contract stage. |
| `FORM_PERFORMANCE_SECURITY` | Performance security is required from the successful bidder. |
| `FORM_ADVANCE_PAYMENT_SECURITY` | Advance payment security is required where advance payment applies. |
| `FORM_BENEFICIAL_OWNERSHIP_DISCLOSURE` | Beneficial ownership disclosure is required for this tender. |
| `FORM_PERSONNEL_SCHEDULE` | Key personnel information is required for qualification assessment. |
| `FORM_EQUIPMENT_SCHEDULE` | Equipment information is required for qualification assessment. |

If no specific mapping exists, fallback reason:

```text
Required by the selected template and current tender configuration.
```

## 13. Evidence Policy Display

The officer-facing checklist shall explain evidence posture in plain language.

| Internal evidence policy | Officer-facing display |
|---|---|
| `STRUCTURED_ONLY` | Structured response in the system; routine document upload should not be required. |
| `STRUCTURED_WITH_OPTIONAL_EVIDENCE` | Structured response is primary; controlled evidence may be requested where needed. |
| `STRUCTURED_WITH_ISSUER_VERIFICATION` | Structured response is primary; issuer or instrument verification may be required later. |
| `POST_AWARD_SECURITY_INSTRUMENT` | Applies after award or contract stage; controlled security instrument handling may be required. |

If policy value is unknown, display:

```text
Evidence approach not classified in the current POC package.
```

This shall produce a warning for admin review but shall not necessarily block officer workflow.

## 14. Required Form Groups

The checklist shall group forms into officer-friendly groups.

| Group | Meaning |
|---|---|
| Core Bidder Forms | Forms normally required from every bidder |
| Eligibility and Declarations | Declarations, confidential questionnaire, independent tender determination |
| Security Forms | Tender security or tender-securing declaration |
| Technical and Qualification Forms | Technical proposal, experience, financial capacity, personnel, equipment |
| Special Participation Forms | JV, foreign tenderer/local input, alternatives |
| Beneficial Ownership | Beneficial ownership disclosure |
| Contract Stage Forms | Performance security, advance payment security, retention money security |

The exact grouping may follow `forms.json.checklist_display.display_group` where already defined, with officer-friendly labels added where necessary.

## 15. Conditional Form Behavior

The checklist must reflect configuration choices.

### 15.1 Tender Security Mode

| Configuration | Expected checklist behavior |
|---|---|
| Tender Security | Include Tender Security form; exclude Tender-Securing Declaration form |
| Tender-Securing Declaration | Include Tender-Securing Declaration form; exclude Tender Security form |
| None | Exclude both unless rules specify otherwise; show warning if legally/policy-sensitive |

### 15.2 Joint Ventures

| Configuration | Expected checklist behavior |
|---|---|
| JV allowed | Include JV Information form |
| JV not allowed | Exclude JV Information form |

### 15.3 Alternatives

| Configuration | Expected checklist behavior |
|---|---|
| Alternatives allowed | Include Alternative Technical Proposal form |
| Alternatives not allowed | Exclude Alternative Technical Proposal form |

### 15.4 International Scope

| Configuration | Expected checklist behavior |
|---|---|
| International scope | Include Foreign Tenderer Local Input form where applicable |
| National scope | Exclude Foreign Tenderer Local Input form unless separately required |

### 15.5 Retention Money Security

| Configuration | Expected checklist behavior |
|---|---|
| Retention money security required | Include Retention Money Security form |
| Not required | Exclude Retention Money Security form |

## 17. Empty State UX

If the checklist has not been generated, show:

```text
No required forms checklist has been generated yet. Complete the tender configuration, validate it, then generate the required forms checklist.
```

Required action shown:

```text
Generate Required Forms
```

If validation has blockers, show:

```text
The forms checklist may change after validation blockers are resolved. Review and fix validation messages before relying on this checklist.
```

---

## 18. Stale Checklist UX

If configuration changes after the checklist is generated, show:

```text
This required forms checklist may be out of date because the tender configuration has changed. Regenerate the checklist before generating the tender-pack preview.
```

Required behavior:

1. checklist remains visible for reference;
2. status shows `Possibly Stale`;
3. preview generation shall require current validation/checklist state or regenerate/check it;
4. officer is prompted to regenerate forms.

---

## 19. Blocked Checklist UX

If validation blockers exist, the checklist may be shown as provisional but shall not be treated as final.

Message:

```text
The tender has unresolved validation blockers. The checklist shown may be provisional and must be regenerated after blockers are resolved.
```

The system shall not allow a final-looking preview to be generated from blocked state.

---

## 20. Checklist Relationship to Preview

The tender-pack preview shall include or reflect the required forms checklist.

Before preview generation, the system shall ensure:

1. configuration is synced;
2. validation has no blockers/errors;
3. required forms are generated or regenerated if stale;
4. required form count is available;
5. preview audit summary includes required form count.

If checklist generation fails, preview generation shall not proceed as valid.

---

## 21. Checklist Relationship to Admin Console

The officer checklist shall be simplified.

| Officer UI | Admin Console |
|---|---|
| Shows form title and plain-language reason | Shows form code, activation source, rules, evidence policy |
| Shows evidence approach in officer language | Shows internal evidence policy and metadata |
| Shows current generated checklist | Shows expected/observed comparison and technical trace |
| Hides rule internals | Exposes rule trace for administrators |

Officer UI may include an admin-only link:

```text
View technical form activation details in Admin Console
```

This link shall be visible only to users with appropriate permission.

---

## 22. Required Actions

| Action | Purpose | Writes data? |
|---|---|---:|
| Generate Required Forms | Generate checklist from current saved configuration | Yes |
| Regenerate Required Forms | Re-run generation after configuration changes | Yes |
| View Form Checklist | Show generated checklist | No |
| View Technical Details | Admin-only details, if available | No |

There shall be no action for officer to manually add, remove, or edit required forms in this POC.

---

## 23. Server Method Requirements

This UX requires the following capabilities.

| Method / capability | Purpose |
|---|---|
| `generate_officer_required_forms(tender_name)` | Sync, validate/check, generate required forms checklist |
| `get_officer_required_forms_checklist(tender_name)` | Return officer-facing checklist summary and rows |
| `get_required_form_reason(form_code, activation_source, config)` | Resolve plain-language reason |
| `is_required_forms_checklist_stale(tender_name)` | Determine whether checklist may be stale |
| `ensure_required_forms_current_for_preview(tender_name)` | Used before preview generation |

Methods may be merged, but capabilities must exist.

---

## 24. Required Return Shape

`get_officer_required_forms_checklist(tender_name)` shall return a structured object.

```json
{
  "ok": true,
  "tender": "...",
  "checklist_status": "Generated",
  "summary": {
    "required_form_count": 15,
    "bid_submission_forms": 0,
    "contract_signature_forms": 0,
    "default_required_forms": 0,
    "configuration_triggered_forms": 0,

    {
      "display_group": "Core Bidder Forms",
      "form_title": "Form of Tender",
      "required_because": "Required for all tenders using this Standard Tender Document.",
      "respondent_type": "Bidder",
```

Technical details may be omitted from ordinary officer display but may exist in returned data for debugging.

## 25. Client/UI Requirements

The UI shall:

1. show checklist status and summary;
2. show empty, stale, blocked, and generated states clearly;
3. show main checklist table using officer-friendly labels;
4. show plain-language required-because reasons;
5. show evidence approach labels;
6. show regenerate action when stale;
7. prevent manual officer editing of generated checklist rows;
8. link checklist status to preview readiness;
9. avoid exposing form codes/rule codes as the main officer-facing content.

## 26. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| Generate forms | Officer must have write permission on tender |
| View checklist | Officer must have read permission on tender |
| Manual checklist editing | Not allowed in officer POC |
| Form catalogue editing | Not allowed |
| Rule editing | Not allowed |
| Technical details | Restricted or secondary |
| Preview generation | Requires current checklist or system-generated refresh |

---

## 27. Error Handling Requirements

| Scenario | Required behavior |
|---|---|
| Tender missing | Clear error |
| User lacks permission | Permission error |
| Template missing | Show template linkage error |
| Package forms missing | Show package/admin error |
| Required forms generation fails | Show error and do not claim checklist is current |
| Unknown form code | Show warning and mark form metadata incomplete |
| Unknown evidence policy | Show fallback evidence message and admin warning |
| Checklist empty after generation | Show warning; likely package/rule issue |
| Checklist stale | Show regenerate prompt |

## 28. Manual Verification Scenarios

### 28.1 Primary Sample Checklist

Steps:

1. Configure primary sample-equivalent tender.
2. Validate.
3. Generate required forms.
4. Open checklist.

Expected:

1. Checklist status is `Generated`.
2. Required form count is 15.
3. Tender Security form is present.
4. Tender-Securing Declaration is absent.
5. Foreign Tenderer Local Input is absent.
6. Retention Money Security is absent.
7. Required-because reasons are readable.
8. Evidence approach is visible.

### 28.2 Tender-Securing Declaration Switch

Steps:

1. Select Tender Security mode and generate forms.
2. Switch to Tender-Securing Declaration.
3. Revalidate/regenerate forms.

Expected:

1. Tender Security form is removed.
2. Tender-Securing Declaration form appears.
3. Reason explains the switch.

### 28.3 International Scope

Steps:

1. Set tender scope to International.
2. Validate and generate forms.

Expected:

1. Foreign Tenderer Local Input form appears.
2. Reason explains international-scope requirement.

### 28.4 Alternatives Disabled

Steps:

1. Set alternatives allowed to false.
2. Generate forms.

Expected:

1. Alternative Technical Proposal form is absent.
2. Checklist remains otherwise valid.

### 28.5 Stale Checklist

Steps:

1. Generate forms.
2. Change a controlling field such as tender security mode or JV allowed.
3. Save.

Expected:

1. Checklist is marked stale or needs regeneration.
2. Officer is prompted to regenerate before preview.

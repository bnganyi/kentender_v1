## 2. Purpose

The purpose of this document is to define the formal decision point after the Procurement Officer Tender Configuration POC.

The decision determines the next highest-impact workstream after the project has tested whether a procurement officer can configure a draft Works tender from a controlled STD template package using guided UI, validation, generated required forms, representative BoQ handling, preview, and audit outputs.

This document does not design the next workstream. It defines the evidence required, the decision options, the go/no-go criteria, residual risks, and recommended direction.

---

## 3. Decision Principle

The project shall move to the next workstream only after confirming that the procurement officer workflow is sufficiently safe, understandable, bounded, and technically coherent.

The decision shall follow this principle:

> Do not proceed downstream to publication, bidder access, evaluation, award, or contract execution until the officer-facing tender configuration workflow has proven that controlled STD packages can be used safely without exposing template internals or creating uncontrolled documents.

---

## 4. Decision Question

The decision question is:

> After the Procurement Officer Tender Configuration POC, what is the highest-impact and safest next workstream for the public e-Procurement system?

Possible decisions are:

| Decision | Meaning |
|---|---|
| `Proceed to Tender Publication and Internal Approval POC` | Design controlled approval and publication readiness after officer preview |
| `Proceed to Structured Bidder Response POC` | Begin bidder-facing structured response and submission design |
| `Proceed to Works BoQ and Pricing POC` | Deepen Works BoQ, pricing, and bid pricing structures before bidder workflow |
| `Proceed to STD Governance and Lifecycle POC` | Strengthen template versioning, approval, activation, retirement, and audit governance |
| `Extend Officer Configuration POC` | Improve officer workflow before moving downstream |
| `Stop / Reconsider` | Pause because the approach is not sufficiently proven |

## 10. Candidate Next Workstreams

### 10.1 Option A — Tender Publication and Internal Approval POC

Purpose:

Design the controlled transition from officer-generated preview to reviewed, approved, and publication-ready tender package.

This option includes:

1. internal review workflow;
2. maker/checker state transitions;
3. approval gates;
4. publication readiness checks;
5. final tender-pack lock before publication;
6. audit trail of approval decisions;
7. separation between preview and public issuance.

This option still does not need to include full bidder submission.

High-impact reason:

The system cannot safely expose tenders to bidders until internal approval and publication controls exist.

---

### 10.2 Option B — Structured Bidder Response POC

Purpose:

Design bidder-facing structured response, form completion, and controlled submission using generated forms checklist.

This option includes:

1. bidder tender access;
2. structured form responses;
3. controlled evidence posture;
4. bid submission draft/final states;
5. basic submission completeness checks;
6. no evaluation yet.

High-impact reason:

It directly advances the goal of eliminating uncontrolled bidder document uploads.

Risk:

It may be premature if publication and approval controls are not yet defined.

---

### 10.3 Option C — Works BoQ and Pricing POC

Purpose:

Deepen Works-specific BoQ authoring, bidder pricing, pricing validation, and pricing schedule structures.

This option includes:

1. production-grade BoQ model exploration;
2. item structure;
3. lot linkage;
4. bidder pricing input model;
5. arithmetic extension;
6. provisional sums/dayworks handling;
7. price summary;
8. evaluation-ready pricing outputs.

High-impact reason:

Works procurement cannot be production-ready without a serious BoQ/pricing model.

Risk:

It is a large technical and domain-specific workstream that can delay full tender workflow learning.

---

### 10.4 Option D — STD Governance and Lifecycle POC

Purpose:

Design governance for STD template package import, approval, activation, versioning, retirement, audit, and controlled update handling.

This option includes:

1. template version approval;
2. activation/deactivation;
3. allowed-for-tender-creation status;
4. hash/version audit;
5. restricted re-import;
6. template retirement;
7. migration impact analysis for existing tenders.

High-impact reason:

Production use requires strong template governance.

Risk:

It does not advance officer-to-bidder operational workflow immediately.

---

### 10.5 Option E — Extend Officer Configuration POC

Purpose:

Improve the officer workflow before moving downstream.

Use this if:

1. smoke test shows usability gaps;
2. validation feedback is not clear enough;
3. conditional behavior is unreliable;
4. preview/audit is difficult to understand;
5. state transitions are confusing.

## 11. Recommended Decision Logic

The recommended next workstream depends on the smoke test outcome.

| Smoke outcome | Recommended decision |
|---|---|
| Officer POC passes cleanly | Proceed to Tender Publication and Internal Approval POC |
| Officer POC passes but form/upload-elimination learning is urgent | Proceed to Structured Bidder Response POC with publication controls explicitly bounded |
| Officer POC passes but BoQ is the main risk | Proceed to Works BoQ and Pricing POC |
| Officer POC passes but template lifecycle risk is highest | Proceed to STD Governance and Lifecycle POC |
| Officer POC has major usability/validation gaps | Extend Officer Configuration POC |
| Officer POC fails safety or architecture checks | Stop / Reconsider |

## 12. Recommended Next Workstream

If the Procurement Officer Tender Configuration POC passes without unresolved blocker or major issues, the recommended next workstream is:

```text
Tender Publication and Internal Approval POC
```

Reason:

The officer workflow will have proven draft tender configuration and preview generation. The next highest-risk boundary is the transition from internal draft/preview to approved publication-ready tender. That boundary is legal, procedural, and system-critical. It must be designed before exposing bidder access or bid submission.

This is safer than jumping directly to bidder response because bidder access depends on knowing exactly what has been approved, locked, published, and made available.

---

## 13. Why Not Bidder Response Immediately

Structured Bidder Response is strategically critical, but it should normally follow publication/approval readiness.

Reason:

1. bidders should not access draft or unapproved tender packs;
2. bidder response requirements depend on the final locked tender package;
3. addenda, clarifications, publication dates, and submission windows require publication state controls;
4. legal defensibility requires audit trail from officer draft to approved tender to public issue;
5. bid submission integrity depends on a known published tender baseline.

Bidder response may be started only if explicitly bounded to a mock published tender package, but that would be less legally coherent than designing publication controls first.

---

## 14. Recommended Scope of Tender Publication and Internal Approval POC

If selected, the next workstream should be bounded to:

### 14.1 In Scope

1. Review-ready tender package state.
2. Internal approval workflow.
3. Maker/checker review.
4. Approval comments and decision log.
5. Locking tender configuration after approval.
6. Publication readiness checklist.
7. Approved tender package snapshot.
8. Publication status model.
9. Audit trail from draft to approved publication-ready package.
10. POC publication record or internal published state.

### 14.2 Out of Scope Initially

1. Public portal display.
2. Bidder registration/access.
3. Bid submission.
4. Bid opening.
5. Evaluation.
6. Award.
7. Contract execution.
8. Final PDF generation, unless explicitly approved as a narrow rendering subtask.
9. External integrations.

---

## 15. Governance and State-Transition Requirements for Next Workstream

The next workstream must not defer governance as an afterthought.

At minimum, it shall define:

1. Draft → Ready for Review;
2. Ready for Review → Returned for Correction;
3. Ready for Review → Approved;
4. Approved → Publication Ready;
5. Publication Ready → Published Internally or Publicly, depending on POC boundary;
6. Approved/Published locks;
7. who can approve;
8. who can return;
9. who can publish;
10. what fields become immutable;
11. what audit entries are recorded;
12. how configuration/package hashes are snapshotted;
13. how a changed tender requires re-approval.

---

## 16. Residual Risks Before Next Workstream

Even if the officer POC passes, these risks remain.

| Risk | Impact | Recommended handling |
|---|---|---|
| No formal approval workflow yet | Cannot legally publish | Address in publication/internal approval POC |
| No final publication baseline snapshot | Bidder response would lack authoritative baseline | Address before bidder portal |
| PDF/final tender document not yet designed | Publication may need rendering strategy | Keep POC publication internal or HTML snapshot first |
| BoQ remains representative | Works production readiness incomplete | Decide whether to branch into BoQ POC later |
| Template lifecycle governance remains minimal | Production rollout risk | Schedule STD Governance POC before production |
| Bidder structured response pending | Upload-elimination not fully realized yet | Start after publication baseline exists |

## 19. Recommended Decision Statement if Smoke Test Passed

If the Officer POC smoke test passes without unresolved blocker or major issues, use this decision statement:

```text
Proceed to the Tender Publication and Internal Approval POC. The Procurement Officer Tender Configuration POC has proven that a procurement officer can configure a draft Works tender from a controlled STD template package through guided UI, validation, generated required forms, representative BoQ handling, preview generation, and audit outputs. The next system-critical boundary is the controlled transition from draft preview to internally approved and publication-ready tender package.
```

Recommended conditions:

1. publication must not expose bidder access until bidder-facing workflow is separately designed;
2. approval state transitions must be defined before implementation;
3. approved tender package snapshot and hash/audit strategy must be included;
4. configuration changes after approval must force re-validation and re-approval;
5. PDF/final rendering shall remain out of scope unless explicitly scoped;
6. public portal publication shall remain out of scope unless explicitly scoped.

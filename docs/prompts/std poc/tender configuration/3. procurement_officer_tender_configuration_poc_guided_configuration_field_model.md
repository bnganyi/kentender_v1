## 2. Purpose

The purpose of this document is to define how STD package configuration fields shall be translated into a procurement officer-facing guided configuration model.

The officer shall not edit raw `configuration_json`. The officer shall interact with business-friendly field groups, labels, help text, conditional fields, validation indicators, and guided actions. The system shall then map the officer's inputs into the controlled configuration structure used by the STD engine.

This specification defines:

1. officer-facing field groups,
2. field mapping principles,
3. field metadata requirements,
4. required field groups and field inventory,
5. conditional field dependencies,
6. save/sync behavior,
7. validation relationship,
8. implementation boundaries.

---

## 3. Design Principle

The guided configuration field model shall implement the principle:

> Procurement officers configure procurement decisions and tender parameters; the system translates those values into the controlled STD configuration model.

This means:

1. officers see procurement concepts, not package schema mechanics;
2. field codes remain stable internally;
3. labels, help text, defaults, and options come from the controlled package where possible;
4. officer input is validated before preview generation;
5. conditional fields are visible only when relevant;
6. server-side validation remains authoritative;
7. raw JSON remains hidden from ordinary officer workflow.

## 6. Field Model Overview

The guided configuration model shall consist of:

| Layer | Description |
|---|---|
| STD package field definition | Controlled field metadata in `fields.json` |
| Officer field group | Business grouping shown in UI |
| Officer input field | UI control bound to an internal field code |
| Configuration store | `Procurement Tender.configuration_json` |
| Validation engine | Server-side rules from `rules.json` |
| Generated outputs | Required forms, BoQ status, preview, audit |

The field model shall preserve the package field code as the canonical key. Officer labels may be friendlier, but internal mappings must remain deterministic.

## 8. Field Metadata Requirements

Each officer-facing field shall have a minimum metadata model.

| Metadata field | Purpose | Required |
|---|---|---:|
| `field_code` | Stable internal configuration key | Yes |
| `group_code` | Officer UI group | Yes |
| `label` | Officer-facing label | Yes |
| `field_type` | UI/data type | Yes |
| `required_mode` | Always, conditional, derived, optional | Yes |
| `default_value` | Initial/default value where applicable | No |
| `options` | Controlled options for select fields | Yes, for select fields |
| `help_text` | Officer guidance | Recommended |
| `placeholder` | UI hint | Optional |
| `conditional_visibility` | Visibility dependency | Conditional fields only |
| `legal_sensitivity` | Whether field affects legal-sensitive output | Recommended |
| `validation_notes` | Notes for validation/resolution | Optional |
| `source_field_code` | Package field code if different from UI binding | Optional |

The POC may derive much of this from `fields.json`. Where `fields.json` lacks display metadata, the implementation may provide a small hard-coded officer display map for the POC, but it must not replace the package field codes as the source of truth.

## 13. Defaults and Initialization

When a procurement officer creates a tender from an STD template, the system shall initialize configuration values.

### 13.1 Source of Defaults

Defaults may come from:

1. `fields.json` default values;
2. `sample_tender.json` for POC sample/demo initialization;
3. safe system defaults for derived fields;
4. empty values for officer-supplied fields.

### 13.2 Recommended POC Approach

For the officer POC, use a controlled initializer:

1. Select template.
2. Initialize derived system fields from the template.
3. Initialize known safe defaults for field types and common values.
4. Optionally provide a `Load POC Sample Values` action for demonstration.
5. Do not silently load sample values into a real officer-created tender unless the action is explicitly labelled as POC/sample.

This avoids confusing sample data with real officer input.

---

## 14. Configuration Storage Model

The system shall persist guided values into `Procurement Tender.configuration_json`.

### 14.1 Required Behavior

1. Officer fields map to stable `field_code` keys.
2. Saving guided fields updates `configuration_json` deterministically.
3. Derived audit fields are written by server methods, not manually by officer.
4. Raw JSON is not the primary editing surface.
5. Server-side validation uses the saved `configuration_json`.
6. Generated outputs are based on saved configuration, not unsaved UI state.

### 14.2 Recommended Implementation Options

| Option | Description | Recommendation |
|---|---|---|
| Direct DocType fields | Add visible DocType fields mapped to config | Good for core fields if manageable |
| Dynamic child table | Store field code/value rows | Flexible but may be less officer-friendly |
| Custom page with generated controls | Most flexible but higher effort |
| Hybrid | Core DocType fields plus config JSON sync | Recommended for POC |

Recommended POC approach:

Use a hybrid model: add or use officer-facing DocType fields where practical, and maintain `configuration_json` as canonical engine input through a sync method.

## 15. Save and Sync Behavior

The guided field model shall include explicit save/sync behavior.

### 15.1 Required Actions

| Action | Behavior |
|---|---|
| Save Draft | Saves officer-visible fields and syncs to configuration JSON |
| Validate Configuration | Syncs latest field values, runs validation, stores messages |
| Generate Required Forms | Uses latest saved/validated configuration |
| Generate Preview | Uses latest saved/validated configuration and blocks on blockers |

### 15.2 Required Sync Rule

Before validation or generation actions, the system shall synchronize officer-facing field values into `configuration_json`.

This prevents stale configuration from being validated or rendered.

---

## 16. Validation Relationship

The guided field model does not replace rule validation.

Validation relationship:

| Layer | Role |
|---|---|
| Client UI required indicators | Helps officer complete obvious fields |
| Client conditional visibility | Improves usability |
| Server sync | Writes canonical config |
| Server validation engine | Authoritative safety check |
| Validation messages | Officer-facing resolution guidance |

If client-side visibility and server validation disagree, server validation wins.

## 17. Officer-Friendly Help Text Requirements

Every group shall have guidance text. Important conditional/legal-sensitive fields shall have help text.

Examples:

| Field | Required help text intent |
|---|---|
| Tender security mode | Explain choice between security, declaration, or none |
| Tender security amount | Explain amount is validated against estimated cost limit |
| JV allowed | Explain that allowing JV activates JV information requirements |
| Alternative tenders allowed | Explain that alternative tender details are required if allowed |
| Tender scope | Explain international scope may require additional foreign tenderer/local input information |
| Dayworks included | Explain that Dayworks BoQ category must be present |
| Provisional sums included | Explain that Provisional Sums BoQ category must be present |
| Preview generation | Explain preview is not final legal publication |

## 18. Officer-Facing Field Status Indicators

Each group should show a status where practical.

| Status | Meaning |
|---|---|
| `Not Started` | No key fields completed |
| `In Progress` | Some fields completed |
| `Needs Attention` | Validation blockers/errors exist in group |
| `Complete` | Required fields appear complete and no known blockers for group |
| `Not Applicable` | Group is inactive due to configuration |

This can be derived progressively. The first implementation may show validation-based group status only.

## 19. Field-Level Validation Display

Validation messages shall be mapped to field groups and fields.

Required display behavior:

1. message appears in validation panel;
2. message identifies affected group;
3. message identifies affected field where possible;
4. field or group is visually marked as needing attention where practical;
5. resolution hint is shown.

## 22. Permission and Safety Requirements

| Requirement | Rule |
|---|---|
| Officer can edit guided fields | Only on draft/configuring tender records they may write |
| Officer cannot edit package internals | No access through officer UI |
| Officer cannot edit raw configuration JSON | Hidden or read-only technical field |
| Server sync validates field codes | Unknown field codes are rejected or ignored with warning according to implementation policy |
| Server validation authoritative | Preview blocked on blockers/errors |
| Derived audit fields not officer-editable | System-controlled |

## 23. Error Handling Requirements

| Scenario | Required behavior |
|---|---|
| Template missing | Prompt to select valid template |
| Template not allowed | Show unavailable template message |

| Required package metadata missing | Show package validation warning and block if critical |
| Sync fails | Show clear save/sync error |
| Validation fails | Show officer-friendly messages |

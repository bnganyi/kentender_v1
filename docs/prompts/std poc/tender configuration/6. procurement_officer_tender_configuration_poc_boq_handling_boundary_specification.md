## 2. Purpose

The purpose of this document is to define how Bills of Quantities shall be handled in the Procurement Officer Tender Configuration POC.

The Works STD is the most complex template family because Works procurement depends heavily on BoQ, lots, pricing structure, provisional sums, dayworks, specifications, drawings, and downstream contract execution. However, a production-grade BoQ module is a separate major workstream and shall not be built inside this officer configuration POC.

This specification defines the safe POC boundary: the officer workflow shall show BoQ awareness, representative structured BoQ rows, category coverage, lot linkage, validation messages, and preview output, without claiming to provide full production BoQ authoring, pricing, evaluation, measurement, or contract execution capability.

---

## 3. Design Principle

The BoQ handling design shall implement the principle:

> The officer POC must prove that Works BoQ requirements can be represented as structured system data, while clearly avoiding the false claim that a production BoQ module has been completed.

This means:

1. BoQ shall be visible as structured data, not as an uncontrolled tender-document upload.
2. The officer shall understand whether BoQ is required.
3. The officer shall see representative BoQ categories and rows.
4. The system shall validate core POC category expectations, such as Dayworks and Provisional Sums where configured.
5. The BoQ panel shall clearly state that production BoQ authoring and pricing are out of scope.
6. The preview shall include a BoQ summary, not a full legal/production BoQ schedule.
7. Bidder pricing, arithmetic checks, and evaluated price calculations are deferred.

## 5. Non-Scope

The following are explicitly out of scope:

1. production BoQ authoring UI;
2. full quantity take-off;
3. standard method of measurement configuration;
4. Excel BoQ import/export;
5. item library or rate library;
6. engineer's estimate management;
7. bidder pricing entry;
8. arithmetic extension and total checks;
9. discount, correction, or evaluated price logic;
10. bid comparison sheets;
11. contract bills;
12. measurement sheets;
13. interim payment certificates;
14. variations;
15. final account;
16. integration with project management or contract execution;
17. PDF BoQ schedule generation;
18. external QS/cost-estimation integrations.

Any of these shall require a dedicated Works BoQ and Pricing module design.

## 6. POC BoQ Position

The officer POC shall treat BoQ as a **representative structured data output**.

The POC shall prove:

| Proven capability | Meaning |
|---|---|
| BoQ rows can exist as system data | Rows are stored in `Tender BoQ Item`, not only uploaded as documents |
| BoQ can be linked to lots | Rows may reference `Tender Lot` |
| BoQ categories can be checked | System can verify expected POC categories |
| BoQ requirements can be rule-sensitive | Dayworks and Provisional Sums can be required by configuration |
| BoQ summary can render in preview | Preview can include BoQ status and summary |

The POC shall not prove:

| Not proven | Reason |
|---|---|
| Full BoQ authoring | Requires separate QS-grade design |
| Bidder pricing | Bidder portal and pricing workflow are out of scope |
| Arithmetic evaluation | Requires pricing/evaluation module |
| Legal final BoQ schedule | PDF/publication not in scope |
| Contract measurement | Contract execution not in scope |

## 8. Officer-Facing BoQ Panel Purpose

The BoQ panel shall answer these officer questions:

1. Is a BoQ required for this tender?
2. Has representative structured BoQ data been generated or provided?
3. Which BoQ categories are represented?
4. Are Dayworks and Provisional Sums present when configured?
5. Do BoQ rows reference valid lots where applicable?
6. Which rows are bidder-priced versus fixed/provisional/summary rows?
7. Is the BoQ ready for preview in this POC?
8. What remains out of scope before production Works tendering?

---

## 9. Required BoQ Boundary Warning

The officer-facing BoQ panel shall display this warning or materially equivalent text:

```text
POC BoQ Boundary: This workflow uses representative structured BoQ rows to prove that Works BoQ requirements can be handled as system data. It is not a production BoQ authoring, bidder pricing, arithmetic evaluation, measurement, or payment-certification module.
```

This warning shall be visible whenever BoQ details are shown.

---

## 10. BoQ Status Model

The BoQ panel shall show a status.

| Status | Meaning |
|---|---|
| `Not Required` | Current configuration does not require BoQ |
| `Required - Not Generated` | BoQ is required but no representative rows exist |
| `Generated` | Representative rows exist and required POC category checks pass |
| `Generated with Warnings` | Rows exist but non-blocking issues are present |
| `Blocked` | BoQ-related validation blockers exist |
| `Possibly Stale` | Configuration changed after BoQ generation/checking |
| `Unknown` | Status cannot be determined |

Recommended POC behavior:

If `WORKS.BOQ_REQUIRED = true` and no rows exist, status shall be `Required - Not Generated`.

## 15. Dayworks Handling

### 15.1 Configuration Relationship

If `WORKS.DAYWORKS_INCLUDED = true`, the system shall expect at least one `DAYWORKS` row.

### 15.2 Officer Feedback

If Dayworks are included and category is missing, show:

```text
Dayworks are included in this tender, but the representative BoQ does not contain a Dayworks row. Generate or correct the POC BoQ rows before preview.
```

### 15.3 Validation Relationship

Missing Dayworks row may be treated as blocker if the existing POC rules classify it as blocking.

---

## 16. Provisional Sums Handling

### 16.1 Configuration Relationship

If `WORKS.PROVISIONAL_SUMS_INCLUDED = true`, the system shall expect at least one `PROVISIONAL_SUMS` row.

### 16.2 Officer Feedback

If Provisional Sums are included and category is missing, show:

```text
Provisional sums are included in this tender, but the representative BoQ does not contain a Provisional Sums row. Generate or correct the POC BoQ rows before preview.
```

### 16.3 Validation Relationship

Missing Provisional Sums row may be treated as blocker if the existing POC rules classify it as blocking.

---

## 17. Lot Linkage Handling

### 17.1 Purpose

When a tender is divided into lots, BoQ rows should reference valid lot codes where applicable.

### 17.2 Required Checks

| Check | Expected behavior |
|---|---|
| Lot code exists | If a BoQ row has a lot code, it must match a `Tender Lot` row |
| Summary row exception | `GRAND_SUMMARY` may be unlinked |
| Missing lot rows | If multiple lots are allowed but no lots exist, validation should flag issue |
| Lot title display | If lot code matches, display lot title |

### 17.3 Officer Feedback

If invalid lot references exist, show:

```text
Some BoQ rows reference lot codes that do not exist in the tender lot list. Review the Lots section and BoQ rows.
```

## 18. Row Type Classification

The BoQ panel shall classify rows for officer understanding.

| Condition | Row type |
|---|---|
| `item_category = GRAND_SUMMARY` | Summary |
| `item_category = PROVISIONAL_SUMS` | Provisional / fixed allowance |
| `item_category = DAYWORKS` | Dayworks |
| `is_priced_by_bidder = true` | Bidder priced |
| `amount` present and not bidder-priced | Fixed amount |
| Otherwise | Other |

This classification is explanatory only and does not implement pricing logic.

|---|---|---:|

| Check BoQ Status | Recompute category and lot checks | No |

| Action | Purpose | Notes |

## 20. Empty State UX

If BoQ is required but no rows exist, show:

```text
BoQ is required for this Works tender, but no representative BoQ rows have been generated. Generate representative BoQ rows for this POC before preview.
```

Required action:

```text
Generate Representative BoQ
```

If BoQ is not required, show:

```text
The current configuration does not require BoQ rows. This is unusual for Works procurement and should be reviewed if unintended.
```

---

## 21. Stale BoQ UX

If configuration changes after BoQ generation, show:

```text
The BoQ status may be out of date because tender configuration has changed. Recheck or regenerate representative BoQ rows before generating preview.
```

Potential triggers:

1. `WORKS.BOQ_REQUIRED` changes.
2. `WORKS.DAYWORKS_INCLUDED` changes.
3. `WORKS.PROVISIONAL_SUMS_INCLUDED` changes.
4. `LOTS.MULTIPLE_LOTS_ALLOWED` changes.
5. Lot rows are added/removed/changed.

---

## 22. BoQ Relationship to Validation

Validation shall remain authoritative.

The BoQ panel shall show BoQ-related validation messages where available.

Required behavior:

1. If validation blockers exist for BoQ, BoQ status shall be `Blocked`.
2. If BoQ warnings exist but no blockers, status may be `Generated with Warnings`.
3. If validation has not been run, BoQ status may be provisional.
4. Preview generation shall be blocked if BoQ-related blockers exist.

---

## 23. BoQ Relationship to Preview

The tender-pack preview shall include a BoQ summary.

Before preview generation, the system shall ensure:

1. configuration is synced;
2. validation has no blockers/errors;
3. BoQ status is checked;
4. required categories are present where configured;
5. BoQ row count is available for audit summary.

If BoQ is required and rows are missing, preview shall not be generated as valid.

---

## 24. BoQ Relationship to Required Forms

BoQ does not directly replace the required forms checklist, but it influences the tender-pack completeness posture.

Expected relationship:

| BoQ condition | Effect |
|---|---|
| BoQ required and present | Preview may include BoQ summary if validation passes |
| Dayworks included and present | BoQ status supports preview readiness |
| Provisional sums included and present | BoQ status supports preview readiness |
| BoQ missing | Validation/preview should block or warn according to rules |

## 25. Server Method Requirements

This UX requires the following capabilities.

| Method / capability | Purpose |
|---|---|
| `generate_officer_representative_boq(tender_name)` | Populate POC BoQ rows using existing/sample logic |

## 30. Manual Verification Scenarios

### 30.1 Primary BoQ Generation

Steps:

1. Create officer tender from STD-WORKS-POC template.
2. Configure Works values equivalent to primary sample.
3. Generate representative BoQ.
4. Open BoQ panel.

Expected:

1. BoQ status is `Generated`.
2. Row count is 9.
3. Required categories are present.
4. Dayworks present.
5. Provisional Sums present.
6. Grand Summary present.
7. Lot references are valid except permitted summary row.
8. Boundary warning is visible.

### 30.2 Dayworks Missing

Steps:

1. Set Dayworks included to true.
2. Remove or omit Dayworks row in a test scenario.
3. Check BoQ status or validate.

Expected:

1. BoQ panel shows Dayworks missing.
2. Validation blocks or warns according to rules.
3. Preview is not generated as valid if blocker exists.

### 30.3 Provisional Sums Missing

Steps:

1. Set Provisional Sums included to true.
2. Remove or omit Provisional Sums row in a test scenario.
3. Check BoQ status or validate.

Expected:

1. BoQ panel shows Provisional Sums missing.
2. Validation blocks or warns according to rules.
3. Preview is not generated as valid if blocker exists.

### 30.4 Invalid Lot Reference

Steps:

1. Create BoQ row referencing unknown lot code.
2. Check BoQ status.

Expected:

1. Invalid lot reference count is greater than zero.
2. Row-level validation note identifies problem.
3. Officer is directed to review Lots and BoQ.

### 30.5 BoQ Stale After Configuration Change

Steps:

1. Generate representative BoQ.
2. Change Dayworks, Provisional Sums, or Multiple Lots setting.
3. Save configuration.

Expected:

1. BoQ panel shows `Possibly Stale` or prompts recheck/regeneration.
2. Preview requires current BoQ readiness.

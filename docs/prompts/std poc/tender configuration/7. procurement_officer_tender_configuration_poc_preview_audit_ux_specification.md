## 2. Purpose

The purpose of this document is to define the procurement officer-facing Preview and Audit UX for the Procurement Officer Tender Configuration POC.

The preview and audit surface shall allow a procurement officer to generate and review a draft tender-pack preview from a controlled STD template package and guided configuration. It shall also show the reproducibility metadata needed to prove which template package, configuration, validation result, required forms checklist, and BoQ state produced the preview.

This UX shall not create a final legally certified tender document, PDF, publication record, bidder-facing tender, or public tender notice. It is a POC preview and internal review artefact only.

---

## 3. Design Principle

The Preview and Audit UX shall implement the principle:

> A procurement officer may generate a draft preview only when the configured tender is validated, traceable, and clearly marked as non-public POC output.

This means:

1. preview generation shall be blocked when validation blockers or errors exist;
2. preview generation shall use the latest synced configuration;
3. required forms and BoQ readiness shall be checked before preview;
4. preview output shall be read-only;
5. preview output shall be clearly labelled as POC/non-public output;
6. audit metadata shall be visible with full hash values available;
7. officers shall not publish, issue, or download a final tender document from this POC workflow;
8. any legal/publication step shall remain deferred to a future workstream.

## 6. Preview UX Objective

The officer-facing preview UX shall answer:

1. Can the tender-pack preview be generated now?
2. If not, what blocks preview generation?
3. If preview exists, was it generated from the latest validated configuration?
4. Which STD template package produced the preview?
5. Which configuration hash produced the preview?
6. How many required forms and BoQ rows were included?
7. Does the preview contain the expected POC sections?
8. Is the preview clearly non-public and non-final?
9. What must happen before any future publication workflow?

---

## 7. Preview Generation Preconditions

Preview generation shall require the following conditions.

| Precondition | Required behavior |
|---|---|
| STD template selected | Tender must be linked to an allowed `STD Template` |
| Configuration synced | Guided field values must be synced to `configuration_json` |
| Validation completed | Server-side validation must run before preview |
| No blockers/errors | Preview generation must stop if blockers/errors exist |
| Required forms current | Required forms checklist must be generated or current |
| BoQ readiness checked | BoQ must be checked if required |
| Audit values available | Template/package/configuration metadata should be available |

Recommended POC behavior:

The `Generate Preview` action shall automatically sync configuration and run validation. If validation passes, it shall ensure required forms and BoQ readiness before calling the existing preview renderer. If blockers/errors exist, it shall stop and display officer-facing messages.

## 8. Preview Status Model

The preview panel shall show one of the following statuses.

| Status | Meaning |
|---|---|
| `Not Generated` | No preview exists yet |
| `Blocked` | Validation or readiness checks prevent preview generation |
| `Generated` | Preview exists and is current enough for POC review |
| `Generated with Warnings` | Preview exists but non-blocking warnings remain |
| `Possibly Stale` | Configuration, forms, BoQ, or validation may have changed after preview generation |
| `Unknown` | System cannot determine preview state |

The first implementation may use conservative stale detection based on timestamps, validation status, configuration hash presence, checklist status, and BoQ status.

## 10. Required Preview Boundary Warning

The preview panel shall display this warning or materially equivalent text:

```text
POC Preview Only: This preview is generated for internal review from a controlled STD template package and tender configuration. It is not a final legally certified tender document, not a published tender, and not a production PDF.
```

The warning shall be visible even when preview content itself contains a warning.

## 12. Preview Actions

Required action:

| Action | Purpose | Writes data? |
|---|---|---:|
| Generate Preview | Sync, validate, check forms/BoQ, generate read-only POC preview | Yes |

Optional action:

| Action | Purpose | Notes |
|---|---|---|
| Regenerate Preview | Re-run preview after changes | Same as Generate Preview |

Explicitly prohibited actions:

2. Generate Final PDF.
3. Issue Tender.
4. Invite Bidders.
5. Open Bid Submission.

## 13. Blocked Preview UX

If preview generation is blocked, show:

```text
Tender-pack preview cannot be generated because the tender configuration has unresolved blockers or readiness issues. Review the messages below, correct the highlighted areas, validate again, and then regenerate the preview.
```

The blocked state shall display:

| Item | Required |
|---|---:|
| Blocker count | Yes |
| Error count | Yes |
| Affected configuration areas | Yes |
| Resolution hints | Yes |
| Links/buttons to relevant sections | Recommended |

Preview shall not be generated or updated as valid in blocked state.

---

## 14. Warnings UX

If validation passes with warnings, show:

```text
The tender configuration has warnings. Review them before relying on this preview. For this POC, preview generation may continue because there are no blockers or errors.
```

Warnings shall not be hidden. They shall be visible in the preview/audit panel and in the validation panel.

---

## 15. Empty Preview UX

If no preview exists, show:

```text
No tender-pack preview has been generated. Complete the guided configuration, validate the tender, generate required forms, check BoQ status, then generate the preview.
```

Show the `Generate Preview` action if the officer has write permission.

---

## 16. Possibly Stale Preview UX

If the preview may be stale, show:

```text
This preview may be out of date because the tender configuration or generated outputs may have changed after the preview was generated. Regenerate the preview before review.
```

Possible stale triggers:

1. configuration modified after `last_generated_at`;
2. validation run after preview generation;
3. required forms regenerated after preview generation;
4. BoQ regenerated or changed after preview generation;
5. configuration hash missing;
6. validation status reset to `Not Validated`.

The first implementation may use conservative timestamp/hash logic.

---

The preview shall display the generated HTML content from the existing renderer.

Required behavior:

3. Preview cannot be published from this panel.
4. Preview cannot be converted to production PDF from this POC panel.
5. Preview shall preserve POC warning text.
6. Preview shall be scrollable or displayed in a readable section.
7. Preview shall remain linked to audit metadata.

## 21. Next Step Boundary Display

The panel shall show a boundary message after preview generation:

```text
This POC stops at internal preview and review readiness. Publication, final PDF generation, bidder access, bid submission, evaluation, award, and contract execution require separate workflows and approvals.
```

This avoids users mistaking preview generation for tender issuance.

---

## 22. Relationship to Validation

Validation is the first gate for preview generation.

Required behavior:

1. Generate Preview action shall sync configuration.
2. Generate Preview action shall run validation or require current validation.
3. If blockers/errors exist, preview generation stops.
4. If warnings only exist, preview may proceed with warning in the POC.
5. Validation messages remain visible after preview generation.
6. If configuration changes after preview, preview becomes possibly stale.

---

## 23. Relationship to Required Forms Checklist

Preview generation shall ensure required forms are current.

Required behavior:

1. If required forms are not generated, generate them or prompt officer.
2. If checklist is stale, regenerate or block until regenerated.
3. Required form count shall appear in audit summary.
4. Preview shall include required forms checklist summary.
5. Failure to generate forms shall block valid preview generation.

---

## 24. Relationship to BoQ Readiness

Preview generation shall respect BoQ readiness.

Required behavior:

1. If BoQ is required and rows are missing, preview shall be blocked or prevented from being treated as valid.
2. If Dayworks/Provisional Sums are configured but categories are missing, preview shall respect validation/readiness blockers.
3. BoQ row count shall appear in audit summary.
4. Preview shall include BoQ summary.
5. BoQ boundary warning shall remain visible in the BoQ panel and may also appear in preview notes.

---

## 25. Relationship to State Indicators

The preview action may update POC-level tender state.

| Existing state | Event | Resulting state |
|---|---|---|
| Draft / Configuring | Preview generation blocked | Validation Failed |
| Validated | Preview generated | Preview Generated |
| Preview Generated | Officer changes configuration | Configuring or Not Validated |
| Preview Generated | Officer marks ready | Ready for Review |

This is not production workflow. It is a POC state model.

---

## 26. Server Method Requirements

This UX requires the following capabilities.

| Method / capability | Purpose |
|---|---|
| `get_officer_preview_audit_summary(tender_name)` | Return preview status, readiness, outputs summary, audit metadata |
| `generate_officer_preview(tender_name)` | Sync, validate, ensure forms/BoQ readiness, generate preview or block |
| `get_preview_completeness_check(tender_name)` | Return expected/observed POC preview sections |
| `is_preview_stale(tender_name)` | Determine whether preview may be stale |
| `get_officer_preview_blockers(tender_name)` | Return blocker summary for preview panel |

Methods may reuse Admin Console preview/audit summary and existing preview renderer, but officer-facing messages and boundary copy shall be appropriate to officer workflow.

## 30. Error Handling Requirements

| Scenario | Required behavior |
|---|---|
| Tender missing | Clear error |
| User lacks permission | Permission error |
| Template missing | Show template linkage error |
| Configuration sync fails | Show sync error and do not generate preview |
| Validation blockers exist | Show blocker summary and stop |
| Required forms generation fails | Stop and show forms error |
| BoQ readiness fails | Stop and show BoQ readiness error |
| Preview renderer fails | Show render error and do not update preview as valid |
| Package hash missing | Show audit warning |
| Configuration hash missing | Show audit warning |
| Completeness detection unknown | Show Unknown, not false pass |

Silent failure is not acceptable.

## 31. Manual Verification Scenarios

### 31.1 Generate Preview for Valid Tender

Steps:

1. Create officer tender.
2. Complete valid primary-sample-equivalent configuration.
3. Validate.
4. Generate required forms.
5. Generate/check BoQ.
6. Generate preview.
7. Open preview/audit panel.

Expected:

1. Preview status is `Generated`.
2. Preview is visible read-only.
3. POC warning is visible.
4. Required form count is 15.
5. BoQ row count is 9.
6. Package hash is visible.
7. Configuration hash is visible.
8. Last preview generated timestamp is visible.
9. No publication/PDF action is present.

### 31.2 Block Preview with Validation Error

Steps:

1. Configure site visit mandatory.
2. Leave site visit date/time missing.
3. Attempt Generate Preview.

Expected:

1. Validation runs or is checked.
2. Preview is blocked.
3. Blocker appears with officer-facing resolution hint.
4. Existing valid preview, if any, is marked stale or not current.

### 31.3 Block Preview with Missing BoQ Category

Steps:

1. Set Dayworks included.
2. Remove Dayworks row in test setup or skip BoQ generation.
3. Attempt Generate Preview.

Expected:

1. Preview is blocked or not treated as valid according to validation/readiness result.
2. BoQ readiness message is visible.
3. Officer is directed to generate/correct representative BoQ.

### 31.4 Stale Preview After Configuration Change

Steps:

1. Generate preview.
2. Change tender security mode or JV allowed.
3. Save configuration.
4. Open preview/audit panel.

Expected:

1. Preview status is `Possibly Stale` or validation resets to `Not Validated`.
2. Officer is prompted to revalidate/regenerate.

### 31.5 Hash Missing Warning

Steps:

1. Simulate missing configuration hash in development/test.
2. Open preview/audit panel.

Expected:

1. Audit warning is visible.
2. System does not claim full reproducibility.

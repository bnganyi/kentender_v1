## 2. Purpose

The purpose of this implementation pack is to provide a precise, bounded, Cursor-ready implementation plan for the Procurement Officer Tender Configuration POC.

This pack converts Officer POC Steps 1 through 7 into an implementation sequence using the existing STD-WORKS-POC and STD Administration Console foundation.

The implementation shall allow a procurement officer to create and configure a draft Works tender from a controlled STD template package using guided UI, conditional fields, validation feedback, generated required forms, representative BoQ status, tender-pack preview, and audit outputs.

This pack shall not expand the system into tender publication, bidder workflows, evaluation, award, contract execution, production BoQ, PDF generation, external integrations, or full production governance.

---

## 3. Implementation Principle

The implementation shall follow this principle:

> Build a guided procurement officer workflow over the controlled STD engine; do not expose the officer to package internals or convert the POC into production tender management.

This means:

1. reuse `STD Template` and `Procurement Tender`;
2. reuse the template engine, validation, required forms, BoQ, and preview services;
3. provide officer-facing guided configuration groups;
4. hide raw package internals and raw configuration JSON from ordinary officer workflow;
5. preserve server-side validation as authoritative;
6. block preview generation when blockers/errors exist;
7. expose audit values and POC boundaries;
8. avoid building downstream procurement lifecycle modules in this step.

---

## 4. Required Implementation Outcome

At the end of this implementation, a procurement officer shall be able to:

1. start a new draft tender from an available STD template;
2. see a non-technical template summary;
3. complete guided configuration groups;
4. experience conditional field visibility for major Works branches;
5. save/sync configuration without editing raw JSON;
6. run validation and understand blockers/warnings;
7. generate required forms checklist;
8. view required forms in officer-friendly language;
9. generate/check representative BoQ status;
10. view representative BoQ rows and category coverage;
11. generate a read-only POC tender-pack preview when valid;
12. see audit metadata including package hash and configuration hash;
13. mark the tender as ready for review or keep it in draft/configuring state;
14. remain blocked from publication, final PDF, bidder workflow, evaluation, award, and contract execution.

## 8. Field Addition Policy

The fastest implementation may use existing `Procurement Tender` fields and `configuration_json`. However, the guided officer workflow may require visible fields that are easier for officers than raw JSON.

### 8.1 Recommended POC Approach

Use a hybrid model:

1. add officer-facing fields only where necessary for the guided workflow;
2. sync those fields into `configuration_json` before validation/generation;
3. keep `configuration_json` hidden or read-only for ordinary officer users;
4. keep package internals hidden entirely.

### 8.2 Candidate Officer-Facing Fields on `Procurement Tender`

The implementation may add fields corresponding to the field model, including but not limited to:

| Field area | Candidate examples |
|---|---|
| Tender Identity | tender title, reference, procuring entity, works location, estimated cost, currency |
| Method and Scope | procurement method, tender scope |
| Participation | reserved tender, reservation group, JV allowed, max JV members |
| Dates and Meetings | publication date, clarification deadline, submission deadline, opening datetime, site visit fields, pre-tender meeting fields |
| Securities | tender security mode/type/amount/currency, performance security, advance security, retention security |
| Alternatives | alternatives allowed, alternative type, alternative scope |
| Lots | multiple lots allowed, lot evaluation method, lot award method |
| Qualification | turnover, liquid assets, similar contracts, personnel/equipment/compliance toggles |
| Works | scope summary, BoQ required, dayworks, provisional sums, drawings, ESHS |
| Contract/SCC | completion period, defects liability, LDs, retention, advance payment, dispute forum |

If the project already stores some of these as fields, reuse them.

Do not add fields that belong to bidder submission, evaluation, award, contract execution, or production BoQ.

## 13. Configuration Sync Requirements

### 13.1 Sync Source

Configuration sync shall read guided officer fields from `Procurement Tender` and build a dictionary keyed by stable STD package field codes.

### 13.2 Sync Target

The target is:

```text
Procurement Tender.configuration_json
```

### 13.3 Required Behavior

1. Sync shall include all relevant visible and stored guided values.
2. Sync shall include derived system values such as template code, package version, and package hash.
3. Sync shall preserve stable field codes expected by the engine.
4. Unknown or unsupported fields shall be logged or returned as warnings.
5. Sync shall run before validation, required forms generation, BoQ readiness check, and preview generation.
6. Sync shall not allow officer to write package internals.

## 15. Validation Feedback Implementation Requirements

`validate_officer_configuration(tender_name)` shall:

1. sync guided fields;
2. run existing engine validation;
3. write/update `Tender Validation Message` rows;
4. update validation status and validation timestamp;
5. compute configuration hash;
6. return officer-friendly grouped feedback;
7. mark preview/forms/BoQ as stale where relevant if configuration changed.

`get_officer_validation_feedback(tender_name)` shall return:

1. validation status;
2. blocker count;
3. error count;
4. warning count;
5. grouped messages;
6. resolution hints;
7. technical details only as secondary data.

---

## 16. Required Forms Implementation Requirements

`generate_officer_required_forms(tender_name)` shall:

1. sync configuration;
2. validate or check validation status;
3. generate required forms using existing engine logic;
4. populate `Tender Required Form` rows;
5. compute checklist status;
6. return officer-friendly checklist summary.

`get_officer_required_forms_checklist(tender_name)` shall:

1. read generated rows;
2. map internal form metadata to officer labels;
3. generate plain-language `Required Because` text;
4. map evidence policy to officer-facing evidence approach;
5. detect stale/empty/blocked state;
6. return structured rows.

Officers shall not manually add or remove checklist rows in this POC.

---

## 17. BoQ Implementation Requirements

`generate_officer_representative_boq(tender_name)` shall:

1. use existing POC BoQ generation logic or sample package data;
2. populate representative `Tender BoQ Item` rows;
3. not implement production BoQ authoring;
4. return BoQ summary.

`get_officer_boq_status(tender_name)` shall:

1. inspect BoQ rows;
2. inspect lots;
3. compute required category coverage;
4. check Dayworks and Provisional Sums where configured;
5. check invalid lot references;
6. classify rows;
7. return officer-facing status and warning messages.

---

## 18. Preview Implementation Requirements

`generate_officer_preview(tender_name)` shall:

1. sync configuration;
2. run validation;
3. stop if blockers/errors exist;
4. ensure required forms are current;
5. ensure BoQ readiness if BoQ is required;
6. call existing preview renderer;
7. update `generated_tender_pack_html`;
8. update `last_generated_at`;
9. update tender state to `Preview Generated` or equivalent POC status;
10. return preview/audit summary.

It shall not generate PDF, publish the tender, or expose bidder access.

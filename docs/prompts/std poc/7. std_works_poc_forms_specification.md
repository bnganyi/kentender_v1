## 2. Purpose

The purpose of this step is to define the full content of the `forms.json` file for the PPRA Works Standard Tender Document POC template package.

`forms.json` shall define the bidder response forms and downstream contract/security forms represented in the POC. The file shall support generation of a required-forms checklist based on tender configuration and rule activation.

This file shall not implement a full bidder portal, actual bid submission, electronic signatures, encryption, form rendering screens, or document upload workflows. It shall define form metadata and minimal field schemas sufficient for the POC engine to determine which forms are required for a tender.

---

## 3. Design Principle

The `forms.json` file shall implement the principle:

> Standard bidder response forms should be generated from structured schemas and activated by tender configuration, not uploaded as manually completed documents.

For the POC, each form shall be represented as a structured schema with:

1. stable form code,
2. source STD relationship,
3. form category,
4. respondent type,
5. workflow stage,
6. activation behavior,
7. minimum form fields,
8. evidence/upload posture,
9. render and checklist behavior,
10. traceability to rules and sections.

The POC shall not attempt to capture every field contained in the full source STD forms. It shall include enough fields to prove the structured-form architecture and scenario-driven form activation model.

## 7. Form Categories

The POC shall define the following form categories:

| Form category | Meaning |
|---|---|
| `CORE_BID_SUBMISSION` | Core forms required for most bid submissions |
| `ELIGIBILITY_DECLARATION` | Declarations regarding eligibility, corruption, conflict, independence, or statutory status |
| `SECURITY_INSTRUMENT` | Tender security or tender-securing declaration forms |
| `PARTICIPATION_STRUCTURE` | Forms related to joint ventures, foreign tenderer participation, or local input commitments |
| `QUALIFICATION_EVIDENCE` | Forms related to experience, finance, personnel, equipment, or qualifications |
| `TECHNICAL_PROPOSAL` | Forms related to methodology, technical proposal, alternatives, ESHS, or works execution approach |
| `OWNERSHIP_DISCLOSURE` | Beneficial ownership or ownership/control disclosure forms |
| `CONTRACT_SECURITY` | Forms generated or required after award or before contract signature |
| `DOWNSTREAM_CONTRACT_FORM` | Contract forms that are downstream from bid submission |

---

## 8. Respondent Types

The POC shall define the following respondent types:

| Respondent type | Meaning |
|---|---|
| `BIDDER` | Form is completed by the tenderer/bidder during bid submission |
| `BIDDER_OR_JV` | Form is completed by an individual bidder or joint venture |
| `JOINT_VENTURE` | Form is completed only when a joint venture participates |
| `FOREIGN_TENDERER` | Form is completed when foreign tenderer/local input requirements apply |
| `SUCCESSFUL_BIDDER` | Form is completed only by the successful bidder after award or before contract signing |
| `PROCURING_ENTITY` | Form or generated record belongs to the procuring entity |
| `SYSTEM_GENERATED` | Form/checklist item is generated by the system from workflow data |

---

## 9. Workflow Stages

The POC shall define the following workflow stages:

| Workflow stage | Meaning |
|---|---|
| `BID_PREPARATION` | Form is visible to bidder before submission |
| `BID_SUBMISSION` | Form is part of the bid submission package |
| `EVALUATION` | Form supports evaluation or post-qualification |
| `AWARD` | Form is used at intention-to-award or award stage |
| `CONTRACT_SIGNATURE` | Form is required before or during contract signature |
| `CONTRACT_EXECUTION` | Form is relevant after contract signature |
| `POC_CHECKLIST_ONLY` | Form is represented only as a checklist item in this POC |

For this POC, all forms shall be checklist/schema definitions. The workflow stage indicates future placement, not full workflow implementation.

---

## 10. Evidence Policies

The POC shall define the following evidence policies:

| Evidence policy | Meaning |
|---|---|
| `STRUCTURED_ONLY` | Form should be completed as structured fields; no routine upload expected |
| `STRUCTURED_WITH_OPTIONAL_EVIDENCE` | Form is structured but may later allow controlled evidence attachments |
| `STRUCTURED_WITH_REQUIRED_EVIDENCE_LATER` | Form is structured at bid stage; evidence may be required later for verification |
| `ISSUER_VERIFICATION_FUTURE` | Instrument should later be verified through issuer/bank/insurer workflow rather than scanned upload |
| `GENERATED_AFTER_AWARD` | Form is generated later from award or contract data |
| `POC_CHECKLIST_ONLY` | POC only records the form as required or not required |

## 11. Form Object Schema

Each form object in the `forms` array shall follow this structure:

```json
{
  "form_code": "...",
  "title": "...",
  "source_reference": "...",
  "category": "...",
  "respondent_type": "...",
  "workflow_stage": "...",
  "default_required": true,
  "activation_rule_codes": [],
  "section_targets": [],
  "evidence_policy": "...",
  "poc_treatment": "...",
  "ordinary_user_can_edit_schema": false,
  "bidder_upload_replacement_goal": true,
  "minimum_schema_fields": [],
  "checklist_display": {},
  "notes": []
}
```

### 11.1 Required Form Object Attributes

| Attribute | Type | Required | Purpose |
|---|---:|---:|---|
| `form_code` | String | Yes | Stable system identifier for the form |
| `title` | String | Yes | Human-readable form title |
| `source_reference` | String | Yes | Source STD form or conceptual reference |
| `category` | String | Yes | One of the defined form categories |
| `respondent_type` | String | Yes | One of the defined respondent types |
| `workflow_stage` | String | Yes | One of the defined workflow stages |
| `default_required` | Boolean | Yes | Whether form is required by default before rules are applied |
| `activation_rule_codes` | Array | Yes | Rules that can activate this form |
| `section_targets` | Array | Yes | Sections where the form is relevant |
| `evidence_policy` | String | Yes | Evidence/upload treatment |
| `poc_treatment` | String | Yes | Explanation of POC handling |
| `ordinary_user_can_edit_schema` | Boolean | Yes | Whether procurement officers can modify the schema |
| `bidder_upload_replacement_goal` | Boolean | Yes | Whether this form is intended to replace routine uploaded documents |
| `minimum_schema_fields` | Array | Yes | Minimal structured fields represented in the POC |
| `checklist_display` | Object | Yes | Display metadata for required-form checklist |
| `notes` | Array | Yes | Implementation notes |

---

## 12. Minimum Schema Field Object

Objects inside `minimum_schema_fields` shall follow this structure:

```json
{
  "field_code": "...",
  "label": "...",
  "field_type": "...",
  "required": true,
  "description": "..."
}
```

These are form-internal schema fields and are separate from procurement-officer configuration fields in `fields.json`.

For the POC, the minimum form schema shall be intentionally concise. It exists to prove the direction of structured bidder response, not to implement complete bidder submission.

---

## 13. Checklist Display Object

Each form shall include a `checklist_display` object:

```json
{
  "display_group": "...",
  "display_order": 0,
  "required_label": "Required",
  "not_required_label": "Not Required",
  "show_in_tender_pack_preview": true
}
```

Required display groups:

| Display group | Meaning |
|---|---|
| `Core Bid Forms` | Core bid submission forms |
| `Declarations` | Eligibility and integrity declarations |
| `Security Forms` | Tender security and declaration forms |
| `Participation Forms` | JV and foreign tenderer forms |
| `Qualification Forms` | Experience, finance, personnel, equipment forms |
| `Technical Forms` | Technical proposal and alternative proposal forms |
| `Ownership Forms` | Beneficial ownership forms |
| `Contract Security Forms` | Post-award security forms |

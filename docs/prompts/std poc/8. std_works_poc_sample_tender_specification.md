## 2. Purpose

The purpose of this step is to define the full content of the `sample_tender.json` file for the PPRA Works Standard Tender Document POC template package.

`sample_tender.json` shall provide a complete sample Works tender configuration that can be used to test package loading, default tender creation, rule validation, scenario activation, required-form generation, BoQ generation, and tender-pack preview rendering.

The sample tender shall exercise the most important POC branches without requiring full system implementation.

---

## 3. Design Principle

The sample tender shall be realistic enough to prove the Works STD configuration model, but it shall remain clearly marked as sample data.

The sample shall deliberately activate several complex branches:

1. multiple lots,
2. joint ventures,
3. alternative technical tenders,
4. site visit,
5. pre-tender meeting,
6. tender security,
7. BoQ,
8. dayworks,
9. provisional sums,
10. key personnel,
11. equipment,
12. beneficial ownership,
13. performance security,
14. advance payment security.

The sample shall also contain a separate international tender variant to test foreign tenderer/local input activation without making the primary sample tender international by default.

---

## 4. Required File Location

The file shall exist at:

```text
{app_name}/procurement/std_templates/ke_ppra_works_building_2022_04_poc/sample_tender.json
```

The placeholder `{app_name}` shall be replaced by the actual Frappe custom app name in the repository.

---

## 5. Relationship to Previous Package Files

`sample_tender.json` shall use the same template code as prior package files:

```text
KE-PPRA-WORKS-BLDG-2022-04-POC
```

The sample tender shall use only field codes defined in `fields.json`.

The sample tender shall use lot and BoQ structures that will later map to the Frappe child DocTypes:

1. `Tender Lot`
2. `Tender BoQ Item`

The sample tender shall not define new template fields, rules, forms, or sections.

## 9. Sample Lot Structure

The sample shall include two lots.

| Lot code | Lot title | Description |
|---|---|---|
| `LOT-001` | Main Building Works | Construction of ward administration block superstructure, substructure, finishes, and associated works |
| `LOT-002` | External Works and Services | Drainage, paving, boundary works, water connection, and external electrical/service works |

| `FINISHES` | Yes | Tests ordinary works rows |
| `EXTERNAL_WORKS` | Yes | Supports Lot 2 |
| `DAYWORKS` | Yes | Tests dayworks rule |
| `PROVISIONAL_SUMS` | Yes | Tests provisional sums rule |
| `GRAND_SUMMARY` | Yes | Tests summary behavior |

The sample BoQ rows shall be representative only. They are not intended to form a legally complete BoQ.

## 12. Scenario Variants

The sample file shall include scenario variants to support future tests without changing the main sample.

Required scenario variants:

| Variant code | Purpose |
|---|---|
| `VARIANT-INTERNATIONAL` | Tests international tender and foreign tenderer local input activation |
| `VARIANT-TENDER-SECURING-DECLARATION` | Tests tender-securing declaration instead of tender security |
| `VARIANT-RESERVED-TENDER` | Tests reservation and reserved group requirement |
| `VARIANT-MISSING-SITE-VISIT-DATE` | Tests blocker when site visit is enabled but date/location are missing |
| `VARIANT-MISSING-ALTERNATIVE-SCOPE` | Tests blocker when alternatives are enabled but scope is missing |
| `VARIANT-SINGLE-LOT` | Tests configuration where multiple lots are disabled |
| `VARIANT-RETENTION-MONEY-SECURITY` | Tests retention money security form activation |

The variants shall be expressed as override patches against the primary sample configuration.

## 2. Purpose

The purpose of this step is to define the formal end-to-end smoke path for STD-WORKS-POC.

This step verifies that the complete POC workflow works as an integrated system, from developer-controlled template package import through creation of a configured Works tender and generation of a tender-pack preview.

The smoke path is not a full QA test suite. It is a disciplined end-to-end proof that the architecture holds together and that the POC demonstrates its central legal and product claim:

> A Works tender can be configured and generated from a controlled Standard Tender Document package without requiring stakeholders to upload manually assembled tender documents.

---

## 3. Design Principle

The end-to-end smoke path shall verify integration, not merely individual components.

The smoke path shall confirm that:

1. the STD template package can be imported,
2. the imported package is stored with identity, version, and hash,
3. a tender can be linked to the imported package,
4. the sample tender can populate controlled configuration fields,
5. lots and BoQ rows can be populated as structured child data,
6. configuration validation can detect pass and failure states,
7. required forms can be generated from `forms.json` and `rules.json`,
8. sample BoQ rows satisfy BoQ-related rules,
9. tender-pack preview can be generated from render context,
10. audit fields expose template code, package hash, configuration hash, and validation status.

## 17. Smoke Pass / Fail Criteria

### 17.1 Smoke Pass Criteria

The POC smoke path passes only if all of the following are true:

1. Template package imports successfully.
2. `STD Template` record is created or updated without duplication.
3. Package hash is populated.
4. Package JSON is populated.
5. A `Procurement Tender` can be created and linked to the template.
6. Primary sample tender loads successfully.
7. Configuration JSON is populated.
8. Lots are populated with at least 2 rows.
9. BoQ has exactly 9 representative rows after generation.
10. Validation passes for the primary sample.
11. Required forms generate exactly the expected primary-sample checklist.
12. Tender-pack preview generates successfully for the primary sample.
13. Preview contains all required sections.
14. Audit summary contains template code, package version, package hash, configuration hash, and validation status.
15. Missing site visit negative scenario blocks preview generation.
16. Missing alternative scope negative scenario blocks preview generation.
17. Tender security amount above 2% blocks preview generation.
18. Tender-securing declaration variant switches security forms correctly.
19. International variant activates foreign tenderer local input form.
20. No final PDF, bidder portal, bid submission, evaluation workflow, or contract workflow is accidentally introduced.

### 17.2 Smoke Fail Criteria

The smoke path fails if any of the following occurs:

1. Package import fails.
2. Package import creates duplicate template records.
3. Package hash is missing.
4. `Procurement Tender` cannot link to `STD Template`.
5. Sample tender cannot load.
6. Configuration JSON is missing after sample load.
7. Lots or BoQ rows are missing when required.
8. Primary sample validation is blocked.
9. Required forms checklist is missing expected forms.
10. Required forms checklist contains duplicate forms.
11. Tender security and tender-securing declaration forms appear together incorrectly.
12. Tender-pack preview fails for the valid primary sample.
13. Preview omits the audit summary.
14. Preview omits the POC warning.
15. Negative scenarios fail to block generation.
16. Errors are silent or unclear.

| Field | Description |
|---|---|
| Issue ID | Sequential identifier, e.g. `SMOKE-001` |
| Severity | `Blocker`, `Major`, `Minor`, `Observation` |
| Area | Import, Template Record, Tender, Validation, Forms, BoQ, Preview, Audit, Variant |
| Description | Clear issue description |
| Steps to reproduce | Minimal reproduction steps |
| Expected behavior | What should have happened |
| Actual behavior | What happened |
| Evidence | Screenshot/log reference |
| Decision | Fix now, defer to Step 17, defer to production gap analysis, or no action |
| Owner | Developer/product/SME |
| Status | Open, Fixed, Deferred, Accepted |

Blocker and Major issues must be resolved before the POC is considered smoke-passed unless explicitly accepted by the product owner as a known limitation.

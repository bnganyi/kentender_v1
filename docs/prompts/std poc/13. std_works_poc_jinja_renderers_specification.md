## 2. Purpose

The purpose of this step is to define the Jinja/HTML renderers required to generate a readable tender-pack preview for the STD-WORKS-POC.

The renderer shall transform the render context prepared by the Step 11 template engine and exposed through Step 12 actions into an HTML preview stored on the `Procurement Tender` document.

This preview is intended to prove the template-package rendering model. It is not a final legally certified tender document, not a production PDF, and not a replacement for the full PPRA STD rendering that may be implemented later.

---

## 3. Design Principle

The tender-pack preview shall implement the principle:

> The system can generate a coherent tender package from an approved STD template package and controlled tender configuration without allowing ordinary users to edit legal template text.

The renderer shall be:

1. deterministic,
2. readable,
3. section-based,
4. traceable to template code and configuration hash,
5. clearly marked as a POC preview,
6. safe from direct legal-clause editing,
7. simple enough to inspect during development.

The renderer shall not attempt to reproduce the full PPRA Works STD formatting, pagination, legal wording, clause numbering, or PDF layout in this step.

## 5. Required Controller/Server Updates

This step shall update the Step 12 server method:

```text
prepare_render_context(tender_name)
```

or add a new server method:

```text
generate_tender_pack_preview(tender_name)
```

Recommended approach:

1. Keep `prepare_render_context(tender_name)` as a context/debug action.
2. Add `generate_tender_pack_preview(tender_name)` as the actual HTML rendering action.
3. Add a Desk button for `Generate Tender Pack Preview` under the `STD POC` button group.

The render method shall:

1. load the tender,
2. load the linked template package,
3. parse configuration JSON,
4. validate configuration,
5. block generation if there are `ERROR` or `BLOCKER` messages,
6. resolve required forms,
7. build render context,
8. render `templates/std_works_poc/tender_pack.html`,
9. store generated HTML in `Procurement Tender.generated_tender_pack_html`,
10. update `last_generated_at`,
11. update `tender_status` to `Tender Pack Generated`,
12. save the tender.

## 8. Required Render Context Shape

The renderer shall expect the render context produced by Step 11.

Required context shape:

```json
{
  "template": {},
  "manifest": {},
  "configuration": {},
  "active_sections": [],
  "lots": [],
  "boq_items": [],
  "required_forms": [],
  "validation": {},
  "audit": {}
}
```

### 8.1 Required Context Values

The renderer shall be able to access at least:

| Context path | Purpose |
|---|---|
| `manifest.template_code` | Template code |
| `manifest.template_name` | Formal template name |
| `manifest.template_short_name` | Short display name |
| `manifest.versioning.package_version` | Package version |
| `manifest.source_document.source_document_code` | Source STD document code |
| `template.package_hash` or `audit.package_hash` | Package hash |
| `audit.configuration_hash` | Configuration hash |
| `validation.status` | Validation status |
| `validation.messages` | Validation messages |
| `configuration` | Tender field values |
| `active_sections` | Sections included in preview |
| `required_forms` | Required forms checklist |
| `lots` | Tender lots |
| `boq_items` | Tender BoQ rows |

If any value is missing, the renderer shall show a clear placeholder such as `Not specified` rather than failing with an unhandled template error.

## 9. Formatting Requirements

The preview shall use simple HTML suitable for display in a Frappe HTML/Long Text field.

Required formatting conventions:

1. Use one wrapper element with class `std-poc-preview`.
2. Include a visible POC warning at the top.
3. Use `<h1>` for the main preview title.
4. Use `<h2>` for major sections.
5. Use `<h3>` for subsections.
6. Use tables for structured summaries.
7. Escape values by default using Jinja autoescaping conventions where applicable.
8. Avoid inline JavaScript.
9. Avoid external CSS dependencies.
10. Use minimal inline CSS only if necessary.
11. Keep the HTML readable when copied from the field.

---

Required behavior:

## 12. Required Server Method: `generate_tender_pack_preview`

Add a server method callable from the `Procurement Tender` form.

### 12.1 Signature

```python
@frappe.whitelist()
def generate_tender_pack_preview(tender_name: str) -> dict:
    ...
```

### 12.2 Expected Behavior

The method shall:

1. Load the `Procurement Tender` document.
2. Check write permission.
3. Confirm `std_template` is selected.
4. Confirm `configuration_json` exists.
5. Load the template using `std_template_engine.load_template()`.
6. Parse configuration JSON.
7. Convert `lots` and `boq_items` child rows to dictionaries.
8. Validate configuration using `std_template_engine.validate_config()`.
9. Write validation messages back to the tender.
10. If validation blocks generation, save validation results and return without rendering.
11. Resolve required forms.
12. Write required forms back to the tender.
13. Build render context.
14. Render `std_works_poc/tender_pack.html` using Frappe/Jinja rendering.
15. Store HTML in `generated_tender_pack_html`.
16. Set `last_generated_at`.
17. Set `tender_status` to `Tender Pack Generated`.
18. Save the document.
19. Return structured result.

### 12.3 Required Return Shape on Success

```json
{
  "ok": true,
  "message": "Tender-pack preview generated.",
  "tender": "...",
  "validation_status": "Passed",
  "required_form_count": 15,
  "configuration_hash": "..."
}
```

### 12.4 Required Return Shape when Blocked

```json
{
  "ok": false,
  "message": "Tender-pack preview was not generated because validation blocks generation.",
  "tender": "...",
  "validation_status": "Blocked",
  "blocks_generation": true,
  "message_count": 2
}
```

## 14. Rendering Safety Requirements

The renderer shall follow these safety requirements:

1. Do not render unescaped user input as raw HTML.
2. Do not use Jinja `safe` filter on configuration values unless explicitly justified.
3. Do not execute user-supplied expressions.
4. Do not include external scripts.
5. Do not include external CSS links.
6. Do not assume every field exists.
7. Do not fail if optional fields are missing.
8. Do not display `package_json` raw contents in the preview.
9. Do not display full source STD legal text not included in the POC package.
10. Do not modify template package data during rendering.

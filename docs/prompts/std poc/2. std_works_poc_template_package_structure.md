## 2. Purpose

The purpose of this step is to define the repository structure for the PPRA Works Standard Tender Document POC template package.

This package shall be the developer-controlled source of truth for the POC template. It shall contain structured JSON files that describe the template identity, sections, fields, rules, bidder forms, render mapping, and sample tender configuration.

The package shall not be edited by ordinary procurement officers. Procurement officers shall interact only with tender configuration screens generated from the imported package.

---

## 3. Design Principle

The POC shall use a **template-package-as-code** model.

This means:

1. The template package is stored in the code repository.
2. The package is version-controlled through Git.
3. Changes are made by developers or authorized template engineers.
4. Changes are reviewed before deployment.
5. The Frappe application imports the package into an `STD Template` record.
6. Runtime tender creation uses the imported package, not loose files directly.
7. Ordinary procurement officers cannot edit the package content through the application UI.

This approach is intentionally selected instead of building a template administration UI, because PPRA templates are relatively stable and the legal risk of uncontrolled editing is high.

## 4. Required Repository Location

The POC template package shall be created inside the custom Frappe app repository.

The exact app name may differ by implementation. In this document, the placeholder `{app_name}` shall represent the Frappe custom app name.

## 7. Folder Naming Standard

The package folder name shall follow this standard:

```text
{country}_{authority}_{procurement_category}_{template_family}_{version}_{purpose}
```

For this POC, the folder name shall be:

```text
ke_ppra_works_building_2022_04_poc
```

Where:

| Segment | Meaning |
|---|---|
| `ke` | Kenya |
| `ppra` | Public Procurement Regulatory Authority |
| `works` | Procurement category |
| `building` | Template family: building and associated civil engineering works |
| `2022_04` | April 2022 version/revision |
| `poc` | Proof-of-concept package |

The folder name shall use lowercase letters, underscores, and numeric version segments only. It shall not contain spaces, punctuation other than underscores, or environment-specific names.

---

## 8. Template Code Standard

The package folder name and the internal template code shall be related but not identical.

The internal template code shall use a formal uppercase code:

```text
KE-PPRA-WORKS-BLDG-2022-04-POC
```

Where:

| Segment | Meaning |
|---|---|
| `KE` | Kenya |
| `PPRA` | Public Procurement Regulatory Authority |
| `WORKS` | Procurement category |
| `BLDG` | Building and associated civil engineering works |
| `2022-04` | April 2022 source version |
| `POC` | Proof-of-concept package |

This code shall be defined in `manifest.json` and used by the seed loader to identify the template uniquely.

## 9. JSON File Standards

All JSON files in the package shall follow these standards:

1. Files shall contain valid JSON only.
2. Files shall use UTF-8 encoding.
3. Files shall use two-space indentation.
4. Object keys shall use `snake_case`.
5. Codes and enum values shall use uppercase where they represent system constants.
6. Human-readable labels shall use title case or sentence case as appropriate.
7. Dates shall use ISO format: `YYYY-MM-DD`.
8. Date-time values shall use ISO format where needed: `YYYY-MM-DDTHH:MM:SS`.
9. Monetary values in sample data shall be numeric, with currency captured separately.
10. No comments shall be placed inside JSON files.

Because JSON does not support comments, explanatory notes shall be placed in `README.md` or in explicit `description`, `help_text`, or `notes` fields where appropriate.

## 11. Package README Requirements

The package-level `README.md` shall be created with the following minimum content:

1. Package name.
2. Template code.
3. Source STD reference.
4. Statement that the package is a POC representation, not a full legal digitization.
5. Statement that ordinary procurement officers must not edit this package.
6. List of package files.
7. Note that the package is imported through the seed loader.
8. Note that changes require review.

Required initial README content:

```markdown
# KE-PPRA-WORKS-BLDG-2022-04-POC

This folder contains the developer-controlled template package for the PPRA Works Standard Tender Document POC.

This package represents the PPRA Standard Tender Document for Procurement of Works — Building and Associated Civil Engineering Works, April 2022 revision, as a minimal but complete proof-of-concept configuration package.

This package is not a full legal digitization of the entire PPRA STD. It is used to prove the template-package, safe-configuration, validation, bidder-form activation, BoQ hook, and tender-pack rendering architecture.

Ordinary procurement officers must not edit this package. Tender-specific values shall be entered through controlled Frappe tender configuration screens.

Package files:

- manifest.json
- sections.json
- fields.json
- rules.json
- forms.json
- render_map.json
- sample_tender.json

Changes to this package must be reviewed before import or deployment.
```

## 13. Package Integrity Requirements

The seed loader to be built in a later step shall depend on this folder structure. Therefore, this step must ensure that file names and locations are stable.

The package shall be considered structurally valid only if:

1. The folder exists at the required path.
2. All required files exist.
3. All JSON files parse successfully.
4. `manifest.json` contains `template_code`.
5. `sample_tender.json` references the same `template_code` as `manifest.json`.
6. No required file is empty.
7. No non-JSON content is placed inside JSON files.
8. The package folder name matches the intended STD family and version.

The seed loader shall later compute a package hash using the required JSON files. The package hash shall be used to detect changes and support auditability.

---

## 14. Package Hashing Policy

The package hash shall be computed later by the seed loader, but the folder structure shall support deterministic hashing.

Hashing shall include the following files in this exact order:

1. `manifest.json`
2. `sections.json`
3. `fields.json`
4. `rules.json`
5. `forms.json`
6. `render_map.json`
7. `sample_tender.json`

`README.md` shall not be included in the package hash because it is explanatory documentation and does not affect runtime behavior.

The future seed loader shall normalize file content before hashing if necessary. For the POC, it is acceptable to hash raw UTF-8 file bytes in the stated order.

---

## 15. Git and Review Requirements

All package files shall be committed to Git.

Any change to the package shall be reviewed as code. Review shall check:

1. JSON validity.
2. Naming consistency.
3. Template code consistency.
4. Scope compliance.
5. No unauthorized expansion into full STD digitization at Step 2.
6. No addition of runtime business logic inside JSON that belongs in Python services.
7. No introduction of environment-specific values.

The package shall not contain secrets, credentials, private keys, API tokens, or local machine paths.

---

## 16. Development Boundary for Step 2

This step is limited to creating the package folder and skeleton files.

The following shall not be implemented in Step 2:

1. Frappe DocTypes.
2. Seed loader.
3. Template engine.
4. Validation logic.
5. Render templates.
6. Full manifest content.
7. Full section content.
8. Full field registry.
9. Full rule registry.
10. Full form schema.
11. BoQ generation.
12. Tender-pack rendering.

Those items belong to later steps.

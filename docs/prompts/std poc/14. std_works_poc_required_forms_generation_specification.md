## 2. Purpose

The purpose of this step is to make required-form checklist generation a validated operating feature of the STD-WORKS-POC.

Earlier steps defined forms declaratively in `forms.json`, referenced form activation from `rules.json`, implemented the template engine, added server methods, and rendered a preview. Step 14 formalizes and hardens the feature that determines which bidder and contract forms are required for a specific tender configuration.

This step shall ensure that changing tender configuration reliably changes the `Tender Required Form` child table in a transparent, auditable, and repeatable way.

---

## 3. Design Principle

The required-forms checklist shall implement the principle:

> The system determines required bidder and contract forms from the STD template package and tender configuration; procurement officers do not manually assemble form packs.

The checklist shall be generated from:

1. `forms.json` default form requirements,
2. `rules.json` activation rules,
3. `Procurement Tender.configuration_json`,
4. `Procurement Tender` child rows where relevant,
5. validation output from the Step 11 template engine.

The checklist shall not be manually curated by ordinary procurement officers. If a form is wrong, the correction should normally be made by changing tender configuration or correcting the template package, not by editing the generated child table.

---

## 4. Scope of Step 14

Step 14 shall refine and verify the required-form generation feature.

This step includes:

1. Review and harden `resolve_required_forms()` in `std_template_engine.py`.
2. Review and harden `write_required_forms_to_tender()` in `std_template_engine.py`.
3. Review and harden the `generate_required_forms(tender_name)` server method.
4. Ensure the `Tender Required Form` child table is deterministic and auditable.
5. Ensure generated rows include activation source and evidence policy.
6. Ensure default required forms are always included.
7. Ensure scenario-activated forms are included when rules apply.
8. Ensure inactive alternative forms are excluded.
9. Ensure the generated checklist is ordered by `checklist_display.display_order`.
10. Ensure generation is idempotent.
11. Provide scenario acceptance tests for manual verification.

This step does not implement bidder submission screens.

## 6. Required Inputs

The required-form generation feature shall use the following inputs:

| Input | Source | Purpose |
|---|---|---|
| Form definitions | `forms.json` | Defines available forms, default required status, checklist metadata, evidence policy |
| Rule definitions | `rules.json` | Activates conditional forms based on configuration |
| Tender configuration | `Procurement Tender.configuration_json` | Supplies values used by rule conditions |
| Lots | `Procurement Tender.lots` | Supports multiple-lot validation where required |
| BoQ items | `Procurement Tender.boq_items` | Supports BoQ-related validation where required |
| Validation result | `std_template_engine.validate_config()` | Supplies active forms and blocking status |

## 7. Required Outputs

The feature shall populate the `Tender Required Form` child table on `Procurement Tender`.

Each generated row shall include:

| Field | Source |
|---|---|
| `form_code` | `forms.json.forms[].form_code` |
| `form_title` | `forms.json.forms[].title` |
| `display_group` | `forms.json.forms[].checklist_display.display_group` |
| `workflow_stage` | `forms.json.forms[].workflow_stage` |
| `respondent_type` | `forms.json.forms[].respondent_type` |
| `required` | Always `1` for generated rows in this POC |
| `activation_source` | `DEFAULT` or one or more rule codes |
| `evidence_policy` | `forms.json.forms[].evidence_policy` |
| `display_order` | `forms.json.forms[].checklist_display.display_order` |
| `notes` | POC notes or empty string |

Inactive forms shall not be added as rows in the child table in Step 14. Inactive forms may be shown later in UI/reporting, but this POC child table is the required checklist only.

## 8. Required Form Resolution Rules

### 8.1 Default Required Forms

Forms with `default_required = true` in `forms.json` shall always be included unless a later production exclusion rule is explicitly introduced.

Required default forms from Step 7:

| Form code | Display group |
|---|---|
| `FORM_OF_TENDER` | Core Bid Forms |
| `FORM_CONFIDENTIAL_BUSINESS_QUESTIONNAIRE` | Declarations |
| `FORM_CERTIFICATE_INDEPENDENT_TENDER_DETERMINATION` | Declarations |
| `FORM_SELF_DECLARATION` | Declarations |
| `FORM_TECHNICAL_PROPOSAL` | Technical Forms |
| `FORM_SIMILAR_EXPERIENCE` | Qualification Forms |
| `FORM_FINANCIAL_CAPACITY` | Qualification Forms |

### 8.2 Conditional Forms

Conditional forms shall be included when one or more applicable rules activate the form.

Required conditional behavior:

| Scenario | Trigger | Required form |
|---|---|---|
| Tender security selected | `SECURITY.TENDER_SECURITY_MODE = TENDER_SECURITY` | `FORM_TENDER_SECURITY` |
| Tender-securing declaration selected | `SECURITY.TENDER_SECURITY_MODE = TENDER_SECURING_DECLARATION` | `FORM_TENDER_SECURING_DECLARATION` |
| Joint ventures allowed | `PARTICIPATION.JV_ALLOWED = true` | `FORM_JV_INFORMATION` |
| International tender | `METHOD.TENDER_SCOPE = INTERNATIONAL` | `FORM_FOREIGN_TENDERER_LOCAL_INPUT` |
| Alternative tenders allowed | `ALTERNATIVES.ALTERNATIVE_TENDERS_ALLOWED = true` | `FORM_ALTERNATIVE_TECHNICAL_PROPOSAL` |
| Key personnel required | `QUALIFICATION.KEY_PERSONNEL_REQUIRED = true` | `FORM_PERSONNEL_SCHEDULE` |
| Equipment required | `QUALIFICATION.EQUIPMENT_REQUIRED = true` | `FORM_EQUIPMENT_SCHEDULE` |
| Beneficial ownership required | `QUALIFICATION.BENEFICIAL_OWNERSHIP_REQUIRED = true` | `FORM_BENEFICIAL_OWNERSHIP_DISCLOSURE` |
| Performance security required | `SECURITY.PERFORMANCE_SECURITY_REQUIRED = true` | `FORM_PERFORMANCE_SECURITY` |
| Advance payment security required | `SECURITY.ADVANCE_PAYMENT_SECURITY_REQUIRED = true` | `FORM_ADVANCE_PAYMENT_SECURITY` |
| Retention money security required | `SECURITY.RETENTION_MONEY_SECURITY_REQUIRED = true` | `FORM_RETENTION_MONEY_SECURITY` |

### 8.3 Mutual Exclusion: Tender Security vs Tender-Securing Declaration

The tender security instrument forms shall behave as alternatives.

| Tender security mode | Included form | Excluded form |
|---|---|---|
| `TENDER_SECURITY` | `FORM_TENDER_SECURITY` | `FORM_TENDER_SECURING_DECLARATION` |
| `TENDER_SECURING_DECLARATION` | `FORM_TENDER_SECURING_DECLARATION` | `FORM_TENDER_SECURITY` |
| `NONE` | Neither security form | Both security forms |

The engine shall not include both `FORM_TENDER_SECURITY` and `FORM_TENDER_SECURING_DECLARATION` for the same configuration unless a future rule explicitly permits it.

### 8.4 Form Deduplication

If a form is activated by multiple sources, it shall appear only once.

The `activation_source` field shall record all sources if practical.

Acceptable formats:

```text
DEFAULT
```

```text
RULE_JV_ACTIVATE_FORM_AND_REQUIRE_MAX_MEMBERS
```

```text
DEFAULT; RULE_REQUIRE_WORKS_REQUIREMENTS
```

For the POC, semicolon-separated activation sources are acceptable.

### 8.5 Sorting

Generated form rows shall be sorted by:

1. `display_order` ascending,
2. `form_code` ascending as tie-breaker.

This ensures stable output and predictable preview rendering.

## 9. Required Engine Behavior

The Step 11 function `resolve_required_forms(template, config, validation_result=None)` shall be confirmed or updated to satisfy this behavior:

1. Load all forms from `forms.json`.
2. Add all `default_required` forms.
3. Add all forms present in `validation_result.active_forms`.
4. If `validation_result` is not supplied, call or require validation first.
5. Deduplicate by `form_code`.
6. Build row dictionaries matching the `Tender Required Form` child table.
7. Include activation source.
8. Include evidence policy.
9. Include display group and display order.
10. Sort rows deterministically.
11. Return only required forms.

---

## 10. Required Persistence Behavior

The Step 11 function `write_required_forms_to_tender(tender_doc, required_forms)` shall be confirmed or updated to satisfy this behavior:

1. Clear existing `required_forms` child rows.
2. Append exactly one row per required form.
3. Preserve generated order.
4. Set `required = 1` for every generated row.
5. Do not save automatically unless explicitly requested by caller.
6. Do not modify `configuration_json`.
7. Do not modify `forms.json` or package data.

The Step 12 server method `generate_required_forms(tender_name)` shall save the tender document after writing child rows.

---

## 11. Required Server Method Behavior

The server method `generate_required_forms(tender_name)` shall follow this sequence:

1. Load the `Procurement Tender` document.
2. Check write permission.
3. Confirm `std_template` is selected.
4. Confirm `configuration_json` exists.
5. Load the template package.
6. Parse configuration JSON.
7. Convert lot and BoQ child rows to dictionaries.
8. Run validation through `validate_config()`.
9. Write validation messages to `validation_messages`.
10. Resolve required forms.
11. Write required forms to `required_forms`.
12. Update validation status and configuration hash.
13. Save the tender document.
14. Return structured result.

Required return shape:

```json
{
  "ok": true,
  "message": "Required forms generated.",
  "tender": "PT-2026-00001",
  "required_form_count": 15,
  "validation_status": "Passed",
  "blocks_generation": false,
  "configuration_hash": "..."
}
```

If validation has blockers, required forms may still be generated, but the response shall clearly indicate `blocks_generation = true`.

---

## 12. Idempotency Requirement

Required-form generation shall be idempotent.

Running `generate_required_forms(tender_name)` repeatedly against the same tender configuration shall produce:

1. the same number of rows,
2. the same form codes,
3. the same ordering,
4. the same activation sources,
5. no duplicate rows.

The method shall clear and regenerate rows rather than appending to existing generated rows.

## 15. Required Auditability Behavior

The generated required-forms checklist shall be auditable.

At minimum:

1. The tender record shall show the current `configuration_hash` after generation.
2. The `activation_source` column shall identify whether the form was required by default or by rule.
3. The required forms shall be regenerated when tender configuration changes.
4. The preview shall display the same forms as stored in the child table.
5. The tender-pack preview audit summary shall show validation status and configuration hash.

For the POC, a separate form-generation hash is not required.

---

## 16. Error Handling Requirements

The feature shall fail clearly for:

| Scenario | Required behavior |
|---|---|
| Missing tender name | Raise clear error |
| Tender not found | Raise clear error |
| User lacks write permission | Raise permission error |
| Missing `std_template` | Raise clear message to select template |
| Missing `configuration_json` | Raise clear message to load defaults or sample tender first |
| Invalid `configuration_json` | Raise clear JSON parse error |
| Missing form definition | Raise clear error identifying form code |
| Rule activates unknown form | Raise clear error identifying rule and form code |
| Missing `forms.json` component | Raise clear package error |

Do not silently ignore unknown activated forms. This is critical because silent omission would create legal risk.

## 2. Purpose

The purpose of this step is to define the full content of the `sections.json` file for the PPRA Works Standard Tender Document POC template package.

`sections.json` shall define the major document sections represented by the POC template package, their ordering, their mutability, their runtime role, their relationship to the source STD, and how they behave during tender creation and tender-pack preview generation.

This file does not define field-level configuration, validation rules, bidder form schemas, or rendering templates. Those are handled in later package files.

---

## 3. Design Principle

The `sections.json` file shall provide a formal section map of the Works STD.

The POC shall represent all major Works STD sections without attempting to digitize the full content of each section. This enables the system to prove that the entire tender-package structure can be generated and governed, while keeping the POC tractable.

Each section shall be classified by mutability so the system can distinguish between:

1. locked legal text,
2. configurable tender-specific data,
3. controlled bidder-response form areas,
4. native system modules such as BoQ,
5. controlled PE-authored requirements,
6. controlled attachments or references,
7. generated downstream contract documents.

---

## 4. Required File Location

The file shall exist at:

```text
{app_name}/procurement/std_templates/ke_ppra_works_building_2022_04_poc/sections.json
```

The placeholder `{app_name}` shall be replaced by the actual Frappe custom app name in the repository.

## 5. Relationship to the Source STD

The Works STD is structurally organized into the following major areas:

1. Invitation to Tender.
2. Part 1: Tendering Procedures.
3. Section I: Instructions to Tenderers.
4. Section II: Tender Data Sheet.
5. Section III: Evaluation and Qualification Criteria.
6. Section IV: Tendering Forms.
7. Part 2: Procuring Entity's Requirements / Works' Requirements.
8. Section V: Bills of Quantities.
9. Section VI: Specifications.
10. Section VII: Drawings.
11. Part 3: Conditions of Contract and Contract Forms.
12. Section VIII: General Conditions of Contract.
13. Section IX: Special Conditions of Contract.
14. Section X: Contract Forms.

The POC shall preserve this structure at section level.

## 7. Section Mutability Types

The file shall define the permitted mutability classifications used by sections.

Required mutability types:

| Mutability type | Meaning |
|---|---|
| `PARAMETERIZED_GENERATED` | Generated by the system from controlled tender-specific fields |
| `LOCKED_REPRESENTATIVE_TEXT` | Representative locked legal text or section summary in POC; not editable by ordinary users |
| `CONFIGURABLE_SCHEMA` | Configurable through predefined fields only |
| `CONTROLLED_CONFIGURABLE_SCHEMA` | Configurable only through approved options, criteria, thresholds, or rules |
| `STRUCTURED_BIDDER_RESPONSE_SCHEMA` | Defines bidder response forms or form checklist, not free-form uploads |
| `NATIVE_SYSTEM_MODULE` | Implemented through a system module or child table rather than static document text |
| `CONTROLLED_PE_AUTHORED_CONTENT` | Procuring entity content entered through controlled fields or controlled text areas |
| `CONTROLLED_ATTACHMENT_REFERENCE` | Controlled references or attachments, not arbitrary document substitution |
| `DOWNSTREAM_GENERATED_DOCUMENTS` | Generated after later workflow events such as award or contract formation |

These types shall be represented in `section_mutability_types` as structured objects, not merely implied by section values.

## 8. Section Object Schema

Each section object in the `sections` array shall follow this structure:

```json
{
  "section_code": "...",
  "source_section_code": "...",
  "title": "...",
  "part": "...",
  "render_order": 0,
  "mutability": "...",
  "runtime_role": "...",
  "included_in_poc_render": true,
  "ordinary_user_can_edit": false,
  "ordinary_user_input_allowed": false,
  "input_source": "...",
  "output_behavior": "...",
  "source_coverage": "...",
  "poc_treatment": "...",
  "depends_on_fields": [],
  "activates_forms": [],
  "notes": []
}
```

### 8.1 Required Section Object Fields

| Field | Type | Required | Purpose |
|---|---:|---:|---|
| `section_code` | String | Yes | System section identifier |
| `source_section_code` | String/null | Yes | Source STD section identifier or null if generated cover area |
| `title` | String | Yes | Human-readable section title |
| `part` | String | Yes | STD part or logical grouping |
| `render_order` | Number | Yes | Display/render order |
| `mutability` | String | Yes | One of the defined mutability types |
| `runtime_role` | String | Yes | Role of the section in runtime tender creation/rendering |
| `included_in_poc_render` | Boolean | Yes | Whether section appears in tender-pack preview |
| `ordinary_user_can_edit` | Boolean | Yes | Whether ordinary procurement officers can edit the section itself |
| `ordinary_user_input_allowed` | Boolean | Yes | Whether ordinary users provide controlled input affecting the section |
| `input_source` | String | Yes | Where section content comes from |
| `output_behavior` | String | Yes | How section appears in generated output |
| `source_coverage` | String | Yes | Level of coverage in POC |
| `poc_treatment` | String | Yes | Explanation of POC handling |
| `depends_on_fields` | Array | Yes | Field codes that can affect section activation/content |
| `activates_forms` | Array | Yes | Form codes related to this section, where applicable |
| `notes` | Array | Yes | Implementation notes |

## 10. Section Treatment Rules

### 10.1 Locked Sections

The following sections shall not be editable by ordinary procurement officers:

1. `ITT`
2. `GCC`
3. `PART_1_HEADER`
4. `PART_2_HEADER`
5. `PART_3_HEADER`

For the POC, `ITT` and `GCC` shall use representative locked text or summary blocks rather than full verbatim content.

### 10.2 Configurable Sections

The following sections shall be populated through controlled configuration fields:

1. `COVER`
2. `INVITATION_TO_TENDER`
3. `TDS`
4. `EVALUATION_QUALIFICATION`
5. `SPECIFICATIONS`
6. `SCC`
7. `AUDIT_SUMMARY`

### 10.3 Structured Bidder Response Section

`TENDERING_FORMS` shall not be treated as a free-form document upload area. It shall be represented as a structured bidder form checklist derived from `forms.json` and scenario rules.

### 10.4 Native Module Section

`BOQ` shall be represented by the `Tender BoQ Item` child table and rendered in summary form.

### 10.5 Controlled Attachment Section

`DRAWINGS` shall be represented as a controlled attachment/reference placeholder. It shall not be used to allow arbitrary replacement of the STD package.

### 10.6 Downstream Generated Section

`CONTRACT_FORMS` shall be represented by a checklist or sample generated-form summary. Full award and contract generation are outside this POC.

## 2. Purpose

The purpose of this step is to define the full content of the `manifest.json` file for the PPRA Works Standard Tender Document POC template package.

The manifest is the identity and control file for the template package. It tells the system what the package is, where it comes from, which procurement category it applies to, which procurement methods it supports, which source STD it represents, what its POC limitations are, and how it should be treated during import.

The manifest shall not define template sections, fields, validation rules, bidder forms, or rendering behavior. Those are handled in later package files.

---

## 3. Design Principle

The `manifest.json` file shall answer the following questions unambiguously:

1. What is this STD template package?
2. Which legal/source document does it represent?
3. Which procurement category and procurement methods does it apply to?
4. Is it a production template or a POC template?
5. Is it intended for ordinary procurement officer editing?
6. Which package files are required for a complete import?
7. Which assumptions and limitations apply to this package?
8. Which system behaviors should the seed loader apply during import?

The manifest shall be declarative. It shall not contain executable logic.

---

## 4. Required File Location

The file shall exist at:

```text
{app_name}/procurement/std_templates/ke_ppra_works_building_2022_04_poc/manifest.json
```

The placeholder `{app_name}` shall be replaced by the actual Frappe custom app name in the repository.

---

## 5. Manifest Role in the POC

The manifest shall be used by later implementation steps for the following purposes:

| Consumer | Use of manifest |
|---|---|
| Seed loader | Identifies, validates, imports, and updates the `STD Template` record |
| Template engine | Confirms category, version, status, and runtime behavior |
| Procurement Tender creation | Ensures the selected template is available for tender creation |
| Audit output | Displays template code, version, source reference, and package status |
| QA tests | Confirms package identity and required files |
| Future governance | Supports versioning, supersession, and source traceability |

This step is limited to completing `manifest.json`.

The following shall not be implemented in Step 3:

1. `sections.json` content beyond the skeleton from Step 2.
2. `fields.json` content beyond the skeleton from Step 2.
3. `rules.json` content beyond the skeleton from Step 2.
4. `forms.json` content beyond the skeleton from Step 2.
5. `render_map.json` content beyond the skeleton from Step 2.
6. `sample_tender.json` content beyond the skeleton from Step 2.
7. Frappe DocTypes.
8. Seed loader.
9. Template engine.
10. Render templates.
11. Validation service.
12. Tests.

Those items belong to later steps.

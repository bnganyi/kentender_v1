## Rules of engagement (non-optional)

1. **Reference intent** — Before implementing or extending work for steps **2–18**, carefully read **R1** ([`1. std_works_poc_scope_document.md`](1.%20std_works_poc_scope_document.md)) so objectives, layers, and **non-objectives** stay in frame. Use **R0** (PPRA Works STD PDF, DOC. 1) when the step depends on legal/source alignment. Do not build blindly past scope.
2. **Step document is law for that step** — Each numbered spec contains rationale, inputs, outputs, and steps. The **“Cursor Implementation Prompt”** is the **immediate instruction set** for implementation. Follow and **document** deliverables, **development boundaries**, **checklists**, and **acceptance criteria** from that document (summarise in **Notes**, point to test files / commands in **Evidence**, or attach pointers to step-17-style evidence when applicable). Adjust this tracker (columns/rows) if a step’s doc requires clearer traceability.
3. **TDD; Playwright for UI** — Tests first (or minimal red-green cycle). Any **Desk/UI** work for the POC must have **Playwright** coverage; “Done” for UI is not claimed without a passing run or a logged environment **blocker** per repo guidance.
4. **Exit conditions** — A step may be marked **`Done`** only when **all exit conditions** defined in **that step’s** specification are met **and** evidenced. **`Partial`** when some acceptance or exit items remain; **`Blocked`** when progress depends on an issue logged in **ISSUES_LOG**. Deliberate spec gaps require a **`deviation`** row in **ISSUES_LOG**.

### Per-step completion (copy into Notes or tick locally before setting Done)

- R1 (and R0 if relevant) read for this tranche  
- Step spec: rationale, inputs, outputs reviewed  
- **Cursor Implementation Prompt** followed  
- Deliverables + development boundaries respected  
- In-doc **checklists** satisfied (reference)  
- In-doc **acceptance criteria** satisfied (reference)  
- In-doc **exit conditions** satisfied (list which / how evidenced)  
- TDD evidence (+ Playwright for UI) recorded in **Evidence**  

---

**Status values:** `Not started` | `In progress` | `Partial` | `Blocked` | `Done`

**Evidence column:** Commands (e.g. `bench run-tests …`), test module paths, Playwright spec paths, or artefact paths. **Notes column:** acceptance criteria / exit condition refs, `STD-POC-NNN` issue ids, checklist “see §X” pointers.

1. Open the step’s specification; execute from the **Cursor Implementation Prompt** outward (full doc for boundaries and acceptance).
2. Pick the next **Not started** row in order (or parallelise package JSON steps 2–8 if desired).
3. When work starts, set **In progress**; when exit conditions and acceptance criteria are met, set **Done** with **Evidence**; otherwise **Partial** or **Blocked**.

5. If this tracker is insufficient to trace acceptance or exit conditions for a step, **extend the tracker** in the same change set as the implementation.

## 1. Purpose

This UI/UX Model defines the user-facing behavior, screen structure, navigation rules, interaction patterns, validation surfaces, and implementation-facing UI requirements for **STD Engine — Works Configuration**.

The purpose of this artifact is to prevent five classes of failure:

1. **Legal content failure** — official PPRA Works STD structure is edited or represented inconsistently.
2. **Source-of-truth failure** — users revert to uploading PDFs, DOCX files, spreadsheets, or drawings as the rule source instead of configuring structured STD data.
3. **Downstream rule failure** — Tender Management, Bid Submission, Opening, Evaluation, or Contract Management receives incomplete, stale, or manually altered derived models.
4. **Versioning failure** — active template versions, published STD instances, generated outputs, or addendum-regenerated outputs are overwritten instead of versioned/superseded.
5. **Usability failure** — configuration users cannot understand what is incomplete, what is locked, what must be reviewed, what generated outputs exist, or why readiness is blocked.

This document is intentionally detailed because it will be used by Cursor and developers as an implementation guide.

## 2. UI Design Principles

### 2.1 STD Engine is a governance/configuration workbench, not a file library

The primary internal interface must support legal template governance, structured configuration, validation, generation, versioning, and evidence.

Correct model:

```text
STD Workbench → Template Family / Version → Structured Sections → Mappings → Outputs → Readiness → Activation / Handoff
```

Incorrect model:

```text
Document Library → Upload PPRA PDF → Attach Excel BOQ → Use Uploaded Files as Tender Rules
```

### 2.2 Structured STD status must be visible everywhere it matters

Users must always understand whether a template version or STD instance is:

1. draft;
2. structurally incomplete;
3. validation blocked;
4. legally/policy review pending;
5. approved;
6. active;
7. used by tender instances;
8. published and locked;
9. superseded;
10. affected by addendum impact and regeneration.

### 2.3 Legal configuration actions must be visually distinct

The following actions must never look like ordinary save/edit buttons:

1. Submit Template Version for Legal Review;
2. Clear Legal Review;
3. Clear Policy Review;
4. Approve Template Version;
5. Activate Template Version;
6. Supersede Template Version;
7. Create Tender-Specific STD Instance;
8. Lock STD Instance for Tender Handoff;
9. Generate Bundle / DSM / DOM / DEM / DCM;
10. Run Readiness;
11. Approve Addendum Impact;
12. Regenerate Outputs After Addendum;
13. Confirm Supersession Chain;
14. Export Evidence.

They require confirmation, summary review, reason where required, and audit messaging.

### 2.4 The UI must be state-aware and role-aware

Buttons must appear only when current user, object state, version status, instance status, readiness status, addendum impact state, and permissions allow the action.

If an action is blocked, the UI must explain why in plain business language and expose the denial/blocker code for support/audit users.

### 2.5 Template configuration and tender-instance configuration must be separated

The UI must not confuse:

1. configuring an official reusable Works template version; and
2. completing a tender-specific STD instance from an active version.

Template version configuration is governance/configuration work.

Tender-specific STD instance completion is operational tender preparation work.

### 2.6 Attachments are secondary and section-bound

The UI may allow supporting source documents, drawings, specifications, and imported BOQ files, but these must be visually and functionally subordinate to structured records.

The UI must never present:

```text
Upload STD Document
```

as the way to create the authoritative STD structure.

Correct labels:

```text
Add Source Document Reference
Attach Supporting Drawing to Section VII
Import BOQ into Structured Items
Upload Specification Attachment for Section VI
```

### 2.7 Derived outputs are previews and generated artifacts, not editable forms

DSM, DOM, DEM, and DCM previews must be read-only. Users must modify the structured source data that generates them, not edit the generated outputs directly.

### 2.8 Every displayed business object must use business codes

No user-facing screen should rely on raw database IDs.

Display:

1. template code;
2. version code;
3. profile code;
4. instance code;
5. section code;
6. clause code;
7. parameter code;
8. BOQ code;
9. generated output code;
10. readiness run code;
11. addendum impact code;
12. audit event code.

## 3. Navigation Architecture

### 3.1 Procurement App Placement

STD Engine is a governance/configuration module. It must not be mixed into the ordinary Tender Management operational workbench.

Recommended Procurement navigation placement:

```text
Procurement
├── Procurement Home
├── Demand Intake & Approval
├── Procurement Planning
├── Tendering & Solicitation
├── Submissions & Opening
├── Evaluation & Award
├── Contracts & Execution
├── Complaints & Oversight
└── Governance & Configuration
    ├── STD Engine
    ├── Approval Rules
    ├── Publication Channels
    ├── Notification Templates
    └── Audit Settings
```

### 3.2 STD Engine Entry Point

Clicking STD Engine must open:

```text
STD Engine Workbench
```

It must not open:

1. a raw DocType list;
2. a file library;
3. an upload page;
4. a nested sidebar of technical tables;
5. a Tender Management operational queue.

### 3.3 Primary STD Engine Routes

Recommended routes:

```text
/desk/std-engine
/desk/std-engine/templates
/desk/std-engine/templates/<template_code>
/desk/std-engine/versions/<version_code>
/desk/std-engine/instances/<instance_code>
/desk/std-engine/generation-jobs/<generation_job_code>
/desk/std-engine/readiness/<readiness_run_code>
/desk/std-engine/addendum-impact/<impact_analysis_code>
/desk/std-engine/audit
```

Final routes may follow framework conventions, but behavior must preserve the workbench model.

### 3.4 Relationship to Tender Management Navigation

Tender Management may link to a tender-specific STD instance detail page, but it must not expose template-authoring controls to ordinary tender users.

A Tender Management user opening an STD instance from a tender sees a role-filtered instance view, not the full STD template library administration workbench unless they also have STD governance permissions.

## 6. KPI / Risk Strip

### 6.1 KPI Cards

Recommended KPI cards:

| KPI | Meaning | Click Behavior |
|---|---|---|
| Draft Versions | Template versions being configured. | Filter to Draft / Structure In Progress versions. |
| Validation Blocked | Versions or instances blocked by validation. | Filter to blockers. |

| Generation Failures | Failed generation jobs. | Filter failed jobs. |

### 6.2 KPI Rules

1. KPI cards must be interactive filters.
2. KPI cards must not be decorative only.
3. Counts must not truncate.
4. High-risk cards such as Validation Blocked, Generation Failures, and Addendum Impact Pending must be visually distinct.
5. Clicking a KPI must update the list and active queue state.

## 12. State / Readiness / Risk Summary

### 12.1 Summary Cards

The detail panel must show compact cards depending on object type.

For Template Version:

| Card | Meaning |
|---|---|
| Version State | Draft, Validation Passed, Legal Review Pending, Active, etc. |
| Structure Completion | Parts, sections, clauses, forms, mappings. |
| Review Status | Legal and policy review state. |
| Activation Readiness | Whether activation blockers exist. |
| Usage | Whether used by any STD instances/tenders. |
| Locks | Whether version is editable or immutable. |

For STD Instance:

| Card | Meaning |
|---|---|
| Instance State | Draft, In Progress, Ready, Published Locked, etc. |
| Readiness | Not Run, Blocked, Ready, Invalidated. |
| Generated Outputs | Bundle, DSM, DOM, DEM, DCM status. |
| Works Configuration | Parameters, Works Requirements, BOQ completion. |
| Addendum Impact | Pending or active impact workflows. |
| Locks | Pre-publication or published lock. |

### 12.2 Blocker Display Rules

1. Blockers must be visible near the top of the detail panel.
2. Critical blockers must be visually distinct.
3. Each blocker must show owner, source object, rule code, and resolution action.
4. Support/audit users must see technical codes.
5. Ordinary users must see plain business explanation.

### 12.3 Example Blocker Display

```text
Activation blocked: Section III has evaluation-driving clauses without extraction mappings.
Owner: STD Engine Configuration
Required action: Complete DEM extraction mappings before legal review.
Blocker code: STD_AUTH_MAPPING_MISSING
Severity: Critical
```

## 13. Primary Action Bar

### 13.1 Action Rules

The action bar must show only actions valid for:

1. current user role;
2. object type;
3. object state;
4. review status;
5. readiness status;
6. generation status;
7. separation-of-duties rules;
8. lock status.

### 13.2 Common Actions by Object/State

| Object / State | Primary Actions |
|---|---|
| Template Version Draft | Configure Structure, Configure Parameters, Configure Forms, Configure Mappings, Run Validation. |
| Template Version Validation Blocked | View Findings, Correct Configuration, Rerun Validation. |
| Template Version Validation Passed | Submit Legal Review, Submit Policy Review. |
| Template Version Legal Review Pending | Clear Legal Review, Return for Correction. |
| Template Version Approved | Activate Version. |
| Template Version Active | View, Supersede, Retire, Evidence Export. |
| Applicability Profile Draft | Configure Works Rules, Validate Profile. |
| Applicability Profile Approved | Activate Profile. |
| STD Instance Draft/In Progress | Complete Parameters, Complete Works Requirements, Configure BOQ, Generate Outputs, Run Readiness. |
| STD Instance Blocked | View Findings, Resolve Blockers, Regenerate Outputs, Rerun Readiness. |
| STD Instance Ready | Lock for Tender Handoff, View Outputs. |
| STD Instance Published Locked | View, Addendum Impact Path, Evidence Export. |
| Generation Job Failed | View Error, Retry if authorized. |
| Addendum Impact Analysis Complete | Approve Impact, Return for Correction, Regenerate Outputs. |
| Regenerated Addendum Outputs | View Supersession, Confirm Regeneration. |

### 13.3 Disabled Action Behavior

If an action is shown but disabled, the UI must explain why.

Example:

```text
Activate Version is disabled because the policy review has not been cleared and two mandatory DSM mappings are missing.
```

### 13.4 Action Availability Source

The UI must use a server-provided action availability service.

Each action availability response must include:

1. action code;
2. allowed true/false;
3. denial reason code if false;
4. user-facing explanation;
5. required state;
6. current state;
7. risk level;
8. reason required true/false;
9. confirmation required true/false;
10. audit event type to be created.

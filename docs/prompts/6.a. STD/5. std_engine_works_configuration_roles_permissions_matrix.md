## 1. Purpose

This document defines the roles, permissions, authorization rules, separation-of-duties requirements, governance restrictions, administrative boundaries, audit access controls, and system enforcement rules for **STD Engine — Works Configuration**.

The STD Engine is the legal definition layer of the procurement platform. Its permissions model must therefore be stricter than ordinary role-based access control. Authorization must evaluate:

```text
Actor + Role + Action + Object Type + Object State + Template Version State + STD Instance State + Context + Separation of Duties + Governance Preconditions
```

Role alone is insufficient.

This artifact defines:

1. actor categories;
2. role catalogue;
3. role boundary rules;
4. action catalogue;
5. high-level permission matrix;
6. state-aware permission rules;
7. template-version governance permissions;
8. STD instance governance permissions;
9. BOQ and Works Requirements permissions;
10. derived model permissions;
11. addendum impact and regeneration permissions;
12. separation-of-duties rules;
13. administrative override restrictions;
14. audit visibility and evidence permissions;
15. denial codes;
16. server-side enforcement requirements;
17. acceptance criteria and completion gate.

## 2. Access Control Principles

### 2.1 Legal STD Content Requires Explicit Governance

Template families, template versions, official section structures, locked clauses, parameter definitions, evaluation rule templates, BOQ rules, and extraction mappings are legally consequential configuration. They may only be created, reviewed, approved, activated, superseded, or retired through explicit governed actions.

### 2.2 State-Aware Authorization Is Mandatory

An actor may perform an action only if the relevant object is in a valid state for that action.

The system must evaluate at least:

1. template family status;
2. template version status;
3. applicability profile status;
4. STD instance status;
5. readiness status;
6. generated output status;
7. addendum impact status;
8. generation job status;
9. downstream use state.

### 2.3 Version-Aware Authorization Is Mandatory

A user may edit Draft structures but must not edit Active, Published, Superseded, Retired, or Archive-only legal structures directly.

### 2.4 Source-of-Truth Protection Is Mandatory

No actor may bypass the structured STD model by uploading documents as the source of tender rules.

Attachments may support sections, specifications, or drawings only when section-bound, versioned, and controlled.

### 2.5 Locked Official Text Is Protected

Instructions to Tenderers and General Conditions of Contract must not be modified in a tender-specific STD instance. System administrators and business users are equally prohibited from bypassing this rule.

### 2.6 Evaluation Criteria Protection Is Mandatory

Section III evaluation and qualification rules must be structured in the STD Engine and generated into DEM. Tender Management, Evaluation, or any user must not add hidden criteria outside the DEM.

### 2.7 Works BOQ Controls Are Mandatory

For Works BOQ profiles:

1. BOQ quantities are owned by the Procuring Entity;
2. supplier quantity edits are prohibited;
3. supplier rate-only input is controlled by DSM;
4. arithmetic correction is an Evaluation-stage rule generated into DEM;
5. contract price source is defined by DCM.

### 2.8 Separation of Duties Is Mandatory

The same actor must not normally configure, review, approve, activate, and publish legally consequential STD configuration without independent review. Exceptions require explicit delegated authority, reason, and audit, and some actions must never be overrideable.

### 2.9 System Administrator Privilege Is Not Business Authority

System administrators may maintain technical operation but must not use technical access to alter legal STD content, approve template versions, override source-of-truth constraints, or delete audit evidence.

### 2.10 All Material Actions Must Be Audited

Every legally material action and sensitive denial must create append-only audit evidence.

## 3. Role Catalogue

### 3.1 STD Engine Governance Roles

| Role | Description | Typical Scope |
|---|---|---|
| STD Administrator | Operational owner responsible for creating draft template families, template versions, profiles, sections, parameters, forms, mappings, and readiness rules before legal approval. | STD Engine configuration scope. |
| Senior STD Administrator | Senior configuration owner with authority to coordinate complex STD structures, submit versions for validation/review, and manage non-active configuration corrections. | STD Engine configuration scope; may cover multiple template families. |
| STD Legal Reviewer | Reviews legal sufficiency of template version structure, locked clauses, mandatory sections, evaluation source mappings, contract carry-forward, and deviations. | Legal review scope. |
| STD Policy Reviewer | Reviews alignment with procurement policy, PPRA-style structure, internal policy, delegation thresholds, and method applicability. | Policy/compliance review scope. |
| STD Governance Approver | Approves template versions and profiles for activation after required reviews and validations. | Formal governance authority scope. |
| STD Configuration Auditor | Reviews configuration, state transitions, mappings, generated outputs, and audit evidence without mutation rights. | Read-only oversight scope. |
| STD Technical Maintainer | Maintains technical configuration infrastructure, background jobs, storage, and integration health without legal-content approval authority. | Technical maintenance scope only. |
| STD Support Analyst | Provides limited support visibility into errors, generation failures, and blocker messages without authority to alter legal content. | Support-only scope. |
| System | Automated actor executing validations, generation, regeneration, locking, supersession, and audit creation. | System-defined actions only. |

### 3.2 Tender-Specific STD Instance Roles

| Role | Description | Typical Scope |
|---|---|---|
| Procurement Officer | Creates or requests tender-specific STD instance through Tender Management integration and completes permitted tender-specific parameters. | Assigned procuring entity / tender scope. |
| Technical Officer / Engineer | Completes Works Requirements, specifications, drawings register, technical components, and technical attachments. | Assigned tender technical scope. |
| Quantity Surveyor / BOQ Preparer | Configures Works BOQ bills, items, quantities, units, provisional sums, and pricing structure. | Assigned tender BOQ scope. |
| Procurement Manager | Reviews STD instance readiness, blockers, and outputs before Tender Management publication workflow. | Assigned procuring entity / delegated tender scope. |
| Legal Reviewer | Reviews legal sufficiency of tender-specific addendum impacts, SCC parameters, and non-standard risks. | Review scope only unless separate approval authority exists. |
| Compliance Reviewer | Reviews compliance findings, readiness blockers, and exception risks. | Compliance review scope. |
| Auditor / Oversight User | Read-only visibility into STD instance, outputs, addenda, readiness, audit trail, and evidence package. | Oversight scope. |

### 3.3 Downstream Consumer Roles Referenced but Not Owned Here

| Role / Module | Relationship |
|---|---|
| Tender Management User | Consumes STD instance, bundle, DSM, DOM, DEM, DCM; must not redefine STD-owned rules. |
| Supplier User | Does not configure STD Engine; consumes supplier-facing outputs through Supplier Portal. |
| Bid Opening User | Consumes DOM only; cannot alter opening model. |
| Evaluation User | Consumes DEM only; cannot inject evaluation criteria. |
| Contract Officer | Consumes DCM only; cannot override carry-forward source rules without Contract Management governance. |

## 4. Role Boundary Rules

| Rule ID | Rule |
|---|---|
| STD-RBAC-001 | STD Administrator may create and edit Draft template structures but may not activate their own template version without independent review and approval. |
| STD-RBAC-002 | Senior STD Administrator may submit versions for review but may not bypass legal/policy review. |
| STD-RBAC-003 | STD Legal Reviewer may approve legal sufficiency but may not independently activate a template version unless separately assigned governance approval authority. |
| STD-RBAC-004 | STD Policy Reviewer may approve policy sufficiency but may not alter template content directly during approval. |
| STD-RBAC-005 | STD Governance Approver may activate approved versions but may not edit content during activation. |
| STD-RBAC-006 | STD Configuration Auditor is read-only and may not mutate records. |
| STD-RBAC-007 | STD Technical Maintainer may restart failed jobs or inspect technical logs but may not change legal content, mappings, readiness outcomes, or audit events. |
| STD-RBAC-008 | Procurement Officer may complete tender-specific parameters but may not edit template definitions or locked clauses. |
| STD-RBAC-009 | Technical Officer may complete Works Requirements but may not alter evaluation mappings or BOQ supplier input rules. |
| STD-RBAC-010 | Quantity Surveyor may configure BOQ before publication but may not make supplier-owned inputs or post-publication BOQ changes outside addendum. |
| STD-RBAC-011 | Procurement Manager may review readiness and outputs but may not edit generated DSM, DOM, DEM, or DCM directly. |
| STD-RBAC-012 | System Administrator / Technical Maintainer cannot bypass locked ITT/GCC, active version immutability, published instance immutability, or derived model locks. |
| STD-RBAC-013 | System may generate, validate, lock, supersede, and audit only according to configured services and state transitions. |

## 7. State-Aware Permission Rules

### 7.1 Template Family State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Edit metadata, submit review | Activate, archive if dependencies exist |
| Under Review | Review, return, approve | Content edits except governed return/correction |
| Approved | Activate, suspend, archive if unused | Ordinary edits |
| Active | Create versions, suspend | Edit legal identity fields, delete |
| Suspended | Reactivate, archive if no active dependency | New version use where suspended prohibits use |
| Archived | View/audit only | All mutation |

### 7.2 Template Version State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Edit structure, configure sections, parameters, forms, mappings | Activate, create STD instances |
| Structure In Progress | Configure structure, run validation | Activate, approve final use |
| Validation Blocked | Correct structure, rerun validation | Legal review, activation |
| Validation Passed | Submit legal/policy review | Structural edits unless returned |
| Legal Review Pending | Legal/policy review, return, clear | Structure edits |
| Approved | Activate | Structure edits |
| Active | Create STD instances, view, supersede/retire | Direct edits, deletion |
| Superseded | View, audit, use for existing linked instances only | New instances unless explicitly grandfathered |
| Retired | View, audit | New instances, edits |
| Archived | View/audit only | All mutation |

### 7.3 Applicability Profile State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Edit, validate | Use for STD instance creation |
| Validation Blocked | Correct, validate | Activate |
| Approved | Activate | Edit without return |
| Active | Use for compatible STD instances | Direct edits, deletion |
| Suspended | Reactivate, retire | New instance use |
| Retired | View/audit | New instance use |

### 7.4 STD Instance State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Configure parameters, Works Requirements, BOQ, attachments | Publish, lock as ready |
| In Progress | Continue configuration, run generation/readiness | Publish, addendum path |
| Ready for Validation | Run readiness, generate outputs | Publish without readiness pass |
| Blocked | View findings, correct source data, rerun readiness | Lock, publish handoff |
| Ready | Lock pre-publication, handoff to Tender | Direct mutation without invalidating readiness |
| Locked (Pre-Publication) | View, publish handoff, governed unlock if policy permits | Ordinary edits |
| Published Locked | View, audit, addendum impact path | Direct edits |
| Superseded | View/audit, trace lineage | Mutation |
| Cancelled | View/audit | Mutation except archive/evidence |

### 7.5 Generated Output State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Review, regenerate, discard if unused | Publish as legal output without current designation |
| Current | Handoff, readiness, publication use | Manual edits |
| Published | View, audit, supersede by addendum | Manual edits, overwrite |
| Superseded | View, audit, lineage | Use as current output |
| Archived | Audit only | Operational use |

### 7.6 Addendum Impact State Rules

| State | Allowed Action Families | Restricted Action Families |
|---|---|---|
| Draft | Define change request, request analysis | Issue addendum |
| Analysis Pending | System analysis | Manual issue/regeneration bypass |
| Analysis Complete | Review, return, approve | Issue without approval/regeneration |
| Approved | Regenerate outputs | Change impact silently |
| Regeneration Required | Generate affected outputs | Issue addendum |
| Regenerated | Confirm outputs, return to Tender addendum issue | Direct source edits |
| Issued | View/audit, enforce supersession | Mutation |
| Cancelled | View/audit | Regeneration/issue |

## 8. Separation of Duties Matrix

| Rule ID | Prohibited Combination | Scope | Exception |
|---|---|---|---|
| STD-SOD-001 | Template version creator activates same template version. | Same version. | Never without independent governance approver. |
| STD-SOD-002 | Template structure editor provides final legal clearance. | Same version. | Legal Reviewer may comment but final clearance requires separate reviewer if they edited content. |
| STD-SOD-003 | Policy reviewer edits configuration during policy review and clears same review. | Same version/profile. | Must return to Draft/Correction first. |
| STD-SOD-004 | Governance approver edits content during activation. | Same version/profile. | Not permitted; must return for correction. |
| STD-SOD-005 | Procurement Officer configures and independently approves tender-specific STD readiness where approval workflow requires manager. | Same STD instance. | Delegated exception with reason only if policy allows. |
| STD-SOD-006 | Technical Officer who authored Works Requirements is sole approver of the same technical readiness. | Same instance/component. | Independent review required for high-risk tenders. |
| STD-SOD-007 | Quantity Surveyor who created BOQ approves BOQ-impacting addendum alone. | Same tender/addendum. | Requires Procurement Manager and/or technical/legal review according to risk. |
| STD-SOD-008 | User requests and approves same addendum impact analysis. | Same addendum. | Not allowed for material impacts. |
| STD-SOD-009 | System Administrator grants self business role to approve STD content. | System-wide. | Never through STD Engine. |
| STD-SOD-010 | Actor modifies active/published legal content and approves the modification. | Same version/instance. | Not permitted; change must create new version or addendum path. |
| STD-SOD-011 | Evaluation actor modifies DEM source rules for active tender. | Same tender. | Never permitted downstream. |
| STD-SOD-012 | Contract actor modifies DCM source rules for active tender. | Same tender. | Never permitted downstream. |

## 1. Purpose

This Smoke Contract defines the minimum non-negotiable system invariants and automated checks required before the **STD Engine — Works Configuration** can be considered implementable or safe for integration with Tender Management v2.

The Smoke Contract is not a general QA checklist. It is the legal and architectural acceptance contract for the Works STD Engine implementation.

It proves that:

1. DOC 1 seed data loads as structured STD data;
2. the Works STD is not treated as an attachment or upload-only document;
3. PPRA Works Parts and Sections are represented structurally;
4. locked standard sections are immutable;
5. TDS and SCC are parameter-only;
6. Section III generates the Evaluation Model and cannot be extended downstream;
7. Section IV generates the Submission Model;
8. Section V BOQ is first-class structured data;
9. Works BOQ quantities are Procuring Entity-owned;
10. supplier input is rate-only where permitted;
11. arithmetic correction exists only in DEM, not DSM or DOM;
12. contract carry-forward exists in DCM;
13. addendum impact analysis regenerates affected outputs;
14. Tender Management consumes STD outputs without redefining rules;
15. audit evidence can reconstruct the full lineage.

## 3. Smoke Contract Philosophy

The system must pass three classes of tests.

### 3.1 Structural Tests

Structural tests verify that the required objects exist and are shaped correctly.

Examples:

1. `STD-WORKS` template family exists;
2. DOC 1 source document exists;
3. all required sections exist;
4. ITT and GCC are locked;
5. TDS and SCC are parameter-only;
6. BOQ definition exists;
7. mappings exist.

### 3.2 Behavioral Tests

Behavioral tests verify that the system acts correctly.

Examples:

1. generated bundle is produced from STD instance;
2. DSM is produced from forms, parameters, BOQ, and requirements;
3. DOM is produced from opening clauses and parameters;
4. DEM is produced from Section III and evaluation-driving sources;
5. DCM is produced from GCC, SCC, BOQ, contract forms, and award/carry-forward rules;
6. readiness blocks missing DEM;
7. addendum triggers regeneration.

### 3.3 Negative Enforcement Tests

Negative tests prove that legally unsafe behavior is impossible.

Examples:

1. cannot edit ITT in a tender-specific instance;
2. cannot edit GCC in a tender-specific instance;
3. cannot add manual evaluation criteria in Tender Management or Evaluation;
4. cannot bypass DSM for bid submission;
5. cannot open tenders using fields outside DOM;
6. cannot create contract terms inconsistent with DCM;
7. cannot modify published STD instance without addendum;
8. cannot overwrite published outputs.

## 5. Global Invariants

| STD-SMOKE-SOT-002 | Tender Management must not define submission requirements, opening fields, evaluation criteria, or contract carry-forward independently. | Critical |
| STD-SMOKE-SOT-003 | Supplier Portal must consume supplier-facing bundle and DSM; it must not infer submission requirements from uploaded files. | Critical |
| STD-SMOKE-SOT-004 | Bid Opening must consume DOM; it must not define independent opening register fields. | Critical |
| STD-SMOKE-SOT-005 | Evaluation must consume DEM; it must not inject hidden or manual criteria. | Critical |
| STD-SMOKE-SOT-006 | Contract Management must consume DCM; it must not override tender-derived contract terms. | Critical |

## 24. Required Pass Criteria

### 24.1 Minimum Pass Rule

The Works STD Engine implementation is not acceptable unless:

```yaml
minimum_pass_rule:
  critical_tests_failed: 0
  high_tests_failed_allowed: 0 before production handoff
  warnings_allowed: only documented non-blocking UX or reporting gaps approved by project owner
```

### 24.2 Blocking Failures

Any of the following failures blocks implementation acceptance:

1. DOC 1 not loaded or not source-traced;
2. missing required sections;
3. ITT editable;
4. GCC editable;
5. TDS not parameter-only;
6. SCC not parameter-only;
7. Section III not mapped to DEM;
8. Section IV not mapped to DSM;
9. BOQ not structured;
10. supplier can edit BOQ quantities;
11. arithmetic correction appears in DSM or DOM;
12. DEM missing or manually editable;
13. DCM missing or contract can override it;
14. addendum bypasses impact/regeneration;
15. published outputs overwritten;
16. Tender Management defines manual submission/evaluation rules;
17. audit evidence cannot reconstruct lineage.

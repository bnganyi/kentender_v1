## 2. Purpose and Problem Statement

### 2.1 Problem

The prior system treated Standard Tender Documents (STDs) as:

- file uploads
- attachments
- unstructured documents

This resulted in:

- inability to enforce submission requirements
- inability to generate opening registers
- inability to bind evaluation models
- inability to enforce contract carry-forward
- inability to support controlled addenda

This violates legal and operational requirements under PPRA-style procurement.

### 2.2 Purpose

The STD Engine — Works Configuration establishes:

- a structured, parameterized STD model
- machine-readable sections and clauses
- deterministic generation of:
  - submission requirements
  - opening models
  - evaluation models
  - contract carry-forward
- controlled addendum impact and regeneration

This module becomes the **source of truth** for all tender behavior.

## 4. Core Principles (Non-Negotiable)

### 4.1 STD is the Source of Truth

STD Engine defines and owns:

- submission requirements
- opening register structure
- evaluation criteria
- contract terms

No downstream module may redefine these.

---

### 4.2 Structured, Not Document-Based

STD is composed of:

- sections
- clauses
- parameters
- forms
- BOQ

Attachments are:

- secondary
- section-bound
- version-controlled

---

### 4.3 Deterministic Outputs

From a given STD instance, the system must deterministically generate:

- submission model
- opening model
- evaluation model
- contract model

No manual intervention allowed.

---

### 4.4 Legal Integrity

- Published outputs immutable
- Changes only via addendum
- Full version lineage
- Full auditability

---

### 4.5 Works BOQ Principle

- quantities fixed by procuring entity
- supplier inputs rates only
- amounts computed by system
- arithmetic correction occurs only at evaluation

---

## 5. Key Concepts

### 5.1 STD Template

Defines:

- procurement method applicability
- procurement category (Works)
- base structure (sections, clauses)

Example:

- STD-WORKS

---

### 5.2 Template Version

Defines:

- specific legal version
- clause text
- parameter schema

Example:

- STDTV-WORKS-2020-REV1

---

### 5.3 Applicability Profile

Defines configuration for a context:

- building works
- civil works
- road works

Example:

- WORKS-PROFILE-BUILDING-CIVIL

---

### 5.4 STD Instance (Tender-Specific)

Created per tender:

- binds template version
- stores parameter values
- stores section data
- stores BOQ
- stores forms configuration

Example:

- STDINST-TND-MOH-2026-001

---

### 5.5 Generated Bundle

System-generated tender document output:

- compiled structured STD
- versioned
- immutable post-publication

---

### 5.6 Derived Models

#### Submission Model (DSM)
Defines:
- required forms
- required documents
- validation rules

#### Opening Model (DOM)
Defines:
- opening register fields
- disclosure rules

#### Evaluation Model (DEM)
Defines:
- evaluation criteria
- scoring rules

#### Contract Model (DCM)
Defines:
- contract clauses
- SCC values
- BOQ carry-forward

---

### 5.7 Addendum Impact Engine

Determines:

- affected sections
- affected parameters
- affected BOQ items
- affected derived models
- need for regeneration
- need for deadline extension
- need for supplier acknowledgement

## 7. High-Level Workflow

1. Select STD template + version + profile
2. Create STD instance
3. Populate parameters
4. Configure Works Requirements
5. Configure BOQ
6. Configure forms
7. Run STD readiness validation
8. Generate outputs:
   - bundle
   - DSM
   - DOM
   - DEM
   - DCM
9. Pass outputs to Tender Management

---

## 8. Key Risks and Controls

| Risk | Control |
|---|---|
| Reverting to file-based STD | Block upload-as-source pattern |
| Manual criteria injection | Derived models only |
| Inconsistent outputs | Deterministic generators |
| Addendum inconsistency | Impact engine + regeneration |
| BOQ manipulation | Quantity lock enforcement |

## 1. Purpose

This Seed Data Specification defines the first legally controlled Works STD seed package for the STD Engine.

The seed package is based on:

**DOC 1 — Standard Tender Document for Procurement of Works (Building and Associated Civil Engineering Works), Rev. April 2022.**

This artifact is not a generic example. It is the implementation specification for loading one real PPRA Works STD into the STD Engine as structured, versioned, source-traced data.

The seed package must support:

1. STD template family creation;
2. template version creation;
3. applicability profile creation;
4. source document registration;
5. Part / Section / Clause structure;
6. locked standard text sections;
7. parameterized tender-specific sections;
8. Works Requirements structure;
9. BOQ schedule structure;
10. Tendering Forms structure;
11. Evaluation and Qualification Criteria structure;
12. General Conditions of Contract structure;
13. Special Conditions of Contract parameters;
14. Contract Forms structure;
15. extraction mappings for generated bundle, DSM, DOM, DEM, and DCM;
16. readiness rules;
17. versioning and supersession metadata;
18. audit/evidence references.

## 2. Non-Invention Rule

The seed data must not invent legal content.

The implementation may add technical metadata required by the system, such as:

1. business codes;
2. order indexes;
3. editability classification;
4. source page ranges;
5. source text hashes;
6. extraction mapping identifiers;
7. readiness rule codes;
8. generated model rule codes;
9. version lineage metadata;
10. implementation comments.

However, the following must come from the PPRA source document:

1. section names;
2. clause numbering;
3. clause headings;
4. clause text;
5. form names;
6. form text and field labels;
7. BOQ schedule headings and required schedule structure;
8. GCC clause group headings and text;
9. SCC parameter labels and GCC sub-clause references;
10. Contract Form names and text;
11. instructions on what may or may not be modified;
12. evaluation criteria structure;
13. submission requirements stated in ITT and forms.

Where a field is a system-derived interpretation, it must be explicitly marked:

```yaml
source_type: system_classification
legal_text_source: null
```

Where a field is copied or extracted from the source document, it must be explicitly marked:

```yaml
source_type: source_document
legal_text_source: DOC1_WORKS_BUILDING_REV_APR_2022
source_page_start: <page>
source_page_end: <page>
source_anchor: <section_or_clause_reference>
```

### 3.1 Why Modular Files Are Required

A modular seed format is mandatory because:

1. PPRA documents can be revised;
2. individual sections may be superseded;
3. forms may be amended without changing all sections;
4. addendum logic must trace affected parts precisely;
5. legal reviewers must inspect manageable units;
6. Cursor and developers must not edit a 10,000-line fixture blindly;
7. future Works variants must reuse common structures.

### 4.2 Source Inclusion / Exclusion Rules

The source document contains preliminary material that must be handled differently from the tender document issued to tenderers.

```yaml
source_inclusion_rules:
  - source_area: Cover Page
    include_in_supplier_tender_bundle: false
    seed_usage: source_metadata_only
    reason: Source cover identifies PPRA document but is not part of issued tender document.

  - source_area: Preface
    include_in_supplier_tender_bundle: false
    seed_usage: legal_configuration_guidance
    reason: Preface contains rules on mandatory use, latest version, non-modification of ITT/GCC, and customization through TDS/SCC.

  - source_area: Appendix to the Preface / Guidelines
    include_in_supplier_tender_bundle: false
    seed_usage: legal_configuration_guidance
    reason: Guidelines define preparation rules, including no TDS additions outside the standard tender document and no additional evaluation criteria.

  - source_area: Beginning Page to be Issued to Bidders
    include_in_supplier_tender_bundle: true
    seed_usage: generated_bundle_template
    reason: Source states this page begins the Standard Tender Document to be issued to bidders.

  - source_area: Invitation to Tender
    include_in_supplier_tender_bundle: true
    seed_usage: generated_bundle_template_parameterized

  - source_area: Sections I to X
    include_in_supplier_tender_bundle: true
    seed_usage: structured_std_sections
```

## 22. Versioning and Supersession Rules

### 22.1 Template Version Supersession

If PPRA releases a later Works Building STD revision, do not edit `STDTV-WORKS-BUILDING-REV-APR-2022`.

Create a new version:

```yaml
version_code: STDTV-WORKS-BUILDING-REV-<NEW-REVISION>
supersedes_version_code: STDTV-WORKS-BUILDING-REV-APR-2022
```

### 22.2 Section Supersession

If only one section changes, the new template version may reuse unchanged section seed files by reference and override changed section files.

Recommended pattern:

```yaml
section_versioning:
  unchanged_section_strategy: reuse_by_hash
  changed_section_strategy: new_section_code_or_new_versioned_source_ref
  source_text_hash_required: true
```

### 22.3 Form Supersession

Because the source document itself states the April 2022 update amended the Form of Tender and Beneficial Ownership Information Disclosure Form, form-level supersession must be supported.

Required metadata:

```yaml
form_versioning_required: true
form_fields:
  - form_code
  - version_code
  - source_revision_label
  - supersedes_form_code
  - source_text_hash
```

### 22.4 Addendum Supersession

Tender-specific changes after publication do not create a new template version. They create:

1. addendum impact analysis;
2. new tender-specific generated bundle version;
3. new affected derived model versions;
4. supersession links to previous outputs.

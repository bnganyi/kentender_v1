## 1. Purpose

This document defines the governance and state model for the **STD Engine — Works Configuration**.

It ensures that:

1. template structures are legally controlled;
2. template versions are immutable once active;
3. STD instances are governed and validated before use;
4. derived outputs (bundle, DSM, DOM, DEM, DCM) are deterministically generated;
5. post-publication changes follow addendum + impact analysis;
6. all transitions are server-enforced and auditable;
7. downstream modules cannot operate without valid STD outputs.

---

## 2. Governance Principles

### 2.1 Structured STD is the Legal Source of Truth
No downstream module may define submission, opening, evaluation, or contract rules outside STD-derived models.

### 2.2 Immutability After Activation / Publication
- Template Version → immutable after Active
- STD Instance → immutable after publication (except via addendum)
- Generated Outputs → versioned; never overwritten

### 2.3 Deterministic Generation
Same STD instance state → same outputs (bundle, DSM, DOM, DEM, DCM).

### 2.4 Addendum as Sole Change Mechanism (Post-Publication)
All material changes must:
- trigger impact analysis
- regenerate affected outputs
- create new version chain

### 2.5 Server-Side Enforcement
No UI-only validation is accepted.
All transitions enforced via services.

### 2.6 Full Auditability
Every transition produces append-only audit evidence.

## 4. Template Family State Model

### 4.1 States

| State | Meaning | Editable |
|---|---|---|
| Draft | Initial creation of the STD template family; structure not yet validated or reviewed. | Yes |
| Under Review | Template family undergoing legal, regulatory, or policy review. | Limited |
| Approved | Template family approved for controlled use; no structural edits allowed without new versioning. | No structure edits |
| Active | Template family is available for template version creation and usage. | No |
| Suspended | Temporarily disabled due to policy, legal, or operational reasons. | No |
| Archived | Retained only for audit and historical reference; not usable. | No |

### 4.2 Transitions

| From | Action | To | Actor | Preconditions | Audit |
|---|---|---|---|---|---|
| None | Create family | Draft | STD Admin | Authority | Family Created |
| Draft | Submit review | Under Review | STD Admin | Metadata complete | Submitted |
| Under Review | Approve | Approved | Legal | Review passed | Approved |
| Approved | Activate | Active | Governance | No blockers | Activated |
| Active | Suspend | Suspended | Governance | Reason | Suspended |
| Suspended | Reactivate | Active | Governance | Valid | Reactivated |
| Any | Archive | Archived | Governance | No active dependency | Archived |

## 5. Template Version State Model

### 5.1 States

| State | Meaning |
|---|---|
| Draft | Version created but no structural definition completed. |
| Structure In Progress | Sections, clauses, mappings, and BOQ structures are being defined. |
| Validation Blocked | Validation failed due to structural or mapping errors. |
| Validation Passed | Structural validation passed but not yet legally reviewed. |
| Legal Review Pending | Awaiting legal/compliance approval. |
| Approved | Approved for activation; no further structural edits allowed. |
| Active | Legally enforceable version available for STD instance creation. |
| Superseded | Replaced by a newer active version; remains valid for existing instances. |
| Retired | Withdrawn from future use but preserved for audit. |
| Archived | Fully decommissioned and retained for record only. |

### 5.2 Key Rules

- Active version is immutable
- Only one Active version per profile context (configurable)
- Supersession does not mutate existing STD instances

### 5.3 Critical Blockers

| Code | Meaning |
|---|---|
| MISSING_SECTION | Required section absent |
| ITT_EDITABLE | ITT not locked |
| GCC_EDITABLE | GCC not locked |
| MISSING_MAPPING | DSM/DOM/DEM/DCM mapping missing |
| MISSING_BOQ_DEF | BOQ definition missing for required profile |

## 6. STD Instance State Model

### 6.1 States

| State | Meaning |
|---|---|
| Draft | STD instance created but no configuration applied. |
| In Progress | Parameters, BOQ, and section-level inputs are being configured. |
| Ready for Validation | Configuration complete and ready for validation execution. |
| Blocked | Validation failed due to missing or invalid configuration. |
| Ready | Validation passed and instance is complete for tender use. |
| Locked (Pre-Publication) | Instance frozen and handed off to Tender before publication. |
| Published Locked | Instance bound to a published tender and fully immutable. |
| Superseded | Replaced through addendum-driven regeneration. |
| Cancelled | Invalidated due to tender cancellation. |

### 6.2 Transitions

| From | Action | To | Actor | Preconditions | Audit |
|---|---|---|---|---|---|
| None | Create instance | Draft | System | Valid template/profile | Instance Created |
| Draft | Configure | In Progress | User | — | Updated |
| In Progress | Validate | Ready / Blocked | System | All inputs present | Validation Run |
| Blocked | Fix + revalidate | Ready / Blocked | User/System | Fix applied | Validation Run |
| Ready | Lock for tender | Locked | System | Tender submission | Locked |
| Locked | Publish (via Tender) | Published Locked | System | Tender published | Bound |
| Published Locked | Addendum change | Superseded | System | Impact approved | Superseded |

## 7. Readiness State Model

### 7.1 States

| State | Meaning |
|---|---|
| Not Run | Validation has not been executed. |
| Incomplete | Required data or outputs missing. |
| Blocked | Critical validation failure preventing progression. |
| Warning | Non-critical issues present but may allow progression with waiver. |
| Ready | All validation checks passed successfully. |
| Invalidated | Previously valid state invalidated due to changes. |
| Superseded | Replaced by a newer validation run. |

### 7.2 Rules

- Cannot manually set Ready
- Any change → Invalidated
- Only latest run is current

### 7.3 Critical Blockers

- Missing BOQ
- Missing DEM
- Missing DSM
- Missing DCM
- Missing required parameters

## 8. Generation State Model

### 8.1 States

| State | Meaning |
|---|---|
| Pending | Generation job queued but not yet started. |
| Running | Generation process actively executing. |
| Completed | Outputs successfully generated. |
| Failed | Generation failed due to system or data error. |
| Superseded | Outputs replaced by a newer generation cycle. |

### 8.2 Rules

- All outputs must exist before readiness passes
- Regeneration creates new version
- Previous outputs remain immutable

## 9. Output State Model (Bundle, DSM, DOM, DEM, DCM)

### 9.1 States

| State | Meaning |
|---|---|
| Draft | Output generated but not yet designated as active/current. |
| Current | Latest valid output used by the system pre-publication. |
| Published | Output tied to a published tender and immutable. |
| Superseded | Replaced by a newer output version. |
| Archived | Retained for audit only. |

### 9.2 Rules

- Published outputs immutable
- Addendum → new version → old becomes Superseded

## 10. Addendum Impact Governance

### 10.1 States

| State | Meaning |
|---|---|
| Draft | Addendum change defined but not analyzed. |
| Analysis Pending | Awaiting STD Engine impact analysis. |
| Analysis Complete | Impact analysis completed and available. |
| Approved | Addendum approved for regeneration. |
| Regeneration Required | Changes require regeneration of outputs. |
| Regenerated | New outputs successfully generated. |
| Issued | Addendum formally issued and applied to tender. |
| Cancelled | Addendum abandoned before issuance. |

### 10.2 Transition Highlights

- Draft → Analysis Pending (request)
- Analysis Pending → Complete (STD Engine)
- Complete → Approved (governance)
- Approved → Regenerated (generation jobs)
- Regenerated → Issued (Tender)

### 10.3 Critical Rules

- Cannot issue without regeneration
- BOQ changes always require regeneration
- Evaluation-impacting changes require DEM regeneration

## 13. Audit Rules

Every transition must record:

- actor
- role
- timestamp
- object
- previous state
- new state
- reason (if required)
- STD references

Critical events include:

- template activation
- instance creation
- validation runs
- generation jobs
- addendum impact
- regeneration
- illegal edit attempts
